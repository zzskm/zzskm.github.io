## 토픽 22: 예외 처리 vs UE5 에러 핸들링

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 5

### 1. 핵심 개념

핵심 용어: `RAII`, `ensure`, `check`, `스택 언와인딩`.

C++의 `exception handling(익셉션 핸들링, 예외 처리)`은 `try` / `catch` / `throw`를 통해 오류를 호출자에게 전파하는 메커니즘이다. 장점은 정상 경로와 오류 경로를 분리해 코드를 읽기 좋게 만들고, 스택 언와인딩 중 소멸자를 호출하므로 RAII와 자연스럽게 맞물린다는 점이다.

하지만 예외를 지원한다고 해서 모든 코드가 예외 중심으로 설계되는 것은 아니다. 게임 엔진과 실시간 애플리케이션에서는 빌드 일관성, 숨은 control flow 최소화, unwind 비용 관리, 디버깅 편의성 때문에 예외보다 `check` / `ensure` / `verify`, `bool`, `TOptional`, 반환 코드 기반 실패 보고를 더 자주 쓴다.

중요한 것은 예외를 쓰느냐보다 실패 모델을 분리해 생각하는 것이다. `basic guarantee(베이직 개런티, 기본 예외 안전 보장)`는 실패 뒤에도 객체가 유효함을 뜻하고, `strong guarantee(스트롱 개런티, 강한 예외 안전 보장)`는 실패 시 상태가 바뀌지 않음을 뜻하며, `nothrow guarantee(노스로 개런티, 예외 없음 보장)`는 애초에 예외를 던지지 않겠다는 계약이다. `noexcept(노익셉트, 예외를 던지지 않음)`는 이 계약을 함수 수준에서 드러낸다.

쉽게 말해 예외는 "실패를 상위 호출자로 넘기는 방식"이고, 반환값 기반 처리는 "호출자가 실패를 직접 확인하는 방식"이다. `check`는 내부 불변식이 깨졌다는 뜻에 가깝고, `ensure`는 개발 중 경고를 남기면서 흐름을 조금 더 관찰하고 싶을 때 쓰는 도구다. 반대로 UB는 실패 경로를 보고하는 것이 아니라 언어 계약이 깨진 상태라는 점에서 성격이 완전히 다르다.

### 2. 코드 예시

```cpp
void Read(const char* path) {
    FILE* f = fopen(path, "r");
    if (!f) return;
    Parse();
    fclose(f);
}
```

이 예시는 예외 자체보다 raw 자원을 수동으로 닫는 방식이 취약하다는 점을 보여 준다. `Parse()`에서 예외가 나거나 중간 return 경로가 늘어나면 `fclose`에 도달하지 못하므로, 파일 핸들을 RAII 객체로 감싸는 편이 훨씬 안전하다.

### 3. 왜 중요한가 (게임/UE5 맥락)

게임과 엔진 코드에서는 치명적 버그, 복구 가능한 실패, 외부 입력 오류를 같은 방식으로 다루면 안 된다. 면접에서도 예외 문법 자체보다 "왜 UE5가 예외보다 명시적 실패 처리와 assert 계열 매크로를 자주 쓰는가"를 설명할 수 있는지가 더 중요하고, 실무에서도 그 구분이 디버깅 비용과 코드 일관성을 크게 좌우한다.

### 4. 면접 예상 Q&A

**Q1: 표준 C++ 예외 처리와 UE5식 실패 처리의 차이는 무엇인가?**
> A: 표준 C++은 `throw` / `catch`로 깊은 호출 스택을 따라 오류를 전파하지만, UE5는 대개 예외를 끄고 `check`, `ensure`, 로그, 반환값으로 실패를 드러낸다. 즉 자동 전파보다 명시적인 실패 경로를 선호하는 편에 가깝다.

**Q2: 예외를 쓰지 않더라도 RAII가 왜 중요한가요?**
> A: 실패를 반환값으로 보고하더라도 함수는 중간에 여러 경로로 빠져나간다. RAII를 써야 조기 return이나 오류 처리 분기에서도 파일 핸들, 락, 메모리가 자동으로 정리된다.

**Q3: basic / strong / nothrow guarantee의 차이는 무엇인가?**
> A: basic guarantee는 실패 뒤에도 객체가 유효한 상태를 유지함을 뜻하고, strong guarantee는 실패 시 관찰 가능한 상태가 바뀌지 않음을 뜻한다. nothrow guarantee는 아예 예외를 던지지 않겠다는 약속이므로 컨테이너 최적화나 정리 코드 설계와도 연결된다.

**Q4: `check`, `ensure`, 반환값 기반 처리는 어떻게 구분해 쓰나요?**
> A: `check`는 내부 불변식이 깨졌을 때 즉시 버그를 드러내는 용도이고, `ensure`는 개발 중 경고를 남기면서 흐름을 더 관찰하고 싶을 때 쓴다. 파일 없음, 잘못된 입력, 네트워크 실패처럼 예상 가능한 오류는 반환값이나 `TOptional` 같은 명시적 실패 모델로 처리하는 편이 맞다.

### 5. 흔한 실수

- **예외를 안 쓴다고 RAII까지 버리기**: 반환값 기반 처리라도 함수는 중간에 여러 경로로 빠져나간다. 정리 코드를 수동으로 흩뿌리면 조기 return이나 실패 분기에서 누수가 생기기 쉬우므로, 자원 정리는 별도로 자동화해야 한다.
- **`check`를 사용자 입력 검증에 사용**: `check`는 내부 버그를 드러내는 도구에 가깝다. 파일 없음, 잘못된 입력, 네트워크 실패처럼 예상 가능한 오류는 반환값과 로그로 명시적으로 처리해야 한다.
