## 토픽 22: 예외 처리 vs UE5 에러 핸들링

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 5

### 1. 문제 상황

핵심 용어: `RAII`, `ensure`, `check`, `스택 언와인딩`.

함수가 실패할 때 어떻게 알릴 것인가? C++ 표준은 예외(throw/catch)를 제공하지만, UE5를 포함한 대부분의 게임 엔진은 예외를 끈다(-fno-exceptions). 예외를 끄면 실패 전파 수단이 사라지는데, 그 대안을 모르면 실패 경로가 방치된다.

더 근본적인 문제는 '버그'와 '예상 가능한 실패'를 같은 방식으로 처리하는 것이다. check로 사용자 입력을 검증하거나, 반환값으로 내부 불변식 위반을 알리면 Shipping 빌드에서 검사가 사라지거나 호출자가 확인을 빼먹는다.

### 2. 핵심 개념

#### 정의

예외 처리(exception handling)는 throw/catch로 오류를 호출 스택 위로 전파하는 메커니즘이다. 스택 언와인딩(stack unwinding) 중 지역 객체 소멸자가 호출되므로 RAII와 자연스럽게 맞물린다.

예외 안전 보장은 세 단계다. basic guarantee: 실패 후에도 객체가 유효한 상태. strong guarantee: 실패 시 상태가 호출 전과 동일(copy-and-swap). nothrow guarantee: 예외를 던지지 않겠다는 계약(noexcept).

UE5의 실패 처리 도구는 역할이 다르다. check: 내부 불변식 위반 → 즉시 중단(Shipping에서 제거). ensure: 경고 로그 + 실행 계속(개발 중 관찰용). verify: 조건 평가는 Shipping에서도 실행, 위반 시 중단은 Development만. 반환값/TOptional: 예상 가능한 실패의 명시적 전달.

#### 실패 모델 선택 플로우

- 내부 불변식이 깨졌는가? → check (버그를 즉시 드러낸다)
- 개발 중 조건을 관찰하고 싶은가? → ensure (경고 + 계속 실행)
- 외부 입력·네트워크·파일 같은 예상 가능한 실패인가? → bool/TOptional/에러 코드 반환
- 표준 C++ 라이브러리 호출이 예외를 던질 수 있는가? → try/catch 또는 noexcept 래퍼

#### 이 개념을 이해하려면

- RAII (토픽 5): 예외/반환값 무관하게 자원 정리를 자동화하는 기반
- 스마트 포인터 (토픽 4): RAII의 대표 구현체, 예외 안전 코드의 핵심 도구

#### 이 개념이 영향을 주는 것

- 멀티스레딩 기초 (토픽 21): 실패 경로에서 락 해제와 상태 복구
- UObject 생명주기 (카테고리 2 / 토픽 5): 엔진 객체의 생성 실패 처리 패턴

### 3. 내부 동작 / 메커니즘

#### 예외 전파 과정

throw가 실행되면 런타임은 현재 스코프부터 호출 스택을 역순으로 탐색하며 매칭되는 catch를 찾는다. 탐색 중 각 스코프의 지역 객체 소멸자가 호출된다(스택 언와인딩). catch를 찾지 못하면 std::terminate가 호출된다.

noexcept 함수에서 예외가 탈출하면 스택 언와인딩 없이 곧바로 std::terminate다. 이것은 '예외를 던지지 않겠다'는 타입 수준 계약이다. 컴파일러는 이 계약을 활용해 코드를 최적화하고, 컨테이너는 noexcept 여부에 따라 이동/복사 전략을 결정한다.

#### UE5가 예외를 끄는 이유

- 플랫폼 일관성: 콘솔 플랫폼에서 예외 지원이 제한적이거나 비용이 크다
- 숨은 제어 흐름 최소화: throw가 어디서든 날 수 있으면 코드 리뷰와 디버깅이 어려워진다
- unwind 테이블 비용: 바이너리 크기와 런타임 오버헤드 증가
- 디버깅 편의: check/ensure는 즉시 중단점을 잡을 수 있지만, 예외는 catch까지 흘러간 뒤에야 상태를 볼 수 있다

#### check / ensure / verify 내부 차이

check(조건): 조건 실패 → 에디터/게임 즉시 중단. Shipping 빌드에서 매크로 자체가 제거되므로 조건 평가도 사라진다. 따라서 부수 효과가 있는 표현식을 check에 넣으면 Shipping에서 동작이 달라진다.

verify(조건): 조건 평가는 모든 빌드에서 실행. 위반 시 중단은 Development/Debug에서만. Shipping에서도 조건의 부수 효과가 필요하면 verify를 쓴다.

ensure(조건): 조건 실패 → 경고 로그 + 실행 계속. 첫 실패만 보고하므로 반복 호출에서 로그가 폭주하지 않는다. 개발 중 문제를 관찰하면서 흐름을 유지하고 싶을 때 쓴다.

### 4. 자주 틀리는 포인트

- 예외를 안 쓴다고 RAII까지 버린다: 반환값 기반 처리라도 함수는 여러 경로로 빠져나간다. 수동 정리 코드를 흩뿌리면 조기 return에서 누수가 생긴다. 자원 정리는 예외 사용 여부와 무관하게 RAII로 자동화해야 한다.
- check를 사용자 입력 검증에 쓴다: check는 내부 버그를 드러내는 도구다. 외부 입력(플레이어/네트워크)에 check를 쓰면 Shipping 빌드에서 검사가 사라져 잘못된 값이 통과한다. 예상 가능한 실패는 if + 반환값으로 처리한다.
- check에 부수 효과 있는 표현식을 넣는다: check(ptr = Allocate()) 같은 코드는 Shipping에서 할당 자체가 사라진다. 할당을 먼저 하고 check(ptr != nullptr)로 분리해야 한다.
- 예외 안전 보장 수준을 구분하지 않는다: 컨테이너 push_back이 실패하면 원소가 추가되지 않아야 하는데(strong), 정리 코드에서는 절대 예외가 나면 안 된다(nothrow). 모든 함수를 같은 수준으로 설계하면 과도한 비용이거나 불충분한 보장이 된다.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

#### 예외(throw/catch)를 쓰는 경우

- 표준 C++ 라이브러리 호출이 예외를 던질 수 있고, 예외가 활성화된 빌드일 때
- 깊은 호출 스택을 따라 실패를 전파해야 하고, 중간 함수가 실패를 몰라도 될 때
- 판단 기준: 예외가 켜진 환경에서, 정상 경로와 오류 경로를 분리해 코드 가독성을 높일 수 있으면 쓴다

#### 예외를 쓰지 않는 경우 (UE5 기본)

- -fno-exceptions 빌드: throw가 컴파일되지 않거나 std::terminate를 호출한다
- 실시간 코드(게임 루프, 렌더링): unwind 비용과 숨은 제어 흐름이 부담이 된다
- 판단 기준: 예외가 꺼진 코드베이스에서는 check/ensure/반환값 조합이 기본이다

#### check vs ensure vs 반환값 선택 기준

- 절대 깨져서는 안 되는 내부 불변식 → check
- 개발 중 관찰하고 싶지만 중단까지는 필요 없는 조건 → ensure
- 파일 없음, 네트워크 실패, 잘못된 입력 같은 예상 가능한 실패 → bool/TOptional/에러 코드 반환

### 6. 코드 예시 (bad → good)

#### Bad: UE5에서 throw 사용

```cpp
// UE5는 -fno-exceptions가 기본이므로 throw가 동작하지 않는다
double Divide(double A, double B) {
    if (B == 0.0)
        throw std::runtime_error("division by zero");
    return A / B;
}
```

이 예시는 실패를 호출자에게 되돌려 보내는 대신 예외 전파에 기대는 방식이다. UE5 기본 빌드에서는 이런 실패 모델이 자연스럽지 않으므로 반환값 기반 모델로 바꾸는 편이 안전하다.

#### Good: TOptional로 실패를 명시적으로 표현

```cpp
TOptional<double> Divide(double A, double B) {
    if (B == 0.0)
        return {};  // 빈 optional = 실패
    return A / B;
}

// 호출자
if (TOptional<double> Result = Divide(X, Y))
{
    UseResult(*Result);
}
else
{
    UE_LOG(LogTemp, Warning, TEXT("Division failed"));
}
```

좋은 점은 실패가 타입에 드러난다는 것이다. 호출자는 값을 쓸지 로그를 남길지 직접 결정해야 하므로 실패 경로를 빠뜨릴 가능성이 줄어든다.

#### Bad: check를 외부 입력 검증에 사용

```cpp
void AMyActor::SetHealth(int32 NewHealth) {
    // Shipping에서 check가 제거되면 검증 자체가 사라진다
    check(NewHealth >= 0 && NewHealth <= MaxHealth);
    Health = NewHealth;
}
```

이 코드는 외부 입력 검증을 assert 계열에 맡긴다. 개발 빌드에서는 버그를 빨리 드러내지만 Shipping에서는 검사 자체가 사라질 수 있어 예상 가능한 실패 처리와 맞지 않는다.

#### Good: 외부 입력은 if + 클램프로 처리

```cpp
void AMyActor::SetHealth(int32 NewHealth) {
    if (NewHealth < 0 || NewHealth > MaxHealth)
    {
        UE_LOG(LogTemp, Warning, TEXT("Invalid health: %d"), NewHealth);
        NewHealth = FMath::Clamp(NewHealth, 0, MaxHealth);
    }
    Health = NewHealth;
}
```

외부 입력은 예상 가능한 실패이므로 정상 제어 흐름 안에서 처리하는 편이 맞다. 이 방식은 로그와 복구 정책을 함께 남길 수 있고 빌드 설정에 따라 의미가 바뀌지 않는다.

#### Bad: raw 자원 수동 정리

```cpp
void Read(const char* Path) {
    FILE* F = fopen(Path, "r");
    if (!F) return;
    Parse();       // 예외 발생 시 fclose에 도달 못함
    fclose(F);
}
```

문제는 성공 경로에서는 맞아 보여도 조기 return이나 예외 경로에서 정리 코드가 쉽게 빠진다는 점이다. 실패 모델이 무엇이든 자원 정리는 제어 흐름과 분리해야 한다.

#### Good: RAII로 자원 정리 자동화

```cpp
#include <fstream>
void Read(const char* Path) {
    std::ifstream File(Path);
    if (!File.is_open()) return;
    Parse();  // 예외가 나든 조기 return이든 소멸자가 정리
}
```

RAII를 쓰면 성공 경로와 실패 경로를 따로 기억하지 않아도 된다. 함수가 어떤 이유로 종료되든 소멸자가 정리를 담당하므로 누수와 중복 정리 위험이 크게 줄어든다.

### 7. 판단 기준 요약

- 실패 모델을 먼저 분류한다: 버그(check)인가, 예상 가능한 실패(반환값)인가, 관찰 대상(ensure)인가
- 예외가 꺼진 코드베이스에서는 throw 대신 반환값/TOptional + check/ensure 조합이 기본이다
- 예외 사용 여부와 무관하게 자원 정리는 RAII로 자동화한다
- check에 부수 효과를 넣지 않는다 — Shipping에서 평가 자체가 사라진다
- 예외 안전 보장 수준(basic/strong/nothrow)은 함수 역할에 맞춰 정한다 — 모든 함수를 같은 수준으로 설계하지 않는다
