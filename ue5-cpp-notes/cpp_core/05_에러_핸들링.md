## 토픽 22: 예외 처리 vs UE5 에러 핸들링

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 5

### 1. 핵심 개념

C++의 `exception handling(익셉션 핸들링, 예외 처리)`은 `try` / `catch` / `throw`를 통해 오류를 호출자에게 전파하는 메커니즘이다. 장점은 정상 경로와 오류 경로를 분리해 코드를 읽기 좋게 만들고, 스택 언와인딩 중 소멸자를 호출해 RAII와 잘 결합된다는 점이다.

하지만 게임 엔진과 실시간 애플리케이션에서는 예외를 무조건 선호하지 않는다. UE5는 보통 예외를 끄고, `check` / `ensure` / `verify` 같은 assert 계열 매크로와 `bool`, `TOptional`, 반환 코드 기반 실패 보고를 더 자주 사용한다. 이유는 빌드 일관성, 숨은 control flow 최소화, 디버깅 용이성 때문이다.

예외 안전 보장은 보통 세 단계로 설명한다. `basic guarantee(베이직 개런티, 기본 예외 안전 보장)`는 객체가 유효한 상태를 유지함을, `strong guarantee(스트롱 개런티, 강한 예외 안전 보장)`는 실패 시 상태가 바뀌지 않음을, `nothrow guarantee(노스로 개런티, 예외 없음 보장)`는 아예 예외를 던지지 않음을 뜻한다. 여기에 `noexcept(노익셉트, 예외를 던지지 않음)`는 "이 함수는 예외를 던지지 않는다"는 계약을 타입 수준에서 표현해 주며, 이동 생성자나 소멸자 설계에도 중요하다.

**쉽게 감 잡기**
- 예외는 "문제가 생기면 상위 호출자로 점프한다"
- 반환값 기반 에러 처리는 "호출자가 실패를 직접 확인한다"
- `check`는 "이 조건이 깨지면 버그다", `ensure`는 "버그지만 개발 중 경고를 남기고 진행할 수도 있다"에 가깝다

**실무 기준**
- 라이브러리 코드나 일반 C++ 도메인에서는 예외가 자연스러울 수 있다
- UE5 게임플레이 코드에서는 `bool`, `TOptional`, 로그, `ensure`가 더 흔하다
- 자원 정리는 예외 사용 여부와 무관하게 RAII로 해결하는 편이 가장 안전하다

### 2. 코드 예시

```cpp
#include <iostream>
#include <optional>
#include <stdexcept>
#include <string>

int divide(int a, int b) {
    if (b == 0) {
        throw std::runtime_error("divide by zero");
    }
    return a / b;
}

std::optional<int> parseLevel(const std::string& text) {
    if (text.empty()) {
        return std::nullopt;      // ← 예외 대신 명시적 실패 반환
    }

    return std::stoi(text);
}

int main() {
    try {
        std::cout << divide(10, 2) << "\n";
        std::cout << divide(10, 0) << "\n";
    } catch (const std::exception& e) {
        std::cout << "error: " << e.what() << "\n";
    }

    if (auto level = parseLevel("")) {
        std::cout << *level << "\n";
    } else {
        std::cout << "invalid level input\n";
    }
}
```

### 3. 왜 중요한가 (게임/UE5 맥락)

게임 면접에서는 "예외 문법"보다 "왜 UE5가 예외보다 명시적 실패 처리와 assert 계열 매크로를 더 선호하는가"를 설명하는 능력이 더 중요하다. 실무에서도 치명적 버그, 복구 가능한 실패, 사용자 입력 오류를 같은 방식으로 다루면 디버깅과 유지보수가 무너진다.

### 4. 면접 예상 Q&A

**Q1: 예외 처리의 장점은 무엇인가요?**
> A: 정상 경로와 오류 경로를 분리해 코드 가독성을 높이고, 깊은 호출 스택에서도 오류를 자연스럽게 상위로 전파할 수 있다는 점이 장점이다. 또한 스택 언와인딩과 소멸자 호출 덕분에 RAII와 잘 맞는다.

**Q2: UE5는 왜 예외를 거의 쓰지 않나요?**
> A: 빌드 설정에서 예외를 끄는 경우가 많고, 실시간 게임 코드에서는 숨은 제어 흐름보다 명시적 실패 경로를 선호하기 때문이다. 치명적 버그는 즉시 드러내고, 복구 가능한 실패는 `bool`/`ensure`/로그로 처리하는 쪽이 엔진 철학과 잘 맞는다.

**Q3: `check`, `ensure`, `verify`의 차이를 어떻게 설명하나요?**
> A: `check`는 깨지면 바로 중단해야 하는 내부 불변식 검증에 가깝다. `ensure`는 문제를 기록하되 개발 중 계속 진행해 후속 상황을 관찰할 수 있게 해 주며, `verify`는 표현식 평가 자체가 중요할 때도 사용한다.

**Q4: 예외를 쓰지 않으면 자원 정리는 어떻게 하나요?**
> A: 예외를 쓰든 안 쓰든 자원 정리는 RAII로 해결하는 것이 기본이다. 스마트 포인터, 락 가드, 파일 핸들 래퍼를 쓰면 조기 return이나 실패 경로에서도 정리가 자동화된다.

**Q5: 예외 안전 보장 레벨은 무엇인가요?**
> A: `basic guarantee`는 실패 후에도 객체가 유효하다는 보장, `strong guarantee`는 실패 시 상태가 바뀌지 않는 보장, `nothrow guarantee`는 실패해도 예외를 던지지 않는 보장이다. 면접에서는 copy-and-swap이나 transactional update 예시와 함께 설명하면 좋다.

**Q6: `noexcept(노익셉트, 예외를 던지지 않음)`는 왜 중요한가요?**
> A: 이 함수가 예외를 던지지 않는다는 계약을 명시해 최적화와 안전한 호출 경로 선택에 영향을 주기 때문이다. 특히 이동 생성자나 소멸자에 `noexcept`가 있으면 표준 컨테이너가 복사 대신 이동을 더 적극적으로 사용할 수 있다.

**Q7: `std::optional(에스티디 옵셔널, 값이 없을 수 있는 반환)`이나 `std::expected(에스티디 익스펙티드, 값 또는 오류 반환)` 같은 타입은 언제 쓰나요?**
> A: 실패가 "예상 가능한 비정상 결과"일 때 유용하다. 값이 없을 수 있는 조회는 `optional`, 실패 이유까지 함께 전달하고 싶으면 `expected`가 적합하다. 즉 예외를 켜지 않는 코드베이스에서 명시적 실패 모델로 특히 강력하다.

### 5. 흔한 실수

- **예외를 안 쓴다고 RAII까지 버리기**: 반환값 기반 처리라고 해서 정리 코드를 수동으로 흩뿌리면 조기 return 경로에서 누수가 생긴다. 에러 전파 방식과 자원 정리 방식은 분리해서 생각해야 한다.
- **`check`를 사용자 입력 검증에 사용**: `check`는 내부 버그를 드러내는 도구에 가깝다. 파일 없음, 잘못된 입력, 네트워크 실패처럼 예상 가능한 오류는 명시적으로 처리해야 한다.
- **모든 실패를 같은 수준으로 다루기**: 치명적 버그, 재시도 가능한 실패, 값 부재 상황은 성격이 다르다. `check` / `ensure` / `optional` / 반환 코드 / 로그를 상황에 맞게 구분해야 한다.
