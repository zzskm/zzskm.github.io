## 토픽 22: 예외 처리 vs UE5 에러 핸들링

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 5

### 1. 핵심 개념

핵심 용어: `RAII`, `ensure`, `check`, `스택 언와인딩`.

C++의 `exception handling(익셉션 핸들링, 예외 처리)`은 `try` / `catch` / `throw`를 통해 오류를 호출자에게 전파하는 메커니즘이다. 장점은 정상 경로와 오류 경로를 분리해 코드를 읽기 좋게 만들고, 스택 언와인딩 중 소멸자를 호출해 RAII와 잘 결합된다는 점이다.

하지만 게임 엔진과 실시간 애플리케이션에서는 예외를 무조건 선호하지 않는다. UE5는 보통 예외를 끄고, `check` / `ensure` / `verify` 같은 assert 계열 매크로와 `bool`, `TOptional`, 반환 코드 기반 실패 보고를 더 자주 사용한다. 이유는 빌드 일관성, 숨은 control flow 최소화, unwind 비용/바이너리 정책 관리, 디버깅 용이성 때문이다.

예외 안전 보장은 보통 세 단계로 설명한다. `basic guarantee(베이직 개런티, 기본 예외 안전 보장)`는 객체가 유효한 상태를 유지함을, `strong guarantee(스트롱 개런티, 강한 예외 안전 보장)`는 실패 시 상태가 바뀌지 않음을, `nothrow guarantee(노스로 개런티, 예외 없음 보장)`는 아예 예외를 던지지 않음을 뜻한다. 여기에 `noexcept(노익셉트, 예외를 던지지 않음)`는 "이 함수는 예외를 던지지 않는다"는 계약을 타입 수준에서 표현해 주며, 이동 생성자나 소멸자 설계에도 중요하다. 반대로 UB(Undefined Behavior, 정의되지 않은 동작)는 예외처럼 "정의된 실패 경로"가 아니라 언어 계약이 깨진 상태라는 점에서 완전히 다르다.

**쉽게 감 잡기** - 예외는 "문제가 생기면 상위 호출자로 점프한다" - 반환값 기반 에러 처리는 "호출자가 실패를 직접 확인한다" - `check`는 "이 조건이 깨지면 버그다", `ensure`는 "버그지만 개발 중 경고를 남기고 진행할 수도 있다"에 가깝다 - UB는 "실패를 보고한다"가 아니라 "무슨 일이든 일어날 수 있다"에 가깝다

### 2. 코드 예시

```cpp
void Read(const char* path) {
    FILE* f = fopen(path, "r");
    if (!f) return;
    Parse();
    fclose(f);
}
```

Parse에서 예외가 나거나 중간 return 경로가 추가되면 fclose에 도달하지 못해 자원 누수가 생길 수 있다.

### 3. 왜 중요한가 (게임/UE5 맥락)

게임 면접에서는 "예외 문법"보다 "왜 UE5가 예외보다 명시적 실패 처리와 assert 계열 매크로를 더 선호하는가"를 설명하는 능력이 더 중요하다. 실무에서도 치명적 버그, 복구 가능한 실패, 사용자 입력 오류를 같은 방식으로 다루면 디버깅과 유지보수가 무너진다.

### 4. 면접 예상 Q&A

**Q1: 표준 C++ 예외 처리와 UE5식 에러 핸들링의 차이는 무엇인가?**
> A: 표준 C++은 throw/catch로 오류 전파를 모델링하지만, UE5는 대개 예외를 끄고 check/ensure/로그/반환값으로 실패를 다룬다.

**Q2: C++ 예외(try/catch/throw)의 가장 큰 장점과, UE5가 예외를 끄는 주된 이유는?**
> A: 예외는 에러 경로와 정상 경로를 분리해 가독성을 높이지만, 스택 언와인딩 비용·코드 크기 증가·결정론적 실행 보장 어려움 때문에 UE5는 예외를 끈다.

**Q3: 예외 처리의 장점은 무엇인가요?**
> A: 정상 경로와 오류 경로를 분리해 코드 가독성을 높이고, 깊은 호출 스택에서도 오류를 자연스럽게 상위로 전파할 수 있다는 점이 장점이다. 또한 스택 언와인딩과 소멸자 호출 덕분에 RAII와 잘 맞는다.

**Q4: UE5는 왜 예외를 거의 쓰지 않나요?**
> A: 빌드 설정에서 예외를 끄는 경우가 많고, 실시간 게임 코드에서는 숨은 제어 흐름보다 명시적 실패 경로를 선호하기 때문이다. unwind 정책과 모듈 경계의 일관성을 맞추기 쉬운 점도 크다. 치명적 버그는 즉시 드러내고, 복구 가능한 실패는 `bool`/`ensure`/로그로 처리하는 쪽이 엔진 철학과 잘 맞는다.

### 5. 흔한 실수

- **예외를 안 쓴다고 RAII까지 버리기**: 반환값 기반 처리라고 해서 정리 코드를 수동으로 흩뿌리면 조기 return 경로에서 누수가 생긴다. 에러 전파 방식과 자원 정리 방식은 분리해서 생각해야 한다.
- **`check`를 사용자 입력 검증에 사용**: `check`는 내부 버그를 드러내는 도구에 가깝다. 파일 없음, 잘못된 입력, 네트워크 실패처럼 예상 가능한 오류는 명시적으로 처리해야 한다.
