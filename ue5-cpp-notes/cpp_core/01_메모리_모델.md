## 토픽 1: 스택 vs 힙

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 없음

### 1. 문제 상황

핵심 용어: `SP (스택 포인터)`, `stack frame`, `범위(scope)`, `메모리 누수(memory leak)`.

게임 코드에는 한 프레임 안에서만 쓰이는 임시 벡터와, 레벨이 끝날 때까지 살아야 하는 액터 데이터가 공존한다. 이 둘의 수명 차이를 인식하지 못하면 두 가지 사고가 반복된다.

첫째, 함수 안에서만 유효한 지역 변수의 주소를 외부로 넘겨 dangling pointer(댕글링 포인터)가 발생한다. 프로그램은 즉시 죽지 않고 몇 프레임 뒤에 랜덤 크래시를 일으킨다. 재현이 어려워 디버깅 시간이 폭증한다.

둘째, 모든 객체를 습관적으로 new로 힙에 올리면 불필요한 할당 비용·캐시 미스·메모리 누수가 누적된다. 프레임 예산을 갉아먹고, 장시간 플레이 시 메모리 단편화로 OOM(Out Of Memory) 크래시가 터진다.

스택과 힙의 차이는 '빠르다 vs 느리다'가 아니다. ownership(소유권)과 lifetime(수명)이 판단 기준이다. 이 기준을 모르면 매번 같은 버그를 반복한다.

### 2. 핵심 개념

스택(stack): 함수 호출과 함께 자동 할당되고, 함수 반환 시 자동 해제되는 LIFO 메모리 영역이다.

힙(heap): 프로그래머가 런타임에 명시적으로 할당하고, 명시적으로(또는 스마트 포인터를 통해) 해제하는 동적 메모리 영역이다.

핵심 판단 기준은 단 하나다. 이 데이터의 수명이 현재 스코프 안에서 끝나는가, 아닌가. 스코프 안에서 끝나면 스택, 넘어가면 힙이다.

#### 이 개념을 이해하려면

- 함수 호출 스택(call stack)의 LIFO 구조를 이해해야 한다.
- 포인터(pointer)와 주소(address)의 기본 개념이 필요하다.
- C++에서 변수의 스코프(scope)와 자동 소멸 규칙을 알아야 한다.

#### 이 개념이 영향을 주는 것

- 동적 메모리 관리(new/delete)의 올바른 사용 패턴.
- 스마트 포인터(unique_ptr, shared_ptr) 선택 기준.
- RAII 패턴 — 스택 기반 자동 정리를 자원 관리에 일반화한 설계.
- 메모리 할당자(allocator) 커스터마이징 전략.

### 3. 내부 동작 / 메커니즘

#### 스택 포인터 이동

스택 할당은 SP(Stack Pointer) 레지스터를 이동시키는 단일 명령어로 완료된다. x86-64에서는 `sub rsp, N`으로 N바이트를 확보하고, 함수 반환 시 `add rsp, N`으로 되돌린다. O(1) 연산이다.

할당과 해제 모두 포인터 산술 한 번이므로 분기도 없고 시스템 콜도 없다. 이것이 스택이 빠른 이유의 전부다.

```asm
; 함수 프롤로그 — 32바이트 지역 변수 확보
push rbp
mov  rbp, rsp
sub  rsp, 32       ; SP를 32바이트 아래로 이동

; 함수 에필로그 — 복원
mov  rsp, rbp
pop  rbp
ret
```

#### 힙 할당자 메커니즘

힙 할당(malloc/new)은 free list(프리 리스트)에서 요청 크기 이상의 빈 블록을 탐색한다. 적합한 블록을 찾으면 분할하고 메타데이터를 기록한다. 못 찾으면 OS에 페이지를 요청한다(sbrk/mmap).

해제(free/delete)는 블록을 free list에 반환하고, 인접 빈 블록과 병합(coalescing)을 시도한다. 멀티스레드 환경에서는 이 과정 전체가 락 또는 CAS(Compare-And-Swap)로 보호된다.

결과적으로 힙 할당은 스택 대비 수십~수백 배 느리다. 비용의 본질은 탐색 + 동기화 + 시스템 콜이다.

#### 수명(lifetime) vs 저장 기간(storage duration)

C++ 표준은 네 가지 저장 기간을 정의한다: automatic, dynamic, static, thread_local. 스택 변수는 automatic storage duration이고, 힙 변수는 dynamic storage duration이다.

수명(lifetime)은 객체가 유효한 구간이다. automatic 객체는 스코프가 수명을 결정한다. dynamic 객체는 프로그래머(또는 스마트 포인터)가 수명을 결정한다. 저장 기간과 수명은 겹치지만 동일하지 않다. 예를 들어, placement new로 스택 버퍼에 객체를 생성하면 저장 기간은 automatic이지만 수명은 수동 관리한다.

#### 소유권(ownership) 모델

스택 객체의 소유자는 해당 스코프다. 스코프가 끝나면 소유권도 소멸도 자동이다. 별도 관리가 필요 없다.

힙 객체의 소유자는 명확하지 않다. new를 호출한 코드가 소유자인가, 포인터를 전달받은 코드가 소유자인가? 이 모호함이 누수와 이중 해제의 근본 원인이다.

해결책은 소유권을 타입으로 표현하는 것이다. unique_ptr은 단독 소유, shared_ptr은 공유 소유를 컴파일 타임에 강제한다. 이것이 RAII로 이어지는 핵심 아이디어다.

### 4. 자주 틀리는 포인트

- 수명 분석 없이 습관적으로 new부터 쓴다. 지역 변수로 충분한 경우에도 힙을 쓰면 불필요한 비용과 누수 위험이 생긴다.
- "크기가 크면 힙"이라고 외운다. 기준은 크기가 아니라 수명이다. 큰 배열도 스코프 안에서만 쓰이면 스택(또는 alloca)이 적합하다.
- 지역 변수의 주소를 함수 밖으로 반환한다. 함수가 끝나면 스택 프레임이 사라지므로 반환된 포인터는 dangling pointer가 된다.
- delete를 호출했으니 안전하다고 생각한다. delete 후에도 포인터 변수 자체는 여전히 같은 주소를 가리킨다. 이후 역참조하면 use-after-free다. delete 직후 nullptr로 초기화하는 습관이 필요하다.

#### dangling pointer 발생 과정 (단계별)

1단계: 함수 foo() 안에서 지역 변수 int x = 10;이 스택에 할당된다. x의 주소는 예를 들어 0x7ffd1234이다.

2단계: return &x;로 0x7ffd1234를 호출자에게 반환한다.

3단계: foo()가 반환되면서 스택 프레임이 해제된다. 0x7ffd1234는 더 이상 x에 속하지 않는다.

4단계: 호출자가 반환받은 포인터를 역참조한다. 이 시점에 0x7ffd1234에는 다음 함수 호출의 지역 변수가 덮어씌워져 있을 수 있다.

5단계: 결과는 정의되지 않은 동작(Undefined Behavior)이다. 0이 나올 수도 있고, 쓰레기 값이 나올 수도 있고, 크래시할 수도 있다. 가장 위험한 경우는 '정상처럼 보이는' 것이다. 버그가 숨어서 나중에 터진다.

```cpp
int* foo() {
    int x = 10;      // 스택에 할당
    return &x;       // x의 주소 반환 — 함수 종료 후 무효
}

void bar() {
    int* p = foo();  // p는 dangling pointer
    // 이 사이에 다른 함수 호출이 스택을 덮어쓸 수 있다
    int val = *p;    // Undefined Behavior
}
```

### 5. 언제 쓰는가 / 언제 쓰지 않는가

#### 스택을 쓰는 경우

- 객체 수명이 현재 함수(또는 블록) 안에서 끝난다.
- 크기가 컴파일 타임에 결정되고, 스택 크기 제한(보통 1~8MB)을 넘지 않는다.
- 소유권 이전이 필요 없다. 생성한 스코프에서 소비하고 버린다.
- 예: FVector 임시 계산, 루프 인덱스, 함수 매개변수.

#### 힙을 쓰는 경우

- 객체 수명이 생성 스코프를 넘어야 한다. 함수 반환 후에도 살아 있어야 한다.
- 크기가 런타임에 결정된다. 가변 길이 배열, 사용자 입력 기반 버퍼 등.
- 소유권 이전이 필요하다. 생성자와 소비자가 다른 모듈에 있다.
- 다형성(polymorphism)이 필요하다. 베이스 클래스 포인터로 파생 객체를 다루려면 힙 할당이 일반적이다.
- 예: AActor 서브클래스, 비동기 로딩 리소스, 레벨 전환 후에도 유지되는 데이터.

#### 힙을 쓰지 않아야 하는 경우

- 한 함수 안에서 생성하고 같은 함수 안에서 소멸하는 객체. 스택이면 충분하다.
- 매 프레임 수천 번 호출되는 경로에서의 소량 할당. 힙 할당 비용이 누적되어 프레임 드롭을 유발한다.
- 명확한 수명 분석 없이 '혹시 모르니까' new를 쓰는 경우. 이것은 설계 부재다.

### 6. 코드 예시 (bad → good)

#### 사례 1: 지역 변수 주소 반환

Bad: 스택 지역 변수의 주소를 반환하면 dangling pointer가 된다.

```cpp
// BAD: dangling pointer 반환
int* GetValue() {
    int local = 42;
    return &local;  // local은 함수 종료 시 소멸
}
```

Good: 값을 복사로 반환한다. 힙이 필요하면 스마트 포인터를 쓴다.

```cpp
// GOOD: 값 복사 반환 — 안전하고 빠르다 (RVO 적용)
int GetValue() {
    int local = 42;
    return local;
}

// GOOD: 힙이 필요한 경우 — 소유권을 unique_ptr로 명시
std::unique_ptr<int> GetValueOnHeap() {
    return std::make_unique<int>(42);
}
```

#### 사례 2: 포인터 재할당에 의한 메모리 누수

Bad: 이전 힙 블록의 주소를 잃어버려 delete할 수 없다.

```cpp
// BAD: 메모리 누수
void Process() {
    int* p = new int(5);
    p = new int(10);  // 첫 번째 할당(5)의 주소 유실 → 누수
    delete p;         // 두 번째 할당(10)만 해제됨
}
```

Good: 스마트 포인터가 이전 블록을 자동 해제한다. 수동 delete가 필요 없다.

```cpp
// GOOD: unique_ptr로 자동 관리
void Process() {
    auto p = std::make_unique<int>(5);
    p = std::make_unique<int>(10);  // 이전 블록 자동 해제
    // 함수 종료 시 마지막 블록도 자동 해제
}
```

#### 사례 3: 불필요한 힙 할당

Bad: 한 함수 안에서만 쓰는 객체를 힙에 올리면 비용만 늘고 이점은 없다.

```cpp
// BAD: 불필요한 힙 할당
void CalculateDamage() {
    int* damage = new int(100);   // 왜 힙?
    int result = *damage * 2;
    delete damage;
}
```

Good: 스택 변수로 충분하다. 할당/해제 비용 제로, 누수 위험 제로.

```cpp
// GOOD: 스택이면 충분하다
void CalculateDamage() {
    int damage = 100;
    int result = damage * 2;
}
```

### 7. 판단 기준 요약

- 질문 1: 이 데이터의 수명이 현재 스코프 안에서 끝나는가? → Yes면 스택, No면 힙(또는 스마트 포인터).
- 질문 2: 크기가 컴파일 타임에 결정되는가? → No면 힙 할당이 불가피하다.
- 질문 3: 소유권 이전이 필요한가? → Yes면 unique_ptr/shared_ptr로 힙 객체를 관리한다.
- 질문 4: 다형성이 필요한가? → Yes면 베이스 포인터 + 힙 할당이 일반적이다.
- 기본 원칙: 스택을 기본으로 쓰고, 위 질문 중 하나라도 Yes일 때만 힙을 쓴다.

#### RAII로의 연결

스택의 자동 소멸 특성을 자원 관리에 일반화한 것이 RAII(Resource Acquisition Is Initialization)다. 힙 메모리뿐 아니라 파일 핸들, 뮤텍스 락, GPU 리소스 등 모든 자원을 스코프 기반으로 관리할 수 있다. 스택 vs 힙을 이해하면 RAII는 자연스러운 다음 단계다. 토픽 5(RAII)에서 이 패턴을 본격적으로 다룬다.

## 토픽 2: 포인터 vs 참조

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 없음

### 1. 문제 상황

핵심 용어: `별칭(alias)`, `const FType&`, `주소(address)`.

함수 매개변수로 T*를 받으면 호출자는 여러 가지를 의심한다. null을 넘겨도 되는가? 이 함수가 delete를 호출하는가? 반환 후에도 유효한가? 이 모호함이 버그의 시작이다.

포인터와 참조는 단순히 문법 차이가 아니라 API 계약의 차이다. 어떤 것을 선택하느냐에 따라 null 허용 여부, 소유권 기대, 수명 책임이 달라진다.

### 2. 핵심 개념

#### 정의

포인터(pointer)는 메모리 주소를 값으로 담는 독립적인 변수다. null일 수 있고, 재할당 가능하며, *와 ->로 간접 참조를 명시한다.

참조(reference)는 기존 객체의 별칭(alias)이다. 선언 시 초기화가 필수이고, null이 될 수 없으며, 한 번 바인딩되면 다른 대상으로 바꿀 수 없다.

내부 구현은 유사하지만, 언어 수준의 보장이 다르다. 참조는 "반드시 유효한 대상이 있다"는 계약을 코드로 표현한다.

#### 최소 동작 모델

- 포인터: 선언 → 주소 대입(&obj) → 역참조(*ptr) → 재대입 가능 → null 가능
- 참조: 선언과 동시에 초기화(int& r = x) → 원본과 동일한 문법으로 사용 → 재바인딩 불가 → null 불가

#### 이 개념을 이해하려면

- 스택 vs 힙 (토픽 1): 포인터가 가리키는 대상의 수명을 판단하는 기반
- 메모리 주소의 기본 개념

#### 이 개념이 영향을 주는 것

- const correctness (토픽 7): const 참조 vs const 포인터 선택
- 스마트 포인터 (토픽 4): raw pointer의 소유권 문제를 해결
- 캐스팅 4종 (토픽 11): 포인터 타입 변환과 strict aliasing

### 3. 내부 동작 / 메커니즘

#### 참조의 구현

대부분의 컴파일러는 참조를 포인터와 동일한 기계어로 구현한다. 차이는 언어 수준에서만 존재한다. sizeof(참조)는 원본 타입의 크기를 반환하고, sizeof(포인터)는 주소 크기(64-bit에서 8바이트)를 반환한다.

이 구현 동일성 때문에 '참조는 포인터와 같다'고 오해하기 쉽다. 핵심은 컴파일러가 참조에 대해 null 불가와 재바인딩 불가를 강제한다는 점이다. 이 보장이 코드의 안전성과 가독성을 높인다.

#### strict aliasing 규칙

C++은 서로 호환되지 않는 타입의 포인터로 같은 메모리를 접근하는 것을 금지한다(strict aliasing rule). float*를 int*로 재해석하여 역참조하면 UB다.

예외는 char, unsigned char, std::byte 타입뿐이다. 비트 패턴을 안전하게 복사하려면 std::memcpy를 사용해야 한다.

#### 비소유 관찰자 패턴

현대 C++에서 raw pointer의 주된 역할은 '비소유 관찰자(non-owning observer)'다. 소유권은 스마트 포인터나 컨테이너가 담당하고, raw pointer는 '대상을 가리키되 수명에 관여하지 않는다'는 의도를 표현한다. 이 패턴을 따르면 raw pointer에서 delete를 호출할 일이 없다.

### 4. 자주 틀리는 포인트

- 반드시 유효한 대상을 받는 함수에 포인터를 쓴다 → API 계약이 약해지고, 호출 측에서 null 검사를 강요받는다. const T&가 맞다.
- 참조를 재바인딩 가능한 포인터처럼 오해한다 → 참조에 대입하면 대상 객체의 값이 바뀌는 것이지, 참조 자체가 다른 곳을 가리키는 것이 아니다.
- 널 포인터를 역참조하여 참조를 만든다 → int& ref = *static_cast<int*>(nullptr)은 즉시 UB다. 참조는 항상 유효한 객체로 초기화해야 한다.
- 소유권이 있는 포인터와 관찰용 포인터를 구분하지 않는다 → raw pointer가 delete 책임까지 지면 누수와 이중 해제가 생긴다.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

참조를 쓰는 경우: 대상이 항상 유효하고, 재바인딩이 불필요하고, null이 의미 없을 때. 함수 매개변수의 기본 선택은 const T&다.

포인터를 쓰는 경우: null이 유효한 상태('없음'을 표현)이거나, 가리키는 대상을 런타임에 바꿔야 하거나, 배열/오프셋 연산이 필요할 때.

소유권이 있는 포인터가 필요하면: raw pointer 대신 std::unique_ptr 또는 std::shared_ptr을 사용한다.

선택 흐름: ① null이 의미 있는가? → 포인터 ② 아니면 → const T& ③ 수정이 필요하면 → T& ④ 소유권이 필요하면 → 스마트 포인터

### 6. 코드 예시 (bad → good)

#### Bad: 널 참조 생성

```cpp
int& ref = *static_cast<int*>(nullptr);  // UB!
```

#### Good: 유효한 객체로 초기화

```cpp
int value = 10;
int& ref = value;  // 항상 유효
```

#### Bad: strict aliasing 위반

```cpp
float f = 3.14f;
int* ip = reinterpret_cast<int*>(&f);
int bits = *ip;  // UB: strict aliasing 위반
```

#### Good: memcpy로 비트 패턴 복사

```cpp
float f = 3.14f;
int bits;
std::memcpy(&bits, &f, sizeof(bits));  // 안전
```

#### Bad: 불필요한 포인터 매개변수

```cpp
void PrintName(const std::string* name) {
    if (!name) return;  // null 검사 강제
    std::cout << *name;
}
```

#### Good: const 참조 매개변수

```cpp
void PrintName(const std::string& name) {
    std::cout << name;  // null 불가, 검사 불필요
}
```

### 7. 판단 기준 요약

- 대상이 항상 존재하는가? → 참조
- null이 유효한 상태인가? → 포인터
- 재바인딩이 필요한가? → 포인터
- 읽기 전용 전달인가? → const T&
- 소유권이 필요한가? → 스마트 포인터 (raw pointer로 소유하지 마라)
- 의도가 불분명하면 const T&를 기본값으로 시작하라.

## 토픽 3: 동적 메모리 관리

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 1, 2

### 1. 문제 상황

핵심 용어: `placement new`, `delete[]`, `해제`.

게임은 런타임에 적, 아이템, 이펙트를 생성하고 파괴한다. 컴파일 타임에 개수를 알 수 없으므로 동적 할당이 필수다.

문제는 할당 자체가 아니라 해제다. 누가 delete를 호출하는지 불명확하면 누수, 이중 해제, use-after-free가 발생한다. 장시간 플레이 시 누수가 누적되면 메모리 부족으로 크래시한다.

### 2. 핵심 개념

#### 정의

동적 메모리 관리(dynamic memory management)는 런타임에 메모리를 할당하고, 적절한 시점에 해제하는 일이다. C++에서는 new/delete, new[]/delete[], placement new가 기본 도구이고, 현대 C++에서는 스마트 포인터와 컨테이너가 이 부담을 대체한다.

핵심 질문은 '누가 해제 책임을 지는가'다. 이 질문에 코드로 답하지 못하면 버그가 생긴다. 따라서 소유권을 타입으로 표현하는 것이 현대 C++의 기본 원칙이다.

#### 최소 동작 모델

- new T(args): 힙에 sizeof(T) 할당 → 생성자 호출 → 주소 반환
- delete p: 소멸자 호출 → 힙에 메모리 반납
- new T[n] / delete[] p: 배열 버전. 짝이 맞아야 한다. new[]에 delete를 쓰면 UB
- placement new: 이미 확보된 메모리 위에 생성자만 호출. 할당 없이 객체를 배치하는 기법. 해제 시 소멸자를 수동 호출해야 한다

#### 이 개념을 이해하려면

- 스택 vs 힙 (토픽 1): 왜 동적 할당이 필요한지의 배경
- 포인터 (토픽 2): 동적 할당의 결과물은 포인터다

#### 이 개념이 영향을 주는 것

- 스마트 포인터 (토픽 4): new/delete를 자동화하는 RAII 래퍼
- RAII (토픽 5): 소유권을 객체 수명에 묶는 패턴
- 메모리 단편화 & 커스텀 할당 (토픽 25): 반복 할당의 성능 문제 해결

### 3. 내부 동작 / 메커니즘

#### new vs malloc

C의 malloc은 메모리 블록만 할당한다. C++의 new는 할당 + 생성자 호출을 한 번에 수행한다. delete는 소멸자 호출 + 메모리 반납이다. 따라서 new/delete와 malloc/free를 섞으면 생성자/소멸자 호출이 빠지거나 힙 메타데이터가 꼬인다.

new는 내부적으로 operator new를 호출하여 메모리를 확보한 뒤 생성자를 실행한다. 이 과정을 분리하고 싶을 때 placement new를 사용한다.

#### 소유권 표현의 진화

raw new/delete → 해제 책임이 주석이나 문서에만 존재. 코드 리뷰와 유지보수가 어렵다.

std::unique_ptr → 단독 소유. 복사 불가. 소멸자에서 자동 delete. 대부분의 동적 할당에 적합하다.

std::shared_ptr → 공유 소유. 참조 카운트로 마지막 소유자가 해제. 공유가 반드시 필요할 때만 사용한다.

std::vector → 동적 배열의 기본 선택. new[]를 직접 쓸 이유가 거의 없다.

### 4. 자주 틀리는 포인트

- new[]로 할당하고 delete로 해제한다 → 배열 원소의 소멸자가 호출되지 않고 힙 메타데이터가 손상된다. 반드시 delete[]를 써야 한다.
- 해제 책임을 코드가 아닌 주석에만 남긴다 → 주석은 컴파일러가 강제하지 않는다. 소유권은 타입으로 표현해야 한다.
- 예외 경로에서 delete에 도달하지 못한다 → new 후 예외가 발생하면 누수. 스마트 포인터로 감싸면 예외 경로에서도 소멸자가 호출된다.
- placement new로 생성한 객체를 delete로 해제한다 → placement new는 메모리를 할당하지 않았으므로 소멸자를 수동 호출해야 한다.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

raw new/delete를 직접 쓰는 경우: 커스텀 할당자 구현, placement new, 레거시 코드 유지보수. 그 외에는 스마트 포인터나 컨테이너를 우선 사용한다.

std::vector를 쓰는 경우: 동적 크기 배열이 필요할 때. new[]를 직접 쓸 이유가 거의 없다.

풀/아레나 할당을 고려하는 경우: 같은 크기의 객체를 반복 생성/파괴할 때. 일반 힙 할당의 단편화와 성능 오버헤드를 줄인다.

### 6. 코드 예시 (bad → good)

#### Bad: 예외 시 누수

```cpp
void foo() {
    int* p = new int(42);
    risky();         // 예외 발생 시 delete 도달 불가
    delete p;
}
```

#### Good: 스마트 포인터

```cpp
void foo() {
    auto p = std::make_unique<int>(42);
    risky();  // 예외 발생해도 소멸자에서 자동 해제
}
```

#### Bad: 소유권 불명확

```cpp
Widget* CreateWidget() {
    return new Widget();  // 호출자가 delete 해야 하나?
}
```

#### Good: 반환 타입으로 소유권 명시

```cpp
std::unique_ptr<Widget> CreateWidget() {
    return std::make_unique<Widget>();  // 소유권이 호출자에게 이전됨
}
```

### 7. 판단 기준 요약

- 동적 배열이 필요한가? → std::vector
- 단독 소유 힙 객체가 필요한가? → std::unique_ptr
- 공유 소유가 반드시 필요한가? → std::shared_ptr
- 기존 메모리 위에 객체를 배치해야 하는가? → placement new
- raw new/delete를 써야 하는가? → 커스텀 할당자나 레거시 코드가 아니면 No
- '누가 해제하는가'를 코드로 답하지 못하면 설계가 잘못된 것이다.

## 토픽 4: 스마트 포인터

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 3

### 1. 문제 상황

핵심 용어: `weak_ptr`, `TSharedPtr`, `단독`, `참조 카운트(reference count)`.

raw new/delete의 근본 문제는 소유권이 코드에 드러나지 않는다는 점이다. Widget*를 반환하는 함수가 있을 때, 호출자가 delete 해야 하는지 알 수 없다.

예외, 조기 반환, 복잡한 분기에서 delete 누락은 시간문제다. 반대로 이중 delete는 힙 손상으로 이어진다. 소유권을 사람이 기억하는 설계는 규모가 커질수록 반드시 실패한다.

### 2. 핵심 개념

#### 정의

스마트 포인터(smart pointer)는 소유권 정책을 타입으로 표현하는 RAII 래퍼다. '포인터를 편리하게 만든 것'이 아니라 '소유권을 코드로 강제하는 것'이 본질이다.

std::unique_ptr — 단독 소유. 복사 불가, 이동만 가능. 기본 deleter 시 원시 포인터와 동일 크기(제로 오버헤드).

std::shared_ptr — 공유 소유. 참조 카운트로 마지막 소유자가 해제. 제어 블록(control block) 오버헤드가 있다.

std::weak_ptr — 공유 소유를 관찰만 하는 비소유 참조. 참조 카운트를 증가시키지 않으므로 순환 참조를 끊는 데 사용한다.

#### 최소 동작 모델

- unique_ptr: make_unique로 생성 → 이동(std::move)으로 소유권 이전 → 스코프 끝에서 자동 delete
- shared_ptr: make_shared로 생성(할당 1회) → 복사 시 카운트 증가 → 소멸 시 카운트 감소 → 0이 되면 delete
- weak_ptr: shared_ptr로부터 생성 → lock()으로 유효성 검사 → 유효하면 shared_ptr 획득, 소멸됐으면 빈 포인터

#### 이 개념을 이해하려면

- 동적 메모리 관리 (토픽 3): 소유권 문제의 배경
- 포인터 (토픽 2): raw pointer의 한계를 알아야 스마트 포인터의 가치를 이해한다

#### 이 개념이 영향을 주는 것

- RAII (토픽 5): 스마트 포인터는 RAII의 대표적 구현
- move semantics (토픽 16): unique_ptr은 이동으로만 소유권 이전
- UE5 스마트 포인터 (카테고리 2 / 토픽 9): TSharedPtr, TWeakObjectPtr 등 엔진 자체 구현

### 3. 내부 동작 / 메커니즘

#### unique_ptr의 구현

unique_ptr은 내부에 raw pointer 하나만 들고 있다. 복사 생성자와 복사 대입이 삭제(= delete)되어 있으므로 단독 소유가 컴파일 타임에 강제된다. 기본 deleter(std::default_delete)는 빈 클래스이므로 EBO(Empty Base Optimization)에 의해 추가 크기가 없다.

custom deleter를 지정하면 deleter 타입이 unique_ptr의 템플릿 인자에 포함되므로, deleter 크기만큼 unique_ptr 크기가 커진다.

#### shared_ptr의 구현

shared_ptr은 raw pointer + 제어 블록 포인터 두 개를 가진다. 제어 블록에는 강한 참조 카운트, 약한 참조 카운트, deleter, allocator가 포함된다.

make_shared는 객체와 제어 블록을 한 번의 할당으로 연속 배치한다. 할당 횟수가 줄고 캐시 효율이 좋아진다. 단점은 약한 참조가 남아 있으면 객체 메모리까지 유지된다는 점이다.

참조 카운트 증감은 atomic 연산이므로 스레드 안전하다. 그러나 가리키는 객체 자체의 접근은 별도 동기화가 필요하다.

#### custom deleter

스마트 포인터는 메모리뿐 아니라 파일 핸들, GPU 리소스, C API 자원도 관리할 수 있다. unique_ptr<FILE, decltype(&fclose)>는 fclose를 자동 호출한다. 이렇게 RAII를 일반 자원에 확장하는 것이 custom deleter의 핵심 가치다.

### 4. 자주 틀리는 포인트

- 공유가 필요 없는데 shared_ptr을 기본값처럼 쓴다 → 참조 카운트 오버헤드 + 소유권이 모호해진다. 기본은 unique_ptr이다.
- 같은 raw pointer로 shared_ptr을 여러 개 만든다 → 제어 블록이 중복 생성되어 이중 해제(double free)가 발생한다. make_shared를 사용하거나 기존 shared_ptr을 복사해야 한다.
- 순환 참조를 만들고 weak_ptr로 끊지 않는다 → A→B, B→A 양방향 shared_ptr은 카운트가 0이 되지 않아 영원히 해제되지 않는다.
- unique_ptr을 복사하려 한다 → 컴파일 에러. std::move로 이동해야 한다. 이동 후 원래 포인터는 nullptr이 된다.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

선택 순서: ① 스택 값 타입으로 충분한가? → 동적 할당 자체를 피한다 ② 단독 소유인가? → unique_ptr ③ 공유 소유가 반드시 필요한가? → shared_ptr ④ 순환 관계가 있는가? → 한쪽을 weak_ptr로

비소유 관찰이 필요하면: raw pointer 또는 weak_ptr을 사용한다. raw pointer는 수명이 보장된 대상을 관찰할 때, weak_ptr은 수명이 불확실한 shared_ptr 대상을 관찰할 때 쓴다.

UE5에서는: UObject 계열은 GC가 관리하므로 std 스마트 포인터를 쓰지 않는다. non-UObject 자원에 TSharedPtr, TUniquePtr을 사용한다.

### 6. 코드 예시 (bad → good)

#### Bad: 이중 제어 블록

```cpp
int* p = new int(10);
std::shared_ptr<int> a(p);
std::shared_ptr<int> b(p);  // 제어 블록 2개 → double free!
```

#### Good: make_shared 또는 복사

```cpp
auto a = std::make_shared<int>(10);
auto b = a;  // 같은 제어 블록 공유
```

#### Bad: 순환 참조

```cpp
struct Node {
    std::shared_ptr<Node> next;
};
auto a = std::make_shared<Node>();
auto b = std::make_shared<Node>();
a->next = b;
b->next = a;  // 순환 → 영원히 해제 안 됨
```

#### Good: weak_ptr로 순환 끊기

```cpp
struct Node {
    std::weak_ptr<Node> next;  // 비소유 참조
};
```

### 7. 판단 기준 요약

- 스택 값 타입 > unique_ptr > shared_ptr — 왼쪽이 기본, 오른쪽은 정당한 이유가 있을 때만
- unique_ptr: 단독 소유, 제로 오버헤드, 대부분의 동적 할당에 적합
- shared_ptr: 공유 소유가 반드시 필요한 경우에만. 참조 카운트 비용을 감수할 이유가 있어야 한다
- weak_ptr: 순환 참조 끊기 + 수명 불확실한 대상 관찰
- raw pointer: 비소유 관찰용. delete를 호출하면 안 된다

## 토픽 5: RAII(알에이아이아이, 자원 획득은 초기화)

> 난이도: ★★★ | 예상 시간: 1시간 | 선행: 토픽 4

### 1. 문제 상황

핵심 용어: `Resource Acquisition Is Initialization`, `수명(lifetime)`, `생성자`, `소멸자`.

뮤텍스를 잠그고 unlock을 잊으면 데드락이 발생한다. 파일을 열고 close를 빠뜨리면 핸들이 고갈된다. GPU 리소스를 생성하고 해제를 누락하면 VRAM이 차올라 프레임이 떨어진다.

이 문제들의 공통점은 '나중에 꼭 정리해야 하는 것'을 사람이 기억하는 구조라는 점이다. 예외, 조기 반환, 복잡한 분기 — 정리 코드에 도달하지 못하는 경로는 언제든 생긴다.

### 2. 핵심 개념

#### 정의

RAII(Resource Acquisition Is Initialization)는 자원의 획득을 객체의 생성에, 자원의 해제를 객체의 소멸에 묶는 패턴이다. 생성자에서 자원을 획득하고, 소멸자에서 자원을 반납한다.

C++은 스코프를 벗어날 때 지역 객체의 소멸자를 반드시 호출한다. 예외가 발생하면 스택 되감기(stack unwinding) 과정에서도 소멸자가 호출된다. 이 언어 보장이 RAII를 강력하게 만든다.

RAII는 메모리에 국한되지 않는다. 뮤텍스 락(lock_guard), 파일 핸들(ifstream), GPU 리소스, 임시 상태 복구 — 소멸자로 정리할 수 있는 모든 자원에 적용된다.

#### 최소 동작 모델

- RAII 객체 생성 → 생성자에서 자원 획득(lock, fopen, new 등)
- 스코프 내에서 자원 사용
- 스코프 종료 → 소멸자 자동 호출 → 자원 해제(unlock, fclose, delete 등)
- 예외 발생 시에도 스택 되감기가 소멸자를 호출하므로 자원이 안전하게 정리된다

#### 이 개념을 이해하려면

- 스마트 포인터 (토픽 4): RAII의 대표적 구현
- 스택 vs 힙 (토픽 1): 스코프 기반 자동 소멸의 원리

#### 이 개념이 영향을 주는 것

- 에러 핸들링 (토픽 22): 예외 안전성의 기반
- 멀티스레딩 (토픽 21): lock_guard/scoped_lock 등 락 관리
- UE5 GC 심화 (카테고리 2 / 토픽 6): GC 환경에서 RAII의 역할

### 3. 내부 동작 / 메커니즘

#### 소멸자 호출 순서

같은 스코프에 여러 RAII 객체가 있으면 선언의 역순으로 소멸된다. 이 순서는 자원 간 의존성이 있을 때 중요하다. 예: 파일을 열고 그 위에 버퍼를 만들었다면, 버퍼가 먼저 소멸되고 파일이 닫혀야 한다.

#### 스택 되감기와 예외 안전

예외가 발생하면 catch 블록을 찾을 때까지 스택 프레임이 하나씩 해제된다. 이 과정에서 각 프레임의 지역 RAII 객체 소멸자가 호출된다. 이것이 RAII가 try/catch 없이도 자원을 안전하게 정리하는 원리다.

단, 소멸자에서 예외를 던지면 이중 예외가 발생하여 std::terminate가 호출된다. 소멸자는 반드시 noexcept를 유지해야 한다.

#### GC와의 비교

가비지 컬렉션(GC)은 주로 메모리 회수에 초점이 있고, 소멸 시점이 비결정적이다. RAII는 임의의 자원을 스코프 끝에서 즉시 정리한다. 뮤텍스 해제처럼 정확한 시점이 중요한 자원 관리에는 RAII가 필수다.

### 4. 자주 틀리는 포인트

- RAII 객체를 이름 없는 임시 객체로 만든다 → std::lock_guard<std::mutex>(mtx); 는 즉시 소멸되어 락을 전혀 보호하지 못한다. 반드시 변수에 담아야 한다.
- 자원 획득과 해제를 서로 먼 위치에 둔다 → 이것은 RAII를 쓰지 않는다는 신호다. lock()과 unlock()이 떨어져 있으면 lock_guard로 감싸야 한다.
- RAII를 메모리 관리에만 한정 짓는다 → 파일, 락, 소켓, 임시 플래그 복구, 트랜잭션 롤백까지 모두 RAII로 관리할 수 있다.
- 소멸자에서 예외를 던진다 → 스택 되감기 중 이중 예외는 프로그램을 즉시 종료시킨다.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

RAII를 쓰는 경우: '나중에 반드시 정리/복구해야 하는 것'이 있을 때. 메모리, 락, 파일, 소켓, GPU 리소스, 임시 상태 변경 모두 해당한다.

RAII를 쓰지 않아도 되는 경우: 정리가 필요 없는 단순 값 타입. int, float 같은 POD 타입은 소멸자 자체가 필요 없다.

기존 수동 관리를 RAII로 전환하는 신호: lock()/unlock(), open()/close(), begin()/end() 쌍이 떨어져 있으면 래퍼 클래스로 감싸라.

### 6. 코드 예시 (bad → good)

#### Bad: 수동 락 관리

```cpp
std::mutex mtx;
void Unsafe(int& counter) {
    mtx.lock();
    ++counter;
    // 여기서 예외 발생 시 unlock 누락 → 데드락
    mtx.unlock();
}
```

#### Good: lock_guard로 RAII

```cpp
void Safe(int& counter) {
    std::lock_guard<std::mutex> lock(mtx);
    ++counter;  // 예외 발생해도 소멸자에서 자동 unlock
}
```

#### Bad: 이름 없는 임시 RAII 객체

```cpp
std::lock_guard<std::mutex>(mtx);  // 즉시 소멸! 락 보호 안 됨
// 임계 구역 작업
```

#### Good: 변수에 담기

```cpp
std::lock_guard<std::mutex> lock(mtx);  // 스코프 끝까지 유지
// 임계 구역 작업
```

#### 응용: 임시 상태 복구

```cpp
class ScopedFlag {
    bool& flag_;
    bool original_;
public:
    ScopedFlag(bool& flag, bool value)
        : flag_(flag), original_(flag) { flag_ = value; }
    ~ScopedFlag() { flag_ = original_; }
};

void Process() {
    ScopedFlag guard(bIsProcessing, true);
    // 작업 수행... 예외 발생해도 원래 값 복구
}
```

### 7. 판단 기준 요약

- '나중에 정리해야 하는 것'이 있으면 RAII로 감싸라
- lock()/unlock(), open()/close() 쌍이 떨어져 있으면 래퍼가 필요하다는 신호
- 소멸자에서 예외를 던지지 마라 — noexcept를 유지하라
- RAII 객체는 반드시 이름 있는 변수에 담아야 한다
- GC가 있어도 정확한 해제 시점이 필요한 자원은 RAII가 필수다

## 토픽 6: 메모리 레이아웃

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 1, 2

### 1. 문제 상황

핵심 용어: `패딩(Padding)`, `false sharing`, `패딩(padding)`, `정렬(alignment)`.

struct { char A; int B; char C; }의 sizeof를 6이라고 답하면 틀린다. 실제로는 12바이트다. 소스 코드의 순서와 메모리 배치는 다르다.

캐시 효율, 직렬화, 네트워크 패킷, SIMD, 저수준 디버깅은 모두 메모리 레이아웃에 의존한다. 이를 모르면 sizeof 오판, 직렬화 버그, 성능 병목을 놓친다.

### 2. 핵심 개념

#### 정의

메모리 레이아웃(memory layout)은 프로그램의 코드와 데이터가 메모리에 배치되는 구조다. 프로세스 메모리는 text(코드), data(초기화된 전역), bss(미초기화 전역), stack, heap 영역으로 나뉜다.

객체 수준에서는 멤버 순서, 정렬(alignment), 패딩(padding), vtable 포인터 등이 실제 배치를 결정한다.

#### 최소 동작 모델

- alignment: CPU는 자연 정렬된 주소(4바이트 타입은 4의 배수 주소)에서 가장 효율적으로 접근한다
- padding: 컴파일러는 alignment를 맞추기 위해 멤버 사이에 빈 바이트를 삽입한다
- 배열: 원소가 연속 배치되어 cache line에 함께 로드된다 (cache locality의 근거)
- cache line: CPU가 메모리를 가져오는 최소 단위 (보통 64바이트)

#### 이 개념을 이해하려면

- 스택 vs 힙 (토픽 1): 데이터가 어느 영역에 위치하는지
- 포인터 (토픽 2): 주소와 메모리 접근의 기초

#### 이 개념이 영향을 주는 것

- cache locality & data-oriented design (토픽 26): 배치가 성능에 미치는 영향
- 직렬화 (카테고리 2 / 토픽 18): 메모리 배치와 저장 형식의 관계
- 메모리 단편화 & 커스텀 할당 (토픽 25): 정렬과 단편화의 상호작용

### 3. 내부 동작 / 메커니즘

#### 정렬과 패딩 규칙

각 타입에는 alignment 요구사항이 있다. char은 1, short은 2, int/float은 4, double/pointer는 8(64-bit). 컴파일러는 멤버를 그 타입의 alignment 배수 주소에 배치한다.

struct { char A; int B; char C; } → A(1) + pad(3) + B(4) + C(1) + pad(3) = 12바이트. 구조체 전체 alignment는 가장 큰 멤버의 alignment(여기서는 4)를 따른다.

멤버를 크기 내림차순으로 배치하면 패딩이 줄어든다. struct { int B; char A; char C; } → B(4) + A(1) + C(1) + pad(2) = 8바이트.

#### 프로세스 메모리 영역

- text: 컴파일된 기계어 코드. 읽기 전용
- data: 초기화된 전역/static 변수. 프로그램 시작 시 로드
- bss: 미초기화 전역/static. 0으로 초기화. 실행 파일 크기에 포함되지 않음
- stack: 함수별 지역 변수. 높은 주소 → 낮은 주소로 성장
- heap: 동적 할당 영역. 낮은 주소 → 높은 주소로 성장

#### false sharing

서로 다른 스레드가 같은 cache line 내의 다른 변수를 수정하면 cache line 무효화가 반복되어 성능이 급격히 떨어진다. 이를 false sharing이라 한다.

해결: alignas(64)로 스레드별 데이터를 별도 cache line에 배치한다.

### 4. 자주 틀리는 포인트

- sizeof를 멤버 크기 합으로 계산한다 → 패딩을 고려하지 않은 것. 반드시 sizeof(T)로 확인해야 한다.
- 멤버 선언 순서 = 메모리 배치라고 가정한다 → 같은 접근 지정자 안에서는 순서가 보장되지만, 패딩이 삽입되므로 연속 배치와는 다르다.
- reinterpret_cast로 무관한 타입 간 변환한다 → strict aliasing 위반이며 UB다. char/byte로의 변환만 예외.
- volatile을 동기화 도구로 사용한다 → volatile은 캐시를 무시하지 않는다. false sharing 해결은 alignment로 한다.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

레이아웃을 의식해야 하는 경우: 네트워크 패킷 구조체 설계, 직렬화/역직렬화, SIMD 연산, 캐시 최적화, 크로스 플랫폼 바이너리 호환.

레이아웃을 무시해도 되는 경우: 일반적인 비즈니스 로직, 높은 수준의 게임플레이 코드. 프로파일링 결과가 없다면 멤버 정렬 최적화는 후순위다.

판단 기준: '이 구조체가 대량으로 순회되거나, 바이너리로 읽고 쓰이는가?' — 그렇다면 레이아웃을 설계하라.

### 6. 코드 예시 (bad → good)

#### Bad: 패딩 낭비

```cpp
struct Bad { char A; int B; char C; };  // 12 bytes
```

#### Good: 멤버 재배치

```cpp
struct Good { int B; char A; char C; };  // 8 bytes
```

#### Bad: false sharing

```cpp
struct Counters {
    int thread1_count;  // 같은 cache line
    int thread2_count;  // → false sharing!
};
```

#### Good: cache line 분리

```cpp
struct alignas(64) Counter { int value; };
Counter thread1_count;
Counter thread2_count;  // 별도 cache line
```

### 7. 판단 기준 요약

- sizeof를 직접 확인하라 — 수동 계산은 패딩을 놓친다
- 대량 순회 구조체는 멤버를 크기 내림차순으로 배치하라
- 멀티스레드에서 공유 데이터는 cache line 단위로 분리하라
- 비트 패턴 재해석이 필요하면 memcpy를 사용하라 (reinterpret_cast는 UB 위험)
- 프로파일링 없이 레이아웃 최적화를 하지 마라 — 정확한 병목을 먼저 확인해야 한다
