## 토픽 1: 스택 vs 힙

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 없음

### 1. 핵심 개념

핵심 용어: `SP (스택 포인터)`, `stack frame`, `범위(scope)`, `메모리 누수(memory leak)`.

스택과 힙은 단순히 "빠른 메모리 vs 느린 메모리"로 외우기보다 수명과 관리 방식이 다르다고 이해하는 것이 정확하다. 스택은 함수 호출과 함께 자동으로 할당/해제되는 영역이고, 힙은 필요 시점에 동적으로 확보하고 명시적으로 관리하는 영역이다.

스택 객체는 범위를 벗어나면 자동으로 정리되어 수명 추적이 쉽다. 반면 힙 객체는 유연하지만 누수, 이중 해제, dangling pointer 같은 문제가 생기기 쉽다. 따라서 "크기가 커서 힙", "작아서 스택" 같은 단순 기준보다 ownership(오너십, 소유권)과 lifetime(라이프타임, 수명)이 판단 기준이 된다.

### 2. 코드 예시

```cpp
int* foo() {
    int x = 10;
    return &x;
}
```

지역 변수 x는 함수 종료 시 스택에서 사라진다.
반환된 포인터는 댕글링 포인터가 된다.

### 3. 왜 중요한가 (게임/UE5 맥락)

게임 코드에서는 수많은 임시 값과 긴 수명의 데이터가 섞인다. 어떤 데이터가 자동 수명으로 충분한지, 어떤 데이터가 동적 수명을 가져야 하는지 구분하지 못하면 성능과 안정성 모두 흔들린다.

### 4. 면접 예상 Q&A

**Q1: 스택과 힙의 가장 큰 차이를 '수명과 비용' 관점에서 설명하라.**
> A: 스택은 스코프 기반으로 자동 해제되어 매우 빠르고, 힙은 런타임에 임의 수명을 갖지만 할당/해제 비용과 단편화 비용이 더 크다.

**Q2: 스택 메모리와 힙 메모리의 수명 관리 방식 차이를 설명하라.**
> A: 스택은 함수 호출/반환에 따라 자동으로 할당·해제되고, 힙은 프로그래머가 명시적으로 할당·해제해야 한다.

**Q3: 스택과 힙의 가장 큰 차이는 무엇인가요?**
> A: 수명과 관리 방식이다. 스택은 범위 기반 자동 수명, 힙은 동적 수명과 명시적 관리가 핵심이다.

**Q4: 스택이 항상 더 좋다고 볼 수는 없나요?**
> A: 아니다. 수명이 함수 범위를 넘거나 크기를 실행 중 결정해야 하면 힙이 필요하다. 결론적으로 성능보다 수명 모델이 선택 기준이다.

### 5. 흔한 실수

- 수명 요구를 보지 않고 습관적으로 힙 할당부터 한다.
- 지역 변수 주소를 함수 밖으로 넘긴다.

## 토픽 2: 포인터 vs 참조

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 없음

### 1. 핵심 개념

핵심 용어: `별칭(alias)`, `const FType&`, `주소(address)`, `별칭(alias)`.

포인터는 주소를 값으로 담는 변수이고, 참조는 기존 객체의 별칭(alias)이다. 포인터는 null일 수 있고 재할당 가능하며, 간접 참조가 명시적이다. 참조는 선언 시 초기화가 필요하고 일반적으로 null이 없으며, 한 번 바인딩되면 다른 대상을 가리키게 바꿀 수 없다.

핵심은 "선택지가 두 개 있다"보다 의도를 드러내는 것이다. 선택적 대상이면 포인터, 반드시 존재하는 입력이면 참조가 더 자연스럽다. 실무에서는 raw pointer를 "비소유 관찰자(non-owning observer)"로 쓰는 경우가 많으며, 소유권 표현은 스마트 포인터나 컨테이너 쪽이 더 적절하다. 또한 포인터 연산과 별개로 strict aliasing 규칙도 이해해야 한다. 서로 무관한 타입 포인터로 같은 메모리를 함부로 해석하면 정의되지 않은 동작이 될 수 있다.

### 2. 코드 예시

```cpp
int& ref = *static_cast<int*>(nullptr);
```

널 포인터를 역참조하여 참조를 만들고 있다.
참조는 항상 유효한 객체를 가리켜야 하므로 즉시 UB다.

### 3. 왜 중요한가 (게임/UE5 맥락)

API 설계에서 포인터와 참조 선택은 곧 null 허용 여부와 소유권 기대를 드러낸다. 이 신호가 불분명하면 호출자도 수명과 유효성 책임을 오해하기 쉽다.

### 4. 면접 예상 Q&A

**Q1: 포인터와 참조를 언제 각각 선택해야 하나?**
> A: 항상 유효한 대상과 재바인딩 불필요라면 참조를, null 가능성·재대상화·선택적 전달이 필요하면 포인터를 쓴다.

**Q2: 포인터와 참조의 핵심 차이를 설명하라.**
> A: 포인터는 주소를 값으로 담는 변수로 null 가능·재할당 가능하고, 참조는 기존 객체의 별칭으로 선언 시 초기화 필수·null 불가·재바인딩 불가이다.

**Q3: 포인터와 참조의 가장 큰 차이는 무엇인가요?**
> A: 포인터는 독립적인 주소 값이고, 참조는 기존 객체의 별칭이다. 결론적으로 선택성과 재바인딩 가능성이 포인터 쪽에 있다.

**Q4: 함수 인자로 언제 포인터보다 참조가 더 좋은가요?**
> A: null이 허용되지 않고 반드시 유효한 객체를 받아야 할 때다. 호출자 입장에서 계약이 더 명확해진다.

### 5. 흔한 실수

- 반드시 존재해야 하는 입력에도 포인터를 써 계약을 약하게 만든다.
- 참조를 재바인딩 가능한 포인터처럼 오해한다.

## 토픽 3: 동적 메모리 관리

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 1, 2

### 1. 핵심 개념

핵심 용어: `placement new`, `delete[]`, `해제`, `delete[]`.

동적 메모리 관리는 실행 중 필요한 만큼 메모리를 할당하고 적절한 시점에 해제하는 일이다. C++에서는 `new/delete`, `new[]/delete[]`, placement new, 그리고 modern C++에서는 스마트 포인터와 컨테이너를 통해 이 부담을 줄인다.

핵심은 할당 그 자체보다 소유권이다. 누가 해제 책임을 가지는지 불명확하면 누수와 이중 해제가 생긴다. 따라서 raw `new/delete`를 직접 쓰는 경우는 점점 줄이고, 자원 소유를 타입으로 표현하는 방식이 기본이 되었다. 또한 반복적인 생성/파괴 패턴이 보이면 pool, arena allocator 같은 별도 전략을 고려해야 한다.

### 2. 코드 예시

```cpp
void foo() {
    int* p = new int(42);
    risky();
    delete p;
}
```

risky()에서 예외 발생 시 delete p에 도달하지 못해 메모리 누수가 발생한다.

### 3. 왜 중요한가 (게임/UE5 맥락)

동적 할당은 필요하지만, 잘못 관리하면 장시간 플레이에서 메모리 누수와 파편화로 이어진다. 게임 엔진은 할당이 잦기 때문에 "할당을 덜 하거나, 더 명확하게 하거나, 풀링한다"는 관점이 중요하다.

### 4. 면접 예상 Q&A

**Q1: C++에서 동적 메모리 관리는 무엇을 직접 책임지는 문제인가?**
> A: 할당 자체보다 소유권, 해제 시점, 예외 안전성, 배열/단일 객체의 짝 맞춤까지 모두 책임지는 문제다.

**Q2: Modern C++에서 raw new/delete 사용이 줄어드는 이유는?**
> A: 스마트 포인터와 컨테이너가 소유권과 해제를 타입으로 표현하므로 raw new/delete가 불필요해졌다.

**Q3: `new/delete`를 직접 쓰는 것이 점점 줄어드는 이유는 무엇인가요?**
> A: 수동 해제가 실수하기 쉽기 때문이다. 결론적으로 스마트 포인터와 컨테이너가 더 안전한 기본값이 되었다.

**Q4: `new[]`와 `delete[]`를 짝으로 써야 하는 이유는 무엇인가요?**
> A: 배열 원소 개수와 소멸 규칙이 다르기 때문이다. `delete`로 해제하면 정의되지 않은 동작이 될 수 있다.

### 5. 흔한 실수

- `new[]`로 할당한 메모리를 `delete`로 해제한다.
- 해제 책임을 문서와 코드 어느 쪽에도 명확히 남기지 않는다.

## 토픽 4: 스마트 포인터

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 3

### 1. 핵심 개념

핵심 용어: `weak_ptr`, `TSharedPtr`, `단독`, `참조 카운트(reference count)`.

스마트 포인터는 포인터와 소유권 관리를 결합한 도구다. `std::unique_ptr`는 단독 소유, `std::shared_ptr`는 공유 소유, `std::weak_ptr`는 공유 소유를 관찰만 하는 약한 참조를 표현한다.

핵심은 "포인터를 더 편하게 만든 것"이 아니라 "소유권을 타입으로 표현한다"는 점이다. 또한 `unique_ptr`와 `shared_ptr`는 custom deleter를 받을 수 있어 일반 `delete`가 아닌 해제 방식도 안전하게 캡슐화할 수 있다. 파일 핸들, 커스텀 풀, C API 리소스와 연결할 때 특히 중요하다.

### 2. 코드 예시

```cpp
int* p = new int(10);
std::shared_ptr<int> a(p);
std::shared_ptr<int> b(p);
```

같은 원시 포인터로 별도의 제어 블록 2개가 생성된다.
각각 독립적으로 해제를 시도하여 이중 해제(double free)가 발생한다.

### 3. 왜 중요한가 (게임/UE5 맥락)

소유권이 명확해지면 메모리 버그가 줄고 리뷰도 쉬워진다. 게임 코드에서는 메모리뿐 아니라 핸들, 파일, 그래픽 리소스 같은 비메모리 자원도 많아 custom deleter와 RAII가 실전적으로 중요하다.

### 4. 면접 예상 Q&A

**Q1: 스마트 포인터를 도입하는 핵심 이유는 무엇인가?**
> A: 소유권을 타입으로 표현해 누수, 이중 해제, 예외 경로 누락을 줄이기 위해서다.

**Q2: unique_ptr과 shared_ptr의 차이를 설명하라.**
> A: unique_ptr은 단독 소유(하나의 포인터만 자원 보유), shared_ptr은 공유 소유(참조 카운트로 마지막 소유자가 해제)이다.

**Q3: weak_ptr은 왜 필요한가?**
> A: shared_ptr 간 순환 참조를 끊고, 대상 객체의 수명에 영향을 주지 않으면서 관찰하기 위해 필요하다.

**Q4: `unique_ptr`와 `shared_ptr`의 핵심 차이는 무엇인가요?**
> A: 단독 소유냐 공유 소유냐의 차이다. 기본 선택은 `unique_ptr`이고, 정말 공유 책임이 필요할 때만 `shared_ptr`를 고려하는 편이 안전하다.

### 5. 흔한 실수

- 공유가 필요하지 않은데도 `shared_ptr`를 기본값처럼 쓴다.
- 순환 참조를 만들고 `weak_ptr`로 끊지 않는다.

## 토픽 5: RAII(알에이아이아이, 자원 획득은 초기화)

> 난이도: ★★★ | 예상 시간: 1시간 | 선행: 토픽 4

### 1. 핵심 개념

핵심 용어: `Resource Acquisition Is Initialization`, `수명(lifetime)`, `생성자`, `소멸자`.

RAII(Resource Acquisition Is Initialization, 알에이아이아이, 자원 획득은 초기화)는 자원 획득과 객체 수명을 묶는 기법이다. 객체가 생성될 때 자원을 획득하고, 범위를 벗어날 때 소멸자에서 자동 정리한다. 메모리뿐 아니라 락, 파일, 소켓, 임시 상태 복구에도 적용된다.

핵심은 "정리 코드를 나중에 기억하는 방식"을 버리는 것이다. 범위 기반 수명에 묶어 두면 예외, 조기 반환, 조건 분기에서도 자원이 안전하게 정리된다.

### 2. 코드 예시

```cpp
std::lock_guard<std::mutex>(mtx);
// 임계 구역 작업
```

이름 없는 임시 객체라서 문장 끝에서 즉시 소멸된다.
락을 잡자마자 풀어 버려 임계 구역을 보호하지 못한다.

### 3. 왜 중요한가 (게임/UE5 맥락)

게임 코드는 예외보다 조기 반환이 더 흔하고, 락과 핸들, 등록/해제가 널리 퍼져 있다. RAII를 이해하면 메모리뿐 아니라 "나중에 꼭 원상복구해야 하는 것들"을 안전하게 관리할 수 있다.

### 4. 면접 예상 Q&A

**Q1: RAII가 C++에서 강력한 이유는 무엇인가?**
> A: 자원 정리 시점을 사람이 기억하지 않고 객체 수명에 결박할 수 있어서 예외와 early return에도 안정적으로 동작하기 때문이다.

**Q2: RAII의 핵심 아이디어를 한 문장으로 설명하라.**
> A: 생성자에서 자원을 획득하고, 소멸자에서 자동으로 해제한다(Resource Acquisition Is Initialization).

**Q3: RAII가 예외 안전성(exception safety)에 중요한 이유는?**
> A: 예외가 발생해도 스택 되감기(stack unwinding) 중 소멸자가 호출되어 자원이 자동 해제된다.

**Q4: RAII(알에이아이아이, 자원 획득은 초기화)의 핵심 아이디어는 무엇인가요?**
> A: 자원 수명을 객체 수명에 묶는 것이다. 결론적으로 정리를 사람이 기억하지 않아도 되게 만드는 기법이다.

### 5. 흔한 실수

- 자원 획득과 해제를 서로 먼 위치에 둔다.
- 해제 함수를 사람이 직접 호출하는 구조를 유지한다.

## 토픽 6: 메모리 레이아웃

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 1, 2

### 1. 핵심 개념

핵심 용어: `패딩(Padding)`, `false sharing`, `패딩(padding)`, `정렬(alignment)`.

메모리 레이아웃은 객체가 메모리에 어떻게 배치되는지를 다룬다. 변수 주소, 배열의 연속성, 구조체의 멤버 배치, 정렬(alignment), 패딩(padding), 가상 함수가 있는 객체의 숨은 포인터, 그리고 cache line 단위의 배치까지 이해해야 한다.

핵심은 "소스 코드에 보이는 순서와 메모리 배치 규칙은 같지 않을 수 있다"는 점이다. 정렬을 맞추기 위해 패딩이 들어갈 수 있고, 잘못된 재해석 캐스팅은 strict aliasing과 정렬 제약을 동시에 깨뜨릴 수 있다. 또한 멀티스레드 환경에서는 서로 다른 변수가 같은 cache line을 공유해 false sharing을 일으킬 수도 있다. 따라서 로우 레벨 최적화는 반드시 언어 규칙 안에서 해야 한다.

### 2. 코드 예시

```cpp
struct S { char a; int b; char c; };
size_t size = sizeof(char) + sizeof(int) + sizeof(char);
// size == 6 ?
```

멤버 크기 합(6)에는 패딩이 반영되지 않는다.
실제 sizeof(S)는 정렬 규칙 때문에 12바이트가 된다.

### 3. 왜 중요한가 (게임/UE5 맥락)

캐시 효율, 직렬화, 네트워크 패킷, SIMD, 저수준 디버깅은 모두 메모리 레이아웃과 연결된다. 면접에서는 "왜 sizeof가 예상보다 큰가요?" 같은 질문으로 자주 확인한다.

### 4. 면접 예상 Q&A

**Q1: 메모리 레이아웃을 이해해야 하는 이유는 무엇인가?**
> A: 데이터가 코드/전역/스택/힙 중 어디에 놓이는지 알면 수명, 초기화 비용, 캐시 친화성, 버그 원인을 더 정확히 판단할 수 있기 때문이다.

**Q2: 구조체의 sizeof가 멤버 크기 합보다 큰 이유를 설명하라.**
> A: alignment 요구사항을 충족하기 위해 컴파일러가 멤버 사이에 패딩(padding)을 삽입하기 때문이다.

**Q3: 배열의 메모리 연속성이 중요한 이유는?**
> A: 캐시 지역성(cache locality)이 좋아져 순차 접근 성능이 크게 향상된다.

**Q4: 구조체 크기가 멤버 크기 합보다 커질 수 있는 이유는 무엇인가요?**
> A: 정렬 제약을 맞추기 위해 패딩이 들어갈 수 있기 때문이다. 결론적으로 메모리 접근 효율을 위한 규칙이 크기에 영향을 준다.

### 5. 흔한 실수

- 멤버 선언 순서와 실제 패딩 효과를 고려하지 않는다.
- 임의의 `reinterpret_cast`로 메모리 레이아웃을 직접 해석한다.
