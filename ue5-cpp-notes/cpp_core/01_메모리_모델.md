## 토픽 1: 스택 vs 힙

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 없음

### 1. 핵심 개념

스택과 힙은 단순히 "빠른 메모리 vs 느린 메모리"로 외우기보다 수명과 관리 방식이 다르다고 이해하는 것이 정확하다. 스택은 함수 호출과 함께 자동으로 할당/해제되는 영역이고, 힙은 필요 시점에 동적으로 확보하고 명시적으로 관리하는 영역이다.

스택 객체는 범위를 벗어나면 자동으로 정리되어 수명 추적이 쉽다. 반면 힙 객체는 유연하지만 누수, 이중 해제, dangling pointer 같은 문제가 생기기 쉽다. 따라서 "크기가 커서 힙", "작아서 스택" 같은 단순 기준보다 수명과 소유권이 판단 기준이 된다.

### 2. 코드 예시

```cpp
void Example()
{
    int LocalValue = 10;         // 스택
    int* DynamicValue = new int(20); // 힙

    std::cout << LocalValue << ", " << *DynamicValue << "\n";

    delete DynamicValue;
}
```

### 3. 왜 중요한가 (게임/UE5 맥락)

게임 코드에서는 수많은 임시 값과 긴 수명의 데이터가 섞인다. 어떤 데이터가 자동 수명으로 충분한지, 어떤 데이터가 동적 수명을 가져야 하는지 구분하지 못하면 성능과 안정성 모두 흔들린다.

### 4. 면접 예상 Q&A

**Q1: 스택과 힙의 가장 큰 차이는 무엇인가요?**
> A: 수명과 관리 방식이다. 스택은 범위 기반 자동 수명, 힙은 동적 수명과 명시적 관리가 핵심이다.

**Q2: 스택이 항상 더 좋다고 볼 수는 없나요?**
> A: 아니다. 수명이 함수 범위를 넘거나 크기를 실행 중 결정해야 하면 힙이 필요하다. 결론적으로 성능보다 수명 모델이 선택 기준이다.

**Q3: 힙 할당이 위험한 이유는 무엇인가요?**
> A: 해제를 직접 책임져야 하고, 소유권이 흐려지면 누수와 이중 해제가 생기기 때문이다. 코드가 커질수록 위험이 빠르게 증가한다.

**Q4: 큰 객체를 무조건 힙에 둬야 하나요?**
> A: 꼭 그렇지는 않다. 다만 매우 큰 지역 객체는 스택 사용량 문제를 만들 수 있어 상황에 맞게 판단해야 한다.

**Q5: 함수가 끝난 뒤 지역 변수 주소를 반환하면 왜 안 되나요?**
> A: 스택 메모리는 함수 종료와 함께 무효화되기 때문이다. 반환된 포인터는 dangling pointer가 된다.

**Q6: 실무에서 좋은 기본 원칙은 무엇인가요?**
> A: 가능하면 자동 수명과 값 타입을 우선하고, 동적 수명이 필요할 때만 힙을 선택하는 것이다. 기본값은 단순함이어야 한다.

### 5. 흔한 실수

- 수명 요구를 보지 않고 습관적으로 힙 할당부터 한다.
- 지역 변수 주소를 함수 밖으로 넘긴다.
- 힙 객체 소유권을 코드상에서 명확히 표현하지 않는다.

## 토픽 2: 포인터 vs 참조

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 없음

### 1. 핵심 개념

포인터는 주소를 값으로 담는 변수이고, 참조는 기존 객체의 별칭(alias)이다. 포인터는 null일 수 있고 재할당 가능하며, 간접 참조가 명시적이다. 참조는 선언 시 초기화가 필요하고 일반적으로 null이 없으며, 한 번 바인딩되면 다른 대상을 가리키게 바꿀 수 없다.

핵심은 "선택지가 두 개 있다"보다 의도를 드러내는 것이다. 선택적 대상이면 포인터, 반드시 존재하는 입력이면 참조가 더 자연스럽다. 또한 포인터 연산과 별개로 strict aliasing 규칙도 이해해야 한다. 서로 무관한 타입 포인터로 같은 메모리를 함부로 해석하면 정의되지 않은 동작이 될 수 있다.

### 2. 코드 예시

```cpp
void IncreaseByPointer(int* Value)
{
    if (Value)
    {
        ++(*Value);
    }
}

void IncreaseByReference(int& Value)
{
    ++Value;
}
```

### 3. 왜 중요한가 (게임/UE5 맥락)

API 설계에서 포인터와 참조 선택은 곧 null 허용 여부와 소유권 기대를 드러낸다. 이 신호가 불분명하면 호출자도 수명과 유효성 책임을 오해하기 쉽다.

### 4. 면접 예상 Q&A

**Q1: 포인터와 참조의 가장 큰 차이는 무엇인가요?**
> A: 포인터는 독립적인 주소 값이고, 참조는 기존 객체의 별칭이다. 결론적으로 선택성과 재바인딩 가능성이 포인터 쪽에 있다.

**Q2: 함수 인자로 언제 포인터보다 참조가 더 좋은가요?**
> A: null이 허용되지 않고 반드시 유효한 객체를 받아야 할 때다. 호출자 입장에서 계약이 더 명확해진다.

**Q3: 참조는 내부적으로 포인터처럼 구현될 수 있는데 왜 굳이 구분하나요?**
> A: 언어 차원 계약이 다르기 때문이다. 구현보다 "null 가능 여부와 재바인딩 금지" 같은 의미가 더 중요하다.

**Q4: `const T&`가 자주 쓰이는 이유는 무엇인가요?**
> A: 큰 객체를 복사 없이 읽기 전용으로 전달할 수 있기 때문이다. 성능과 API 의도 표현을 동시에 만족한다.

**Q5: strict aliasing을 간단히 설명해 보세요.**
> A: 서로 관련 없는 타입 포인터로 같은 메모리를 임의 해석하면 컴파일러 최적화 가정이 깨져 정의되지 않은 동작이 될 수 있다는 규칙이다. 로우 메모리 해석은 `std::memcpy`나 허용된 타입 규칙 안에서 해야 한다.

**Q6: 실무에서 포인터 사용이 과하다는 신호는 무엇인가요?**
> A: null이 될 수 없는 값까지 전부 포인터로 받아 매번 null 체크가 필요한 경우다. 의미 없는 선택지는 API를 더 약하게 만든다.

### 5. 흔한 실수

- 반드시 존재해야 하는 입력에도 포인터를 써 계약을 약하게 만든다.
- 참조를 재바인딩 가능한 포인터처럼 오해한다.
- 타입 퍼닝을 위해 임의 포인터 캐스팅을 하며 strict aliasing을 깨뜨린다.

## 토픽 3: 동적 메모리 관리

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 1, 2

### 1. 핵심 개념

동적 메모리 관리는 실행 중 필요한 만큼 메모리를 할당하고 적절한 시점에 해제하는 일이다. C++에서는 `new/delete`, `new[]/delete[]`, 그리고 modern C++에서는 스마트 포인터와 컨테이너를 통해 이 부담을 줄인다.

핵심은 할당 그 자체보다 소유권이다. 누가 해제 책임을 가지는지 불명확하면 누수와 이중 해제가 생긴다. 따라서 raw `new/delete`를 직접 쓰는 경우는 점점 줄이고, 자원 소유를 타입으로 표현하는 방식이 기본이 되었다.

### 2. 코드 예시

```cpp
int* Values = new int[3]{1, 2, 3};

for (int i = 0; i < 3; ++i)
{
    std::cout << Values[i] << "\n";
}

delete[] Values;
```

### 3. 왜 중요한가 (게임/UE5 맥락)

동적 할당은 필요하지만, 잘못 관리하면 장시간 플레이에서 메모리 누수와 파편화로 이어진다. 게임 엔진은 할당이 잦기 때문에 "할당을 덜 하거나, 더 명확하게 하거나, 풀링한다"는 관점이 중요하다.

### 4. 면접 예상 Q&A

**Q1: `new/delete`를 직접 쓰는 것이 점점 줄어드는 이유는 무엇인가요?**
> A: 수동 해제가 실수하기 쉽기 때문이다. 결론적으로 스마트 포인터와 컨테이너가 더 안전한 기본값이 되었다.

**Q2: `new[]`와 `delete[]`를 짝으로 써야 하는 이유는 무엇인가요?**
> A: 배열 원소 개수와 소멸 규칙이 다르기 때문이다. `delete`로 해제하면 정의되지 않은 동작이 될 수 있다.

**Q3: 메모리 누수와 dangling pointer는 어떻게 다른가요?**
> A: 누수는 해제하지 못해 메모리를 잃는 것이고, dangling pointer는 이미 해제된 메모리를 계속 가리키는 것이다. 둘 다 위험하지만 문제 양상이 다르다.

**Q4: 동적 메모리 관리에서 가장 중요한 질문은 무엇인가요?**
> A: "누가 이 자원을 소유하고 언제 해제하는가?"이다. 이 답이 코드상에 드러나야 한다.

**Q5: 컨테이너가 동적 메모리 관리를 어떻게 도와주나요?**
> A: 내부 할당과 해제를 캡슐화해 사용자가 생명주기보다 데이터 의미에 집중하게 해 준다. 범위 종료와 함께 정리도 자연스럽다.

**Q6: 실무에서 동적 할당 문제를 줄이는 기본 전략은 무엇인가요?**
> A: 할당 횟수를 줄이고, 소유권을 타입으로 표현하고, 반복 생성되는 대상은 풀링을 고려하는 것이다. 성능과 안전성 둘 다 여기에 연결된다.

### 5. 흔한 실수

- `new[]`로 할당한 메모리를 `delete`로 해제한다.
- 해제 책임을 문서와 코드 어느 쪽에도 명확히 남기지 않는다.
- 반복 루프 안에서 불필요한 할당을 계속 발생시킨다.

## 토픽 4: 스마트 포인터

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 3

### 1. 핵심 개념

스마트 포인터는 포인터와 소유권 관리를 결합한 도구다. `std::unique_ptr`는 단독 소유, `std::shared_ptr`는 공유 소유, `std::weak_ptr`는 공유 소유를 관찰만 하는 약한 참조를 표현한다.

핵심은 "포인터를 더 편하게 만든 것"이 아니라 "소유권을 타입으로 표현한다"는 점이다. 또한 `unique_ptr`와 `shared_ptr`는 custom deleter를 받을 수 있어 일반 `delete`가 아닌 해제 방식도 안전하게 캡슐화할 수 있다. 파일 핸들, 커스텀 풀, C API 리소스와 연결할 때 특히 중요하다.

### 2. 코드 예시

```cpp
auto FileCloser = [](FILE* Ptr)
{
    if (Ptr)
    {
        fclose(Ptr);
    }
};

std::unique_ptr<FILE, decltype(FileCloser)> FilePtr(fopen("log.txt", "w"), FileCloser);
```

### 3. 왜 중요한가 (게임/UE5 맥락)

소유권이 명확해지면 메모리 버그가 줄고 리뷰도 쉬워진다. 게임 코드에서는 메모리뿐 아니라 핸들, 파일, 그래픽 리소스 같은 비메모리 자원도 많아 custom deleter와 RAII가 실전적으로 중요하다.

### 4. 면접 예상 Q&A

**Q1: `unique_ptr`와 `shared_ptr`의 핵심 차이는 무엇인가요?**
> A: 단독 소유냐 공유 소유냐의 차이다. 기본 선택은 `unique_ptr`이고, 정말 공유 책임이 필요할 때만 `shared_ptr`를 고려하는 편이 안전하다.

**Q2: `weak_ptr`는 왜 필요한가요?**
> A: `shared_ptr`끼리 서로 참조하면 순환 참조가 생길 수 있기 때문이다. 약한 참조로 관찰만 하도록 분리하면 순환을 끊을 수 있다.

**Q3: custom deleter는 언제 유용한가요?**
> A: `delete`가 아닌 전용 해제 함수가 필요한 자원에 유용하다. 예를 들어 `fclose`, 소켓 해제, 커스텀 풀 반환 같은 경우다.

**Q4: `shared_ptr`를 조심해야 하는 이유는 무엇인가요?**
> A: 소유권이 분산돼 누가 언제 자원을 끝내는지 흐려질 수 있기 때문이다. 참조 카운팅 비용도 무시할 수 없다.

**Q5: 스마트 포인터가 있어도 RAII가 여전히 중요한 이유는 무엇인가요?**
> A: 스마트 포인터 자체가 RAII의 한 구현이기 때문이다. 자원 정리를 범위 종료와 묶는 사고방식이 더 본질적이다.

**Q6: 실무에서 스마트 포인터 선택 기준을 한 문장으로 말하면 무엇인가요?**
> A: 소유권이 하나면 `unique_ptr`, 진짜 공유해야 하면 `shared_ptr`, 관찰만 하면 `weak_ptr`다. 애매할수록 더 단순한 모델을 고른다.

### 5. 흔한 실수

- 공유가 필요하지 않은데도 `shared_ptr`를 기본값처럼 쓴다.
- 순환 참조를 만들고 `weak_ptr`로 끊지 않는다.
- custom deleter가 필요한 자원을 일반 `delete` 전제로 감싼다.

## 토픽 5: RAII(알에이아이아이, 자원 획득은 초기화)

> 난이도: ★★★ | 예상 시간: 1시간 | 선행: 토픽 4

### 1. 핵심 개념

RAII(Resource Acquisition Is Initialization, 알에이아이아이, 자원 획득은 초기화)는 자원 획득과 객체 수명을 묶는 기법이다. 객체가 생성될 때 자원을 획득하고, 범위를 벗어날 때 소멸자에서 자동 정리한다. 메모리뿐 아니라 락, 파일, 소켓, 임시 상태 복구에도 적용된다.

핵심은 "정리 코드를 나중에 기억하는 방식"을 버리는 것이다. 범위 기반 수명에 묶어 두면 예외, 조기 반환, 조건 분기에서도 자원이 안전하게 정리된다.

### 2. 코드 예시

```cpp
std::mutex Mutex;

void SafeIncrease(int& Value)
{
    std::lock_guard<std::mutex> Lock(Mutex);
    ++Value;
}
```

### 3. 왜 중요한가 (게임/UE5 맥락)

게임 코드는 예외보다 조기 반환이 더 흔하고, 락과 핸들, 등록/해제가 널리 퍼져 있다. RAII를 이해하면 메모리뿐 아니라 "나중에 꼭 원상복구해야 하는 것들"을 안전하게 관리할 수 있다.

### 4. 면접 예상 Q&A

**Q1: RAII(알에이아이아이, 자원 획득은 초기화)의 핵심 아이디어는 무엇인가요?**
> A: 자원 수명을 객체 수명에 묶는 것이다. 결론적으로 정리를 사람이 기억하지 않아도 되게 만드는 기법이다.

**Q2: RAII가 예외 안전과 어떻게 연결되나요?**
> A: 스코프를 벗어날 때 소멸자가 자동 호출되므로, 중간에 예외나 조기 반환이 있어도 자원이 정리된다. 이것이 예외 안전의 기본 토대다.

**Q3: RAII는 메모리 외에 어디에 적용되나요?**
> A: mutex 락, 파일 핸들, 소켓, 임시 플래그 변경 복구, 프로파일링 구간 측정 등 매우 넓다. 자원이라면 거의 모두 적용 가능하다.

**Q4: `lock_guard`가 RAII 예시로 자주 나오는 이유는 무엇인가요?**
> A: 락 획득과 해제를 범위 수명에 자연스럽게 묶기 때문이다. 수동 `unlock` 누락으로 인한 버그를 크게 줄인다.

**Q5: RAII와 스마트 포인터는 어떤 관계인가요?**
> A: 스마트 포인터는 RAII를 포인터 소유권에 적용한 대표 사례다. 즉 RAII가 더 큰 개념이고 스마트 포인터는 그 하위 응용이다.

**Q6: 실무에서 RAII가 부족하다는 신호는 무엇인가요?**
> A: 함수 말미에 정리 코드가 반복되고, 여러 반환 경로 때문에 해제 누락 우려가 큰 경우다. 자원을 객체로 감싸는 리팩터링이 필요하다는 신호다.

### 5. 흔한 실수

- 자원 획득과 해제를 서로 먼 위치에 둔다.
- 해제 함수를 사람이 직접 호출하는 구조를 유지한다.
- 범위 기반 수명으로 묶을 수 있는 자원을 raw 상태로 둔다.

## 토픽 6: 메모리 레이아웃

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 1, 2

### 1. 핵심 개념

메모리 레이아웃은 객체가 메모리에 어떻게 배치되는지를 다룬다. 변수 주소, 배열의 연속성, 구조체의 멤버 배치, 정렬(alignment), 패딩(padding), 가상 함수가 있는 객체의 숨은 포인터 등을 이해해야 한다.

핵심은 "소스 코드에 보이는 순서와 메모리 배치 규칙은 같지 않을 수 있다"는 점이다. 정렬을 맞추기 위해 패딩이 들어갈 수 있고, 잘못된 재해석 캐스팅은 strict aliasing과 정렬 제약을 동시에 깨뜨릴 수 있다. 따라서 로우 레벨 최적화는 반드시 언어 규칙 안에서 해야 한다.

### 2. 코드 예시

```cpp
struct Example
{
    char A;   // 1 byte
    int B;    // 4 bytes
    char C;   // 1 byte
};

std::cout << sizeof(Example) << "\n"; // 패딩 포함 가능
```

### 3. 왜 중요한가 (게임/UE5 맥락)

캐시 효율, 직렬화, 네트워크 패킷, SIMD, 저수준 디버깅은 모두 메모리 레이아웃과 연결된다. 면접에서는 "왜 sizeof가 예상보다 큰가요?" 같은 질문으로 자주 확인한다.

### 4. 면접 예상 Q&A

**Q1: 구조체 크기가 멤버 크기 합보다 커질 수 있는 이유는 무엇인가요?**
> A: 정렬 제약을 맞추기 위해 패딩이 들어갈 수 있기 때문이다. 결론적으로 메모리 접근 효율을 위한 규칙이 크기에 영향을 준다.

**Q2: 배열이 연속 메모리라는 것은 왜 중요한가요?**
> A: 인덱스 계산이 단순하고 캐시 친화적이기 때문이다. `vector`가 기본 선택이 되기 쉬운 이유도 여기에 있다.

**Q3: alignment는 왜 필요한가요?**
> A: CPU가 특정 타입을 효율적으로 읽고 쓰기 위한 제약이 있기 때문이다. 정렬이 깨지면 성능 저하나 플랫폼별 문제로 이어질 수 있다.

**Q4: strict aliasing과 메모리 레이아웃은 어떤 관계가 있나요?**
> A: 메모리 배치를 안다고 해서 임의 타입 포인터로 재해석해도 된다는 뜻은 아니다. 언어가 허용하지 않는 방식으로 메모리를 읽으면 최적화 가정이 깨질 수 있다.

**Q5: `sizeof`와 실제 의미적 데이터 크기를 어떻게 구분해야 하나요?**
> A: `sizeof`는 패딩과 숨은 구현 요소까지 포함한 저장 크기다. 논리적 필드 크기 합과 항상 같지 않다.

**Q6: 실무에서 메모리 레이아웃을 언제 특히 의식하나요?**
> A: 대량 데이터 구조, 직렬화 포맷, SIMD 최적화, 네트워크 패킷, 캐시 최적화가 중요한 경로다. 일반 코드에서는 과도한 미세 최적화보다 규칙 이해가 우선이다.

### 5. 흔한 실수

- 멤버 선언 순서와 실제 패딩 효과를 고려하지 않는다.
- 임의의 `reinterpret_cast`로 메모리 레이아웃을 직접 해석한다.
- `sizeof` 결과를 논리 데이터 크기와 같은 것으로 오해한다.
