## 토픽 26: cache locality & data-oriented design

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 6, 20

### 1. 핵심 개념

핵심 용어: `cache locality(캐시 로컬리티, 캐시 지역성)`.

`cache locality(캐시 로컬리티, 캐시 지역성)`는 CPU가 곧바로 필요할 가능성이 높은 데이터를 가까이 두는 성질이다. 연속 메모리에서 순차 접근이 빠른 이유도 여기에 있다. 결국 같은 연산량이라도 데이터를 어떻게 배치하느냐에 따라 캐시 미스 비용이 크게 달라진다.

`data-oriented design(데이터 지향 설계)`은 객체 계층보다 실제 접근 패턴과 데이터 배치를 우선해 성능을 끌어올리는 접근이다. 이것은 OOP를 부정한다기보다, 대량 반복 처리 구간에서는 책임 분리보다 메모리 배치가 더 큰 병목이 될 수 있음을 인정하는 설계 방식에 가깝다.

그래서 판단 기준은 "코드가 보기 좋은가"보다 "어떤 필드를 어떤 순서로 읽는가"가 된다. AoS(Array of Structures)와 SoA(Structure of Arrays) 선택도 추상화 취향 문제가 아니라 접근 패턴 문제이며, 멀티스레드 환경에서는 cache line 공유로 인한 false sharing까지 함께 봐야 한다.

### 2. 코드 예시

```cpp
struct ParticleAoS
{
    float X, Y, Z;
    float VX, VY, VZ;
};

std::vector<ParticleAoS> ParticlesAoS;

struct ParticleSoA
{
    std::vector<float> X;
    std::vector<float> Y;
    std::vector<float> Z;
    std::vector<float> VX;
    std::vector<float> VY;
    std::vector<float> VZ;
};

void UpdateAoS(std::vector<ParticleAoS>& Particles, float Dt)
{
    for (ParticleAoS& Particle : Particles)
    {
        Particle.X += Particle.VX * Dt;
        Particle.Y += Particle.VY * Dt;
        Particle.Z += Particle.VZ * Dt;
    }
}

void UpdateSoA(ParticleSoA& Particles, float Dt)
{
    for (size_t i = 0; i < Particles.X.size(); ++i)
    {
        Particles.X[i] += Particles.VX[i] * Dt;
        Particles.Y[i] += Particles.VY[i] * Dt;
        Particles.Z[i] += Particles.VZ[i] * Dt;
    }
}
```

이 예시는 AoS와 SoA가 단순한 문법 차이가 아니라 메모리 접근 방식의 차이라는 점을 보여 준다. 객체 하나를 통째로 다루면 AoS가 자연스럽지만, 같은 필드를 한꺼번에 순회하는 루프에서는 SoA가 캐시와 SIMD 활용 면에서 더 유리해질 수 있다.

### 3. 왜 중요한가 (게임/UE5 맥락)

렌더링 준비, 파티클, AI 감지, 물리 보조 계산처럼 같은 연산을 많은 데이터에 반복 적용하는 구간에서는 알고리즘 복잡도보다 메모리 접근 패턴이 더 큰 병목이 되기 쉽다. 이런 구간에서는 코드 구조보다 캐시 미스와 데이터 배치가 프레임 시간을 좌우한다.

### 4. 면접 예상 Q&A

**Q1: cache locality는 왜 중요한가요?**
> A: CPU는 연속된 메모리를 미리 가져오는 데 강하기 때문이다. 같은 연산량이라도 캐시 미스가 많으면 계산보다 대기 시간이 더 커져 체감 성능이 급격히 떨어진다.

**Q2: data-oriented design은 OOP와 어떻게 다르나요?**
> A: OOP는 책임과 추상화 중심으로 구조를 잡고, data-oriented design은 실제 처리되는 데이터 흐름과 레이아웃을 중심으로 구조를 잡는다. 대량 반복 처리 구간에서는 후자가 더 잘 맞는 경우가 많다.

**Q3: AoS와 SoA는 언제 각각 유리한가요?**
> A: 개별 객체 단위 조작과 캡슐화가 중요하면 AoS가 자연스럽고, 특정 필드만 대량 순회하는 연산이면 SoA가 유리하다. 결국 같이 접근되는 데이터가 무엇인지가 선택 기준이다.

**Q4: false sharing은 무엇인가요?**
> A: 서로 다른 스레드가 논리적으로 독립적인 데이터를 만지더라도 같은 cache line 안에 있으면 캐시 일관성 비용이 커지는 현상이다. 락이 없어도 병목이 생길 수 있어 찾기 까다롭다.

### 5. 흔한 실수

- 객체지향 구조가 예쁘다는 이유로 핫패스까지 포인터 추적이 많은 레이아웃을 유지한다. 설계의 일관성보다 실제 접근 패턴이 더 중요한 구간이 있다는 점을 놓치기 쉽다.
- AoS와 SoA를 이론으로만 외우고 실제 접근 패턴을 측정하지 않는다. 어떤 필드를 얼마나 자주 읽는지 모르면 레이아웃 변경도 추상적인 감에 머문다.

## 토픽 27: vector vs list vs deque / iterator invalidation

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 20, 26

### 1. 핵심 개념

`vector`, `list`, `deque`는 모두 순차 컨테이너지만 내부 레이아웃과 강점이 다르다. `vector`는 연속 메모리와 뛰어난 cache locality가 장점이고, `list`는 노드 기반으로 중간 삽입과 삭제가 쉬워 보이지만 포인터 추적과 개별 할당 비용이 크다. `deque`는 앞뒤 확장에 강하지만 메모리 구조는 `vector`보다 복잡하다.

핵심은 빅오 표만 보고 고르지 않는다는 점이다. 실무에서는 검색, 순회, 삽입 위치, 참조 안정성, iterator invalidation 규칙을 함께 봐야 하며, 같은 O 표기라도 실제 실행 비용은 메모리 배치 때문에 크게 달라질 수 있다.

특히 `vector`는 연속 메모리와 단순한 레이아웃 덕분에 순회와 알고리즘 적용에서 강하지만, 재할당 시 iterator와 reference가 무효가 될 수 있다. 반대로 `list`는 안정성을 주는 대신 순회와 메모리 효율에서 대가를 치른다.

### 2. 코드 예시

```cpp
std::vector<int> Values = {1, 2, 3, 4};
Values.reserve(8); // ← 재할당 시점을 제어

auto It = Values.begin();
Values.push_back(5); // reserve가 부족하면 iterator invalidation 가능

std::deque<int> QueueLike = {1, 2, 3};
QueueLike.push_front(0); // ← 앞쪽 확장이 잦으면 deque가 더 자연스럽다

std::list<int> Linked = {1, 2, 3};
auto NodeIt = Linked.begin();
Linked.insert(std::next(NodeIt), 99); // ← 노드 삽입은 쉽지만 순회 캐시 효율은 낮다
```

이 예시는 같은 순차 컨테이너라도 무엇을 자주 하느냐에 따라 선택이 달라진다는 점을 보여 준다. `vector`는 `reserve` 여부에 따라 iterator가 깨질 수 있고, `deque`는 앞쪽 확장이 자연스럽고, `list`는 삽입은 쉬워도 순회는 느릴 수 있다.

### 3. 왜 중요한가 (게임/UE5 맥락)

게임 코드에서는 삽입 한 번의 이론적 이득보다 프레임마다 반복되는 순회 비용이 더 크게 누적된다. 그래서 컨테이너는 O 표보다 cache locality, invalidation 규칙, 메모리 배치를 함께 보고 골라야 하며, 그 기준을 모르면 성능과 안정성을 동시에 잃기 쉽다.

### 4. 면접 예상 Q&A

**Q1: vector가 list보다 빠른 이유는 무엇인가요?**
> A: `vector`는 연속 메모리라 cache locality가 좋고, 분기와 포인터 추적이 적기 때문이다. `list`는 삽입과 삭제 자체는 쉬워 보여도 순회 시 캐시 미스와 노드 할당 비용이 커서 실제로 느린 경우가 많다.

**Q2: deque는 언제 선택하나요?**
> A: 앞뒤 양쪽에서 push/pop이 자주 일어나고, 중간 삽입은 드문 경우에 잘 맞는다. 큐 비슷한 사용 패턴이지만 `vector`보다 앞쪽 확장이 편해야 할 때 후보가 된다.

**Q3: iterator invalidation은 왜 중요하나요?**
> A: 삽입, 삭제, 재할당 뒤에 iterator, pointer, reference가 더 이상 유효하지 않을 수 있기 때문이다. 이 규칙을 모르면 크래시보다 더 위험한 use-after-free성 UB가 생길 수 있다.

**Q4: list가 맞는 경우도 있나요?**
> A: 노드의 참조 안정성이 정말 중요하고, 중간 삽입과 삭제가 매우 자주 일어나며, 순회보다 구조 변경이 더 핵심이면 후보가 될 수 있다. 다만 이런 경우도 intrusive list나 별도 핸들 구조가 더 맞는지 먼저 보는 편이 보통이다.

### 5. 흔한 실수

- `list`가 중간 삽입 O(1)이라는 이유만으로 순회 비용을 무시하고 선택한다. 실제 병목이 구조 변경인지 순회인지 먼저 구분해야 한다.
- `vector` 원소의 주소나 reference를 오래 들고 다니며 재할당을 잊는다. 이런 코드는 크래시보다 더 찾기 어려운 invalidation 버그로 이어지기 쉽다.

## 토픽 28: 멀티스레딩 심화

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 21

### 1. 핵심 개념

핵심 용어: `memory ordering(메모리 오더링, 메모리 순서 보장)`.

멀티스레딩 심화에서 중요한 것은 도구 이름보다 메모리 가시성과 대기 비용을 이해하는 것이다. `memory ordering(메모리 오더링, 메모리 순서 보장)`은 스레드 사이에서 값이 언제 보이는지를 정의하고, `mutex`와 `spinlock`은 대기 방식이 다르며, `task/job system`은 스레드를 직접 늘리기보다 작은 작업 단위로 나누어 실행하는 구조다.

핵심은 atomic 하나로 모든 동기화가 해결되지 않는다는 점이다. 단일 플래그 공개에는 acquire/release가 맞을 수 있지만, 복합 상태의 일관성은 여전히 락이나 메시지 전달이 필요하다.

또 spinlock은 매우 짧은 임계 구역에서만 의미가 있고, 일반적인 게임플레이 로직에서는 `mutex`나 잡 시스템이 더 안전한 경우가 많다. 결국 멀티스레딩은 코드를 병렬로 돌리는 기술이 아니라, 공유 상태를 어떻게 줄이고 어떤 비용을 감수할지 정하는 설계 문제다.

### 2. 코드 예시

```cpp
std::atomic<bool> bReady = false;
int SharedValue = 0;

void Producer()
{
    SharedValue = 42;
    bReady.store(true, std::memory_order_release);
}

void Consumer()
{
    while (!bReady.load(std::memory_order_acquire))
    {
    }

    std::cout << SharedValue << "\n";
}
```

이 코드는 단일 플래그 공개에는 acquire/release 패턴이 유효하다는 점을 보여 준다. 하지만 공유 상태가 둘 이상으로 늘어나면 이런 패턴만으로는 전체 불변식을 지키기 어려워지고, 그때는 락이나 메시지 전달 같은 더 큰 동기화 모델이 필요해진다.

### 3. 왜 중요한가 (게임/UE5 맥락)

로딩, 애니메이션 준비, 경량 AI 계산, 렌더 보조 작업처럼 병렬화 후보가 많은 구간에서도 실제 병목은 스레드 수보다 작업 분해와 동기화 비용인 경우가 많다. 잘못 설계하면 CPU 코어는 많아도 락 경합과 캐시 경합 때문에 오히려 느려질 수 있다.

### 4. 면접 예상 Q&A

**Q1: `memory_order_seq_cst`와 acquire/release는 어떻게 다르나요?**
> A: `seq_cst`는 가장 강한 전역 순서 모델이라 이해하기 쉽고, acquire/release는 필요한 공개/관찰 관계만 보장해 더 가볍다. 그래서 단일 플래그 공개 패턴에는 acquire/release가 맞고, 복잡한 사고 정리는 먼저 `seq_cst` 기준으로 이해하는 편이 안전하다.

**Q2: `mutex`와 `spinlock`은 어떻게 구분하나요?**
> A: 대기 시간이 짧고 정말 바쁘게 기다릴 가치가 있는 아주 짧은 임계 구역이면 spinlock 후보가 될 수 있다. 하지만 일반적인 코드에서는 선점과 낭비가 커서 `mutex`가 더 안전하다.

**Q3: deadlock은 어떻게 피하나요?**
> A: 락 획득 순서를 고정하고, 임계 구역을 짧게 유지하고, 여러 락은 `std::scoped_lock`처럼 한 번에 잡는 패턴을 쓰는 것이 기본이다. 더 좋은 방법은 애초에 공유 상태를 줄여 데드락 가능성 자체를 설계에서 낮추는 것이다.

**Q4: task/job system이 왜 필요한가요?**
> A: 매 작업마다 스레드를 직접 만들면 생성 비용과 스케줄링 오버헤드가 크기 때문이다. 잡 시스템은 작은 작업을 큐잉하고 worker thread가 소화하게 해 부하 분산과 코어 활용을 더 안정적으로 만든다.

### 5. 흔한 실수

- spinlock을 긴 임계 구역이나 I/O가 섞인 코드에 적용한다. 이런 경우는 대기 시간이 아니라 CPU 낭비가 더 커져 병목이 악화된다.
- atomic 플래그 하나로 복합 상태 일관성까지 해결됐다고 믿는다. 원자 연산과 불변식 보장은 다른 문제라는 점을 놓치기 쉽다.
