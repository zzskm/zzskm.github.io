## 토픽 26: cache locality & data-oriented design

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 6, 20

### 1. 문제 상황

핵심 용어: `cache locality(캐시 로컬리티, 캐시 지역성)`.

같은 알고리즘인데 데이터 배치만 바꿨더니 10배 빨라지는 경우가 있다. 렌더링 준비, 파티클, AI 감지, 물리 보조 계산처럼 같은 연산을 대량 데이터에 반복 적용하는 구간에서는 알고리즘 복잡도보다 메모리 접근 패턴이 더 큰 병목이 된다.

OOP 설계가 예쁘다는 이유로 핫패스까지 포인터 추적이 많은 레이아웃을 유지하면, 캐시 미스가 누적되어 프레임 시간이 튄다. 코드 구조보다 캐시 미스와 데이터 배치가 프레임 시간을 좌우하는 구간이 있다.

### 2. 핵심 개념

#### 정의

cache locality(캐시 지역성)는 CPU가 곧바로 필요할 가능성이 높은 데이터를 가까이 두는 성질이다. 연속 메모리에서 순차 접근이 빠른 이유가 여기에 있다.

data-oriented design(데이터 지향 설계)은 객체 계층보다 실제 접근 패턴과 데이터 배치를 우선해 성능을 끌어올리는 접근이다. OOP를 부정하는 것이 아니라, 대량 반복 처리 구간에서는 책임 분리보다 메모리 배치가 더 큰 병목이 될 수 있음을 인정하는 설계다.

#### cache miss 체감 비용

- L1 cache hit: ~1ns (4 cycles)
- L2 cache hit: ~3-10ns
- L3 cache hit: ~10-30ns
- 메인 메모리: ~50-100ns (L1 대비 약 100배)
- 결론: 같은 연산량이라도 캐시 미스가 많으면 계산보다 대기 시간이 더 커진다

#### 이 개념을 이해하려면

- 메모리 레이아웃 (토픽 6): 스택/힙/데이터 영역과 연속 메모리의 관계
- STL 컨테이너 (토픽 20): vector의 연속 메모리 vs list의 노드 분산

#### 이 개념이 영향을 주는 것

- vector vs list 선택 (토픽 27): 컨테이너 선택의 성능 근거
- 메모리 단편화 (토픽 25): 할당자가 데이터 배치를 결정
- 멀티스레딩 심화 (토픽 28): false sharing은 cache line 공유 문제

### 3. 내부 동작 / 메커니즘

#### AoS vs SoA

AoS(Array of Structures): 객체 하나의 모든 필드가 연속으로 배치된다. 객체 단위로 모든 필드를 함께 읽는 로직에 자연스럽다.

SoA(Structure of Arrays): 같은 필드끼리 별도 배열에 모인다. 특정 필드만 대량 순회하는 루프에서 캐시 효율이 높고 SIMD 적용도 쉽다.

선택 기준: '같이 접근되는 데이터가 무엇인가'가 핵심이다. 위치만 한꺼번에 업데이트하는 루프면 SoA가 유리하고, 객체 하나를 통째로 조작하는 로직이면 AoS가 자연스럽다.

#### false sharing

서로 다른 스레드가 논리적으로 독립적인 데이터를 만지더라도 같은 cache line(보통 64바이트) 안에 있으면 캐시 일관성 프로토콜이 작동해 성능이 떨어진다. 락이 없어도 병목이 생기므로 찾기 까다롭다.

해결: 스레드별 데이터를 cache line 경계에 맞춰 정렬하거나(alignas(64)), 패딩을 넣어 분리한다.

#### 프레임 드랍 구간 예시

- 파티클 업데이트: 수천 개 파티클의 위치/속도만 순회 →SoA + 연속 메모리가 효과적
- AI 감지: 주변 액터 검색 시 위치 데이터만 대량 순회 →위치 배열 분리가 유리
- 물리 보조 계산: 충돌체 AABB만 순회 →충돌 데이터를 별도 연속 배열로

### 4. 자주 틀리는 포인트

- OOP 구조가 예쁘다는 이유로 핫패스까지 포인터 추적이 많은 레이아웃을 유지한다: 상위 구조는 OOP로 두되, 핫패스는 데이터 지향적으로 분리하는 혼합이 실무에서 더 흔하다.
- AoS와 SoA를 이론으로만 외우고 실제 접근 패턴을 측정하지 않는다: 어떤 필드를 얼마나 자주 읽는지 모르면 레이아웃 변경도 추상적인 감에 머문다. 프로파일러로 캐시 미스를 확인한 뒤 결정한다.
- 모든 코드를 데이터 지향적으로 바꾸려 한다: 대량 반복 처리 구간(핫패스)에서만 의미가 있다. 초기화 로직이나 이벤트 처리처럼 호출 빈도가 낮은 코드는 가독성이 더 중요하다.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

#### 데이터 지향 설계가 효과적인 경우

- 같은 연산을 대량 데이터에 반복 적용하는 핫패스 (파티클, AI 감지, 물리)
- 특정 필드만 순회하는 루프가 프레임마다 반복될 때
- 프로파일러에서 캐시 미스가 유의미한 비용으로 나올 때
- 판단 기준: '이 루프가 어떤 필드를 어떤 순서로 읽는가'를 기준으로 배치를 정한다

#### OOP 구조를 유지하는 경우

- 호출 빈도가 낮은 초기화, 이벤트, UI 로직
- 객체 하나를 통째로 조작하는 코드 (모든 필드에 접근)
- 판단 기준: 가독성과 유지보수가 캐시 효율보다 중요한 구간

### 6. 코드 예시 (bad → good)

#### AoS: 모든 필드가 객체에 묶임

```cpp
struct ParticleAoS {
    float X, Y, Z;
    float VX, VY, VZ;
    float R, G, B, A;   // 색상 — 위치 업데이트 시 불필요
};

void UpdateAoS(std::vector<ParticleAoS>& P, float Dt) {
    for (auto& p : P) {   // 색상 필드까지 cache line에 올라온다
        p.X += p.VX * Dt;
        p.Y += p.VY * Dt;
        p.Z += p.VZ * Dt;
    }
}
```

AoS는 객체 단위로 다루기 쉽지만, 특정 필드만 반복해서 읽는 루프에서는 필요 없는 데이터까지 함께 캐시에 실린다. 핫패스에서는 이 낭비가 누적 비용으로 바뀐다.

#### SoA: 필요한 필드만 연속 배치

```cpp
struct ParticleSoA {
    std::vector<float> X, Y, Z;
    std::vector<float> VX, VY, VZ;
    std::vector<float> R, G, B, A;
};

void UpdateSoA(ParticleSoA& P, float Dt) {
    for (size_t i = 0; i < P.X.size(); ++i) {
        P.X[i] += P.VX[i] * Dt;  // 위치+속도만 순회, 캐시 효율 극대화
        P.Y[i] += P.VY[i] * Dt;
        P.Z[i] += P.VZ[i] * Dt;
    }
}
```

SoA는 루프가 실제로 쓰는 필드만 연속 배치한다는 점이 핵심이다. 계산 대상이 좁을수록 프리페치와 캐시 활용률이 좋아져 반복 처리에서 차이가 커진다.

### 7. 판단 기준 요약

- 핫패스에서는 '어떤 필드를 어떤 순서로 읽는가'가 자료구조 선택의 첫 번째 기준이다
- 같은 연산량이라도 캐시 미스가 많으면 100배까지 느려진다. 알고리즘 빅오보다 메모리 접근 패턴이 더 중요한 구간이 있다
- AoS는 객체 단위 조작에, SoA는 필드 단위 대량 순회에 유리하다. 접근 패턴이 선택 기준이다
- false sharing은 락 없이도 멀티스레드 성능을 떨어뜨린다. cache line 단위로 데이터를 분리한다
- 상위 구조는 OOP, 핫패스만 데이터 지향 — 혼합이 실무의 기본이다

## 토픽 27: vector vs list vs deque / iterator invalidation

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 20, 26

### 1. 문제 상황

list가 중간 삽입 O(1)이라는 이유만으로 선택했더니 순회가 느려 프레임 시간이 튄다. vector 원소의 주소를 저장해 뒀더니 push_back 후 크래시가 난다.

컨테이너를 빅오 표만 보고 고르면 실제 성능과 안정성을 모두 잃는다. 게임 코드에서는 삽입 한 번의 이론적 이득보다 프레임마다 반복되는 순회 비용이 더 크게 누적된다.

### 2. 핵심 개념

#### 정의

vector: 연속 메모리, 뛰어난 cache locality. 끝 삽입 amortized O(1), 중간 삽입 O(n). 재할당 시 모든 iterator/pointer/reference가 무효화된다.

list: 노드 기반, 양방향 연결. 어디서든 삽입/삭제 O(1)(iterator가 있으면). 하지만 노드가 힙에 분산되어 순회 시 캐시 미스가 크고 메모리 오버헤드도 크다.

deque: 블록 기반, 앞뒤 확장 모두 O(1). vector보다 메모리 구조가 복잡하지만 앞쪽 삽입이 필요할 때 유리하다.

#### 이 개념을 이해하려면

- STL 컨테이너 & 알고리즘 (토픽 20): 컨테이너 기본 사용법
- cache locality (토픽 26): 연속 메모리가 왜 순회에서 유리한지

#### 이 개념이 영향을 주는 것

- crash 분석 (토픽 30): iterator invalidation이 dangling pointer 크래시의 흔한 원인
- UB & 안정성 (토픽 29): 무효화된 iterator 사용은 UB

### 3. 내부 동작 / 메커니즘

#### iterator invalidation 규칙

- vector push_back: capacity 초과 시 재할당 →모든 iterator 무효. reserve로 재할당 시점을 제어할 수 있다
- vector insert/erase: 삽입/삭제 지점 이후 iterator 무효
- deque push_front/push_back: 모든 iterator 무효(pointer/reference는 유지). 중간 삽입/삭제: 전부 무효
- list insert/erase: 해당 노드만 영향. 다른 iterator/pointer/reference는 유효. 이것이 list의 핵심 장점이다

#### 왜 vector가 대부분의 경우 더 빠른가

vector는 연속 메모리이므로 CPU 프리페치가 잘 동작하고 cache line 활용률이 높다. list는 노드마다 별도 힙 할당이 필요하고 포인터를 따라가야 하므로 순회 시 캐시 미스가 크다.

삽입이 O(1)이라도 순회가 O(n * cache_miss_cost)이면 총 비용은 vector의 O(n * shift + cache_hit)보다 클 수 있다. 빅오만 보고 고르면 실제 성능을 오판한다.

### 4. 자주 틀리는 포인트

- list가 중간 삽입 O(1)이라는 이유만으로 선택한다: 삽입보다 순회가 훨씬 자주 일어나는 게임 코드에서는 순회 비용이 더 크다. 실제 병목이 삽입인지 순회인지 먼저 구분한다.
- vector 원소의 주소나 reference를 오래 들고 다닌다: push_back이나 insert로 재할당이 일어나면 기존 주소가 모두 무효화된다. reserve를 쓰거나, 인덱스를 저장하거나, 참조 안정성이 필요하면 다른 컨테이너를 고려한다.
- deque를 vector와 list의 중간으로 본다: deque는 앞뒤 확장이 필요할 때의 별도 선택지다. 순회 성능은 vector보다 낮고 메모리 구조도 더 복잡하다.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

#### vector를 쓰는 경우 (기본 선택)

- 순회가 잦고 끝에서만 삽입/삭제하는 대부분의 경우
- 연속 메모리와 cache locality가 중요한 핫패스
- 판단 기준: 특별한 이유가 없으면 vector가 기본이다

#### deque를 쓰는 경우

- 앞뒤 양쪽에서 push/pop이 자주 일어나는 큐 패턴
- 판단 기준: 앞쪽 확장이 필요하고 중간 삽입은 드문 경우

#### list를 쓰는 경우 (드묾)

- 노드의 참조 안정성이 절대적으로 중요할 때
- 중간 삽입/삭제가 매우 빈번하고 순회보다 구조 변경이 핵심일 때
- 판단 기준: 다른 선택지가 왜 맞지 않는지 확인한 뒤 제한적으로 채택한다. intrusive list나 핸들 구조도 함께 검토한다

### 6. 대안 비교

#### vector vs deque vs list

- vector: 기본 선택지다. 순회와 cache locality에 강하지만 재할당이 일어나면 iterator, pointer, reference가 한꺼번에 무효화될 수 있다.
- deque: 앞뒤 확장이 필요한 큐 패턴에 맞는다. 연속 메모리는 아니므로 순회 효율은 대체로 vector보다 떨어진다.
- list: 노드 참조 안정성과 중간 삽입/삭제에는 유리하지만, 포인터 추적과 힙 분산 때문에 순회 비용과 메모리 오버헤드가 크다.

#### 선택 기준

- 순회 빈도가 가장 높으면 vector에서 시작한다.
- 앞쪽 push/pop이 구조적으로 필요하면 deque를 고려한다.
- 참조 안정성이 절대 조건이고 다른 선택지가 맞지 않을 때만 list를 제한적으로 쓴다.

### 7. 코드 예시 (bad → good)

#### Bad: vector 재할당 후 무효화된 포인터 사용

```cpp
std::vector<int> Values = {1, 2, 3};
int* Saved = &Values[0];

Values.push_back(4);       // capacity 초과 시 재할당
std::cout << *Saved;       // UB: dangling pointer
```

문제는 push_back 자체가 아니라 그 뒤에 재할당이 숨어 있을 수 있다는 점이다. 주소를 오래 들고 다니는 코드는 이렇게 컨테이너의 lifetime 규칙에 쉽게 깨진다.

#### Good: reserve로 재할당 방지 또는 인덱스 사용

```cpp
std::vector<int> Values = {1, 2, 3};
Values.reserve(8);             // 충분한 capacity 확보
int* Saved = &Values[0];
Values.push_back(4);           // 재할당 없음, Saved 유효

// 또는 인덱스로 안전하게 접근
size_t SavedIdx = 0;
Values.push_back(4);
std::cout << Values[SavedIdx]; // 항상 안전
```

reserve는 재할당 시점을 제어하는 방법이고, 인덱스는 주소 안정성에 덜 의존하는 방법이다. 둘 중 무엇을 고르든 핵심은 vector의 invalidation 규칙을 코드에 반영하는 것이다.

### 8. 판단 기준 요약

- 기본 선택은 vector다. 특별한 이유가 없으면 vector에서 시작한다
- 빅오만 보고 고르지 않는다. 실제 비용은 cache locality와 메모리 배치에 크게 좌우된다
- 컨테이너 선택 시 순회 빈도, 삽입 위치, 참조 안정성 세 가지를 함께 본다
- vector 원소의 주소를 오래 들고 다니지 않는다. reserve를 쓰거나 인덱스를 저장한다
- list는 '다른 선택지가 왜 안 되는지' 확인한 뒤에만 채택한다

## 토픽 28: 멀티스레딩 심화

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 21

### 1. 문제 상황

핵심 용어: `memory ordering(메모리 오더링, 메모리 순서 보장)`.

스레드를 늘렸는데 오히려 느려진다. 락 경합, 캐시 경합, 잘못된 동기화 모델이 원인이다. CPU 코어가 많아도 공유 상태를 잘못 다루면 직렬 실행보다 느려질 수 있다.

atomic 하나로 모든 동기화가 해결됐다고 믿으면 복합 상태의 일관성이 깨진다. 멀티스레딩은 코드를 병렬로 돌리는 기술이 아니라, 공유 상태를 어떻게 줄이고 어떤 비용을 감수할지 정하는 설계 문제다.

### 2. 핵심 개념

#### 정의

memory ordering(메모리 순서 보장)은 스레드 사이에서 값이 언제 보이는지를 정의한다. seq_cst는 가장 강한 전역 순서 모델이고, acquire/release는 필요한 공개/관찰 관계만 보장해 더 가볍다.

mutex는 임계 구역에 하나의 스레드만 진입하게 한다. 대기 중 스레드는 잠든다(OS 스케줄러에 양보). spinlock은 대기 중에도 CPU를 점유하며 루프를 돈다. 임계 구역이 극히 짧을 때만 의미가 있다.

task/job system은 스레드를 직접 만들지 않고 작은 작업을 큐에 넣어 worker thread가 소화하게 하는 구조다. 부하 분산과 코어 활용을 더 안정적으로 만든다.

#### 이 개념을 이해하려면

- 멀티스레딩 기초 (토픽 21): thread, mutex, atomic 기본 사용법
- cache locality (토픽 26): false sharing과 cache line 경합

#### 이 개념이 영향을 주는 것

- crash/병목 분석 (토픽 30): race condition과 데드락 디버깅
- UB & 안정성 (토픽 29): data race는 UB

### 3. 내부 동작 / 메커니즘

#### memory ordering 비교

- memory_order_seq_cst: 모든 스레드가 같은 전역 순서를 본다. 이해하기 가장 쉽고 안전하지만 비용이 가장 크다
- memory_order_acquire/release: store(release)한 값을 load(acquire)한 스레드에서 그 이전 쓰기가 모두 보임을 보장한다. 단일 플래그 공개 패턴에 적합하다
- memory_order_relaxed: 원자성만 보장하고 순서는 보장하지 않는다. 카운터처럼 순서가 중요하지 않은 경우에만 사용한다
- 판단 기준: 먼저 seq_cst로 정확성을 확보하고, 프로파일링 후 필요하면 acquire/release로 낮춘다

#### mutex vs spinlock 선택

mutex: 대기 중 OS에 양보하므로 임계 구역이 길거나 I/O가 있어도 괜찮다. 일반적인 코드의 기본 선택이다.

spinlock: 대기 중 CPU를 계속 점유한다. 임계 구역이 수십 ns 이하이고 잠들지 않아야 하는 경우에만 후보가 된다. 질문은 '짧은가'가 아니라 '이 코드가 잠들지 않고 계속 CPU를 써도 되는가'다.

I/O나 긴 계산이 섞이면 spinlock은 거의 항상 나쁜 선택이다.

#### deadlock 회피 패턴

- 락 획득 순서를 고정한다 (A →B →C, 역순 금지)
- 여러 락은 std::scoped_lock으로 한 번에 잡는다
- 임계 구역을 짧게 유지해 락 보유 시간을 줄인다
- 더 좋은 방법: 애초에 공유 상태를 줄여 데드락 가능성 자체를 설계에서 낮춘다

#### task/job system의 이점

매 작업마다 스레드를 만들면 생성 비용(~수십us)과 스케줄링 오버헤드가 크다. 잡 시스템은 worker thread pool을 미리 만들고, 작업을 큐에 넣어 분배한다. 같은 코어 수로 더 많은 작업을 안정적으로 처리할 수 있다.

중요한 것은 스레드 개수가 아니라 작업 단위다. 작업을 적절히 잘게 나누고 의존성을 명시해야 병렬화 이득이 실제로 난다.

### 4. 자주 틀리는 포인트

- atomic 하나로 복합 상태 일관성까지 해결됐다고 믿는다: atomic은 단일 변수의 원자성만 보장한다. 여러 변수의 불변식을 함께 지켜야 하면 락이나 메시지 전달이 필요하다.
- spinlock을 긴 임계 구역이나 I/O가 섞인 코드에 적용한다: 대기 시간이 길어지면 CPU 낭비가 커져 병목이 악화된다. 일반적인 코드에서는 mutex가 안전하다.
- 스레드를 많이 만들수록 빨라진다고 생각한다: 코어 수를 초과하면 컨텍스트 스위칭 비용이 이득을 넘는다. 잡 시스템으로 작업 단위를 관리하는 것이 더 효율적이다.
- 데드락을 디버깅에서 해결하려 한다: 데드락은 코드 한 줄보다 시스템 구조에서 생긴다. 공유 자원을 줄이는 것이 가장 효과적인 예방이다.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

#### 멀티스레딩이 효과적인 경우

- 독립적인 작업이 많고 공유 상태가 적은 경우 (로딩, 파싱, 비동기 I/O)
- 대량 데이터의 병렬 처리 (애니메이션 준비, 경량 AI, 렌더 보조)
- 판단 기준: 작업 분해가 자연스럽고 동기화 비용이 병렬화 이득보다 작을 때

#### 멀티스레딩을 피하는 경우

- 작업 간 의존성이 강해 대부분 직렬화되는 경우
- 공유 상태가 많아 락 경합이 이득을 상쇄하는 경우
- 판단 기준: 병렬화 이득보다 동기화 복잡도가 더 크면 직렬이 낫다

### 6. 코드 예시 (bad → good)

#### acquire/release 패턴: 단일 플래그 공개

```cpp
std::atomic<bool> bReady{false};
int SharedValue = 0;

void Producer() {
    SharedValue = 42;
    bReady.store(true, std::memory_order_release);  // 이전 쓰기를 공개
}

void Consumer() {
    while (!bReady.load(std::memory_order_acquire)) {}  // 공개된 값을 관찰
    assert(SharedValue == 42);  // 보장됨
}
```

이 패턴은 단일 플래그를 통해 이전 쓰기를 공개하는 가장 전형적인 사례다. 상태가 더 복잡해지면 acquire/release만으로 충분한지 다시 검토해야 한다.

#### Bad: 여러 락을 순서 없이 잡음 →데드락

```cpp
// Thread 1: lock(A) →lock(B)
// Thread 2: lock(B) →lock(A)  →데드락!
```

데드락의 핵심 원인은 락 개수가 아니라 획득 순서가 스레드마다 다르다는 점이다. 이런 문제는 코드 한 줄보다 설계 단계의 잠금 규칙 부재에서 더 자주 나온다.

#### Good: scoped_lock으로 한 번에 잡기

```cpp
std::mutex MutexA, MutexB;

void SafeOperation() {
    std::scoped_lock Lock(MutexA, MutexB);  // 내부적으로 데드락 회피
    // 임계 구역
}
```

여러 mutex를 함께 잡아야 한다면 라이브러리의 deadlock-avoidance 규칙을 활용하는 편이 낫다. 그래도 임계 구역 자체를 짧게 유지하는 원칙은 그대로 중요하다.

### 7. 판단 기준 요약

- 멀티스레딩의 핵심은 공유 상태를 줄이는 것이다. 스레드를 늘리는 것이 아니다
- memory ordering은 먼저 seq_cst로 정확성을 확보하고, 프로파일링 후 필요하면 낮춘다
- 기본 선택은 mutex다. spinlock은 극히 짧은 임계 구역에서만 후보가 된다
- 데드락은 디버깅보다 설계로 예방한다. 공유 자원을 줄이고 락 순서를 고정한다
- 직접 스레드를 만들기보다 task/job system으로 작업 단위를 관리한다
