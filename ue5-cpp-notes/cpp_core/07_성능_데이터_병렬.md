## 토픽 26: cache locality & data-oriented design

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 6, 20

### 1. 핵심 개념

`cache locality(캐시 로컬리티, 캐시 지역성)`는 CPU가 곧바로 필요할 가능성이 높은 데이터를 가까이 두는 성질이다. 연속 메모리에서 순차 접근이 빠른 이유도 여기에 있다. `data-oriented design(데이터 지향 설계)`은 객체 계층보다 실제 접근 패턴과 데이터 배치를 우선해 성능을 끌어올리는 접근이다.

핵심은 "코드가 보기 좋은가"보다 "CPU가 데이터를 어떻게 읽는가"를 먼저 보는 것이다. OOP 구조가 나쁘다는 뜻이 아니라, 대량 반복 처리 구간에서는 AoS(Array of Structures)보다 SoA(Structure of Arrays)가 더 잘 맞을 수 있다는 뜻이다. 멀티스레드 환경에서는 cache line을 공유해 false sharing이 생기는지도 함께 봐야 한다.

### 2. 코드 예시

```cpp
struct ParticleAoS
{
    float X, Y, Z;
    float VX, VY, VZ;
};

std::vector<ParticleAoS> ParticlesAoS;

struct ParticleSoA
{
    std::vector<float> X;
    std::vector<float> Y;
    std::vector<float> Z;
    std::vector<float> VX;
    std::vector<float> VY;
    std::vector<float> VZ;
};

void UpdateAoS(std::vector<ParticleAoS>& Particles, float Dt)
{
    for (ParticleAoS& Particle : Particles)
    {
        Particle.X += Particle.VX * Dt;
        Particle.Y += Particle.VY * Dt;
        Particle.Z += Particle.VZ * Dt;
    }
}

void UpdateSoA(ParticleSoA& Particles, float Dt)
{
    for (size_t i = 0; i < Particles.X.size(); ++i)
    {
        Particles.X[i] += Particles.VX[i] * Dt;
        Particles.Y[i] += Particles.VY[i] * Dt;
        Particles.Z[i] += Particles.VZ[i] * Dt;
    }
}
```

### 3. 왜 중요한가 (게임/UE5 맥락)

렌더링 준비, 파티클, AI 감지, 물리 보조 계산처럼 "많은 데이터를 같은 방식으로 반복 처리"하는 코드는 cache locality 차이를 크게 탄다. 게임에서는 알고리즘 복잡도보다 메모리 접근 패턴이 병목인 경우가 자주 나온다.

### 4. 면접 예상 Q&A

**Q1: cache locality는 왜 중요한가요?**
> A: CPU는 연속된 메모리를 미리 가져오는 데 강하기 때문이다. 같은 연산량이라도 캐시 미스가 많으면 대기 시간이 커져 체감 성능이 급격히 떨어진다. 그래서 자료구조 선택은 연산 자체보다 메모리 접근 패턴과 함께 봐야 한다.

**Q2: data-oriented design은 OOP와 어떻게 다르나요?**
> A: OOP는 책임과 추상화 중심으로 구조를 잡고, data-oriented design은 실제 처리되는 데이터 흐름과 레이아웃을 중심으로 구조를 잡는다. 대량 반복 처리 구간에서는 후자가 더 맞는 경우가 많다. 둘은 대립 개념이 아니라 구간별 우선순위가 다르다고 보는 편이 정확하다.

**Q3: AoS와 SoA는 언제 각각 유리한가요?**
> A: 개별 객체 단위 조작과 캡슐화가 중요하면 AoS가 자연스럽고, 특정 필드만 대량 순회하는 연산이면 SoA가 유리하다. 결국 "같이 접근되는 데이터가 무엇인가"가 선택 기준이다.

**Q4: false sharing은 무엇인가요?**
> A: 서로 다른 스레드가 논리적으로 독립적인 데이터를 만지더라도 같은 cache line 안에 있으면 캐시 일관성 비용이 커지는 현상이다. 락이 없어도 병목이 생길 수 있어 찾기 까다롭다.

**Q5: 모든 시스템을 data-oriented로 바꾸는 것이 맞나요?**
> A: 아니다. 핫패스와 대량 데이터 처리 경로에 집중해야 한다. 빈도가 낮거나 복잡한 비즈니스 로직까지 전부 데이터 지향으로 바꾸면 유지보수성만 나빠질 수 있다.

**Q6: 실무에서 어떤 신호가 나오면 데이터 배치를 재검토하나요?**
> A: 프로파일러에서 같은 루프가 오래 걸리고, 연산 자체보다 메모리 접근 비용이 커 보일 때다. 객체 수가 많고, 필드 일부만 반복 접근하며, 컨테이너가 흩어진 할당을 많이 만들면 재배치 후보로 본다.

### 5. 흔한 실수

- 객체지향 구조가 예쁘다는 이유로 핫패스까지 포인터 추적이 많은 레이아웃을 유지한다.
- AoS와 SoA를 이론으로만 외우고 실제 접근 패턴을 측정하지 않는다.
- false sharing을 "락이 없으니 상관없다"라고 오해한다.

## 토픽 27: vector vs list vs deque / iterator invalidation

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 20, 26

### 1. 핵심 개념

`vector`, `list`, `deque`는 모두 순차 컨테이너지만 내부 레이아웃과 강점이 다르다. `vector`는 연속 메모리와 뛰어난 cache locality가 장점이고, `list`는 노드 기반으로 중간 삽입/삭제가 쉬워 보이지만 포인터 추적과 개별 할당 비용이 크다. `deque`는 앞뒤 확장에 강하고 중앙 삽입/삭제는 여전히 비싸다.

핵심은 "빅오 표만 보고 고르지 않는다"는 점이다. 실무에서는 검색, 순회, 삽입 위치, 참조 안정성, iterator invalidation 규칙을 같이 봐야 한다. 특히 `vector`는 재할당 시 iterator와 reference가 무효가 될 수 있어 성능과 안정성을 함께 고려해야 한다.

### 2. 코드 예시

```cpp
std::vector<int> Values = {1, 2, 3, 4};
Values.reserve(8); // ← 재할당 시점을 제어

auto It = Values.begin();
Values.push_back(5); // reserve가 부족하면 iterator invalidation 가능

std::deque<int> QueueLike = {1, 2, 3};
QueueLike.push_front(0); // ← 앞쪽 확장이 잦으면 deque가 더 자연스럽다

std::list<int> Linked = {1, 2, 3};
auto NodeIt = Linked.begin();
Linked.insert(std::next(NodeIt), 99); // ← 노드 삽입은 쉽지만 순회 캐시 효율은 낮다
```

### 3. 왜 중요한가 (게임/UE5 맥락)

프레임마다 도는 루프에서는 삽입 한 번의 이론적 이득보다 순회 비용이 더 중요한 경우가 많다. 그래서 게임 코드에서는 `list`의 이론적 강점보다 `vector`의 단순한 메모리 배치가 더 큰 이익을 주는 일이 많다.

### 4. 면접 예상 Q&A

**Q1: vector가 list보다 빠른 이유는 무엇인가요?**
> A: `vector`는 연속 메모리라 cache locality가 좋고, 분기와 포인터 추적이 적기 때문이다. `list`는 삽입/삭제 자체는 노드 단위로 쉬워 보여도 순회할 때 캐시 미스와 할당 비용이 커서 실제로 느린 경우가 많다. 결국 게임 코드에서는 순회 빈도가 높아 `vector`가 기본 선택이 되기 쉽다.

**Q2: deque는 언제 선택하나요?**
> A: 앞뒤 양쪽에서 push/pop이 자주 일어나고, 중간 삽입은 드문 경우다. 큐 비슷한 사용 패턴이지만 `vector`보다 앞쪽 확장이 편해야 할 때 적합하다.

**Q3: iterator invalidation은 왜 중요하나요?**
> A: 삽입, 삭제, 재할당 뒤에 iterator, pointer, reference가 더 이상 유효하지 않을 수 있기 때문이다. 이 규칙을 모르면 크래시가 아니라 더 위험한 use-after-free성 UB가 생길 수 있다.

**Q4: list가 맞는 경우도 있나요?**
> A: 노드의 참조 안정성이 정말 중요하고, 중간 삽입/삭제가 매우 자주 일어나며, 순회보다 구조 변경이 더 핵심이면 후보가 될 수 있다. 다만 이런 경우도 intrusive list나 별도 핸들 구조가 더 맞는지 먼저 본다.

**Q5: vector에서 invalidation을 줄이는 방법은 무엇인가요?**
> A: `reserve`로 재할당 시점을 제어하고, 원소 주소를 오래 들고 다니지 않고, erase 이후 반환 iterator를 다시 사용하는 식으로 패턴을 정리하는 것이다. 필요하면 index 기반 접근으로 전환할 수도 있다.

**Q6: 컨테이너 선택 기준을 한 문장으로 말하면 무엇인가요?**
> A: "어디서 얼마나 자주 순회하고, 어디에 삽입/삭제하며, 참조 안정성이 필요한가"를 같이 본다고 답하면 된다. 자료구조는 연산 표 하나가 아니라 실제 접근 패턴으로 고르는 것이다.

### 5. 흔한 실수

- `list`가 중간 삽입 O(1)이라는 이유만으로 순회 비용을 무시하고 선택한다.
- `vector` 원소의 주소나 reference를 오래 들고 다니며 재할당을 잊는다.
- `deque`를 "vector보다 조금 다른 vector"처럼 이해하고 invalidation 규칙을 확인하지 않는다.

## 토픽 28: 멀티스레딩 심화

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 21

### 1. 핵심 개념

멀티스레딩 심화에서 중요한 것은 "도구 이름"보다 메모리 가시성과 대기 비용을 이해하는 것이다. `memory ordering(메모리 오더링, 메모리 순서 보장)`은 스레드 사이에서 값이 언제 보이는지를 정의하고, `mutex`와 `spinlock`은 대기 방식이 다르다. `task/job system`은 스레드를 직접 늘리기보다 작은 작업 단위로 분해해 실행하는 구조다.

핵심은 atomic 하나로 모든 동기화가 해결되지 않는다는 점이다. 단일 플래그 공개에는 acquire/release가 맞을 수 있지만, 복합 상태 일관성은 여전히 락이나 메시지 전달이 필요하다. 또 spinlock은 "매우 짧은 임계 구역"에서만 의미가 있고, 일반적인 게임플레이 로직에서는 `mutex`나 잡 시스템이 더 안전한 경우가 많다.

### 2. 코드 예시

```cpp
std::atomic<bool> bReady = false;
int SharedValue = 0;

void Producer()
{
    SharedValue = 42;
    bReady.store(true, std::memory_order_release);
}

void Consumer()
{
    while (!bReady.load(std::memory_order_acquire))
    {
    }

    std::cout << SharedValue << "\n";
}
```

```cpp
std::mutex A;
std::mutex B;

void SafeLockBoth()
{
    std::scoped_lock Lock(A, B); // ← 고정 락 순서/동시 획득으로 deadlock 회피
}
```

### 3. 왜 중요한가 (게임/UE5 맥락)

로딩, 애니메이션 준비, 경량 AI 계산, 렌더 보조 작업은 스레드보다 "작업 분해와 동기화 비용"이 더 중요해지는 경우가 많다. 잘못 설계하면 CPU 코어는 많아도 락 경합과 캐시 경합 때문에 오히려 느려질 수 있다.

### 4. 면접 예상 Q&A

**Q1: `memory_order_seq_cst`와 acquire/release는 어떻게 다르나요?**
> A: `seq_cst`는 가장 강한 전역 순서 모델이라 이해하기 쉽고, acquire/release는 필요한 공개/관찰 관계만 보장해 더 가볍다. 면접에서는 "플래그 공개 패턴엔 acquire/release, 복잡한 사고 정리는 먼저 seq_cst로 이해" 정도로 말하면 안전하다.

**Q2: `mutex`와 `spinlock`은 어떻게 구분하나요?**
> A: 대기 시간이 짧고 절대 block되면 안 되는 아주 짧은 임계 구역이면 spinlock 후보가 될 수 있다. 하지만 일반적인 코드에서는 선점과 낭비가 커서 `mutex`가 더 안전하다. 따라서 "짧다"보다 "정말 바쁘게 기다릴 가치가 있는가"가 핵심 판단 기준이다.

**Q3: deadlock은 어떻게 피하나요?**
> A: 락 획득 순서를 고정하고, 임계 구역을 짧게 유지하고, 여러 락은 `std::scoped_lock`처럼 한 번에 잡는 패턴을 쓰는 것이 기본이다. 더 좋은 방법은 애초에 공유 상태를 줄여 데드락 가능성 자체를 설계에서 낮추는 것이다.

**Q4: task/job system이 왜 필요한가요?**
> A: 매 작업마다 스레드를 직접 만들면 생성 비용과 스케줄링 오버헤드가 크기 때문이다. 잡 시스템은 작은 작업을 큐잉하고 worker thread가 소화하게 해 부하 분산과 코어 활용을 더 안정적으로 만든다.

**Q5: race condition을 줄이는 가장 좋은 방법은 무엇인가요?**
> A: 락을 더 많이 거는 것보다 공유 mutable state를 줄이는 것이다. 소유권을 분리하고, 읽기 전용 데이터와 메시지 전달 패턴을 늘리면 동기화 자체가 줄어든다.

**Q6: 실무에서 멀티스레딩 디버깅 접근을 어떻게 설명하나요?**
> A: 먼저 재현 조건을 좁히고, 공유 변수와 동기화 경계를 그린 다음, 로그/트레이스/TSan류 도구로 실제 순서를 확인한다고 말하면 된다. 면접에서는 "추측으로 고치지 않고 happens-before 관계를 검증했다"는 태도가 중요하다.

### 5. 흔한 실수

- spinlock을 긴 임계 구역이나 I/O가 섞인 코드에 적용한다.
- atomic 플래그 하나로 복합 상태 일관성까지 해결됐다고 믿는다.
- 스레드를 직접 남발하고 job system이나 task graph 같은 구조화를 늦게 고려한다.
