## 토픽 16: move semantics(무브 세맨틱스, 이동 의미론)

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 3, 4

### 1. 핵심 개념

핵심 용어: `valid but unspecified`, `perfect forwarding`, `rvalue`, `값 카테고리`.

move semantics는 복사 대신 자원의 소유권을 옮겨 불필요한 비용을 줄이려는 방식이다. 여기서 핵심은 `lvalue`, `xvalue`, `prvalue` 같은 값 카테고리를 구분하고, rvalue 문맥에서 move constructor와 move assignment가 어떤 의미로 동작하는지 이해하는 데 있다.

`std::move`는 이동을 수행하는 함수가 아니라 '이 표현식을 rvalue로 취급해도 된다'고 알려 주는 캐스트에 가깝다. 반면 `std::forward`는 전달받은 인자의 원래 값 카테고리를 유지해 perfect forwarding을 가능하게 한다.

### 2. 왜 필요한가

문자열, 동적 버퍼, 컨테이너처럼 내부에 큰 자원을 가진 타입은 복사 비용이 눈에 띄게 크다. 이런 타입을 임시 객체나 반환값으로 자주 주고받는 코드에서는 move semantics만으로도 할당과 복사 비용을 크게 줄일 수 있다.

특히 modern C++는 값 의미를 유지한 채 성능도 확보하는 방향으로 발전해 왔다. 그래서 move semantics는 단순 최적화 트릭이 아니라, 값 타입 설계와 API 설계의 기본 전제가 된다.

### 3. 비용 / 단점

move가 항상 공짜이거나 항상 더 빠른 것은 아니다. moved-from 객체를 어떻게 취급해야 하는지, 컨테이너가 실제로 move를 선택하는 조건이 무엇인지까지 알아야 제대로 쓸 수 있다.

- moved-from 객체는 valid but unspecified 상태이므로, 이전 값을 가정하고 읽으면 버그나 UB로 이어질 수 있다.
- `noexcept`가 빠진 move constructor는 `vector` 같은 컨테이너가 재할당 시 복사를 선택하게 만들어 기대한 성능 이점을 잃게 할 수 있다.
- const 객체에 `std::move`를 써도 실제로는 move constructor가 아니라 복사 생성자가 선택될 수 있다.
- 작은 trivially copyable 타입은 move보다 복사가 더 단순하고 최적화도 잘 되는 경우가 많다.

### 4. 언제 쓰고 언제 안 쓰는가

#### 써야 하는 경우

- 함수가 더 이상 사용하지 않을 지역 자원 객체를 반환하거나 컨테이너에 넘길 때
- 큰 버퍼, 문자열, 핸들처럼 소유권 이전이 복사보다 훨씬 저렴한 타입일 때
- forwarding reference 문맥에서 원래 값 카테고리를 보존해 전달해야 할 때

#### 안 쓰는 편이 나은 경우

- 이후에도 같은 객체를 계속 읽거나 쓸 계획이 있을 때
- const 객체라서 실제 이동이 일어나지 않을 때
- 작은 POD나 trivially copyable 타입처럼 복사가 충분히 싼 경우
- 함수 인자를 단순 읽기 전용으로만 볼 뿐 소유권 이전 의도가 없을 때

### 5. 대안 비교

move 관련 판단은 결국 호출자와 피호출자 사이에 '누가 이후에도 이 값을 소유하고 사용할 것인가'를 정하는 문제다. 값 전달 방식은 성능보다 소유권 의도를 먼저 드러내야 한다.

- 복사: 원본을 보존해야 하고 타입이 작거나 공유 비용이 크지 않을 때 기본 선택이다.
- move: 호출 뒤 원본을 더 이상 의미 있게 쓰지 않을 때 적합하다.
- perfect forwarding: 래퍼 함수가 호출자 의도를 왜곡하지 않고 그대로 전달해야 할 때 쓴다.
- `const T&`: 읽기 전용 입력에 가장 무난하다.
- `T&&`: 명시적으로 소유권 이전을 받거나 forwarding reference 문맥에서만 신중하게 쓴다.
- `T` 값 전달: 복사본이 어차피 필요하고, 내부에서 move로 보관할 수 있을 때 단순한 인터페이스가 된다.

### 6. 실전 사례

#### forwarding reference를 무조건 move한 경우

```cpp
template<typename T>
void Push(std::vector<T>& out, T&& value) {
    out.push_back(std::move(value));
}
```

이 코드는 lvalue 인자를 받아도 무조건 rvalue로 바꿔 버리기 때문에, 호출자가 계속 써야 하는 객체를 의도치 않게 이동시킬 수 있다. forwarding reference에서는 `std::forward<T>(value)`로 원래 값 카테고리를 보존해야 한다.

#### const 객체에 대한 std::move와 moved-from 재사용

`const std::string s`에 `std::move(s)`를 적용해도 `const T&&`가 되므로 실제 move constructor가 아니라 복사가 선택될 수 있다. 또한 move 후 원본 객체를 특정 값으로 가정하고 읽는 것은 위험하므로, 파괴하거나 새 값을 대입하는 용도로만 다루는 편이 안전하다.

### 7. 면접 질문

#### Q1. const 객체에 std::move를 써도 실제 이동이 안 되는 이유는?

대부분의 move constructor는 `T&&`를 받는데, `const T&&`에는 바인딩되지 않는다. 그래서 복사 생성자가 선택되고, `std::move`는 이동 요청일 뿐 이동 보장이 아니라는 점이 드러난다.

#### Q2. 함수 인자를 const T&, T&&, T 값 중 무엇으로 받을지 어떻게 결정하나?

읽기 전용이면 `const T&`, 소유권 이전을 명시적으로 받으면 `T&&`, 내부에서 복사본이 필요하고 호출자 편의가 중요하면 값 전달을 고른다. 핵심은 성능 미세 최적화보다 소유권 의도를 인터페이스에 드러내는 것이다.

#### Q3. move constructor에 noexcept가 중요한 이유는?

표준 컨테이너는 재배치 중 강한 예외 안전성을 유지해야 하므로, move가 예외를 던질 수 있으면 복사를 택하는 경우가 많다. 그래서 `noexcept`는 단순 문법이 아니라 컨테이너 최적화 조건과 연결된다.

## 토픽 17: 람다

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 16

### 1. 핵심 개념

핵심 용어: `함수 객체(functor)`, `즉시 호출(IIFE)`, `mutable`, `초기화`.

람다는 이름 없는 함수 객체를 즉석에서 정의하는 문법이다. 문법은 짧지만 실제로는 상태를 가질 수 있는 closure object를 만든다는 점이 핵심이며, 캡처 방식에 따라 값 복사인지 원본 참조인지 의미가 크게 달라진다.

실무에서는 `[=]`, `[&]`, `[this]`, 초기화 캡처, `mutable`, generic lambda를 얼마나 정확히 이해하느냐가 중요하다. 람다는 콜백을 간결하게 만들지만, 수명과 비용을 함께 보지 않으면 버그를 만들기 쉬운 도구이기도 하다.

### 2. 왜 필요한가

람다는 작은 동작을 정의하기 위해 별도 functor 클래스를 만들지 않아도 되므로, 알고리즘 호출, 필터, 정렬 기준, 비동기 콜백 같은 코드가 훨씬 지역적으로 읽힌다. '이 로직은 여기서만 쓰인다'는 사실을 타입 수준으로도 드러내기 쉽다.

특히 modern C++에서는 `std::algorithm`, 비동기 작업, 이벤트 바인딩이 많아지면서 람다가 사실상 기본 콜백 표현식이 되었다. 그래서 문법보다도 캡처가 만드는 수명과 객체 크기 모델을 이해하는 편이 더 중요하다.

### 3. 비용 / 단점

람다는 편하지만 공짜가 아니다. 캡처가 많아질수록 closure 객체 크기가 커지고, 그 람다를 `std::function` 같은 타입 소거 래퍼에 저장하면 추가 비용까지 붙을 수 있다.

- 큰 값을 여러 개 캡처하면 closure 객체 자체가 커져 복사 비용과 전달 비용이 커진다.
- `std::function`에 담기는 순간 small buffer를 넘으면 힙 할당이 발생할 수 있다.
- 람다는 인라인 최적화가 잘 되는 경우가 많지만, 타입 소거 경계나 복잡한 호출 경로를 지나면 그 이점이 줄어든다.
- 참조 캡처는 비용은 싸지만 수명 모델이 어려워져 비동기 코드에서 특히 위험하다.

### 4. 언제 쓰고 언제 안 쓰는가

#### 써야 하는 경우

- 짧은 로직을 호출 지점 근처에 두는 편이 더 읽기 좋을 때
- 상태를 함께 들고 다니는 일회성 콜백이나 알고리즘 조건식이 필요할 때
- 함수 객체 클래스를 별도로 정의하기엔 책임이 너무 작고 국소적일 때

#### 피해야 하는 경우

- hot path에서 큰 객체를 값으로 캡처해야 할 때
- 비동기 실행, 지연 실행, 저장되는 콜백처럼 수명 관리가 복잡할 때
- 람다 본문이 너무 길어져 오히려 named function이 더 읽기 쉬울 때

### 5. 대안 비교

- lambda: 지역 맥락과 상태를 함께 표현하기 좋고 가독성이 높지만, 캡처가 커지면 비용과 수명 복잡성이 올라간다.
- function pointer: 가장 가볍고 단순하지만 상태를 들고 다닐 수 없고 자유 함수/정적 함수 수준으로 제한된다.
- `std::function`: 다양한 callable을 한 타입으로 저장할 수 있지만, 타입 소거와 잠재적 할당 비용을 감수해야 한다.

대개는 람다부터 검토하고, 저장이 필요 없고 상태도 없으면 함수 포인터를 쓴다. 여러 종류의 callable을 같은 타입으로 보관해야 할 때만 `std::function`으로 한 단계 올려 생각하는 편이 자연스럽다.

### 6. 실전 사례

#### 참조 캡처가 수명 버그를 만든 경우

```cpp
std::function<void()> job;
{
    int value = 10;
    job = [&]() { std::cout << value; };
}
job();
```

이 코드는 `value`를 참조로 캡처한 뒤 원본 스코프가 끝난 다음 호출하므로 dangling reference가 된다. 장기 저장되거나 비동기로 실행되는 callable이라면 값 캡처 등으로 수명을 분리하는 설계를 먼저 검토하는 편이 안전하다.

#### 큰 캡처를 std::function에 저장한 경우

예를 들어 큰 `std::vector`나 `FString`을 값 캡처한 람다를 `std::function`에 넣으면, 캡처 비용과 타입 소거 비용이 함께 붙는다. 이런 경로가 매 프레임 반복된다면 람다 자체보다 저장 방식과 캡처 크기를 먼저 줄여야 한다.

### 7. 면접 질문

#### Q1. lambda, function pointer, std::function 중 무엇을 기본 선택으로 두나?

상태가 있는 국소 콜백이면 람다, 상태가 없고 저장도 필요 없으면 function pointer, 저장 시점에 타입을 통일해야 하면 `std::function`을 고른다. 핵심 기준은 상태 보유 여부와 저장 필요성이다.

#### Q2. 람다의 성능 비용은 어디서 생기나?

캡처 크기가 closure 객체 크기로 그대로 이어지고, 이 객체를 복사하거나 타입 소거 래퍼에 저장할 때 추가 비용이 생긴다. 따라서 람다는 '문법이 짧다'와 '실행 비용이 싸다'를 동일시하면 안 된다.

#### Q3. 비동기 코드에서 [this] 캡처가 위험한 이유는?

`[this]`는 객체 자체가 아니라 포인터를 잡는다. 콜백이 실행될 때 객체가 이미 파괴되었다면 dangling pointer가 되어, 가장 재현하기 어려운 수명 버그로 이어질 수 있다.

## 토픽 18: std::function & std::bind(에스티디 펑션 / 에스티디 바인드, 호출 래퍼와 인자 바인딩)

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 17

### 1. 핵심 개념

핵심 용어: `타입 소거(type erasure)`, `placeholder`, `타입 소거`, `람다`.

`std::function`은 함수 포인터, 람다, functor처럼 서로 다른 callable을 하나의 타입으로 감싸는 범용 호출 래퍼다. 내부적으로는 type erasure를 사용해 구체 타입을 숨기고 공통 호출 인터페이스만 노출한다.

`std::bind`는 함수와 일부 인자를 미리 묶어 새로운 호출 객체를 만드는 도구지만, placeholder 중심 문법 때문에 호출 흐름이 잘 드러나지 않는다. modern C++에서는 같은 의도를 보통 람다로 더 명확하게 표현한다.

### 2. 왜 필요한가

콜백 저장소나 이벤트 등록처럼 '지금은 어떤 callable이 들어올지 모르지만, 나중에 같은 시그니처로 호출하고 싶다'는 요구가 있을 때 `std::function`이 편하다. 호출 대상을 하나의 타입으로 통일하면 인터페이스가 단순해지기 때문이다.

다만 게임 코드나 엔진 코드에서는 호출 빈도, 수명, 리플렉션, UObject 소유권 같은 문제가 함께 붙는다. 그래서 편리하다는 이유만으로 `std::function`을 기본으로 두면 오히려 핫패스와 수명 경계를 흐리게 만들 수 있다.

### 3. 비용 / 단점

- `std::function`은 type erasure로 인해 간접 호출 비용이 생기고, 경우에 따라 힙 할당도 발생한다.
- `std::bind`는 placeholder와 중첩 바인딩 때문에 호출 인자 흐름을 읽기 어렵고 디버깅도 불편하다.
- 핫패스에서는 템플릿이나 직접 호출보다 인라인과 최적화 기회가 줄어들 수 있다.
- UE5 이벤트 흐름에서는 표준 callable보다 엔진 델리게이트가 수명과 바인딩 모델에 더 잘 맞는 경우가 많다.

### 4. 언제 쓰고 언제 안 쓰는가

#### 써야 하는 경우

- 저장 시점에 다양한 callable을 하나의 시그니처로 보관해야 할 때
- 호출 빈도는 낮지만 인터페이스 단순화가 더 중요한 콜백 등록 지점일 때
- 람다 타입을 외부 API에 직접 노출하고 싶지 않을 때

#### 안 쓰는 편이 나은 경우

- 매 프레임 반복되는 hot path처럼 호출 비용이 누적되는 곳
- 상태 없는 자유 함수를 단순 호출하면 충분한 곳
- UE 객체 수명과 리플렉션, 바인딩 해제가 중요한 엔진 이벤트 경로
- bind 표현이 람다보다 더 난해해지는 경우 대부분

### 5. 대안 비교

- lambda: 의도와 캡처가 가장 직접적으로 보이고 인라인 최적화 가능성이 높다.
- `std::function`: 타입 통일과 저장이 쉬우나 type erasure 비용과 잠재적 할당 비용을 감수해야 한다.
- function pointer: 가장 가볍고 빠르지만 상태를 가질 수 없다.
- UE delegate: 엔진 수명 모델과 바인딩/해제 흐름에 잘 맞지만 표준 C++ callable과는 목적이 다르다.

대개는 람다로 의도를 직접 표현하고, 저장이 필요할 때만 `std::function`을 고려한다. 상태가 없으면 function pointer가 더 단순하고, UObject 수명과 엔진 시스템에 묶이면 UE delegate가 더 잘 맞는 경우가 많다.

### 6. 실전 사례

#### std::bind를 람다로 치환한 경우

```cpp
auto f = [&player](int hp) {
    player.SetHP(hp);
};
```

이 표현은 `std::bind(&Player::SetHP, &player, std::placeholders::_1)`와 같은 의미를 더 직접적으로 보여 준다. 어떤 객체를 캡처했고, 어떤 인자를 받아 어디에 넘기는지가 즉시 읽히기 때문에 현대 C++에서는 보통 람다가 우선이다.

#### hot path에서 std::function을 피하는 경우

수백만 번 호출되는 루프에서 `std::function`은 간접 호출과 잠재적 할당 비용이 누적될 수 있다. 이런 경로에서는 callable을 템플릿 매개변수로 받거나, 구체 타입을 그대로 들고 가는 설계가 더 적합하다.

### 7. 면접 질문

#### Q1. std::function을 기본값으로 두지 않는 이유는?

유연성은 높지만 type erasure 비용과 저장 비용을 항상 지불하기 때문이다. 호출 대상을 굳이 숨길 필요가 없다면 구체 타입이나 람다가 더 가볍고 최적화도 잘 된다.

#### Q2. lambda와 std::function의 선택 기준은?

저장 없이 바로 넘기거나 타입을 알 수 있으면 람다를 유지하고, 여러 종류의 callable을 동일 타입으로 저장해야 할 때만 `std::function`을 쓴다. 저장 필요성이 가장 큰 분기점이다.

#### Q3. std::bind가 modern C++에서 지양되는 이유는?

placeholder 기반 문법이 호출 흐름을 숨기고, 디버깅과 가독성을 모두 해치기 때문이다. 같은 의미를 람다로 거의 항상 더 명확하게 표현할 수 있으므로, bind는 특별한 이유가 있을 때만 제한적으로 쓴다.

## 토픽 19: 템플릿

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 10

### 1. 핵심 개념

핵심 용어: `SFINAE`, `concepts`, `컴파일`, `실패`.

템플릿은 타입이나 값을 파라미터로 받아 컴파일 시점에 구체 코드를 생성하는 메커니즘이다. 함수 템플릿, 클래스 템플릿, 특수화, `if constexpr`, SFINAE, concepts까지 모두 '컴파일 타임에 일반화된 규칙을 구체화한다'는 같은 축 위에 있다.

핵심은 하나의 규칙을 여러 타입에 재사용하면서도, 필요한 제약과 분기를 타입 수준에서 표현한다는 점이다. 그래서 템플릿은 단순 편의 문법이 아니라, zero-cost abstraction을 가능하게 하는 언어 핵심 도구다.

### 2. 왜 필요한가

컨테이너, 수학 타입, traits, 범용 유틸리티처럼 같은 패턴을 여러 타입에 반복 적용해야 할 때 템플릿은 가장 강력한 선택지다. 런타임 다형성과 달리 구체 타입이 컴파일 시점에 드러나므로, 인라인과 최적화를 적극적으로 받을 수 있다.

UE와 표준 라이브러리의 많은 공용 도구가 템플릿 기반이기 때문에, 템플릿을 읽고 설계할 수 있어야 현대 C++ 코드베이스를 제대로 다룰 수 있다.

### 3. 비용 / 단점

- 인스턴스화가 많아질수록 컴파일 시간이 크게 늘어날 수 있다.
- 치환 실패와 제약 미스매치가 겹치면 에러 메시지가 매우 난해해진다.
- 타입 조합마다 코드가 복제되어 code bloat가 발생할 수 있다.
- 과도한 일반화는 호출 지점과 구현 의도를 흐려 디버깅과 유지보수를 어렵게 만든다.

### 4. 언제 쓰고 언제 안 쓰는가

#### 써야 하는 경우

- 같은 알고리즘을 여러 타입에 zero-cost로 재사용해야 할 때
- 컴파일 타임 제약과 분기가 성능이나 안정성에 직접 이점을 줄 때
- 호출 대상 타입이 런타임이 아니라 컴파일 타임에 이미 정해질 때

#### 안 쓰는 편이 나은 경우

- 단순한 로직인데 미래 확장 가능성만을 이유로 과도하게 일반화할 때
- 디버깅이 매우 중요하고, 명시적 타입과 단순한 호출 구조가 더 필요한 코드
- 실제 다형성이 런타임에 결정되어야 해서 템플릿만으로는 문제를 풀 수 없을 때

### 5. 대안 비교

- template: 컴파일 타임 다형성으로 가장 빠르고 유연하지만, 컴파일 비용과 코드 부피를 감수한다.
- runtime polymorphism: 타입이 런타임에 바뀌는 문제에 적합하고 인터페이스가 명확하지만 간접 호출 비용이 있다.
- overload: 경우의 수가 적고 지원 타입이 제한적일 때 가장 읽기 쉽고 단순하다.

즉 타입이 컴파일 타임에 정해지고 성능이 중요하면 템플릿이 유리하다. 반대로 교체 가능한 객체 집합을 런타임에 다뤄야 하면 virtual 기반 다형성이 더 자연스럽고, 지원 타입이 몇 개 안 되면 오버로드가 가장 단순하다.

### 6. 실전 사례

#### 일반 if를 써서 템플릿 분기가 깨진 경우

```cpp
template<typename T>
void Print(const T& value) {
    if constexpr (std::is_integral_v<T>) {
        std::cout << value + 1;
    } else {
        std::cout << value.size();
    }
}
```

템플릿 분기에서는 일반 `if`가 아니라 `if constexpr`가 필요한 경우가 많다. 그래야 실제로 선택되지 않는 분기는 인스턴스화되지 않고, 타입마다 유효한 코드만 남는다.

#### 과도한 일반화가 역효과를 내는 경우

지원 타입이 두세 개뿐인데 추상화 욕심으로 복잡한 템플릿 계층을 만들면, 구현보다 제약과 에러 메시지를 읽는 데 더 많은 시간이 든다. 이런 경우는 차라리 명시적 오버로드나 단순한 인터페이스가 더 유지보수에 유리하다.

### 7. 면접 질문

#### Q1. template와 runtime polymorphism은 어떻게 고르나?

타입이 컴파일 시점에 정해지고 성능이 중요하면 template, 타입 선택이 런타임에 바뀌고 인터페이스 안정성이 더 중요하면 runtime polymorphism을 고른다. 선택 기준은 '타입 결정 시점'과 '비용을 어디서 치를 것인가'다.

#### Q2. 템플릿의 가장 큰 비용은 무엇인가?

런타임 비용보다 컴파일 비용과 개발자 비용이 크다. 컴파일 시간 증가, 난해한 에러 메시지, code bloat가 함께 오기 때문에 성능 이점만 보고 남용하면 전체 생산성이 떨어진다.

#### Q3. concepts가 왜 중요한가?

템플릿이 어떤 타입을 요구하는지 선언부에서 문서처럼 드러낼 수 있기 때문이다. 결과적으로 제약이 명확해지고, SFINAE 중심 코드보다 에러 메시지와 의도가 모두 읽기 쉬워진다.

## 토픽 20: STL 컨테이너 & 알고리즘(에스티엘, 표준 라이브러리 자료구조와 알고리즘)

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 19

### 1. 핵심 개념

핵심 용어: `iterator`, `iterator invalidation`, `연속`, `erase`.

STL의 핵심은 컨테이너와 알고리즘을 iterator로 연결하는 설계다. `vector`, `deque`, `list`, `map`, `unordered_map`은 각자 메모리 배치와 접근 패턴이 다르고, 알고리즘은 이 공통 인터페이스 위에서 작동한다.

실무에서 중요한 것은 API 이름보다 데이터 배치와 invalidation 규칙이다. 즉 '이 컨테이너가 이 연산을 지원하는가'보다 '이 접근 패턴에서 실제로 빠르고 안전한가'를 먼저 봐야 한다.

### 2. 왜 필요한가

자료구조 선택은 성능, 메모리 사용, 코드 단순성에 직접 연결된다. 같은 O 표기라도 캐시 locality, 해시 비용, 노드 할당 비용 때문에 실제 성능은 크게 달라질 수 있다.

게임 코드에서는 데이터가 많고 프레임 예산이 빡빡하기 때문에, 컨테이너 선택 실수가 곧바로 캐시 미스, 재할당, 느린 순회, 무효화 버그로 이어진다. 그래서 STL은 문법 지식보다 선택 기준이 더 중요하다.

### 3. 비용 / 단점

- `vector`는 재할당 시 iterator, 포인터, 참조가 무효화될 수 있다.
- `list`는 삽입/삭제 자체는 싸 보이지만 노드 기반 구조라 pointer chasing 때문에 순회가 느리다.
- `unordered_map`은 평균 O(1) 조회가 강점이지만 해시 계산 비용, 충돌, 순서 부재를 감수해야 한다.
- `map`은 정렬과 안정성은 좋지만 트리 탐색과 노드 할당 비용이 붙는다.

### 4. 언제 쓰고 언제 안 쓰는가

#### 기본 선택

- 연속 메모리 순회와 캐시 효율이 중요하면 `vector`를 기본값으로 둔다.
- 키 조회가 핵심이고 순서가 필요 없으면 `unordered_map`을 우선 검토한다.
- 정렬 순서 유지, lower_bound 계열 질의, 안정적인 트리 성질이 필요하면 `map`/`set`을 쓴다.

#### 피해야 하는 선택

- 중간 삽입/삭제가 많다는 이유만으로 `list`를 자동 선택하지 않는다.
- 순서가 중요한데도 습관적으로 `unordered_map`을 고르지 않는다.
- invalidation 규칙을 감당하지 못하면서 `vector` 내부 원소 주소를 오래 들고 있지 않는다.

### 5. 대안 비교

#### 시퀀스 컨테이너

- `vector`: cache locality가 가장 좋아 기본 선택으로 적합하다.
- `deque`: 앞뒤 삽입은 유연하지만 메모리가 연속적이지 않아 `vector`만큼 단순하지 않다.
- `list`: iterator 안정성은 좋지만 순회와 메모리 효율이 나빠 실제 기본 선택이 되기 어렵다.

#### 연관 컨테이너

- `unordered_map`: 평균 조회 성능이 좋지만 해시 비용과 순서 부재를 감수한다.
- `map`: 정렬과 범위 질의가 필요할 때 유리하지만, 일반 조회만 보면 `unordered_map`보다 비쌀 수 있다.

실무에서는 특별한 이유가 없으면 `vector`부터 검토하는 경우가 많다. `list`는 이름만 보면 삽입과 삭제에 강해 보이지만, 실제 CPU 친화성은 대개 훨씬 나쁘다.

### 6. 실전 사례

#### erase-remove와 실제 삭제

```cpp
std::vector<int> values = {1, 2, 3, 4, 5};
values.erase(
    std::remove_if(values.begin(), values.end(), [](int v) { return v % 2 == 0; }),
    values.end()
);
```

`remove_if`는 원소를 앞으로 당겨 놓기만 하고 컨테이너 크기를 줄이지 않는다. 그래서 erase-remove idiom을 이해하지 못하면 '지운 줄 알았는데 안 지워진' 버그가 생긴다.

#### 왜 vector가 기본 선택인가

연속 메모리 덕분에 순회가 빠르고, CPU prefetch와 cache locality의 혜택을 가장 잘 받기 때문이다. `list`는 삽입/삭제 자체는 O(1)이어도, 실제로는 노드 따라가기 비용이 커서 대부분의 실전 데이터 처리에서는 `vector`보다 느리다.

### 7. 면접 질문

#### Q1. 왜 vector를 기본 선택으로 두나?

연속 메모리 구조 덕분에 순회, 탐색, 정렬, 알고리즘 적용 대부분에서 실제 성능이 좋기 때문이다. 특별한 이유가 없으면 `vector`가 가장 단순하고 CPU 친화적이다.

#### Q2. list를 언제 써야 하나?

iterator 안정성이 정말 중요하고, 중간 삽입/삭제가 매우 많으며, 순회 성능보다 링크 조작이 더 중요한 특수한 경우에만 제한적으로 고려한다. 일반적인 성능 문제에서는 '거의 안 쓴다'에 가깝다.

#### Q3. map과 unordered_map은 어떻게 고르나?

순서와 범위 질의가 필요하면 `map`, 평균 빠른 조회와 순서 무관이 핵심이면 `unordered_map`이다. 해시 비용과 key distribution까지 함께 고려해야 실제 선택이 맞아진다.

## 토픽 21: 멀티스레딩 기초

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 5

### 1. 핵심 개념

핵심 용어: `race condition`, `predicate`, `mutex`, `RAII`.

멀티스레딩은 여러 실행 흐름이 동시에 상태를 읽고 쓰는 환경이며, 핵심 위험은 race condition과 deadlock이다. 기본 도구로는 `std::thread`, `std::mutex`, `std::lock_guard`, `std::atomic`, `std::condition_variable`이 있다.

중요한 점은 atomic 하나로 모든 동기화가 해결되지 않는다는 것이다. 원자성은 단일 연산의 찢어짐만 막아 주며, 여러 변수의 일관성이나 더 큰 불변식은 별도의 동기화 전략이 필요하다.

### 2. 왜 필요한가

로딩, 백그라운드 계산, 리소스 준비, 작업 분산처럼 독립적인 일을 병렬화하면 전체 응답성과 처리량을 높일 수 있다. 게임과 엔진 코드에서는 프레임 시간과 CPU 자원을 아끼기 위해 멀티스레딩이 자주 필요해진다.

하지만 멀티스레딩의 진짜 난점은 성능이 아니라 정합성이다. 공유 상태를 잘못 다루면 버그가 비결정적으로 나타나고, 재현과 디버깅 비용이 단일 스레드 코드보다 훨씬 커진다.

### 3. 비용 / 단점

- context switch는 생각보다 비싸며, 작업이 너무 잘게 쪼개지면 병렬화 이득을 상쇄한다.
- lock contention이 심하면 스레드를 늘려도 오히려 대기 시간이 늘어난다.
- false sharing은 서로 다른 변수를 만져도 같은 cache line을 공유할 때 큰 성능 손실을 만든다.
- race condition과 deadlock은 재현이 어렵고 로그만으로 원인을 좁히기 힘들어 디버깅 비용이 매우 높다.

### 4. 언제 쓰고 언제 안 쓰는가

#### 써야 하는 경우

- 작업이 충분히 독립적이고 병렬 실행으로 실제 병목이 줄어드는 경우
- I/O 대기나 긴 계산을 메인 경로에서 분리해야 하는 경우
- 공유 상태를 줄이거나 소유권을 명확히 나눌 수 있는 경우

#### 안 쓰는 편이 나은 경우

- 작업 단위가 너무 작아 context switch와 동기화 비용이 더 큰 경우
- 공유 상태가 너무 많아 락과 경쟁이 병목이 되는 경우
- 문제를 단일 스레드 구조 개선이나 작업 분해만으로 해결할 수 있는 경우

### 5. 대안 비교

- mutex: 여러 연산을 하나의 임계 구역으로 묶어 복합 상태를 안전하게 보호할 수 있다.
- atomic: 단일 변수의 원자적 읽기/쓰기나 CAS에 적합하지만, 복합 불변식 보호에는 부족하다.
- lock-free: contention을 줄일 잠재력은 있지만 설계와 검증, 디버깅 난도가 매우 높다.
- 단일 스레드/작업 분해: 가장 단순하고 디버깅이 쉬우며, 병목이 명확하지 않다면 오히려 더 좋은 선택이다.

실무에서는 먼저 mutex 기반의 단순한 동기화로 문제를 푼다. atomic은 정말 단일 상태에만, lock-free는 contention이 확실한 병목이고 팀이 그 복잡도를 감당할 수 있을 때만 검토하는 편이 현실적이다.

### 6. 실전 사례

#### atomic으로 복합 상태를 보호하려 한 경우

```cpp
std::atomic<int> balance{100};
std::atomic<bool> isActive{true};

void withdraw(int amount) {
    if (isActive.load() && balance.load() >= amount) {
        balance -= amount;
    }
}
```

각 load와 store는 원자적이어도, 두 변수 사이의 일관성은 보장되지 않는다. 이런 복합 상태는 mutex로 묶어 한 번에 보호해야 한다.

#### false sharing과 lock ordering

서로 다른 스레드가 논리적으로 독립된 변수만 수정해도 같은 cache line을 공유하면 false sharing으로 성능이 크게 떨어질 수 있다. 또한 여러 락을 서로 다른 순서로 잡으면 deadlock이 생기므로, 락 획득 순서를 전역 규칙으로 통일해야 한다.

### 7. 면접 질문

#### Q1. mutex와 atomic은 어떻게 고르나?

단일 카운터나 플래그처럼 한 변수의 원자 연산이면 atomic, 여러 연산과 여러 변수의 일관성을 함께 지켜야 하면 mutex를 고른다. 기준은 비용이 아니라 '보호해야 하는 불변식의 범위'다.

#### Q2. lock-free는 언제 고려하나?

contention이 실제 병목이고, mutex 기반 설계가 충분히 단순화된 뒤에도 성능이 안 나올 때만 검토한다. lock-free는 빠를 수도 있지만 구현과 검증 비용이 매우 크므로, 기본 선택지가 되어서는 안 된다.

#### Q3. 멀티스레딩 버그는 어떻게 디버깅하나?

재현 조건을 좁히기 위해 로그, thread sanitizer류 툴, 락 획득 순서 점검, deterministic한 테스트 조건을 함께 사용한다. 중요한 것은 코드 수정 전에 공유 상태와 동기화 경계를 먼저 모델로 그려 보는 것이다.
