## 토픽 16: move semantics(무브 세맨틱스, 이동 의미론)

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 3, 4

### 1. 문제 상황

핵심 용어: `valid but unspecified`, `perfect forwarding`, `rvalue`, `값 카테고리(value category)`.

문자열, 동적 버퍼, 컨테이너처럼 내부 자원을 소유한 타입을 값으로 주고받는 일은 modern C++에서 매우 흔하다. 이때 개발자가 '복사는 비싸니까 일단 `std::move`를 붙이면 되겠지'라고 생각하면, 단순한 성능 문제가 아니라 더 위험한 소유권 버그와 상태 불변식 붕괴를 만들기 쉽다.

실무에서 자주 보이는 사고는 대체로 두 가지다. 첫째, forwarding reference를 받는 래퍼가 `std::move`를 남발해 호출자의 lvalue까지 강제로 rvalue처럼 취급하면서, 호출자 입장에서는 아직 써야 하는 객체를 뜻밖에 moved-from 상태로 만들어 버린다. 둘째, 이동이 끝난 객체를 비어 있거나 `nullptr`에 가까운 상태라고 멋대로 가정하고 다시 읽는다. 하지만 표준이 보장하는 것은 '유효하지만 값은 특정되지 않은 상태(valid but unspecified)'일 뿐이며, 구체적인 내용까지 기대하는 순간 코드 의미가 흐려진다.

결국 move semantics를 제대로 이해하려면 단순히 '복사보다 빠른 기법'으로 볼 것이 아니라, **표현식의 value category가 무엇인지**, **어떤 상황에서 이동 생성자나 이동 대입 연산자가 선택되는지**, **이동 후 객체를 어떤 규칙으로 다뤄야 하는지**를 함께 이해해야 한다.

### 2. 핵심 개념

move semantics는 복사 대신 자원의 소유권이나 내부 표현을 이전하여, 불필요한 깊은 복사 비용을 줄이는 규칙이다. 여기서 핵심은 객체 자체보다 **표현식이 어떤 value category에 속하는가**를 구분하는 데 있다. C++ 표현식은 크게 `lvalue`, `xvalue`, `prvalue`로 나뉘며, 이 중 `xvalue`와 `prvalue`를 포함하는 rvalue 계열 표현식은 대체로 '자원을 가져가도 되는 대상'으로 취급된다.

이동은 `std::move`가 수행하는 것이 아니다. `std::move`는 단지 주어진 표현식을 **rvalue, 더 정확히는 xvalue로 캐스팅**하여 '이 객체는 이제 이동 대상으로 취급해도 된다'고 표시할 뿐이다. 실제 자원 이전은 그 결과를 받은 이동 생성자(move constructor)나 이동 대입 연산자(move assignment)가 수행한다. 반대로 `std::forward`는 forwarding reference 문맥에서 호출자가 넘긴 원래의 value category를 보존해 전달할 때 사용한다.

또한 이동 후 객체는 보통 valid but unspecified 상태로 남는다. 즉 객체는 여전히 파괴 가능하고, 새 값을 대입할 수도 있으며, 클래스 불변식도 유지되어야 한다. 다만 어떤 값을 가지고 있는지, 비어 있는지, 기존 자원이 완전히 사라졌는지까지는 일반적으로 가정할 수 없다. 따라서 move semantics는 단순한 성능 최적화 문법이 아니라, **소유권 이전과 후속 사용 규칙을 타입 차원에서 명확히 드러내는 설계 수단**이다.

### 3. 내부 동작 / 메커니즘

#### 값 카테고리와 오버로드 선택

컴파일러는 표현식의 value category를 바탕으로 복사 생성자와 이동 생성자 중 어떤 후보를 사용할지 결정한다. 중요한 점은 **이름이 있는 변수는 타입과 무관하게 표현식으로 쓰이는 순간 기본적으로 lvalue**라는 것이다. 예를 들어 `T&&` 타입의 지역 변수라도 이름을 붙여 다시 사용하면 그 표현식은 lvalue로 간주된다. 그래서 실제로 이동을 유도하고 싶다면 `std::move`를 통해 xvalue 문맥을 만들어 주어야 한다.

반대로 forwarding reference는 호출자가 넘긴 lvalue와 rvalue를 모두 받을 수 있다. 이 문맥에서 래퍼 함수가 무턱대고 `std::move`를 사용하면, 원래는 lvalue로 전달된 인자까지 강제로 이동 대상으로 바꿔 버린다. 이런 래퍼는 호출자의 의도를 왜곡한다. 따라서 forwarding reference에서는 `std::forward<T>`를 사용해 원래의 value category를 보존해야 한다. move semantics를 제대로 쓰는 핵심은 '무조건 이동시키는 것'이 아니라, **어떤 표현식이 실제로 이동 가능한 문맥인지 정확히 구분하는 것**이다.

#### const와 `noexcept`가 개입하는 이유

`const` 객체에 `std::move`를 적용하면 타입은 `const T&&`가 된다. 하지만 대부분의 이동 생성자와 이동 대입 연산자는 내부 자원을 훔쳐 오기 위해 원본 객체를 수정해야 하므로 `T&&`를 받는다. 따라서 `const T&&`는 일반적인 이동 연산의 대상이 되지 못하고, 결과적으로 복사 생성자나 복사 대입 연산자가 선택되는 경우가 많다. 즉 `std::move`를 썼다는 사실만으로 이동이 보장되지는 않는다.

또한 표준 컨테이너는 재배치 중 예외가 발생하더라도 기존 상태를 보존해야 하는 경우가 많다. 이 때문에 `std::vector` 같은 컨테이너는 원소를 재배치할 때, 이동 생성자가 `noexcept`가 아니면 복사를 선택할 수 있다. 이동 연산이 문법상 존재하더라도 예외 보장이 약하면 컨테이너는 더 안전한 경로를 택하는 것이다. 결국 move semantics의 성능 이점은 단순히 `&&` 문법을 썼는지가 아니라, **타입의 예외 보장, 오버로드 조건, moved-from 상태 설계**까지 포함한 전체 품질에 달려 있다.

### 4. 자주 틀리는 포인트

* `std::move`를 호출하면 즉시 이동이 끝난다고 생각한다. 실제 이동은 move constructor나 move assignment가 선택되어 실행될 때 일어난다.
* 이름이 `T&&` 타입이면 그 표현식도 rvalue라고 생각한다. 하지만 이름이 있는 변수 표현식은 기본적으로 lvalue다.
* moved-from 객체가 항상 비어 있거나 `nullptr` 비슷한 상태일 것이라고 가정한다. 표준이 보장하는 것은 valid but unspecified 상태뿐이다.
* const 객체에 `std::move`를 붙이면 무조건 이동된다고 생각한다. 실제로는 복사 경로가 선택되는 경우가 많다.
* 반환문에서 습관적으로 `return std::move(local);`를 쓴다. 이런 코드는 불필요할 뿐 아니라 반환값 최적화(NRVO, copy elision)에 방해가 될 수 있다.
* 작은 trivially copyable 타입까지 기계적으로 move하려 든다. 이런 경우는 복사가 더 단순하고, 최적화 관점에서도 이득이 거의 없을 수 있다.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

#### 써야 하는 경우

* 함수가 더 이상 사용하지 않을 지역 자원 객체를 다른 객체나 컨테이너에 넘길 때
* 큰 버퍼, 문자열, 핸들처럼 소유권 이전 비용이 복사보다 훨씬 저렴한 타입일 때
* `std::unique_ptr`처럼 복사 자체가 불가능하고, 소유권 이전만 의미 있는 타입을 다룰 때
* forwarding reference 문맥에서 호출자의 value category를 왜곡하지 않고 그대로 전달해야 할 때
* setter나 factory 함수처럼 값 전달 후 내부에 저장하면서 move를 활용할 수 있는 인터페이스를 설계할 때

#### 안 쓰는 편이 나은 경우

* 이후에도 같은 객체를 계속 읽거나 쓸 계획이 있을 때
* const 객체라서 실제 이동이 일어나지 않을 가능성이 클 때
* 작은 POD나 trivially copyable 타입처럼 복사가 충분히 싸고 의미도 단순할 때
* 함수 인자를 단순 읽기 전용으로만 다루고, 소유권 이전 의도가 전혀 없을 때
* 지역 변수를 반환하면서 굳이 `std::move`를 붙여 반환값 최적화를 방해할 수 있을 때

### 6. 대안 비교

move 관련 판단은 결국 호출자와 피호출자 사이에 **누가 이 값을 이후에도 의미 있게 사용할 것인가**를 결정하는 문제다. 따라서 전달 방식은 성능 미세 조정보다 먼저 소유권 의도를 분명히 드러내야 한다.

* 복사: 원본을 그대로 유지해야 하고, 타입이 작거나 복사 비용이 충분히 감당 가능할 때 기본 선택이다.
* move: 호출 뒤 원본을 더 이상 의미 있게 사용하지 않을 때 적합하다. 자원 소유 타입에서 특히 중요하다.
* perfect forwarding: 래퍼 함수가 호출자의 lvalue/rvalue 의도를 바꾸지 않고 그대로 전달해야 할 때 쓴다.
* `const T&`: 읽기 전용 입력에 가장 무난하고 오해가 적다.
* `T&&`: 명시적으로 소유권 이전을 받는 인터페이스에서 사용한다. 의미가 강한 만큼 남용하면 오히려 API 의도가 흐려진다.
* `T` 값 전달: 함수 내부에서 어차피 복사본이 필요하고, 받은 값을 멤버에 저장하면서 move할 수 있을 때 단순하고 실용적인 선택이 된다.

### 7. 코드 예시 (bad → good)

#### 사례 1: forwarding reference를 무조건 move한 경우

Bad: 전달받은 인자가 lvalue여도 무조건 rvalue로 바꿔 버린다. 호출자가 이후에도 객체를 사용해야 한다면 의도치 않은 이동이 발생한다.

```cpp
template<typename T>
void Push(std::vector<T>& out, T&& value) {
    out.push_back(std::move(value));
}
```

Good: forwarding reference에서는 `std::forward<T>`를 사용해야 원래 value category가 유지된다. 래퍼 함수는 자기 판단으로 이동을 강제하지 않는 편이 맞다.

```cpp
template<typename T>
void Push(std::vector<T>& out, T&& value) {
    out.push_back(std::forward<T>(value));
}
```

#### 사례 2: move assignment에서 기존 자원 해제를 누락한 경우

Bad: 기존 `data_`를 먼저 해제하지 않고 새 포인터로 덮어쓰면 메모리 누수가 생긴다. move assignment는 소유권 이전 전에 자신이 들고 있던 자원의 정리부터 처리해야 한다.

```cpp
Buffer& operator=(Buffer&& other) noexcept {
    data_ = other.data_;
    size_ = other.size_;
    other.data_ = nullptr;
    other.size_ = 0;
    return *this;
}
```

Good: 기존 자원을 먼저 정리한 뒤 소유권을 이전한다. `noexcept`를 명시해 컨테이너가 재배치 시 이동 경로를 더 적극적으로 선택할 수 있게 한다.

```cpp
Buffer& operator=(Buffer&& other) noexcept {
    if (this != &other) {
        delete[] data_;

        data_ = other.data_;
        size_ = other.size_;

        other.data_ = nullptr;
        other.size_ = 0;
    }
    return *this;
}
```

#### 사례 3: 반환값에 습관적으로 `std::move`를 붙인 경우

Bad: 지역 변수를 반환하면서 무조건 `std::move`를 붙이면, 코드 의도는 더 나빠지고 반환값 최적화에 불리할 수 있다.

```cpp
std::string MakeName() {
    std::string name = "player";
    return std::move(name);
}
```

Good: 지역 변수 반환에서는 우선 자연스럽게 그대로 반환한다. 컴파일러는 copy elision이나 move를 적절히 선택할 수 있다.

```cpp
std::string MakeName() {
    std::string name = "player";
    return name;
}
```

#### 사례 4: move 후 객체를 계속 의미 있게 사용한 경우

Bad: move한 뒤에도 원래 값이 남아 있을 것이라고 기대하며 다시 읽는다. 이런 코드는 타입 구현 세부사항에 기대게 되어 의미가 불분명하다.

```cpp
std::string s = "boss";
names.push_back(std::move(s));

if (s == "boss") {
    StartFight();
}
```

Good: move 후 객체는 파괴하거나 새 값을 대입하는 용도로만 다루는 편이 안전하다. 계속 써야 한다면 애초에 move하지 않는다.

```cpp
std::string s = "boss";
names.push_back(s);

if (s == "boss") {
    StartFight();
}
```

또는

```cpp
std::string s = "boss";
names.push_back(std::move(s));
s = "next_boss";
```

### 8. 판단 기준 요약

* `std::move`는 이동 실행이 아니라 xvalue 캐스트라는 점부터 기억한다.
* 이동 여부는 타입, 오버로드 선택, 예외 보장에 의해 최종 결정된다.
* 이름이 있는 변수 표현식은 `T&&` 타입이어도 기본적으로 lvalue다.
* 이후에도 값을 의미 있게 써야 하면 move하지 않고, 더 이상 쓰지 않을 때만 move한다.
* forwarding reference에서는 기본값이 `std::forward<T>`다.
* moved-from 객체는 읽어서 의미를 기대하지 말고, 파괴하거나 새 값을 대입하는 용도로만 다룬다.
* 성능 이득을 기대한다면 move constructor의 `noexcept`, 타입 크기, 자원 소유 구조까지 함께 확인한다.
* 가능하면 직접 자원 관리를 늘리기보다 표준 라이브러리 타입을 조합해 Rule of Zero에 가깝게 설계하는 편이 안전하다.

## 토픽 17: 람다

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 16

### 1. 문제 상황

핵심 용어: `함수 객체(functor)`, `즉시 호출(IIFE)`, `mutable`, `초기화`.

알고리즘 호출, 비동기 작업, 이벤트 바인딩에서는 짧은 동작을 지금 이 자리에서 표현하고 싶을 때가 많다. 이때 람다는 별도 functor 클래스를 만들지 않고도 지역 맥락과 함께 callable을 정의하게 해 준다.

문제는 람다가 간결해 보인다는 이유로 캡처와 수명을 가볍게 취급하기 쉽다는 점이다. `[&]`나 `[this]`를 무심코 사용하면 원본 스코프가 끝난 뒤에도 참조나 포인터를 붙잡아 두는 버그가 생길 수 있고, 큰 값을 값 캡처하면 객체 크기와 전달 비용이 빠르게 불어난다.

### 2. 핵심 개념

람다는 이름 없는 함수 객체를 즉석에서 정의하는 문법이다. 표면적으로는 짧은 함수처럼 보이지만 실제로는 캡처한 상태를 멤버로 가지는 closure object를 만든다. 따라서 람다는 '문법 설탕'이면서 동시에 '상태 있는 객체'라는 두 관점을 함께 가져야 한다.

값 캡처는 현재 값을 복사해 람다 내부에 저장하고, 참조 캡처는 원본 변수에 직접 연결된다. `[this]`는 객체 자체가 아니라 포인터를 잡고, 초기화 캡처는 `[x = expr]` 형태로 이동 캡처 같은 패턴을 가능하게 한다. `mutable`은 값 캡처된 복사본을 람다 내부에서 수정할 수 있게 해 준다.

### 3. 내부 동작 / 메커니즘

#### closure object가 만들어지는 방식

컴파일러는 람다를 보면 이름 없는 클래스와 그 `operator()`를 만든다고 생각하면 이해가 쉽다. 값 캡처는 그 클래스의 멤버로 복사되어 저장되고, 참조 캡처는 원본을 가리키는 참조나 포인터 비슷한 연결을 유지한다. 따라서 캡처 목록은 단순 문법이 아니라 람다 객체의 데이터 레이아웃을 결정한다.

#### 수명과 비용이 갈리는 지점

기본적으로 람다의 `operator()`는 `const`이므로 값 캡처 변수는 내부에서 수정할 수 없다. `mutable`을 붙이면 람다 자신의 복사본 상태는 바꿀 수 있지만, 원본 변수까지 바뀌는 것은 아니다. 또한 generic lambda는 매개변수에 `auto`를 사용해 템플릿과 비슷한 `operator()`를 생성한다.

람다를 바로 호출하면 대체로 인라인 최적화가 잘 되지만, `std::function` 같은 타입 소거 래퍼에 저장하면 closure 객체 복사와 간접 호출 비용이 추가될 수 있다. 즉 람다의 실제 비용은 본문보다 캡처 크기와 저장 방식에서 결정되는 경우가 많다.

### 4. 자주 틀리는 포인트

- 비동기 람다나 장기 저장되는 콜백에서 `[&]`를 습관적으로 사용한다. 원본 수명이 끝나면 dangling reference가 된다.
- `[this]`가 객체 자체를 복사한다고 오해한다. 실제로는 포인터를 잡기 때문에 객체 파괴 뒤 호출하면 위험하다.
- 큰 `std::vector`나 `FString`을 값 캡처해 놓고 비용이 없다고 생각한다. closure 객체 크기와 복사 비용이 그대로 늘어난다.
- `mutable`을 붙이면 원본 변수도 같이 바뀐다고 착각한다. 수정되는 것은 람다 내부 복사본이다.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

#### 써야 하는 경우

- 짧은 로직을 호출 지점 근처에 두는 편이 더 읽기 좋을 때
- 상태를 함께 들고 다니는 일회성 콜백이나 알고리즘 조건식이 필요할 때
- 함수 객체 클래스를 별도로 정의하기엔 책임이 너무 작고 국소적일 때

#### 안 쓰는 편이 나은 경우

- hot path에서 큰 객체를 값으로 캡처해야 할 때
- 비동기 실행, 지연 실행, 저장되는 콜백처럼 수명 관리가 복잡할 때
- 람다 본문이 길어져 오히려 named function이 더 읽기 쉬울 때

### 6. 대안 비교

- lambda: 지역 맥락과 상태를 함께 표현하기 좋고 가독성이 높지만, 캡처가 커지면 비용과 수명 복잡성이 올라간다.
- function pointer: 가장 가볍고 단순하지만 상태를 들고 다닐 수 없다.
- `std::function`: 다양한 callable을 한 타입으로 저장할 수 있지만, 타입 소거와 잠재적 할당 비용을 감수해야 한다.

기본 선택은 보통 람다다. 저장이 필요 없고 상태도 없으면 함수 포인터가 더 단순하고, 여러 종류의 callable을 같은 인터페이스로 오래 보관해야 할 때만 `std::function`으로 한 단계 올려 생각하는 편이 자연스럽다.

### 7. 코드 예시 (bad → good)

#### 사례 1: 참조 캡처를 장기 저장한 경우

Bad: `value`를 참조로 캡처한 뒤 원본 스코프가 끝난 다음 호출한다. 저장되는 callable에서는 참조 캡처가 바로 수명 버그로 이어질 수 있다.

```cpp
std::function<void()> job;
{
    int value = 10;
    job = [&]() { std::cout << value; };
}
job();
```

Good: 값 캡처로 바꾸면 람다 객체가 자기 복사본을 들고 있으므로 원본 스코프가 끝나도 안전하다. 장기 저장과 비동기 실행에서는 먼저 수명 분리부터 검토하는 편이 맞다.

```cpp
std::function<void()> job;
{
    int value = 10;
    job = [value]() { std::cout << value; };
}
job();
```

#### 사례 2: `[this]` 캡처를 비동기 수명과 섞은 경우

Bad: `[this]`는 객체를 복사하는 것이 아니라 포인터를 저장한다. 콜백이 더 오래 살아남으면 dangling pointer가 된다.

```cpp
class Widget {
    int value_ = 42;
public:
    auto getCallback() {
        return [this] {
            return value_;
        };
    }
};

auto cb = widget->getCallback();
delete widget;
cb();
```

Good: 객체 복사 캡처가 허용되는 문맥이라면 `[*this]`처럼 수명을 분리하는 쪽이 더 안전하다. 핵심은 람다가 무엇을 들고 있는지를 코드상에서 분명히 하는 것이다.

```cpp
class Widget {
    int value_ = 42;
public:
    auto getCallback() {
        return [*this] {
            return value_;
        };
    }
};
```

### 8. 판단 기준 요약

- 람다는 함수가 아니라 상태 있는 closure object라고 생각한다.
- 장기 저장되거나 비동기로 실행되면 참조 캡처보다 수명 분리 전략을 먼저 본다.
- 큰 값을 캡처할 때는 closure 객체 크기와 복사 비용을 함께 계산한다.
- `mutable`은 원본 변경이 아니라 람다 내부 복사본 변경이라는 점을 구분한다.
- 상태 없는 즉시 사용 callable이면 람다, 저장과 타입 통일이 필요하면 `std::function`을 검토한다.

## 토픽 18: std::function & std::bind(에스티디 펑션 / 에스티디 바인드, 호출 래퍼와 인자 바인딩)

## 토픽 18: std::function & std::bind

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 17

---

### 1. 문제 상황

콜백을 저장해야 하는 상황에서는,  
현재 시점에서는 어떤 callable이 들어올지 알 수 없지만  
나중에는 동일한 방식으로 호출해야 하는 요구가 발생한다.

예를 들어:

* 이벤트 시스템
* 작업 큐
* UI 콜백

이러한 구조에서는 함수 포인터, 람다, functor 등 서로 다른 형태의 callable을  
하나의 타입으로 통일하여 저장할 필요가 있다.

이 문제를 해결하기 위해 `std::function`이 사용된다.

---

### 2. 핵심 개념

#### std::function의 역할

`std::function`은 서로 다른 callable을  
하나의 공통 인터페이스로 감싸는 호출 래퍼이다.

핵심은 **type erasure(타입 소거)**이다.

즉, 실제 타입은 숨기고  
“이 시그니처로 호출할 수 있다”는 정보만 남긴다.

```cpp
std::function<void(int)> f;
````

이 변수에는 다음이 모두 들어갈 수 있다:

* 함수 포인터
* 람다
* functor

---

#### std::bind의 역할

`std::bind`는 함수와 일부 인자를 미리 결합하여
새로운 callable을 생성한다.

```cpp
auto f = std::bind(&Player::SetHP, &player, std::placeholders::_1);
```

이 코드는 “hp 하나를 받아 SetHP를 호출하는 함수”를 만든다.

그러나 modern C++에서는 동일한 기능을
람다로 더 명확하게 표현할 수 있다.

---

### 3. 동작 원리 (핵심)

#### std::function 내부 구조

`std::function`은 내부적으로 다음과 같은 구조를 가진다:

```text
[ std::function 객체 ]

+----------------------+
| small buffer         |  ← 작은 객체는 여기에 저장
+----------------------+
| callable pointer ----|----+
+----------------------+    |
                            v
                      실제 callable (힙 or 내부)
```

핵심 구성:

* **type-erased wrapper**

  * 실제 타입을 숨긴 인터페이스

* **function pointer (dispatcher)**

  * 실제 호출을 수행하는 함수

---

#### 호출 과정

```cpp
f(10);
```

실제 동작:

```text
1. std::function 내부 dispatcher 호출
2. 저장된 callable에 간접 접근
3. 실제 함수 실행
```

즉, 호출은 항상:

> **간접 호출 (indirect call)**

---

#### small object optimization (SOO)

작은 callable은 힙 할당 없이 내부 버퍼에 저장된다.
큰 객체는 힙에 할당된다.

→ 따라서 비용은 다음에 따라 달라진다:

* callable 크기
* 복사 여부
* 호출 빈도

---

### 4. 이 구조의 의미

type erasure는 강력하지만, 반드시 비용을 만든다.

#### 얻는 것

* 타입 통일
* 인터페이스 단순화
* 유연성

#### 잃는 것

* 인라인 최적화 불가
* 간접 호출 비용
* 힙 할당 가능성

즉:

> **유연성과 성능의 트레이드오프**

---

### 5. std::bind의 구조적 한계

`std::bind`는 인자 일부를 고정하고
나머지를 placeholder로 표현한다.

```cpp
std::bind(f, _1, 10, _2);
```

이 표현은 다음 문제를 만든다:

* 인자 흐름이 코드에 드러나지 않음
* 실행 시점이 아닌 “정의 시점”에 묶임
* 디버깅이 어려움

같은 동작을 람다로 표현하면:

```cpp
auto f = [&obj](int x) {
    obj.Func(x);
};
```

→ 캡처, 인자 흐름, 호출이 모두 드러난다

따라서 modern C++에서는:

> **std::bind보다 람다가 기본 선택이다**

---

### 6. 자주 틀리는 포인트

* 모든 콜백을 `std::function`으로 감싼다
  → 불필요한 간접 호출 + 성능 손실

* hot path에서 `std::function`을 사용한다
  → 인라인 불가 + 비용 누적

* `std::bind`를 가독성 없이 사용한다
  → 코드 흐름이 숨겨짐

* callable 저장이 필요 없는 상황에서도 사용한다
  → 과한 추상화

---

### 7. 언제 쓰는가 / 언제 쓰지 않는가

#### std::function을 사용하는 경우

* 다양한 callable을 하나로 저장해야 할 때
* 인터페이스 단순화가 중요한 API 경계
* 호출 빈도가 낮은 경우

---

#### 사용하지 않는 것이 좋은 경우

* hot path (매 프레임, 루프 내부)
* 인라인이 중요한 계산 코드
* 단순 함수 호출이면 충분한 경우

---

#### std::bind 대신

* 거의 항상 람다 사용

---

### 8. 대안 비교

| 방법               | 특징             |
| ---------------- | -------------- |
| lambda           | 가장 직관적, 인라인 가능 |
| std::function    | 타입 통일, 비용 존재   |
| function pointer | 가장 가볍지만 상태 없음  |
| template         | 최고 성능, 타입 고정   |

---

### 9. 코드 예시

#### bad: bind 사용

```cpp
auto f = std::bind(&Player::SetHP, &player, std::placeholders::_1);
```

→ 인자 흐름이 숨겨짐

---

#### good: lambda

```cpp
auto f = [&player](int hp) {
    player.SetHP(hp);
};
```

---

#### bad: hot path + std::function

```cpp
std::function<float(float, float)> dist;
```

---

#### good: template

```cpp
template<typename Func>
void update(Func f) {
    f(1.0f, 2.0f);
}
```

---

### 10. 판단 기준 요약

* callable을 저장해야 하는가? → std::function
* 단순 전달/호출인가? → lambda 또는 함수 포인터
* 성능이 중요한가? → template 또는 구체 타입
* bind를 쓰려는가? → 대부분 lambda로 대체 가능

---

### 11. 한 줄 정리

> std::function은 “유연성”,
> lambda는 “기본 선택”,
> bind는 “거의 필요 없다”

---

## 토픽 19: 템플릿

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 10

### 1. 문제 상황

핵심 용어: `SFINAE`, `concepts`, `컴파일`, `실패`.

같은 알고리즘을 여러 타입에 반복 적용해야 할 때 매번 함수를 복사하거나 `void*`와 매크로로 우회하면 타입 안전성과 가독성이 빠르게 무너진다. 반대로 미래 확장을 핑계로 너무 복잡한 템플릿 계층을 만들면 컴파일 시간과 에러 메시지가 감당하기 어려워진다.

실무에서 템플릿의 어려움은 문법보다 경계 설정에 있다. 무엇을 일반화할지, 어떤 타입만 허용할지, 분기를 런타임에 둘지 컴파일 타임에 둘지를 결정하지 못하면 템플릿은 강력한 도구가 아니라 유지보수 비용이 큰 추상화가 된다.

### 2. 핵심 개념

템플릿은 타입이나 값을 파라미터로 받아 컴파일 시점에 구체 코드를 생성하는 메커니즘이다. 함수 템플릿, 클래스 템플릿, 특수화, `if constexpr`, SFINAE, concepts까지 모두 '일반 규칙을 컴파일 타임에 구체화한다'는 같은 축 위에 있다.

핵심은 하나의 알고리즘을 여러 타입에 재사용하면서도, 필요한 제약과 분기를 타입 수준에서 표현한다는 점이다. 그래서 템플릿은 단순 편의 문법이 아니라 zero-cost abstraction을 구현하는 핵심 수단이 된다.

### 3. 내부 동작 / 메커니즘

#### 인스턴스화와 치환

컴파일러는 템플릿을 보고 바로 기계어를 만드는 것이 아니라, 실제 사용된 타입 조합에 맞춰 인스턴스화를 수행한다. 그래서 템플릿은 일종의 코드 생성기처럼 동작한다. 같은 템플릿이라도 `int`, `double`, 사용자 정의 타입마다 서로 다른 구체 코드가 만들어질 수 있다.

#### 분기와 제약을 다루는 방식

템플릿에서 중요한 것은 '어떤 타입이 들어와도 일단 컴파일해 본다'가 아니라, 유효한 후보만 남기고 나머지를 어떻게 배제하느냐이다. SFINAE는 치환 실패를 전체 에러가 아니라 후보 제외로 처리하는 오래된 규칙이고, `if constexpr`는 타입에 따라 필요 없는 분기를 인스턴스화하지 않도록 해 준다.

concepts는 한 단계 더 나아가 템플릿이 요구하는 조건을 선언부에 직접 적게 한다. 덕분에 제약이 문서처럼 보이고, 에러도 깊은 구현 내부가 아니라 인터페이스 수준에서 더 명확하게 드러난다.

### 4. 자주 틀리는 포인트

- 미래 확장을 핑계로 지원 타입이 몇 개 안 되는 로직까지 과도하게 일반화한다.
- 제약 없는 템플릿을 만든 뒤 난해한 에러 메시지를 감수한다. 지원 조건이 명확하면 concepts나 다른 제약 수단을 먼저 검토해야 한다.
- 타입에 따라 달라지는 분기를 일반 `if`로 처리하려 한다. 템플릿 문맥에서는 `if constexpr`가 필요한 경우가 많다.
- 런타임 다형성이 필요한 문제도 템플릿으로만 해결하려 든다. 타입 결정 시점이 다르면 도구도 달라져야 한다.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

#### 써야 하는 경우

- 같은 알고리즘을 여러 타입에 zero-cost로 재사용해야 할 때
- 컴파일 타임 제약과 분기가 성능이나 안정성에 직접 이점을 줄 때
- 호출 대상 타입이 런타임이 아니라 컴파일 타임에 이미 정해질 때

#### 안 쓰는 편이 나은 경우

- 단순한 로직인데 미래 확장 가능성만을 이유로 과도하게 일반화할 때
- 디버깅이 더 중요하고, 명시적 타입과 단순한 호출 구조가 필요한 코드
- 실제 다형성이 런타임에 결정되어 템플릿만으로는 문제를 풀 수 없을 때

### 6. 대안 비교

- template: 컴파일 타임 다형성으로 가장 빠르고 유연하지만, 컴파일 비용과 code bloat를 감수한다.
- runtime polymorphism: 타입이 런타임에 바뀌는 문제에 적합하고 인터페이스가 명확하지만 간접 호출 비용이 있다.
- overload: 경우의 수가 적고 지원 타입이 제한적일 때 가장 읽기 쉽고 단순하다.

타입이 컴파일 타임에 정해지고 성능이 중요하면 템플릿이 유리하다. 반대로 교체 가능한 객체 집합을 런타임에 다뤄야 하면 virtual 기반 다형성이 더 자연스럽고, 지원 타입이 몇 개 안 되면 오버로드가 가장 단순하다.

### 7. 코드 예시 (bad → good)

#### 사례 1: 타입 의존 분기를 일반 `if`로 처리한 경우

Bad: 일반 `if`는 두 분기 모두 템플릿 인스턴스화 대상이 된다. 특정 타입에서는 실제로 실행하지 않을 분기 때문에도 컴파일 에러가 날 수 있다.

```cpp
template<typename T>
void Print(const T& value) {
    if (std::is_integral_v<T>) {
        std::cout << value + 1;
    } else {
        std::cout << value.size();
    }
}
```

Good: `if constexpr`는 참이 아닌 분기를 인스턴스화하지 않는다. 타입마다 유효한 코드만 남기고 싶을 때 필요한 이유가 여기에 있다.

```cpp
template<typename T>
void Print(const T& value) {
    if constexpr (std::is_integral_v<T>) {
        std::cout << value + 1;
    } else {
        std::cout << value.size();
    }
}
```

#### 사례 2: 제약 없는 템플릿으로 에러를 늦게 터뜨린 경우

Bad: 템플릿 내부 깊숙한 곳에서 에러가 터져 메시지가 길고 난해해지기 쉽다. 호출자는 '왜 안 되는지'보다 구현 내부를 먼저 보게 된다.

```cpp
template<typename T>
void process(const T& x) {
    x.serialize();
}

struct NoSerialize {};
process(NoSerialize{});
```

Good: concepts를 쓰면 요구 조건이 선언부에서 바로 보인다. 템플릿이 어떤 타입을 받는지 문서처럼 드러나고, 에러도 인터페이스 수준에서 더 명확해진다.

```cpp
template<typename T>
concept Serializable = requires(T x) { x.serialize(); };

template<Serializable T>
void process(const T& x) {
    x.serialize();
}
```

### 8. 판단 기준 요약

- 타입 결정이 컴파일 타임이라면 템플릿을 우선 검토한다.
- 지원 조건이 명확하다면 제약 없는 템플릿보다 concepts나 다른 제약 수단을 먼저 둔다.
- 타입 의존 분기는 일반 `if`보다 `if constexpr`가 맞는지 먼저 확인한다.
- 성능 이득만 보지 말고 컴파일 시간, 에러 메시지, code bloat 비용까지 함께 본다.
- 지원 타입이 몇 개 안 되면 오버로드가 더 단순한지 먼저 따진다.

## 토픽 20: STL 컨테이너 & 알고리즘(에스티엘, 표준 라이브러리 자료구조와 알고리즘)

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 19

### 1. 문제 상황

핵심 용어: `iterator`, `iterator invalidation`, `연속`, `erase`.

자료구조 선택은 성능과 안정성을 동시에 결정한다. 그런데 실제 코드에서는 빅오 표만 보고 `list`를 고르거나, 습관적으로 `map`과 `unordered_map`을 섞어 쓰고, `vector` 재할당 뒤에도 예전 주소를 계속 들고 있는 식의 실수가 자주 나온다.

STL에서 중요한 것은 컨테이너 이름보다 접근 패턴, 수정 패턴, invalidation 규칙이다. 같은 O 표기라도 연속 메모리와 캐시 locality, 해시 비용, 노드 할당 비용 때문에 실제 성능은 크게 달라진다.

### 2. 핵심 개념

STL의 핵심은 컨테이너와 알고리즘을 iterator로 연결하는 설계다. `vector`, `map`, `unordered_map`, `set` 같은 컨테이너는 데이터 배치와 조회 특성이 다르고, `sort`, `find_if`, `remove_if` 같은 알고리즘은 컨테이너 자체보다 iterator 범위를 기준으로 동작한다.

실무에서 먼저 봐야 하는 질문은 '이 연산이 가능한가'가 아니라 '이 접근 패턴에서 실제로 빠르고 안전한가'다. 그래서 STL은 문법 암기보다 선택 기준이 중요하다.

### 3. 내부 동작 / 메커니즘

#### 메모리 배치와 실제 성능

`vector`는 연속 메모리에 데이터를 배치하므로 순회와 임의 접근에서 캐시 효율이 높다. 반면 `list`는 노드 기반이라 삽입과 삭제 자체는 싸 보여도, 실제 순회에서는 pointer chasing 때문에 CPU 친화성이 크게 떨어진다. 즉 실전 성능은 단순 빅오보다 메모리 배치에 더 크게 좌우된다.

#### 무효화와 알고리즘 조합

`vector`는 재할당이 일어나면 iterator, 포인터, 참조가 무효화될 수 있다. `unordered_map`도 재해시가 일어나면 내부 배치가 바뀐다. 이런 규칙을 모르면 컨테이너 수정 후 예전 핸들을 계속 사용하는 버그가 생긴다.

또한 많은 알고리즘은 컨테이너 크기를 직접 바꾸지 않는다. 대표적으로 `remove_if`는 남길 원소를 앞으로 모으고 새 논리적 끝 iterator를 돌려줄 뿐, 실제 삭제는 `erase`가 담당한다. STL은 알고리즘과 컨테이너 동작을 분리해 두었기 때문에 이 경계를 함께 이해해야 한다.

### 4. 자주 틀리는 포인트

- `vector`가 재할당될 수 있다는 사실을 잊고 원소 주소나 iterator를 오래 들고 있는다.
- `remove_if`만 호출하고 실제 `erase`를 하지 않아 삭제가 끝났다고 착각한다.
- 중간 삽입/삭제가 많다는 이유만으로 `list`를 자동 선택한다. 실제 성능은 순회 비용까지 같이 봐야 한다.
- 순서가 필요한데도 습관적으로 `unordered_map`을 고르거나, 해시 비용을 무시한다.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

#### 기본 선택

- 연속 메모리 순회와 캐시 효율이 중요하면 `vector`를 기본값으로 둔다.
- 키 조회가 핵심이고 순서가 필요 없으면 `unordered_map`을 우선 검토한다.
- 정렬 순서 유지, `lower_bound` 계열 질의, 안정적인 트리 성질이 필요하면 `map`이나 `set`을 쓴다.

#### 안 쓰는 편이 나은 경우

- 중간 삽입/삭제가 많다는 이유만으로 `list`를 자동 선택하지 않는다.
- 순서 요구가 있는데도 습관적으로 `unordered_map`을 고르지 않는다.
- 무효화 규칙을 감당하지 못하면서 `vector` 내부 원소 주소를 오래 보관하지 않는다.

### 6. 대안 비교

#### 시퀀스 컨테이너

- `vector`: cache locality가 가장 좋아 기본 선택으로 적합하다.
- `deque`: 앞뒤 삽입은 유연하지만 메모리가 연속적이지 않아 `vector`만큼 단순하지 않다.
- `list`: iterator 안정성은 좋지만 순회와 메모리 효율이 나빠 기본 선택이 되기 어렵다.

#### 연관 컨테이너

- `unordered_map`: 평균 조회 성능이 좋지만 해시 비용과 순서 부재를 감수한다.
- `map`: 정렬과 범위 질의가 필요할 때 유리하지만, 단순 조회만 보면 더 비쌀 수 있다.

특별한 이유가 없으면 `vector`부터 검토하는 경우가 많다. 이름만 보면 삽입과 삭제에 강해 보이는 `list`도, 실제 CPU 친화성은 대개 훨씬 나쁘다.

### 7. 코드 예시 (bad → good)

#### 사례 1: `vector` 재할당 뒤 예전 포인터를 사용한 경우

Bad: `push_back` 과정에서 재할당이 일어나면 원소 위치가 바뀔 수 있다. 그 순간 `p`는 dangling pointer가 된다.

```cpp
std::vector<int> v = {1, 2, 3};
int* p = &v[0];
v.push_back(4);
std::cout << *p;
```

Good: 필요한 용량을 미리 확보하면 재할당 가능성을 줄일 수 있다. 더 안전한 기본 원칙은 컨테이너 수정 뒤 예전 포인터나 iterator를 다시 얻는 것이다.

```cpp
std::vector<int> v;
v.reserve(4);
v.push_back(1);
v.push_back(2);
v.push_back(3);
int* p = &v[0];
v.push_back(4);
std::cout << *p;
```

#### 사례 2: `remove_if`만 호출하고 삭제가 끝났다고 생각한 경우

Bad: 원소를 앞으로 당겨 놓기만 하고 컨테이너 크기는 줄이지 않는다. 삭제가 끝난 것처럼 보여도 실제 `size()`는 그대로일 수 있다.

```cpp
std::vector<int> values = {1, 2, 3, 4, 5};
std::remove_if(values.begin(), values.end(), [](int v) { return v % 2 == 0; });
```

Good: erase-remove idiom은 STL에서 가장 자주 나오는 패턴 중 하나다. 알고리즘이 논리적 끝을 계산하고, 컨테이너가 그 뒤 구간을 실제로 지운다고 이해하면 헷갈리지 않는다.

```cpp
std::vector<int> values = {1, 2, 3, 4, 5};
values.erase(
    std::remove_if(values.begin(), values.end(), [](int v) { return v % 2 == 0; }),
    values.end()
);
```

### 8. 판단 기준 요약

- 컨테이너 선택의 첫 기준은 빅오보다 접근 패턴과 수정 패턴이다.
- 특별한 이유가 없으면 `vector`를 기본 선택으로 검토한다.
- 순서와 범위 질의가 필요하면 `map`, 평균 빠른 조회가 중요하면 `unordered_map`을 본다.
- 컨테이너 수정 뒤에는 iterator, 포인터, 참조 무효화를 먼저 의심한다.
- 알고리즘이 실제 크기 변경까지 수행하는지 항상 확인한다.

## 토픽 21: 멀티스레딩 기초

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 5

### 1. 문제 상황

핵심 용어: `race condition`, `predicate`, `mutex`, `RAII`.

로딩, 백그라운드 계산, 리소스 준비처럼 병렬화할 수 있는 일은 많다. 하지만 공유 상태를 여러 스레드가 동시에 만지는 순간, 성능 이득보다 race condition과 deadlock 같은 정합성 문제가 먼저 등장한다.

멀티스레딩 버그가 위험한 이유는 재현성이 낮기 때문이다. 같은 코드도 스케줄링 순서에 따라 정상처럼 보였다가 특정 시점에만 깨질 수 있다. 그래서 핵심은 '스레드를 많이 쓰는 법'이 아니라 '공유 상태를 어떻게 줄이고, 남은 공유 상태를 어떤 규칙으로 보호할 것인가'다.

### 2. 핵심 개념

멀티스레딩은 여러 실행 흐름이 동시에 상태를 읽고 쓰는 환경이다. 기본 도구로는 `std::thread`, `std::mutex`, `std::lock_guard`, `std::atomic`, `std::condition_variable`이 있고, 핵심 위험은 race condition과 deadlock이다.

중요한 점은 atomic 하나로 모든 동기화가 해결되지 않는다는 것이다. 원자성은 단일 연산의 찢어짐을 막아 줄 뿐이며, 여러 변수의 일관성이나 더 큰 불변식은 별도의 동기화 전략이 필요하다. `std::memory_order`는 이런 원자 연산의 가시성과 순서를 조절하는 규칙이다.

### 3. 내부 동작 / 메커니즘

#### race condition과 mutex의 역할

공유 데이터에 두 스레드가 동시에 접근하고, 그중 하나라도 쓰기라면 race condition 가능성이 생긴다. `std::mutex`는 임계 구역을 한 번에 한 스레드만 통과시키고, `std::lock_guard`는 RAII로 lock과 unlock을 스코프에 묶어 예외가 나도 해제가 보장되게 만든다.

#### atomic과 `memory_order`의 경계

`std::atomic`은 단일 변수의 읽기, 쓰기, CAS를 원자적으로 수행한다. 하지만 두 개 이상의 변수 사이 관계까지 자동으로 맞춰 주지는 않는다. 그래서 복합 상태 보호에는 여전히 mutex가 더 자연스럽다.

기본 `memory_order_seq_cst`는 가장 직관적인 순서를 제공하고, `acquire/release`는 생산자와 소비자 패턴처럼 더 느슨하지만 자주 쓰이는 규칙이다. 또한 `condition_variable`은 깨어났다고 바로 조건이 참이라고 가정하면 안 되므로, predicate를 다시 확인하는 표준 패턴이 필요하다.

### 4. 자주 틀리는 포인트

- atomic 하나로 복합 상태 일관성까지 해결된다고 생각한다. atomic은 단일 변수 단위에서만 강하다.
- 여러 락을 제각각 순서로 획득한다. 락 순서가 흔들리면 deadlock 가능성이 생긴다.
- `volatile`이 스레드 동기화에 도움이 된다고 오해한다. C++에서 동기화 의미는 mutex나 atomic이 담당한다.
- 작업 단위가 너무 작은데도 스레드를 늘린다. context switch와 contention이 병렬화 이득을 지워 버릴 수 있다.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

#### 써야 하는 경우

- 작업이 충분히 독립적이고 병렬 실행으로 실제 병목이 줄어드는 경우
- I/O 대기나 긴 계산을 메인 경로에서 분리해야 하는 경우
- 공유 상태를 줄이거나 소유권을 명확히 나눌 수 있는 경우

#### 안 쓰는 편이 나은 경우

- 작업 단위가 너무 작아 context switch와 동기화 비용이 더 큰 경우
- 공유 상태가 너무 많아 락과 경쟁이 병목이 되는 경우
- 단일 스레드 구조 개선이나 작업 분해만으로 문제를 해결할 수 있는 경우

### 6. 대안 비교

- mutex: 여러 연산을 하나의 임계 구역으로 묶어 복합 상태를 안전하게 보호할 수 있다.
- atomic: 단일 변수의 원자적 연산에 적합하지만, 복합 불변식 보호에는 부족하다.
- lock-free: contention을 줄일 잠재력은 있지만 설계와 검증, 디버깅 난도가 매우 높다.
- 단일 스레드나 작업 분해: 가장 단순하고 디버깅이 쉬우며, 병목이 명확하지 않다면 오히려 더 좋은 선택이다.

실무에서는 먼저 mutex 기반의 단순한 동기화로 문제를 푼다. atomic은 정말 단일 상태에만, lock-free는 contention이 확실한 병목이고 팀이 그 복잡도를 감당할 수 있을 때만 검토하는 편이 현실적이다.

### 7. 코드 예시 (bad → good)

#### 사례 1: 공유 카운터를 동기화 없이 증가시킨 경우

Bad: `++counter`는 읽기, 수정, 쓰기가 합쳐진 복합 연산이다. 동기화 없이 두 스레드가 동시에 실행하면 race condition이 발생한다.

```cpp
int counter = 0;
std::thread t1([&]() { for (int i = 0; i < 1000; ++i) ++counter; });
std::thread t2([&]() { for (int i = 0; i < 1000; ++i) ++counter; });
t1.join();
t2.join();
```

Good: 단일 카운터처럼 한 변수의 원자 연산만 필요하다면 atomic이 잘 맞는다. mutex보다 단순하고 의도도 분명하다.

```cpp
std::atomic<int> counter = 0;
std::thread t1([&]() { for (int i = 0; i < 1000; ++i) ++counter; });
std::thread t2([&]() { for (int i = 0; i < 1000; ++i) ++counter; });
t1.join();
t2.join();
```

#### 사례 2: atomic으로 복합 상태를 보호하려 한 경우

Bad: 각 연산은 원자적이어도 두 변수 사이의 관계는 보호되지 않는다. '계좌가 활성 상태이고 잔액이 충분해야 한다'는 복합 불변식은 여전히 깨질 수 있다.

```cpp
std::atomic<int> balance{100};
std::atomic<bool> isActive{true};

void withdraw(int amount) {
    if (isActive.load() && balance.load() >= amount) {
        balance -= amount;
    }
}
```

Good: 복합 상태를 한 번에 검사하고 수정해야 할 때는 mutex가 더 자연스럽다. 보호해야 하는 불변식의 범위가 바로 도구 선택 기준이 된다.

```cpp
std::mutex m;
int balance = 100;
bool isActive = true;

void withdraw(int amount) {
    std::lock_guard<std::mutex> lock(m);
    if (isActive && balance >= amount) {
        balance -= amount;
    }
}
```

### 8. 판단 기준 요약

- 멀티스레딩의 첫 규칙은 공유 상태를 줄이고 소유권 경계를 명확히 하는 것이다.
- 단일 변수 원자 연산이면 atomic, 여러 변수와 여러 연산의 일관성이면 mutex를 먼저 본다.
- 락이 둘 이상이면 획득 순서를 전역 규칙으로 고정한다.
- 기본 `seq_cst`가 필요한지, 생산자-소비자 패턴에 `acquire/release`가 맞는지 구분한다.
- 병렬화는 성능 도구이지만, 정합성 비용이 더 큰 경우에는 단일 스레드 구조가 더 좋은 해법일 수 있다.
