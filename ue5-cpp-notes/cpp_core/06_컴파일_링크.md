## 토픽 23: 컴파일 과정

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 없음

### 1. 핵심 개념

C++ 코드는 한 번에 기계어가 되는 것이 아니라 **전처리 → 컴파일 → 어셈블 → 링크** 단계를 거친다. 전처리는 `#include`, `#define` 같은 지시문을 펼치고, 컴파일은 각 `translation unit(트랜슬레이션 유닛, 번역 단위)`를 검사해 중간 산출물을 만들고, 링크는 그 조각들을 묶어 실행 파일을 만든다.

이 과정을 이해하면 include guard가 왜 필요한지, forward declaration이 언제 가능한지, ODR(One Definition Rule) 위반이 왜 링크 에러로 이어지는지 자연스럽게 설명할 수 있다. 여기에 `template instantiation(템플릿 인스턴시에이션, 템플릿 인스턴스화)`, `header-only(헤더 온리, 헤더만으로 구현된 형태)` 라이브러리, `PCH(피씨에이치, 미리 컴파일된 헤더)`, `unity build(유니티 빌드, 여러 cpp를 묶어 한 번에 컴파일)` 개념도 결국 번역 단위와 빌드 비용 이해 위에서 연결된다.

**쉽게 감 잡기**
- 전처리: 텍스트 붙이기와 치환
- 컴파일: 각 소스 파일을 개별 번역 단위로 검사하고 변환
- 링크: 조각난 심볼을 실제 정의와 연결

**실무 기준**
- 헤더에는 선언 위주, 소스에는 정의 위주로 둔다
- 템플릿은 정의가 보이는 곳에 있어야 인스턴스화가 가능하다
- 공통 include가 지나치게 무거우면 PCH와 include 정리가 빌드 시간을 크게 좌우한다
- unity build는 빌드를 빠르게 만들 수 있지만, 숨어 있던 include 누락을 가릴 수 있어 독립 번역 단위 기준 점검도 필요하다

### 2. 코드 예시

```cpp
// Player.h
#pragma once

class Weapon; // ← forward declaration

class Player {
public:
    void SetWeapon(Weapon* InWeapon);

private:
    Weapon* CurrentWeapon = nullptr;
};
```

```cpp
// Player.cpp
#include "Player.h"
#include "Weapon.h" // ← 실제 정의가 필요한 cpp에서 include

void Player::SetWeapon(Weapon* InWeapon) {
    CurrentWeapon = InWeapon;
}
```

```cpp
// ❌ 잘못된 예: 헤더에 일반 함수 정의를 넣어 ODR 위반 유발
// Utils.h
int Add(int a, int b) {
    return a + b;
}
```

### 3. 왜 중요한가 (게임/UE5 맥락)

UE5 프로젝트는 모듈이 많고 빌드 시간이 길며, UHT/UBT가 추가로 개입한다. 그래서 번역 단위, 심볼, include 구조를 이해하지 못하면 "왜 소스 하나 바꿨는데 전체가 다시 빌드되지?", "왜 선언은 있는데 링크가 깨지지?" 같은 문제를 빠르게 풀 수 없다.

### 4. 면접 예상 Q&A

**Q1: C++ 컴파일 과정을 순서대로 설명해 주세요.**
> A: 전처리 단계에서 include와 매크로가 펼쳐지고, 컴파일 단계에서 각 번역 단위가 검사되고 중간 산출물로 바뀐다. 이후 어셈블을 거쳐 목적 파일이 만들어지고, 링크 단계에서 심볼이 연결돼 최종 실행 파일이 생성된다.

**Q2: forward declaration은 언제 쓸 수 있나요?**
> A: 타입의 존재만 알면 되는 상황, 예를 들어 포인터나 참조 멤버, 함수 선언 정도에서는 쓸 수 있다. 반대로 값 멤버를 직접 들거나 `sizeof`, 멤버 접근, 상속처럼 완전한 정의가 필요하면 불가능하다.

**Q3: ODR이란 무엇인가요?**
> A: One Definition Rule로, 같은 엔티티가 프로그램 전체에서 허용되는 방식으로 단 한 번 정의돼야 한다는 규칙이다. 헤더에 일반 함수 정의를 넣고 여러 번역 단위가 이를 포함하면 대표적인 ODR 문제가 발생한다.

**Q4: 컴파일 에러와 링크 에러의 차이는 무엇인가요?**
> A: 컴파일 에러는 개별 번역 단위 안에서 문법이나 타입 검사가 실패하는 것이고, 링크 에러는 번역은 되었지만 정의를 찾지 못하거나 중복 정의가 발생한 경우다. 즉 하나는 "이 파일을 이해 못함", 다른 하나는 "조각을 연결 못함"에 가깝다.

**Q5: template instantiation은 왜 빌드 구조와 연결되나요?**
> A: 템플릿은 실제 사용된 타입 조합에 대해 컴파일 시점에 구체 코드가 생성되기 때문이다. 그래서 보통 템플릿 정의는 헤더에 둬야 하며, 이 때문에 header-only 라이브러리가 자주 등장한다.

**Q6: PCH는 무엇이고 왜 빌드 시간을 줄이나요?**
> A: 자주 포함되는 무거운 헤더 묶음을 미리 컴파일해 재사용하는 장치다. 파싱 비용을 반복하지 않아도 되므로 대형 프로젝트에서 체감 효과가 크다. 다만 PCH가 과도하게 비대해지면 의존성 파악이 흐려질 수 있다.

**Q7: 빌드 시간을 줄이는 방법을 설명해 보세요.**
> A: 헤더 include dependency를 줄이고, forward declaration이 가능한 곳은 선언으로 대체하고, 무거운 공용 헤더를 PCH로 정리하는 것이 기본이다. 필요하면 unity build로 번역 단위 수를 줄일 수 있지만, 이것이 설계 문제를 숨기지 않는지도 함께 봐야 한다. 즉 빌드 캐시와 구조 개선을 같이 봐야 한다.

### 5. 흔한 실수

- **헤더에 일반 함수 정의를 무심코 넣음**: `inline`이나 `constexpr` 조건 없이 여러 번역 단위에 정의가 복제되면 링크 단계에서 문제가 난다.
- **forward declaration이 가능한데도 무조건 include**: 헤더 결합도가 커져 빌드 시간이 늘고 순환 include 위험도 커진다.
- **템플릿 정의를 소스 파일에만 둠**: 사용하는 번역 단위에서 정의가 보이지 않아 인스턴스화가 실패하거나 링크 문제가 발생할 수 있다.
- **unity build가 켜져 있으니 include 정리를 안 해도 된다고 생각함**: 유니티 환경에서 우연히 보이던 선언이 비유니티 빌드에서는 바로 깨질 수 있다.

## 토픽 24: 정적 / 동적 라이브러리

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 23

### 1. 핵심 개념

라이브러리는 재사용 가능한 코드 묶음이다. **정적 라이브러리(static library, 스태틱 라이브러리)** 는 링크 시점에 실행 파일 안으로 복사되고, **동적 라이브러리(dynamic library / DLL / shared library, 다이내믹 라이브러리)** 는 실행 시점에 별도로 로드된다. 차이는 코드가 포함되는 시점, 배포 방식, 메모리 공유, ABI 안정성에서 드러난다.

정적 링크는 배포가 단순하고 self-contained 바이너리를 만들기 쉽지만 실행 파일이 커질 수 있다. 동적 링크는 모듈 교체와 메모리 공유에 유리하지만, 로딩 실패와 ABI 호환성 문제를 더 엄격하게 관리해야 한다. `symbol visibility(심볼 비저빌리티, 외부 노출 심볼 범위)`와 `name mangling(네임 맹글링, 함수/타입 이름 장식)`도 동적 라이브러리 경계에서 자주 이야기된다.

### 2. 코드 예시

```cpp
// MathLib.h
#pragma once

int Add(int a, int b);
```

```cpp
// MathLib.cpp
#include "MathLib.h"

int Add(int a, int b) {
    return a + b;
}
```

```cpp
// App.cpp
#include <iostream>
#include "MathLib.h"

int main() {
    std::cout << Add(2, 3) << "\n";
}
```

설명:
- 정적 라이브러리라면 `Add` 구현이 링크 단계에서 실행 파일에 포함된다.
- 동적 라이브러리라면 실행 시 별도 모듈에서 심볼을 찾아 연결한다.

### 3. 왜 중요한가 (게임/UE5 맥락)

UE5는 모듈과 플러그인 구조를 적극 사용하고, 서드파티 SDK도 자주 연결한다. 그래서 "무엇을 정적으로 묶고, 무엇을 동적으로 배포할지", "DLL 경계에서 어떤 타입을 노출할지"를 설명할 수 있어야 실무 대응력이 높다.

### 4. 면접 예상 Q&A

**Q1: 정적 라이브러리와 동적 라이브러리의 차이는 무엇인가요?**
> A: 정적 라이브러리는 링크 시 실행 파일에 포함되고, 동적 라이브러리는 실행 시 별도로 로드된다. 정적은 배포가 단순하고, 동적은 교체와 공유가 유연하다는 차이가 있다.

**Q2: 동적 라이브러리의 단점은 무엇인가요?**
> A: 실행 환경에 라이브러리가 없거나 버전이 다르면 로딩 실패가 발생할 수 있다. 또한 ABI가 바뀌면 컴파일은 통과해도 런타임에서 문제가 터질 수 있다.

**Q3: ABI 호환성은 왜 중요한가요?**
> A: DLL 경계에서는 함수 호출 규약, 이름 맹글링, 클래스 레이아웃, STL 구현 차이까지 맞아야 하기 때문이다. 헤더가 같아 보여도 ABI가 다르면 런타임 충돌이 난다.

**Q4: 게임에서 정적/동적 선택 기준은 무엇인가요?**
> A: 배포 단순성, 핫스왑 필요, 플러그인 구조, 메모리 공유 여부를 같이 본다. 독립 실행성과 단순 배포가 중요하면 정적, 별도 모듈 교체와 분리가 중요하면 동적이 더 적합하다.

**Q5: name mangling은 왜 라이브러리 경계에서 자주 언급되나요?**
> A: C++ 함수 이름은 오버로드와 시그니처 정보를 담기 위해 컴파일러가 내부적으로 변형하기 때문이다. 서로 다른 컴파일러나 설정이 섞이면 심볼 이름이 달라져 링크/호출 문제가 생길 수 있다.

**Q6: header-only 라이브러리는 정적/동적과 어떻게 다른가요?**
> A: header-only는 배포 형태가 아니라 구현 배치 방식에 가깝다. 코드가 헤더에 있으므로 사용하는 번역 단위마다 인스턴스화/컴파일되며, 별도 라이브러리 파일 없이도 동작할 수 있다.

### 5. 흔한 실수

- **동적 라이브러리 배포 파일 누락**: 개발 환경에서는 되는데 다른 PC에서 로딩 실패가 나는 대표적인 원인이다.
- **DLL 경계에 STL 타입을 무심코 노출**: 컴파일러/런타임 차이로 ABI 문제가 생길 수 있다.
- **정적 링크면 무조건 더 빠르다고 단정**: 링크 형태보다 병목이 실제 어디 있는지 먼저 봐야 한다.

## 토픽 25: 메모리 단편화 & 커스텀 할당

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 3, 6

### 1. 핵심 개념

메모리 단편화(fragmentation)는 총 메모리는 충분한데도 필요한 형태의 연속 공간을 확보하기 어려워지는 현상이다. **내부 단편화**는 할당받은 블록 내부에 남는 낭비이고, **외부 단편화**는 빈 공간은 많지만 조각나 있어 큰 연속 블록을 못 만드는 상황이다.

게임은 프레임마다 반복적인 할당/해제가 일어나므로 범용 할당자만으로는 지터와 단편화가 커질 수 있다. 그래서 메모리 풀(pool), free list, buddy system, frame allocator, arena allocator 같은 커스텀 할당 전략을 사용한다. 목표는 "평균 속도"보다 **예측 가능한 최악 시간과 안정된 메모리 패턴**이다.

### 2. 코드 예시

```cpp
#include <cstddef>
#include <iostream>
#include <vector>

struct Block {
    Block* next;
};

class SimplePool {
public:
    explicit SimplePool(size_t blockCount, size_t blockSize)
        : memory(blockCount * blockSize)
        , freeList(nullptr)
        , blockSize(blockSize) {
        for (size_t i = 0; i < blockCount; ++i) {
            auto* block = reinterpret_cast<Block*>(&memory[i * blockSize]);
            block->next = freeList;
            freeList = block;
        }
    }

    void* allocate() {
        if (!freeList) {
            return nullptr;
        }
        Block* block = freeList;
        freeList = freeList->next;
        return block;
    }

    void deallocate(void* ptr) {
        auto* block = static_cast<Block*>(ptr);
        block->next = freeList;
        freeList = block;
    }

private:
    std::vector<std::byte> memory;
    Block* freeList;
    size_t blockSize;
};

int main() {
    SimplePool pool(4, 64);

    void* a = pool.allocate();
    void* b = pool.allocate();

    std::cout << "allocated: " << a << ", " << b << "\n";

    pool.deallocate(a);
    pool.deallocate(b);
}
```

### 3. 왜 중요한가 (게임/UE5 맥락)

게임에서는 평균 시간보다 최악 시간 지연이 더 중요하다. 단편화와 범용 할당자 지터는 프레임 드랍과 히치의 원인이 되며, 작은 객체가 대량으로 생성/파괴되는 총알, 파티클, 임시 버퍼에서 특히 눈에 띈다.

### 4. 면접 예상 Q&A

**Q1: 내부 단편화와 외부 단편화의 차이는 무엇인가요?**
> A: 내부 단편화는 할당받은 블록 안에서 사용하지 못하는 낭비가 생기는 것이고, 외부 단편화는 빈 공간은 많지만 조각나 있어 연속 블록을 못 구하는 상황이다. 즉 하나는 블록 내부 낭비, 다른 하나는 공간 배치 문제다.

**Q2: 메모리 풀(pooling)은 어떤 상황에 적합하고 왜 필요한가요?**
> A: 같은 크기의 객체를 매우 자주 생성/해제할 때 적합하다. 블록 크기를 고정해 두면 할당과 해제가 빠르고 외부 단편화도 줄이기 쉽다. 결국 pooling은 평균 성능보다 지터 억제와 메모리 패턴 안정화에 강점이 있다.

**Q3: buddy system은 무엇인가요?**
> A: 메모리를 2의 거듭제곱 크기 블록으로 나누고 필요에 따라 분할/병합하는 방식이다. 병합 규칙이 단순해 관리가 쉽지만, 요청 크기와 블록 크기가 안 맞으면 내부 단편화가 생길 수 있다.

**Q4: 프레임 할당자는 왜 게임에서 매력적인가요?**
> A: 한 프레임 동안만 쓰는 임시 메모리를 매우 빠르게 할당하고, 프레임 끝에 한 번에 초기화할 수 있기 때문이다. 개별 free 비용이 거의 없어 scratch buffer, 렌더링 임시 데이터에 잘 맞는다.

**Q5: 커스텀 할당자를 쓰면 항상 성능이 좋아지나요?**
> A: 아니다. 패턴이 뚜렷할 때만 이득이 크다. 잘못 적용하면 메모리 낭비, 디버깅 난이도, 코드 복잡도만 커질 수 있어 먼저 할당 패턴을 측정해야 한다.

**Q6: 게임에서 범용 할당자 대신 전용 할당 전략을 쓰는 이유를 한 문장으로 말하면?**
> A: 평균 속도보다 예측 가능한 지연 시간과 안정적인 메모리 사용 패턴이 더 중요하기 때문이다. 즉 "빠름"보다 "흔들리지 않음"이 핵심이다.

### 5. 흔한 실수

- **모든 문제를 커스텀 할당자로 해결하려 함**: 병목이 실제로는 참조 구조나 자산 로딩에 있을 수도 있다. 측정 없이 도입하면 복잡도만 는다.
- **객체 수명 패턴을 모른 채 풀을 도입**: 풀은 같은 크기와 비슷한 수명의 반복 객체에 강하다. 아무 객체나 넣으면 오히려 메모리 낭비와 초기화 버그가 늘어난다.
- **단편화를 총 메모리 부족과 같은 개념으로 오해**: 메모리가 남아도 연속 공간이 없어 실패할 수 있다는 점을 구분해야 한다.
