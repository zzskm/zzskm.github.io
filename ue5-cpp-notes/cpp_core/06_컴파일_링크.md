## 토픽 23: 컴파일 과정

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 없음

### 1. 핵심 개념

핵심 용어: `ODR`, `translation unit`.

C++ 코드는 한 번에 기계어가 되는 것이 아니라 **전처리 → 컴파일 → 어셈블 → 링크** 단계를 거친다. 전처리는 `#include`, `#define` 같은 지시문을 펼치고, 컴파일은 각 `translation unit(트랜슬레이션 유닛, 번역 단위)`를 검사해 중간 산출물을 만들며, 링크는 그 조각들을 묶어 최종 실행 파일을 만든다.

이 과정을 단계별로 이해하면 include guard가 왜 필요한지, forward declaration이 언제 가능한지, ODR(One Definition Rule) 위반이 왜 링크 에러로 이어지는지 자연스럽게 설명할 수 있다. `template instantiation(템플릿 인스턴시에이션, 템플릿 인스턴스화)`, `header-only(헤더 온리, 헤더만으로 구현된 형태)` 라이브러리, `PCH(피씨에이치, 미리 컴파일된 헤더)`, `unity build(유니티 빌드, 여러 cpp를 묶어 한 번에 컴파일)` 같은 개념도 결국 번역 단위와 빌드 비용을 이해해야 연결된다.

쉽게 구분하면 전처리는 텍스트 치환과 포함, 컴파일은 각 소스 파일의 문법·타입 검사와 코드 생성, 링크는 흩어진 심볼을 실제 정의와 연결하는 단계다. 빌드 에러를 볼 때도 먼저 어느 단계에서 깨졌는지 나누어 보는 습관이 중요하다.

실무에서는 헤더에는 선언을, 소스에는 정의를 두는 것이 기본이다. 템플릿처럼 정의가 보여야 하는 예외도 있지만, 공통 include가 지나치게 무거우면 PCH와 include 정리가 빌드 시간을 크게 좌우하고, unity build는 빌드를 빠르게 만들면서도 독립 번역 단위 기준의 누락 문제를 가릴 수 있다는 점까지 함께 봐야 한다.

### 2. 코드 예시

```cpp
// Utils.h
int Add(int a, int b) {
    return a + b;
}
```

이 예시는 ODR 위반이 왜 링크 단계에서 드러나는지를 보여 준다. 헤더에 일반 함수 정의를 두면 그 정의가 헤더를 포함한 번역 단위마다 복제되고, 링크 시점에 중복 심볼 문제가 발생한다.

### 3. 왜 중요한가 (게임/UE5 맥락)

UE5 프로젝트는 모듈 수가 많고 빌드 시간이 길며, UHT/UBT가 추가로 개입한다. 그래서 번역 단위와 심볼 연결을 이해하지 못하면 "왜 소스 하나를 고쳤는데 전체가 다시 빌드되는가", "왜 선언은 있는데 링크가 깨지는가" 같은 문제를 체계적으로 설명하기 어렵다. 결국 빌드 문제를 빨리 풀려면 어느 단계에서 깨졌는지부터 나눠서 볼 수 있어야 한다.

### 4. 면접 예상 Q&A

**Q1: 컴파일 과정을 전처리-컴파일-어셈블-링크로 나눠 이해해야 하는 이유는 무엇인가?**
> A: 에러가 어느 단계에서 생겼는지 알아야 include 문제, 문법 문제, 심볼 중복/누락 문제를 정확히 분리할 수 있기 때문이다. 단계 구분이 선명해야 빌드 로그를 봤을 때 어디서부터 의심해야 할지도 정리된다.

**Q2: C++ 소스가 실행 파일이 되기까지의 4단계를 순서대로 설명하라.**
> A: 전처리(매크로 확장·include 삽입) → 컴파일(소스→어셈블리) → 어셈블(어셈블리→오브젝트 파일) → 링크(오브젝트 파일 결합→실행 파일) 순서다.

**Q3: forward declaration은 언제 충분하고, 언제 실제 include가 필요한가요?**
> A: 타입의 존재만 알면 되는 포인터, 참조, 함수 선언 정도에서는 forward declaration으로 충분하다. 반대로 값 멤버를 직접 들거나 `sizeof`, 멤버 접근, 상속처럼 완전한 정의가 필요한 순간에는 실제 헤더를 include해야 한다.

**Q4: 템플릿 정의를 보통 헤더에 두는 이유는 무엇인가요?**
> A: 템플릿은 인스턴스화가 일어나는 번역 단위에서 정의를 볼 수 있어야 하기 때문이다. 선언만 두고 정의를 cpp에 숨기면, 다른 번역 단위에서 필요한 구체 타입 인스턴스를 만들지 못해 링크 문제가 생길 수 있다.

### 5. 흔한 실수

- **헤더에 일반 함수 정의를 무심코 넣음**: `inline`이나 `constexpr` 조건 없이 여러 번역 단위에 정의가 복제되면 링크 단계에서 중복 심볼 문제가 난다. 선언과 정의를 어디에 둘지 구분하지 않으면 원인 파악도 어려워진다.
- **forward declaration이 가능한데도 무조건 include**: 헤더 결합도가 커져 빌드 시간이 늘고 순환 include 위험도 커진다. 타입의 완전한 정의가 정말 필요한 순간까지 include를 미루는 습관이 중요하다.

## 토픽 24: 정적 / 동적 라이브러리

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 23

### 1. 핵심 개념

핵심 용어: `ABI`, `DLL`, `name mangling`, `정적 라이브러리`.

라이브러리는 재사용 가능한 코드 묶음이다. **정적 라이브러리(static library, 스태틱 라이브러리)** 는 링크 시점에 실행 파일 안으로 복사되고, **동적 라이브러리(dynamic library / DLL / shared library, 다이내믹 라이브러리)** 는 실행 시점에 별도로 로드된다. 차이는 코드가 포함되는 시점, 배포 방식, 메모리 공유, ABI 안정성에서 드러난다.

정적 링크는 배포가 단순하고 self-contained 바이너리를 만들기 쉽지만 실행 파일이 커질 수 있다. 동적 링크는 모듈 교체와 메모리 공유에 유리하지만, 로딩 실패와 ABI 호환성 문제를 더 엄격하게 관리해야 한다. `symbol visibility(심볼 비저빌리티, 외부 노출 심볼 범위)`와 `name mangling(네임 맹글링, 함수/타입 이름 장식)`도 동적 라이브러리 경계에서 자주 이야기된다.

실무에서는 성능보다 배포와 ABI 경계를 먼저 보는 경우가 많다. 어떤 모듈을 별도 교체 가능하게 둘 것인지, 어떤 타입을 DLL 바깥으로 노출할 것인지, 플랫폼별 패키징 규칙이 어떤지에 따라 정적/동적 선택이 달라진다.

### 2. 코드 예시

```cpp
// 개발 PC에서는 실행되지만
// 배포 PC에서 프로그램 시작 시 DLL 로드 실패
```

이 예시는 동적 링크가 '컴파일 성공'과 '실행 성공'을 분리한다는 점을 보여 준다. 개발 PC에서는 경로가 맞아도 배포 환경에서는 DLL 누락, 검색 경로 문제, ABI나 버전 불일치 때문에 시작 단계에서 실패할 수 있다.

### 3. 왜 중요한가 (게임/UE5 맥락)

UE5는 모듈과 플러그인 구조를 적극 사용하고, 서드파티 SDK도 자주 연결한다. 그래서 "무엇을 정적으로 묶고 무엇을 동적으로 배포할지", "DLL 경계에서 어떤 타입을 노출할지"를 설명할 수 있어야 실무 대응력이 높다. 이 판단을 잘못하면 빌드는 통과해도 배포 환경에서 로딩 실패나 ABI 충돌이 난다.

### 4. 면접 예상 Q&A

**Q1: 정적 라이브러리와 동적 라이브러리는 어떤 기준으로 고르나요?**
> A: 배포 단순성과 self-contained 바이너리가 중요하면 정적 링크가 유리하고, 모듈 교체와 메모리 공유, 별도 배포가 중요하면 동적 링크가 유리하다. 결국 선택 기준은 속도 미세 차이보다 배포 전략과 ABI 경계 관리에 더 가깝다.

**Q2: 동적 라이브러리의 대표적인 단점은 무엇인가요?**
> A: 실행 환경에 라이브러리가 없거나 버전이 다르면 로딩 실패가 날 수 있고, ABI가 바뀌면 컴파일은 통과해도 런타임에서 충돌할 수 있다. 즉 빌드 성공이 실행 성공을 보장하지 않는다는 점이 가장 큰 부담이다.

**Q3: DLL 경계에 STL 타입을 그대로 노출하는 것이 왜 위험한가요?**
> A: 컴파일러, 표준 라이브러리 구현, CRT 버전이 다르면 `std::string`, `std::vector` 같은 타입의 내부 레이아웃과 메모리 소유 규칙이 달라질 수 있기 때문이다. 그래서 DLL 경계에서는 C 스타일 인터페이스나 더 안정적인 추상화 경계를 선호하는 경우가 많다.

**Q4: 개발 PC에서는 되는데 배포 PC에서만 로드 실패가 난다면 무엇부터 의심하나요?**
> A: DLL 파일 누락, 검색 경로 문제, 의존 라이브러리 누락, ABI나 CRT 버전 불일치를 먼저 본다. 이런 문제는 코드보다 패키징과 실행 환경 차이에서 생기는 경우가 많다.

### 5. 흔한 실수

- **동적 라이브러리 배포 파일 누락**: 개발 환경에서는 우연히 경로가 맞아 돌아가도, 다른 PC에서는 바로 로딩 실패가 난다. 동적 링크는 코드만이 아니라 산출물과 검색 경로까지 함께 관리해야 한다.
- **DLL 경계에 STL 타입을 무심코 노출**: 컴파일러/런타임 차이로 ABI 문제가 생길 수 있다. 인터페이스를 설계할 때는 편의보다 경계 안정성을 먼저 보는 편이 안전하다.

## 토픽 25: 메모리 단편화 & 커스텀 할당

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 3, 6

### 1. 핵심 개념

핵심 용어: `내부`, `buddy`, `외부 단편화`, `메모리 풀`.

메모리 단편화(fragmentation)는 총 메모리는 충분한데도 필요한 형태의 연속 공간을 확보하기 어려워지는 현상이다. **내부 단편화**는 할당받은 블록 내부에 남는 낭비이고, **외부 단편화**는 빈 공간은 많지만 조각나 있어 큰 연속 블록을 못 만드는 상황이다.

게임은 프레임마다 반복적인 할당과 해제가 일어나므로 범용 할당자만으로는 지터와 단편화가 커질 수 있다. 그래서 메모리 풀(pool), free list, buddy system, frame allocator, arena allocator 같은 커스텀 할당 전략을 사용하며, 목표는 평균 속도보다 **예측 가능한 최악 시간과 안정된 메모리 패턴**을 확보하는 데 있다.

핵심은 할당자를 자료구조처럼 고르는 것이다. 고정 크기 반복 객체에는 풀, 프레임 전체를 한 번에 비우는 임시 메모리에는 frame/arena allocator처럼, 수명 패턴과 크기 분포에 맞춰야 효과가 난다.

### 2. 코드 예시

```cpp
for (int i = 0; i < 10000; ++i) {
    Bullet* b = new Bullet();
    Simulate(b);
    delete b;
}
```

문제는 `new` / `delete` 문법 자체보다 이런 할당 패턴이다. 작은 동일 크기 객체를 매우 자주 만들고 버리면 범용 할당자 오버헤드와 단편화 가능성이 커지고, 그 결과 프레임 타임 지터가 눈에 띄기 쉽다.

### 3. 왜 중요한가 (게임/UE5 맥락)

게임에서는 평균 시간보다 최악 시간 지연이 더 중요하다. 단편화와 범용 할당자 지터는 프레임 드랍과 히치의 원인이 되고, 총알, 파티클, 임시 버퍼처럼 작은 객체가 대량으로 생성·파괴되는 구간에서 특히 눈에 띈다. 그래서 할당 전략은 메모리 절약보다 프레임 시간 안정성과 직결되는 설계 문제로 본다.

### 4. 면접 예상 Q&A

**Q1: 게임/엔진 코드에서 커스텀 할당자를 고민하는 이유는 무엇인가?**
> A: 핫패스의 할당 패턴이 예측 가능하면 범용 할당자보다 더 낮은 지연과 더 적은 단편화로 동작시킬 수 있기 때문이다. 결국 목적은 평균 속도보다 프레임마다 튀지 않는 메모리 동작을 만드는 데 있다.

**Q2: 내부 단편화와 외부 단편화의 차이를 설명하라.**
> A: 내부 단편화는 할당된 블록 내부에 사용하지 않는 공간이 남는 것이고, 외부 단편화는 빈 공간은 있지만 조각나 있어 큰 연속 블록을 확보하지 못하는 상태다. 즉 하나는 블록 안의 낭비이고, 다른 하나는 블록 사이 배치의 문제다.

**Q3: 메모리 풀이 게임 엔진에 적합한 상황은?**
> A: 동일 크기 객체가 빈번하게 생성·파괴되는 경우, 예를 들어 총알, 파티클, 이벤트 같은 패턴에서 적합하다. 고정 크기 블록을 미리 마련하면 할당과 해제가 빠르고 외부 단편화도 사실상 없다.

**Q4: frame allocator나 arena allocator는 언제 잘 맞나요?**
> A: 한 프레임이나 한 작업 단위가 끝날 때 메모리를 한꺼번에 비워도 되는 임시 데이터에 잘 맞는다. 개별 해제가 필요 없는 대신, 아주 빠른 할당과 예측 가능한 해제 시점을 얻을 수 있다.

### 5. 흔한 실수

- **모든 문제를 커스텀 할당자로 해결하려 함**: 병목이 실제로는 참조 구조나 자산 로딩에 있을 수도 있다. 측정 없이 도입하면 복잡도만 늘고, allocator가 문제의 본질을 가릴 수 있다.
- **객체 수명 패턴을 모른 채 풀을 도입**: 풀은 같은 크기와 비슷한 수명의 반복 객체에 강하다. 아무 객체나 넣으면 오히려 메모리 낭비와 초기화 버그가 늘어나므로, 패턴을 먼저 분석해야 한다.
