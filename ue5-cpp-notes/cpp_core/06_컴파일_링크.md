## 토픽 23: 컴파일 과정

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 없음

### 1. 핵심 개념

핵심 용어: `ODR`, `translation unit`, `translation unit`, `ODR`.

C++ 코드는 한 번에 기계어가 되는 것이 아니라 **전처리 → 컴파일 → 어셈블 → 링크** 단계를 거친다. 전처리는 `#include`, `#define` 같은 지시문을 펼치고, 컴파일은 각 `translation unit(트랜슬레이션 유닛, 번역 단위)`를 검사해 중간 산출물을 만들고, 링크는 그 조각들을 묶어 실행 파일을 만든다.

이 과정을 이해하면 include guard가 왜 필요한지, forward declaration이 언제 가능한지, ODR(One Definition Rule) 위반이 왜 링크 에러로 이어지는지 자연스럽게 설명할 수 있다. 여기에 `template instantiation(템플릿 인스턴시에이션, 템플릿 인스턴스화)`, `header-only(헤더 온리, 헤더만으로 구현된 형태)` 라이브러리, `PCH(피씨에이치, 미리 컴파일된 헤더)`, `unity build(유니티 빌드, 여러 cpp를 묶어 한 번에 컴파일)` 개념도 결국 번역 단위와 빌드 비용 이해 위에서 연결된다.

**쉽게 감 잡기** - 전처리: 텍스트 붙이기와 치환 - 컴파일: 각 소스 파일을 개별 번역 단위로 검사하고 변환 - 링크: 조각난 심볼을 실제 정의와 연결

**실무 기준** - 헤더에는 선언 위주, 소스에는 정의 위주로 둔다 - 템플릿은 정의가 보이는 곳에 있어야 인스턴스화가 가능하다 - 공통 include가 지나치게 무거우면 PCH와 include 정리가 빌드 시간을 크게 좌우한다 - unity build는 빌드를 빠르게 만들 수 있지만, 숨어 있던 include 누락을 가릴 수 있어 독립 번역 단위 기준 점검도 필요하다

### 2. 코드 예시

```cpp
// Utils.h
int Add(int a, int b) {
    return a + b;
}
```

inline도 아닌 일반 함수 정의를 헤더에 두면, 여러 translation unit에서 중복 정의가 생겨 ODR 위반 및 링크 에러가 날 수 있다.

### 3. 왜 중요한가 (게임/UE5 맥락)

UE5 프로젝트는 모듈이 많고 빌드 시간이 길며, UHT/UBT가 추가로 개입한다. 그래서 번역 단위, 심볼, include 구조를 이해하지 못하면 "왜 소스 하나 바꿨는데 전체가 다시 빌드되지?", "왜 선언은 있는데 링크가 깨지지?" 같은 문제를 빠르게 풀 수 없다.

### 4. 면접 예상 Q&A

**Q1: 컴파일 과정을 전처리-컴파일-어셈블-링크로 나눠 이해해야 하는 이유는 무엇인가?**
> A: 에러가 어느 단계에서 생겼는지 알아야 include 문제, 문법 문제, 심볼 중복/누락 문제를 정확히 분리할 수 있기 때문이다.

**Q2: C++ 소스가 실행 파일이 되기까지의 4단계를 순서대로 설명하라.**
> A: 전처리(매크로 확장·include 삽입) → 컴파일(소스→어셈블리) → 어셈블(어셈블리→오브젝트 파일) → 링크(오브젝트 파일 결합→실행 파일).

**Q3: C++ 컴파일 과정을 순서대로 설명해 주세요.**
> A: 전처리 단계에서 include와 매크로가 펼쳐지고, 컴파일 단계에서 각 번역 단위가 검사되고 중간 산출물로 바뀐다. 이후 어셈블을 거쳐 목적 파일이 만들어지고, 링크 단계에서 심볼이 연결돼 최종 실행 파일이 생성된다.

**Q4: forward declaration은 언제 쓸 수 있나요?**
> A: 타입의 존재만 알면 되는 상황, 예를 들어 포인터나 참조 멤버, 함수 선언 정도에서는 쓸 수 있다. 반대로 값 멤버를 직접 들거나 `sizeof`, 멤버 접근, 상속처럼 완전한 정의가 필요하면 불가능하다.

### 5. 흔한 실수

- **헤더에 일반 함수 정의를 무심코 넣음**: `inline`이나 `constexpr` 조건 없이 여러 번역 단위에 정의가 복제되면 링크 단계에서 문제가 난다.
- **forward declaration이 가능한데도 무조건 include**: 헤더 결합도가 커져 빌드 시간이 늘고 순환 include 위험도 커진다.

## 토픽 24: 정적 / 동적 라이브러리

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 23

### 1. 핵심 개념

핵심 용어: `ABI`, `DLL`, `name mangling`, `정적 라이브러리`.

라이브러리는 재사용 가능한 코드 묶음이다. **정적 라이브러리(static library, 스태틱 라이브러리)** 는 링크 시점에 실행 파일 안으로 복사되고, **동적 라이브러리(dynamic library / DLL / shared library, 다이내믹 라이브러리)** 는 실행 시점에 별도로 로드된다. 차이는 코드가 포함되는 시점, 배포 방식, 메모리 공유, ABI 안정성에서 드러난다.

정적 링크는 배포가 단순하고 self-contained 바이너리를 만들기 쉽지만 실행 파일이 커질 수 있다. 동적 링크는 모듈 교체와 메모리 공유에 유리하지만, 로딩 실패와 ABI 호환성 문제를 더 엄격하게 관리해야 한다. `symbol visibility(심볼 비저빌리티, 외부 노출 심볼 범위)`와 `name mangling(네임 맹글링, 함수/타입 이름 장식)`도 동적 라이브러리 경계에서 자주 이야기된다.

### 2. 코드 예시

```cpp
// 개발 PC에서는 실행되지만
// 배포 PC에서 프로그램 시작 시 DLL 로드 실패
```

동적 라이브러리 배포 파일 누락, 검색 경로 문제, 또는 ABI/버전 불일치가 원인일 수 있다. 컴파일이 성공해도 실행 환경 의존성이 남는다.

### 3. 왜 중요한가 (게임/UE5 맥락)

UE5는 모듈과 플러그인 구조를 적극 사용하고, 서드파티 SDK도 자주 연결한다. 그래서 "무엇을 정적으로 묶고, 무엇을 동적으로 배포할지", "DLL 경계에서 어떤 타입을 노출할지"를 설명할 수 있어야 실무 대응력이 높다.

### 4. 면접 예상 Q&A

**Q1: 정적 라이브러리와 동적 라이브러리의 실무 차이는 무엇인가?**
> A: 정적 라이브러리는 빌드 시 바이너리에 포함되고 배포가 단순하지만 크기가 커질 수 있고, 동적 라이브러리는 교체와 공유가 쉽지만 로딩/배포/ABI 호환성을 더 신경 써야 한다.

**Q2: 정적 라이브러리와 동적 라이브러리의 핵심 차이는?**
> A: 정적 라이브러리는 링크 시 실행 파일에 코드가 포함되어 self-contained 바이너리를 만든다. 동적 라이브러리는 실행 시 별도 파일에서 로드되어 모듈 교체와 메모리 공유가 가능하다.

**Q3: 정적 라이브러리와 동적 라이브러리의 차이는 무엇인가요?**
> A: 정적 라이브러리는 링크 시 실행 파일에 포함되고, 동적 라이브러리는 실행 시 별도로 로드된다. 정적은 배포가 단순하고, 동적은 교체와 공유가 유연하다는 차이가 있다.

**Q4: 동적 라이브러리의 단점은 무엇인가요?**
> A: 실행 환경에 라이브러리가 없거나 버전이 다르면 로딩 실패가 발생할 수 있다. 또한 ABI가 바뀌면 컴파일은 통과해도 런타임에서 문제가 터질 수 있다.

### 5. 흔한 실수

- **동적 라이브러리 배포 파일 누락**: 개발 환경에서는 되는데 다른 PC에서 로딩 실패가 나는 대표적인 원인이다.
- **DLL 경계에 STL 타입을 무심코 노출**: 컴파일러/런타임 차이로 ABI 문제가 생길 수 있다.

- ``Engine/Source/Runtime/Projects/Public/Interfaces/IPluginManager.h` — 플러그인 로딩 구조`

## 토픽 25: 메모리 단편화 & 커스텀 할당

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 3, 6

### 1. 핵심 개념

핵심 용어: `내부`, `buddy`, `외부 단편화`, `메모리 풀`.

메모리 단편화(fragmentation)는 총 메모리는 충분한데도 필요한 형태의 연속 공간을 확보하기 어려워지는 현상이다. **내부 단편화**는 할당받은 블록 내부에 남는 낭비이고, **외부 단편화**는 빈 공간은 많지만 조각나 있어 큰 연속 블록을 못 만드는 상황이다.

게임은 프레임마다 반복적인 할당/해제가 일어나므로 범용 할당자만으로는 지터와 단편화가 커질 수 있다. 그래서 메모리 풀(pool), free list, buddy system, frame allocator, arena allocator 같은 커스텀 할당 전략을 사용한다. 목표는 "평균 속도"보다 **예측 가능한 최악 시간과 안정된 메모리 패턴**이다.

### 2. 코드 예시

```cpp
for (int i = 0; i < 10000; ++i) {
    Bullet* b = new Bullet();
    Simulate(b);
    delete b;
}
```

작은 동일 크기 객체를 매우 자주 할당/해제하면 범용 할당자 오버헤드와 단편화 가능성이 커진다. 프레임 타임 지터도 생길 수 있다.

### 3. 왜 중요한가 (게임/UE5 맥락)

게임에서는 평균 시간보다 최악 시간 지연이 더 중요하다. 단편화와 범용 할당자 지터는 프레임 드랍과 히치의 원인이 되며, 작은 객체가 대량으로 생성/파괴되는 총알, 파티클, 임시 버퍼에서 특히 눈에 띈다.

### 4. 면접 예상 Q&A

**Q1: 게임/엔진 코드에서 커스텀 할당자를 고민하는 이유는 무엇인가?**
> A: 핫패스의 할당 패턴이 예측 가능하면 범용 할당자보다 더 낮은 지연과 더 적은 단편화로 동작시킬 수 있기 때문이다.

**Q2: 내부 단편화와 외부 단편화의 차이를 설명하라.**
> A: 내부 단편화는 할당된 블록 내부에 사용하지 않는 공간이 낭비되는 것이고, 외부 단편화는 해제된 블록 사이에 작은 빈 공간이 흩어져 큰 연속 할당이 불가능해지는 것이다.

**Q3: 메모리 풀이 게임 엔진에 적합한 상황은?**
> A: 동일 크기 객체가 빈번하게 생성·파괴되는 경우(총알, 파티클, 이벤트 등)에 적합하다. 고정 크기 블록을 미리 할당해 두면 할당·해제가 O(1)이고 외부 단편화가 없다.

**Q4: 내부 단편화와 외부 단편화의 차이는 무엇인가요?**
> A: 내부 단편화는 할당받은 블록 안에서 사용하지 못하는 낭비가 생기는 것이고, 외부 단편화는 빈 공간은 많지만 조각나 있어 연속 블록을 못 구하는 상황이다. 즉 하나는 블록 내부 낭비, 다른 하나는 공간 배치 문제다.

### 5. 흔한 실수

- **모든 문제를 커스텀 할당자로 해결하려 함**: 병목이 실제로는 참조 구조나 자산 로딩에 있을 수도 있다. 측정 없이 도입하면 복잡도만 는다.
- **객체 수명 패턴을 모른 채 풀을 도입**: 풀은 같은 크기와 비슷한 수명의 반복 객체에 강하다. 아무 객체나 넣으면 오히려 메모리 낭비와 초기화 버그가 늘어난다.
