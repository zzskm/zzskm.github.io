## 토픽 23: 컴파일 과정

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 없음

### 1. 문제 상황

핵심 용어: `ODR`, `translation unit`.

빌드가 실패했을 때 에러 메시지만 보고 원인을 찾으려 하면 전처리 문제, 문법 문제, 심볼 연결 문제를 뒤섞어 추측하게 된다. 에러가 어느 단계에서 발생했는지 구분하지 못하면 디버깅 시간이 낭비된다.

UE5 프로젝트는 모듈 수가 많고 UHT/UBT가 추가로 개입한다. 번역 단위와 심볼 연결을 이해하지 못하면 '왜 소스 하나를 고쳤는데 전체가 다시 빌드되는가', '왜 선언은 있는데 링크가 깨지는가' 같은 문제를 체계적으로 설명할 수 없다.

### 2. 핵심 개념

#### 정의

C++ 빌드는 전처리 →컴파일 →어셈블 →링크 4단계를 거친다. 전처리는 #include와 #define을 펼치는 텍스트 치환이다. 컴파일은 각 translation unit(번역 단위)의 문법과 타입을 검사해 중간 코드를 만든다. 어셈블은 기계어 오브젝트 파일(.o/.obj)을 생성한다. 링크는 오브젝트 파일들의 심볼을 연결해 최종 실행 파일을 만든다.

실제 파일 흐름: .cpp →(전처리) →.i →(컴파일) →.s →(어셈블) →.o/.obj →(링크) →exe/dll.

ODR(One Definition Rule)은 프로그램 전체에서 하나의 심볼이 정확히 하나의 정의만 가져야 한다는 규칙이다. 위반하면 undefined behavior이다.

#### 이 개념을 이해하려면

- 템플릿 (토픽 19): 인스턴스화가 번역 단위 경계와 맞물린다
- 헤더/소스 분리: 선언과 정의의 위치가 빌드 에러 유형을 결정한다

#### 이 개념이 영향을 주는 것

- 정적/동적 라이브러리 (토픽 24): 링크 결과물이 어떤 형태로 배치되는지
- UHT & GENERATED_BODY() (카테고리 2 / 토픽 3): UE 특유의 빌드 전 코드 생성 단계
- UBT (카테고리 8): UE 빌드 시스템 심화

### 3. 내부 동작 / 메커니즘

#### compile error vs link error 구분

compile error는 단일 번역 단위 내 문법/타입 오류다. 헤더 누락, 타입 불일치, 문법 실수가 여기 해당한다. 에러 메시지에 소스 파일 이름과 라인 번호가 나온다.

link error는 번역 단위 간 심볼 연결 실패다. 'undefined reference', 'unresolved external symbol' 메시지가 대표적이다. 선언은 있지만 정의를 찾지 못하거나, 같은 심볼이 여러 번 정의된 경우에 발생한다.

#### ODR 위반이 생기는 대표 사례

- 헤더에 일반(non-inline, non-template) 함수 정의를 둔다 →여러 번역 단위에 정의가 복제 →중복 심볼
- 같은 이름의 전역 변수를 여러 소스에 정의한다
- 해결: 헤더에는 선언만, 정의는 cpp에. 헤더에 둘 이유가 있으면 inline/constexpr/template을 사용한다

#### forward declaration vs include 판단

타입의 존재만 알면 되는 경우(포인터, 참조, 함수 선언): forward declaration으로 충분하다. 타입의 크기, 멤버, 레이아웃이 필요한 경우(값 멤버, sizeof, 상속, 멤버 접근): 완전한 정의가 필요하므로 include한다.

판단 기준: '이 코드가 타입의 이름만 필요한가, 구체 레이아웃까지 필요한가'를 묻는다.

#### 빌드 최적화 기법

- PCH(Precompiled Header): 자주 바뀌지 않는 헤더를 미리 컴파일해 반복 파싱을 줄인다. 자주 바뀌는 헤더를 넣으면 역효과
- Unity build: 여러 .cpp를 하나로 묶어 컴파일 →빌드 빠르지만 독립 번역 단위 기준의 누락 문제를 가릴 수 있다
- Forward declaration 적극 사용: 헤더 간 결합도를 줄여 변경 시 재빌드 범위를 축소한다

### 4. 자주 틀리는 포인트

- 헤더에 일반 함수 정의를 넣는다: inline이나 constexpr 없이 여러 번역 단위에 정의가 복제되면 링크 단계에서 중복 심볼 에러가 난다. 선언과 정의를 어디에 둘지 구분하는 습관이 필요하다.
- forward declaration이 가능한데도 무조건 include한다: 헤더 결합도가 커져 빌드 시간이 늘고 순환 include 위험도 커진다. 완전한 정의가 정말 필요한 순간까지 include를 미루는 것이 원칙이다.
- 템플릿 정의를 .cpp에만 둔다: 템플릿은 인스턴스화가 일어나는 번역 단위에서 정의를 볼 수 있어야 한다. 헤더에 두거나 명시적 인스턴스화를 사용해야 한다.
- compile error와 link error를 구분하지 않는다: 에러 메시지를 보고 어느 단계에서 깨졌는지 먼저 나누는 습관이 디버깅 속도를 결정한다.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

#### forward declaration을 쓰는 경우

- 헤더에서 타입의 포인터나 참조만 사용할 때
- 함수 선언에서 매개변수/반환 타입의 이름만 필요할 때
- 판단 기준: 해당 타입의 크기나 멤버를 몰라도 코드가 성립하면 forward declaration

#### include가 필요한 경우

- 값 멤버로 타입을 직접 포함할 때 (sizeof 필요)
- 상속받을 때 (기반 클래스 레이아웃 필요)
- 멤버 함수를 호출하거나 멤버에 접근할 때
- 판단 기준: 컴파일러가 타입의 구체적인 정보를 알아야 하면 include

#### PCH에 넣을 헤더 vs 넣지 않을 헤더

- 넣는다: 표준 라이브러리, 엔진 기본 헤더 등 거의 바뀌지 않는 것
- 넣지 않는다: 자주 수정하는 프로젝트 헤더. PCH 재빌드가 잦아지면 역효과

### 6. 코드 예시 (bad → good)

#### Bad: 헤더에 일반 함수 정의 →ODR 위반

```cpp
// Utils.h
int Add(int a, int b) {
    return a + b;
}
// 이 헤더를 포함하는 모든 .cpp에 정의가 복제 →링크 에러
```

문제의 핵심은 헤더를 포함한 번역 단위마다 동일한 정의가 생긴다는 점이다. 컴파일은 통과할 수 있어도 링크 단계에서 ODR 위반으로 드러난다.

#### Good: 선언/정의 분리 또는 inline

```cpp
// Utils.h — 선언만
int Add(int a, int b);

// Utils.cpp — 정의
int Add(int a, int b) { return a + b; }

// 또는 헤더에 inline
inline int Add(int a, int b) { return a + b; }
```

일반 함수는 선언과 정의를 분리하는 것이 기본이다. 헤더에 정의를 둘 이유가 있다면 inline처럼 중복 정의를 허용하는 규칙과 함께 가야 한다.

#### Bad: 템플릿 정의를 .cpp에만 둠 →링크 에러

```cpp
// Math.h
template<typename T> T Add(T a, T b);

// Math.cpp
template<typename T> T Add(T a, T b) { return a + b; }

// main.cpp
#include "Math.h"
int main() { return Add(1, 2); }  // undefined reference
```

템플릿은 사용하는 번역 단위에서 인스턴스화되므로 정의도 그 시점에 보여야 한다. 선언만 헤더에 두고 정의를 .cpp로 숨기면 심볼을 만들 수 없어 링크 에러가 난다.

#### Good: 템플릿 정의를 헤더에

```cpp
// Math.h
#pragma once
template<typename T>
T Add(T a, T b) { return a + b; }
```

이 방식이면 템플릿을 쓰는 쪽에서 정의를 바로 볼 수 있다. 헤더 전개 비용은 늘지만 템플릿의 인스턴스화 규칙과는 가장 자연스럽게 맞는다.

### 7. 판단 기준 요약

- 빌드 에러를 보면 먼저 어느 단계(전처리/컴파일/링크)에서 깨졌는지 나눈다
- 헤더에는 선언만 둔다. 정의를 둘 이유가 있으면 inline/constexpr/template인지 확인한다
- forward declaration이 가능하면 include하지 않는다. 판단 기준은 '타입의 크기가 필요한가'이다
- ODR 위반은 링크 단계에서 드러난다. 헤더에 일반 함수 정의를 넣는 것이 가장 흔한 원인이다
- 템플릿 정의는 사용하는 번역 단위에서 보여야 한다. 헤더에 두는 것이 기본이다

## 토픽 24: 정적 / 동적 라이브러리

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 23

### 1. 문제 상황

핵심 용어: `ABI`, `DLL`, `name mangling`, `정적 라이브러리`.

빌드는 성공했는데 배포 환경에서 DLL을 찾을 수 없다는 에러가 난다. 또는 헤더는 맞는데 런타임에 함수 호출이 이상하게 동작한다. 이런 문제는 정적/동적 링크의 차이와 ABI 경계를 이해하지 못하면 원인을 좁히기 어렵다.

게임 엔진은 모듈과 플러그인 구조를 적극 사용하고, 서드파티 SDK도 자주 연결한다. '무엇을 정적으로 묶고 무엇을 동적으로 배포할지'를 설명할 수 없으면 실무에서 빌드 성공이 실행 성공으로 이어지지 않는 상황에 반복적으로 부딪힌다.

### 2. 핵심 개념

#### 정의

정적 라이브러리(static library, .lib/.a)는 링크 시점에 코드가 실행 파일 안으로 복사된다. 결과물은 self-contained 바이너리다.

동적 라이브러리(dynamic library / DLL / shared library, .dll/.so)는 실행 시점에 별도로 로드된다. 코드가 실행 파일 밖에 있으므로 모듈 교체와 메모리 공유가 가능하다.

ABI(Application Binary Interface)는 바이너리 수준의 호출 규약, 타입 레이아웃, name mangling 규칙이다. 같은 헤더를 쓰더라도 컴파일러나 CRT 버전이 다르면 ABI가 달라져 런타임 충돌이 난다.

#### 이 개념을 이해하려면

- 컴파일 과정 (토픽 23): 링크 단계가 심볼을 어떻게 연결하는지
- 메모리 레이아웃 (토픽 6): 타입의 크기와 정렬이 ABI에 영향

#### 이 개념이 영향을 주는 것

- UE5 모듈 시스템 (카테고리 8 / 토픽 2): 엔진의 모듈 로딩 구조
- 플러그인 구조 (카테고리 8 / 토픽 4): 서드파티 모듈 배포와 연결

### 3. 내부 동작 / 메커니즘

#### 정적 링크 과정

컴파일러가 .cpp →.o를 만들고, 아카이버(ar/lib)가 .o들을 묶어 .lib/.a를 만든다. 링커는 실행 파일을 만들 때 필요한 심볼을 .lib에서 꺼내 실행 파일에 포함시킨다. 배포 시 별도 파일이 필요 없다.

#### 동적 링크 과정

컴파일러가 .dll/.so를 만들 때 export 심볼을 지정한다(Windows: __declspec(dllexport), Linux: 기본 공개 또는 visibility 속성). 실행 파일은 import library(.lib)나 심볼 테이블을 참조해 빌드하고, 실행 시 OS 로더가 .dll/.so를 메모리에 매핑한다.

심볼 가시성(symbol visibility)은 플랫폼마다 기본값이 다르다. Windows는 명시적 export가 필요하고, Linux는 기본 공개(-fvisibility=hidden으로 숨길 수 있다).

#### name mangling과 extern "C"

C++ 컴파일러는 함수 이름에 매개변수 타입 정보를 인코딩한다(name mangling). 덕분에 오버로딩된 함수를 링커가 구별할 수 있다. 하지만 컴파일러마다 mangling 규칙이 달라 DLL 경계에서 문제가 된다.

extern "C"는 mangling을 비활성화해 C 호환 심볼 이름을 만든다. DLL 경계에서 안정적인 인터페이스가 필요하면 extern "C"를 사용한다.

### 4. 자주 틀리는 포인트

- DLL 경계에 STL 타입을 그대로 노출한다: 컴파일러/CRT 버전이 다르면 std::string, std::vector의 내부 레이아웃이 달라져 메모리 손상이 난다. DLL 인터페이스는 C 호환 타입이나 엔진 자체 타입으로 설계한다.
- 동적 라이브러리 배포 파일을 누락한다: 개발 환경에서는 PATH에 DLL이 있어 동작하지만, 배포 PC에서는 로딩 실패가 난다. 동적 링크는 코드뿐 아니라 산출물과 검색 경로까지 함께 관리해야 한다.
- 정적 링크가 항상 빠르다고 단정한다: DLL 로드 오버헤드를 피하긴 하지만, 바이너리 크기 증가로 캐시 효율이 떨어질 수 있다. 선택 기준은 속도보다 배포 전략과 ABI 경계 관리다.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

#### 정적 라이브러리를 쓰는 경우

- self-contained 바이너리가 필요할 때 (배포 단순화)
- 서드파티 코드를 통제할 수 있고 ABI 변경 걱정이 없을 때
- 판단 기준: 배포 환경에 별도 파일을 두는 부담보다 바이너리 크기 증가가 더 수용 가능하면 정적

#### 동적 라이브러리를 쓰는 경우

- 런타임에 모듈을 교체하거나 플러그인 구조가 필요할 때
- 여러 프로세스가 같은 코드를 공유해 메모리를 절약해야 할 때
- 패치 시 전체를 다시 배포하지 않고 모듈 단위로 교체하고 싶을 때
- 판단 기준: 모듈 교체, 별도 배포, 메모리 공유 중 하나라도 필요하면 동적

### 6. 대안 비교

#### 정적 라이브러리 vs 동적 라이브러리

- 정적 라이브러리: 실행 파일에 코드가 함께 묶여 self-contained 배포에 유리하다. 대신 바이너리 크기가 커지고 모듈 교체가 어렵다.
- 동적 라이브러리: 모듈 단위 교체와 공유에 유리하다. 대신 DLL/SO 배포, 검색 경로, 버전 호환성을 함께 관리해야 한다.
- 판단 기준: 속도 차이를 과장하기보다 배포 전략과 런타임 로딩 실패 가능성을 먼저 비교한다.

#### ABI 경계 설계

- C++ 타입을 그대로 노출하면 컴파일러, CRT, STL 구현 차이가 ABI 문제로 이어질 수 있다.
- DLL 경계가 길게 유지되거나 외부 팀/툴체인과 연결되면 C ABI나 엔진 자체 타입처럼 더 안정적인 경계를 우선한다.
- 판단 기준: 헤더가 아니라 바이너리 계약을 배포한다는 관점으로 인터페이스를 설계한다.

### 7. 코드 예시 (bad → good)

#### Bad: DLL 경계에 STL 타입 노출

```cpp
// MyDLL.h
#ifdef MYDLL_EXPORTS
#define MYDLL_API __declspec(dllexport)
#else
#define MYDLL_API __declspec(dllimport)
#endif

// 위험: CRT 버전이 다르면 std::string 레이아웃이 달라진다
MYDLL_API std::string GetName();
```

겉으로는 함수 선언 하나지만 실제 위험은 ABI와 메모리 소유권 계약에 있다. EXE와 DLL의 CRT나 STL 구현이 다르면 문자열 내부 레이아웃이 달라져 런타임 손상으로 이어질 수 있다.

#### Good: C 호환 타입으로 DLL 인터페이스 설계

```cpp
// MyDLL.h
extern "C" MYDLL_API const char* GetName();

// 또는 호출자가 버퍼 제공
extern "C" MYDLL_API void GetName(char* Buffer, int MaxLen);
```

좋은 점은 경계가 단순해진다는 것이다. name mangling과 C++ 타입 레이아웃 의존성을 줄이면 DLL 바깥과의 계약을 더 보수적으로 유지할 수 있다.

#### 정적 라이브러리 사용 예

```cpp
// MathLib.h
#pragma once
int Add(int a, int b);

// MathLib.cpp
#include "MathLib.h"
int Add(int a, int b) { return a + b; }

// 빌드: MathLib.cpp →MathLib.lib (정적 라이브러리)
// App.cpp 링크 시 MathLib.lib을 넘겨 심볼 결합
```

이 예시는 코드를 실행 파일 쪽에 묶어 배포하는 전형적인 흐름이다. 별도 런타임 로딩은 줄지만, 라이브러리 교체는 재링크와 재배포를 전제로 한다.

### 8. 판단 기준 요약

- 선택 기준은 속도가 아니라 배포 전략, ABI 경계, 모듈 교체 필요성이다
- DLL 경계에서는 C 호환 타입이나 엔진 자체 타입을 사용한다. STL 타입은 ABI 불일치 위험이 있다
- 동적 링크는 '빌드 성공 != 실행 성공'이다. 배포 산출물과 검색 경로까지 관리해야 한다
- name mangling이 문제가 되면 extern "C"로 안정적인 심볼 이름을 만든다
- 정적은 단순 배포에 강하고, 동적은 모듈 교체와 공유에 강하다

## 토픽 25: 메모리 단편화 & 커스텀 할당

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 3, 6

### 1. 문제 상황

핵심 용어: `내부`, `buddy`, `외부 단편화`, `메모리 풀`.

총알, 파티클, 임시 버퍼처럼 작은 객체가 프레임마다 대량으로 생성-파괴되면 범용 할당자의 지터와 단편화가 누적된다. 평균 프레임 시간은 괜찮은데 간헐적으로 히치가 튀는 원인이 여기일 때가 많다.

게임에서는 평균 속도보다 최악 시간 지연이 더 중요하다. 범용 할당자는 어떤 패턴이든 동작하지만, 예측 가능한 최악 시간을 보장하지는 않는다. 그래서 수명 패턴과 크기 분포에 맞춘 커스텀 할당 전략이 필요해진다.

### 2. 핵심 개념

#### 정의

메모리 단편화(fragmentation)는 총 메모리는 충분한데 필요한 형태의 연속 공간을 확보하기 어려워지는 현상이다. 내부 단편화는 할당된 블록 내부에 남는 낭비(정렬 요구, 고정 크기 블록)이고, 외부 단편화는 빈 공간은 많지만 조각나 있어 큰 연속 블록을 만들지 못하는 상태다.

커스텀 할당자는 특정 할당 패턴에 최적화된 메모리 관리 전략이다. 핵심 목표는 평균 속도보다 예측 가능한 최악 시간과 안정된 메모리 패턴을 확보하는 것이다.

#### 할당자 종류와 용도

- 메모리 풀(pool): 고정 크기 블록을 미리 마련. 같은 크기 객체의 빈번한 생성/파괴에 최적. O(1) 할당/해제, 외부 단편화 없음
- Free list: 해제된 블록을 연결 리스트로 관리. 풀보다 유연하지만 크기가 다양하면 단편화 가능
- Buddy system: 2의 거듭제곱 크기로 분할/합병. 내부 단편화는 있지만 외부 단편화 관리에 강함
- Frame/Arena allocator: 포인터만 전진시켜 할당하고 프레임/작업 단위 끝에 한꺼번에 리셋. 개별 해제 없음, 임시 데이터에 최적
- 범용 할당자(malloc/new): 어떤 패턴이든 동작하지만 최악 시간을 보장하지 않음. 핫패스가 아닌 일반 코드에서는 충분

#### 이 개념을 이해하려면

- 동적 메모리 관리 (토픽 3): new/delete와 범용 할당의 기초
- 메모리 레이아웃 (토픽 6): 캐시 친화성과 데이터 배치

#### 이 개념이 영향을 주는 것

- cache locality (토픽 26): 할당자가 데이터 배치를 결정하고 캐시 효율에 직접 영향
- crash/병목 분석 (토픽 30): 메모리 관련 히치와 단편화 진단

### 3. 내부 동작 / 메커니즘

#### 내부 단편화 vs 외부 단편화

내부 단편화: 12바이트를 요청했는데 16바이트 블록을 할당받으면 4바이트가 낭비된다. 정렬 요구나 고정 크기 블록에서 발생한다. 개별 낭비는 작지만 대량 객체에서 누적된다.

외부 단편화: 4KB 빈 블록이 세 군데 있어도 10KB 연속 블록은 만들지 못한다. 다양한 크기의 할당과 해제가 반복될수록 심해진다. 총 여유 메모리는 충분해도 할당이 실패하는 원인이 된다.

#### 풀 할당자 동작

초기화 시 큰 메모리 블록을 받아 고정 크기 청크로 나누고 free list에 연결한다. 할당은 free list에서 꺼내기(O(1)), 해제는 free list에 반환하기(O(1)). 블록 크기가 고정이므로 외부 단편화가 원리적으로 없다.

단점: 블록 크기와 다른 객체를 넣으면 내부 단편화가 생기거나 아예 담지 못한다. 사용 패턴이 균일할수록 효과가 크다.

#### 프레임/아레나 할당자 동작

큰 버퍼에서 포인터만 전진시켜 할당한다(bump allocation). 개별 해제는 없고, 프레임이나 작업이 끝나면 포인터를 시작점으로 되돌려 전체를 리셋한다. 할당이 극도로 빠르고 단편화가 없다.

수명이 프레임/작업 단위로 명확한 임시 데이터에만 적합하다. 개별 객체의 수명이 제각각이면 쓸 수 없다.

### 4. 자주 틀리는 포인트

- 병목 확인 없이 커스텀 할당자부터 도입한다: 실제 병목이 할당 패턴이 아니라 참조 구조나 자산 로딩에 있을 수 있다. 측정 없이 도입하면 복잡도만 늘고 효과가 없다.
- 객체 수명 패턴을 모른 채 풀을 도입한다: 풀은 같은 크기와 비슷한 수명의 반복 객체에 강하다. 크기와 수명이 제각각인 객체를 넣으면 오히려 메모리 낭비와 초기화 버그가 늘어난다.
- 단편화를 단순 메모리 부족으로 오해한다: 총 여유 메모리는 충분해도 연속 공간이 없어 할당이 실패할 수 있다. 메모리 사용량보다 할당 패턴을 봐야 한다.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

#### 커스텀 할당자가 필요한 경우

- 동일 크기 객체가 프레임마다 대량 생성/파괴 (총알, 파티클, 이벤트) →풀
- 한 프레임/작업이 끝나면 한꺼번에 비워도 되는 임시 데이터 (렌더 준비, 파싱 중간물) →frame/arena
- 할당 크기가 2의 거듭제곱에 가깝고 합병이 중요한 경우 →buddy system
- 판단 기준: 프로파일러에서 할당/해제가 핫패스의 유의미한 비용일 때

#### 범용 할당자로 충분한 경우

- 할당이 드물거나 핫패스 밖에 있을 때
- 객체 크기와 수명이 다양해서 패턴화할 수 없을 때
- 판단 기준: 측정 결과 할당이 병목이 아니면 범용으로 충분하다

### 6. 코드 예시 (bad → good)

#### Bad: 핫패스에서 매 프레임 new/delete

```cpp
void SpawnBullets(int Count) {
    for (int i = 0; i < Count; ++i) {
        Bullet* B = new Bullet();  // 범용 할당자, 프레임마다 수백 번
        Simulate(B);
        delete B;
    }
}
```

반복되는 범용 할당은 평균 비용보다 지터와 단편화 누적이 더 문제다. 핫패스에서 이런 패턴이 많아지면 프레임 시간 분산이 커진다.

#### Good: 풀 할당자로 고정 크기 객체 재활용

```cpp
// 초기화 시 풀 생성
TMemoryPool<Bullet> BulletPool(MaxBullets);

void SpawnBullets(int Count) {
    for (int i = 0; i < Count; ++i) {
        Bullet* B = BulletPool.Allocate();  // O(1), 단편화 없음
        Simulate(B);
        BulletPool.Free(B);                 // O(1)
    }
}
```

고정 크기 객체가 반복적으로 생성되고 파괴된다면 풀은 가장 직접적인 선택지다. 할당과 해제가 예측 가능해지고 외부 단편화도 크게 줄어든다.

#### Good: 프레임 할당자로 임시 데이터 일괄 해제

```cpp
FFrameAllocator FrameAlloc(FrameBufferSize);

void RenderPrepare() {
    auto* TempData = FrameAlloc.Alloc<RenderTemp>(Count);
    BuildRenderList(TempData);
    SubmitToGPU(TempData);
}  // 프레임 끝에 FrameAlloc.Reset()으로 전체 해제
```

수명이 프레임 경계로 명확하다면 개별 free보다 일괄 reset이 훨씬 단순하다. 이런 allocator는 임시 버퍼처럼 짧고 예측 가능한 lifetime에 특히 잘 맞는다.

### 7. 판단 기준 요약

- 커스텀 할당자는 측정 결과 할당이 핫패스의 병목일 때만 도입한다
- 할당자는 자료구조처럼 고른다: 수명 패턴(프레임/영구) x 크기 분포(고정/가변)가 선택 기준이다
- 풀 = 고정 크기 반복 객체, frame/arena = 일괄 해제 임시 데이터, buddy = 가변 크기 합병 필요
- 단편화는 메모리 부족이 아니라 연속 공간 부족이다. 메모리 사용량보다 할당 패턴을 분석한다
- 목표는 평균 속도가 아니라 프레임 시간 안정성이다
