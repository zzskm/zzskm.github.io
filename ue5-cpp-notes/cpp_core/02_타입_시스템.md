## 토픽 7: const correctness(콘스트 코렉트니스, const 일관성)

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 없음

### 1. 문제 상황

핵심 용어: `API 계약`, `왼쪽`, `const int*`, `mutable`.

대규모 코드베이스에서 함수가 인자를 수정하는지 확인하려면 구현체를 열어봐야 한다. 이 비용은 코드 리뷰, 디버깅, 리팩토링 시마다 반복된다.

const가 빠진 API는 '이 함수가 내 데이터를 건드리는가?'라는 질문에 답하지 못한다. 의도치 않은 수정이 버그로 이어지고, 발견이 늦어진다.

### 2. 핵심 개념

#### 정의

const correctness는 '변경하지 않겠다'는 약속을 타입 시스템에 일관되게 표현하는 원칙이다. 변수, 포인터, 참조, 매개변수, 반환값, 멤버 함수에 const를 붙여 컴파일러가 실수를 잡도록 한다.

const 위치에 따라 의미가 달라진다. * 왼쪽 = 가리키는 대상 불변, * 오른쪽 = 포인터 자신 불변. 멤버 함수 뒤 const = 이 함수는 객체의 논리적 상태를 바꾸지 않는다.

bitwise constness(비트 수준 불변)는 컴파일러가 검사하는 기준이고, logical constness(논리적 불변)는 외부에서 관찰 가능한 상태가 바뀌지 않는다는 의미다. 캐시 같은 내부 상태를 바꿔야 할 때 mutable을 사용한다.

#### 이 개념을 이해하려면

- 포인터 vs 참조 (토픽 2): const가 포인터/참조에 붙는 위치와 의미
- 함수 시그니처의 의미: 매개변수 타입이 곧 API 계약

#### 이 개념이 영향을 주는 것

- auto / decltype (토픽 8): auto가 const를 보존하는 규칙
- 캐스팅 4종 (토픽 11): const_cast의 올바른 사용 범위
- 함수 시그니처 설계: const T& 매개변수가 기본 선택인 이유

### 3. 내부 동작 / 메커니즘

#### const 위치별 해석

- const int* p — 가리키는 값이 const. *p = 10은 에러. p = &other는 허용.
- int* const p — 포인터 자신이 const. p = &other는 에러. *p = 10은 허용.
- const int* const p — 둘 다 const.
- const T& param — 읽기 전용 참조 매개변수. 복사 비용 없이 원본 보호.
- void Foo() const — 멤버 함수가 this->멤버를 수정하지 않음을 보장.

#### 함수 시그니처 설계 기준

매개변수 선택 흐름: (1) 읽기만 하는가? →const T& (큰 타입) 또는 T (작은 타입, int/float 등) (2) 수정이 필요한가? →T& (3) null이 유효한가? →const T* 또는 T*

반환값: const T&를 반환하면 호출자에게 읽기 전용을 강제한다. 단, 임시 객체의 참조를 반환하면 dangling이므로 주의해야 한다.

#### mutable과 logical constness

mutable 멤버는 const 멤버 함수 안에서도 수정 가능하다. 캐시, 뮤텍스, 디버깅 카운터처럼 외부 관찰 가능한 상태에 영향을 주지 않는 내부 상태에만 사용한다.

mutable을 남용하면 const의 의미가 무너진다. '이 멤버를 바꿔도 외부에서 관찰 가능한 동작이 같은가?'가 판단 기준이다.

### 4. 자주 틀리는 포인트

- const를 문법 장식으로만 본다 →const는 컴파일러가 강제하는 읽기 전용 계약이다.
- 포인터에서 const 위치를 혼동한다 →* 왼쪽 = 대상 불변, * 오른쪽 = 포인터 불변으로 외워라.
- const_cast로 const를 벗긴다 →원래 const로 선언된 객체의 const를 벗기면 UB다. const_cast는 레거시 API 호환에만 제한적으로 사용한다.
- const 멤버 함수 안에서 외부 관찰 가능한 상태를 mutable로 바꾼다 →logical constness 위반.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

const를 붙이는 기본 원칙: 변경하지 않는 모든 곳에 붙인다. 나중에 빼는 것보다 처음부터 붙이는 것이 훨씬 쉽다.

const를 안 쓰는 경우: 값을 실제로 수정해야 할 때. int, float 같은 작은 타입의 값 복사 매개변수에는 const가 의미 없다.

함수 시그니처에서 const 선택: 읽기 전용 →const T&, 수정 필요 →T&, 선택적 입력 →const T*

### 6. 코드 예시 (bad →good)

#### Bad: const 없는 읽기 전용 매개변수

```cpp
void PrintName(std::string& name) {  // 수정 안 하는데 non-const
    std::cout << name;
}
```

#### Good: const 참조

```cpp
void PrintName(const std::string& name) {  // 읽기 전용 명시
    std::cout << name;
}
```

#### Bad: const 누락된 멤버 함수

```cpp
class Player {
    int HP;
public:
    int GetHP() { return HP; }  // const 빠짐
};
```

#### Good: const 멤버 함수

```cpp
class Player {
    int HP;
public:
    int GetHP() const { return HP; }  // 상태 불변 보장
};
```

### 7. 판단 기준 요약

- 변경하지 않는 모든 곳에 const를 붙여라 — 기본값은 const
- 큰 타입 읽기 전용 매개변수 →const T&
- 상태를 바꾸지 않는 멤버 함수 →뒤에 const
- mutable은 외부 관찰 불변일 때만 허용
- const_cast는 레거시 API 호환용 최후의 수단

## 토픽 8: auto / decltype(오토 / 디클타입, 타입 추론)

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 7

### 1. 문제 상황

핵심 용어: `decltype(auto)`, `forwarding reference`, `추론`, `타입`.

템플릿과 컨테이너가 많은 코드에서 타입 이름이 길어진다. std::unordered_map<std::string, std::vector<int>>::iterator 같은 타입을 매번 쓰면 가독성이 떨어진다.

그러나 auto를 무분별하게 쓰면 타입이 보이지 않아 의도를 파악하기 어렵다. auto가 const를 벗기거나 참조를 떨어뜨리는 규칙을 모르면 의도치 않은 복사나 수정이 발생한다.

### 2. 핵심 개념

#### 정의

auto는 초기화 표현식에서 타입을 추론하는 키워드다. 컴파일러가 오른쪽 값의 타입을 보고 왼쪽 변수의 타입을 결정한다.

decltype은 표현식의 선언 타입을 그대로 가져온다. auto와 달리 참조성과 const를 보존한다.

decltype(auto)는 auto의 편의성과 decltype의 정확성을 결합한다. 반환 타입 추론에서 참조를 보존해야 할 때 사용한다.

#### 이 개념을 이해하려면

- const correctness (토픽 7): auto가 const를 떨어뜨리는 규칙
- 포인터 vs 참조 (토픽 2): auto가 참조를 떨어뜨리는 규칙

#### 이 개념이 영향을 주는 것

- perfect forwarding (토픽 17): decltype(auto)와 forwarding reference
- 람다 (토픽 18): 캡처에서 auto와 decltype의 동작

### 3. 내부 동작 / 메커니즘

#### auto의 추론 규칙

auto는 템플릿 인자 추론과 동일한 규칙을 따른다. 핵심: auto는 최상위 const와 참조를 떨어뜨린다.

const int& x = val; auto y = x; →y는 int. const도 참조도 사라진다.

const를 유지하려면 const auto&, auto&, auto&&를 명시적으로 써야 한다.

#### auto가 위험한 실제 사례

```cpp
std::vector<bool> flags = {true, false};
auto val = flags[0];  // val은 bool이 아니라 프록시 객체!
// vector<bool>은 특수화되어 operator[]가 프록시를 반환한다
```

프록시 타입을 반환하는 API(vector<bool>, expression template 등)에서 auto는 예상과 다른 타입을 추론한다. 이런 경우 명시적 타입이나 static_cast로 의도를 드러내야 한다.

### 4. 자주 틀리는 포인트

- auto가 const와 참조를 보존한다고 착각한다 →auto는 최상위 const와 참조를 떨어뜨린다. const auto&를 명시해야 한다.
- 중요한 인터페이스 경계에서 auto를 남용한다 →함수 반환 타입, 클래스 멤버 타입 등 의도가 중요한 곳에서는 명시적 타입이 더 안전하다.
- auto&&가 항상 rvalue 참조라고 오해한다 →auto&&는 forwarding reference다. lvalue를 받으면 lvalue 참조가 된다.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

auto를 쓰는 경우: 이터레이터 타입, 람다 저장, 타입 이름이 길고 명확한 초기화 표현식이 있을 때.

auto를 쓰지 않는 경우: 타입이 의도의 핵심일 때 (float vs double 구분 등), API 경계 (함수 매개변수에는 auto 불가), 프록시 타입을 반환하는 API.

decltype을 쓰는 경우: 표현식의 정확한 타입(참조 포함)이 필요할 때. 주로 메타프로그래밍과 perfect forwarding에서 사용.

### 6. 코드 예시 (bad →good)

#### Bad: const 손실

```cpp
const std::string& GetName() const;
auto name = GetName();  // string 복사 발생! const& 사라짐
```

#### Good: const auto& 명시

```cpp
const auto& name = GetName();  // 복사 없음, const 유지
```

### 7. 판단 기준 요약

- auto는 편의 도구지 만능이 아니다 — const와 참조를 명시적으로 붙여라
- 타입이 의도를 전달하는 곳에서는 명시적 타입을 써라
- auto&&는 forwarding reference — rvalue 참조와 혼동하지 마라
- 프록시 타입 반환 API에서는 auto를 피하거나 static_cast를 써라

## 토픽 9: enum vs enum class(이넘 vs 이넘 클래스, 열거형 비교)

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 없음

### 1. 문제 상황

핵심 용어: `스코프 격리`, `기저 타입`, `스코프(scope)`, `암시적(implicit)`.

게임 코드에는 상태(State), 타입(Type), 플래그(Flag) 같은 열거 표현이 많다. 일반 enum을 쓰면 열거자가 바깥 스코프로 새어나와 이름 충돌이 발생한다.

또한 일반 enum은 암시적으로 int로 변환되어 무관한 enum 값끼리 비교해도 컴파일 에러가 나지 않는다. 이런 타입 구멍이 논리 버그의 원인이 된다.

### 2. 핵심 개념

#### 정의

enum(열거형)은 정수 상수의 이름 붙은 집합이다. enum class(범위 있는 열거형)는 C++11에서 추가된 타입 안전한 열거형이다.

enum class는 두 가지를 강제한다: (1) 열거자에 스코프가 있어 EColor::Red처럼 접근해야 한다 (2) int로 암시적 변환이 안 된다.

underlying type(기저 타입)을 명시할 수 있다. enum class EState : uint8 { ... }로 크기를 제어하면 메모리와 직렬화에서 유리하다.

#### 이 개념을 이해하려면

- 정수 타입과 암시적 변환의 위험성

#### 이 개념이 영향을 주는 것

- UE5 UENUM 매크로: 블루프린트 노출 열거형은 enum class + uint8 기저 타입 권장
- 직렬화/네트워크: 기저 타입 크기가 패킷 크기에 영향

### 3. 내부 동작 / 메커니즘

일반 enum의 열거자는 선언된 스코프에 바로 노출된다. enum Color { Red }; 후 Red만으로 접근 가능하다. 다른 enum에도 Red가 있으면 충돌한다.

enum class의 열거자는 enum 이름 안에 격리된다. Color::Red로만 접근 가능하다. int로의 암시적 변환도 차단되어 static_cast가 필요하다.

underlying type을 지정하지 않으면 enum class는 int(4바이트), 일반 enum은 컴파일러가 결정한다. 크기가 중요한 경우 : uint8 등으로 명시하라.

### 4. 자주 틀리는 포인트

- 상태 표현에 일반 enum을 쓴다 →이름 충돌과 타입 구멍. enum class를 기본으로 사용하라.
- underlying type을 지정하지 않는다 →직렬화/네트워크에서 크기가 불확실해진다.
- enum class를 int로 쓰려고 암시적 변환을 기대한다 →static_cast를 명시해야 한다. 이것은 의도적인 안전장치다.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

enum class를 기본으로 사용한다. 이름 충돌 방지 + 타입 안전.

일반 enum은 C 호환 API나 비트 플래그(operator| 오버로드 필요 시)에서 사용을 고려할 수 있다.

UE5에서는 UENUM() + enum class : uint8이 블루프린트 노출의 표준 패턴이다.

### 6. 코드 예시 (bad →good)

#### Bad: 이름 충돌

```cpp
enum Color { Red, Green };
enum TrafficLight { Red, Yellow, Green };  // 에러: Red 재정의
```

#### Good: enum class

```cpp
enum class EColor : uint8 { Red, Green };
enum class ETrafficLight : uint8 { Red, Yellow, Green };  // 충돌 없음
```

### 7. 판단 기준 요약

- 새 코드에서는 enum class가 기본 선택
- underlying type을 명시하라 — 특히 직렬화/네트워크에서 중요
- 일반 enum은 C 호환이나 비트 플래그에서만 고려
- UE5: UENUM() + enum class : uint8이 블루프린트 표준

## 토픽 10: typedef vs using(타입디프 vs 유징, 타입 별칭)

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 8

### 1. 문제 상황

핵심 용어: `템플릿 별칭(template alias)`, `별칭(alias)`, `template alias`, `이름(alias)`.

복잡한 타입 이름(std::unordered_map<FString, TArray<FVector>>)을 반복 사용하면 가독성이 떨어진다. 별칭이 필요하다.

typedef와 using 두 가지 방법이 있지만, 템플릿 별칭은 using만 가능하다. 어떤 상황에서 어떤 것을 쓸지 기준이 필요하다.

### 2. 핵심 개념

#### 정의

typedef와 using은 기존 타입에 별칭(alias)을 부여하는 도구다. 새 타입을 만드는 것이 아니라 이름만 바꾼다.

using은 C++11에서 도입되었고, 가독성이 좋으며 템플릿 별칭을 지원한다. typedef는 레거시 코드에서 주로 보인다.

#### 이 개념을 이해하려면

- 타입 시스템의 기본 개념

#### 이 개념이 영향을 주는 것

- 템플릿 (토픽 19): 템플릿 별칭은 using만 가능
- UE5 코딩 규칙: FName, FString 등 타입 별칭 패턴

### 3. 내부 동작 / 메커니즘

typedef int Score; 와 using Score = int; 는 동일하다. 둘 다 Score를 int의 별칭으로 만든다.

차이가 나는 곳: 템플릿 별칭. template<typename T> using Vec = std::vector<T>; 는 typedef로 불가능하다.

별칭은 새 타입이 아니므로 Score와 int 사이에 암시적 변환이 아닌 동일성이 성립한다. 별칭끼리 실수로 섞여도 컴파일러가 잡지 못한다.

### 4. 자주 틀리는 포인트

- 별칭을 새 타입이라고 착각한다 →using Score = int; 와 int는 완전히 같은 타입이다.
- 의미 없는 짧은 이름을 만든다 →using S = std::string; 은 오히려 가독성을 해친다.
- 함수 포인터에 typedef를 쓴다 →using이 읽기 훨씬 쉽다. using Handler = void(*)(int);

### 5. 언제 쓰는가 / 언제 쓰지 않는가

새 코드에서는 using을 기본으로 사용한다. 가독성과 템플릿 호환성 모두 우위.

typedef는 C 호환 헤더나 레거시 코드에서 유지 목적으로만 사용한다.

별칭 이름은 의미를 담아야 한다. PlayerID, FrameIndex처럼 도메인 의도를 표현하라.

### 6. 코드 예시 (bad →good)

#### Bad: typedef 함수 포인터

```cpp
typedef void (*Handler)(int, float);  // 읽기 어려움
```

#### Good: using 함수 포인터

```cpp
using Handler = void(*)(int, float);  // 왼->오 읽기 가능
```

### 7. 판단 기준 요약

- 새 코드 →using. 레거시/C 호환 →typedef
- 템플릿 별칭 →using만 가능
- 별칭은 새 타입이 아니다 — 타입 안전이 필요하면 강타입 래퍼를 만들어라
- 이름에 도메인 의미를 담아라

## 토픽 11: 캐스팅 4종

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 7, 9

### 1. 문제 상황

핵심 용어: `RTTI`, `reinterpret_cast`, `static_cast`, `dynamic_cast`.

C 스타일 캐스트 (int)val은 어떤 종류의 변환이든 시도한다. const를 벗기면서 동시에 타입을 바꿀 수 있어 의도가 숨는다.

잘못된 캐스팅은 런타임 버그의 주요 원인이다. C++은 4가지 캐스트 연산자로 의도를 명시하도록 설계되었다.

### 2. 핵심 개념

#### 정의

static_cast — 컴파일 타임에 검증 가능한 안전한 변환. 업캐스트, 숫자 타입 변환, void* 변환 등.

dynamic_cast — RTTI를 사용한 런타임 다운캐스트. 실패 시 포인터는 nullptr, 참조는 예외를 던진다.

const_cast — const/volatile 한정자만 추가하거나 제거한다. 원래 const인 객체의 const를 벗기면 UB.

reinterpret_cast — 비트 패턴을 다른 타입으로 재해석한다. strict aliasing 위반 위험이 높다.

#### 이 개념을 이해하려면

- 포인터 vs 참조 (토픽 2): 캐스팅 대상이 주로 포인터/참조
- const correctness (토픽 7): const_cast의 맥락
- OOP / virtual (토픽 14): dynamic_cast의 전제조건

#### 이 개념이 영향을 주는 것

- UE5 캐스팅 (카테고리 2 / 토픽 14): Cast<T>는 dynamic_cast의 UE5 최적화 버전
- 메모리 레이아웃 (토픽 6): reinterpret_cast와 strict aliasing

### 3. 내부 동작 / 메커니즘

#### 캐스트별 비용

- static_cast: 컴파일 타임. 런타임 비용 없음 (숫자 변환은 명령어 1~2개).
- dynamic_cast: RTTI 테이블 조회. 상속 깊이에 비례하는 비용. 게임 엔진에서는 비싸다고 판단하여 자체 캐스팅을 구현하는 경우가 많다.
- const_cast: 런타임 비용 없음. 포인터/참조의 타입 정보만 변경.
- reinterpret_cast: 런타임 비용 없음. 비트 패턴을 그대로 재해석.

#### narrowing conversion

큰 타입에서 작은 타입으로 변환 시 값이 잘릴 수 있다. double →int, int64 →int32 등. 중괄호 초기화({})는 narrowing을 컴파일 에러로 잡아준다.

static_cast<int>(doubleVal)은 의도적 narrowing을 명시한다. 무의식적 narrowing이 진짜 위험이다.

### 4. 자주 틀리는 포인트

- C 스타일 캐스트를 사용한다 →어떤 캐스트가 적용되는지 불명확하다. C++ 캐스트를 써서 의도를 드러내라.
- dynamic_cast를 nullptr 체크 없이 사용한다 →다운캐스트 실패 시 nullptr. 반드시 검사해야 한다.
- reinterpret_cast로 무관한 타입 간 변환한다 →strict aliasing 위반은 UB. memcpy를 사용하라.
- narrowing 가능성을 무시한다 →double을 int에 넣을 때 데이터 손실 여부를 확인하라.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

static_cast: 가장 자주 사용. 안전한 변환의 기본 선택.

dynamic_cast: 다형적 다운캐스트가 필요할 때. 비용이 크므로 게임 코드에서는 최소화. UE5에서는 Cast<T>를 대신 사용.

const_cast: 레거시 API가 const를 받지 않을 때 호환용으로만.

reinterpret_cast: 하드웨어 레지스터 매핑, 메모리 주소 직접 조작 등 극히 제한적.

선택 흐름: (1) static_cast로 가능한가? →사용 (2) 다운캐스트가 필요한가? →dynamic_cast(또는 UE5 Cast<T>) (3) const만 바꾸면 되는가? →const_cast (4) 비트 재해석이 불가피한가? →reinterpret_cast(최후의 수단)

### 6. 코드 예시 (bad →good)

#### Bad: C 스타일 캐스트

```cpp
Base* b = GetObject();
Derived* d = (Derived*)b;  // 어떤 캐스트인지 불명확
```

#### Good: static_cast 또는 dynamic_cast

```cpp
Derived* d = dynamic_cast<Derived*>(b);
if (d) { /* 안전한 사용 */ }
```

### 7. 판단 기준 요약

- C 스타일 캐스트를 쓰지 마라 — 항상 C++ 캐스트로 의도를 드러내라
- 기본 선택은 static_cast
- dynamic_cast는 비용을 인식하고, UE5에서는 Cast<T>를 사용하라
- reinterpret_cast는 최후의 수단
- narrowing이 의도적이면 static_cast로 명시하고, 아니면 중괄호 초기화로 잡아라
