## 토픽 7: const correctness(콘스트 코렉트니스, const 일관성)

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 없음

### 1. 핵심 개념

핵심 용어: `API 계약`, `왼쪽`, `const int*`, `mutable`.

`const`는 "이 값은 안 바꾼다"는 약속을 컴파일러가 대신 확인해 주는 장치다. `const correctness(콘스트 코렉트니스, const 일관성)`는 이 약속을 변수, 포인터, 참조, 매개변수, 반환값, 멤버 함수까지 일관되게 적용하는 습관을 말한다. 쉽게 말해 "읽기 전용이면 타입에서도 읽기 전용이라고 드러내자"는 규칙이다.

핵심은 `const`가 붙는 위치에 따라 의미가 달라진다는 점이다. 값 앞에 붙으면 그 값 자체를 못 바꾸고, 참조나 매개변수에 붙으면 "읽기 전용으로 받겠다"는 뜻이 된다. 포인터에서 `*` 왼쪽에 붙으면 "가리키는 대상"을 못 바꾸며, `*` 오른쪽에 붙으면 "포인터 자신"을 못 바꾼다. 멤버 함수 뒤의 `const`는 "이 함수는 이 객체의 논리적 상태를 바꾸지 않는다"는 뜻이다.

반환값에서도 해석이 조금 다르다. `const T&`나 `const T*`를 반환하면 호출자에게 "읽기 전용으로만 보라"는 의도를 전달할 수 있지만, `const T`를 값으로 반환하는 것은 실익이 크지 않은 경우가 많다. 값으로 복사되어 나간 객체는 호출자 쪽의 별도 값이기 때문에, 보통은 참조/포인터 반환에서 `const` 의미가 더 중요하다.

또 하나 중요한 개념은 `비트 수준 상수성(bitwise constness, 비트 단위 불변)`과 `논리적 상수성(logical constness, 의미상 불변)`의 차이다. C++ 컴파일러는 기본적으로 비트 수준 상수성을 기준으로 검사하지만, 실무에서는 캐시처럼 외부에서 보이는 의미가 안 바뀌는 경우를 논리적 상수성으로 다루기도 한다. 이때 필요한 도구가 `mutable(뮤터블, const 함수 안에서 바꿀 수 있는 예외 멤버)`이다.

### 2. 코드 예시

```cpp
void PrintName(FString Name) {
    UE_LOG(LogTemp, Log, TEXT("%s"), *Name);
}
```

FString을 값으로 받아 불필요한 복사를 만든다.
읽기 전용 입력이라면 const 참조가 더 명확하고 효율적이다.

### 3. 왜 중요한가 (게임/UE5 맥락)

큰 프로젝트에서는 "이 함수가 값을 바꾸는가?"를 매번 구현체로 확인하면 생산성이 급격히 떨어진다. `const`를 잘 붙이면 API만 봐도 읽기 전용 함수와 상태 변경 함수를 빠르게 구분할 수 있어 코드 리뷰, 디버깅, 협업이 쉬워진다.

### 4. 면접 예상 Q&A

**Q1: const correctness는 단순 문법이 아니라 어떤 설계 도구인가?**
> A: 함수가 무엇을 바꾸지 않는지 타입 차원에서 약속해 API 신뢰성과 재사용성을 높이는 설계 도구다.

**Q2: const correctness를 한 문장으로 설명하라.**
> A: 변하지 않아야 할 값과 인터페이스에 const를 붙여 컴파일러가 실수를 잡도록 하는 원칙이다.

**Q3: `const correctness(콘스트 코렉트니스, const 일관성)`를 한 문장으로 설명해 보세요.**
> A: 객체나 값을 수정하지 않겠다는 약속을 타입에 일관되게 표현하는 습관이라고 설명하면 된다. 이렇게 하면 컴파일러가 잘못된 수정 시도를 잡아 주고, API 의도도 더 분명해진다. 결국 읽기 전용과 수정 가능 경계를 코드 차원에서 강제하는 것이다.

**Q4: `const int*`, `int* const`, `const int* const`는 어떻게 읽나요?**
> A: `const int*`는 "상수 int를 가리키는 포인터", `int* const`는 "int를 가리키는 상수 포인터", `const int* const`는 "상수 int를 가리키는 상수 포인터"다. 결론적으로 `*` 왼쪽 const는 대상, 오른쪽 const는 포인터 자신에 붙는다고 기억하면 된다.

### 5. 흔한 실수

- `const`를 문법 장식처럼만 보고, 읽기 전용 계약이라는 의미를 놓친다.
- 포인터에서 `const` 위치를 반대로 읽어 대상과 포인터 자신을 혼동한다.

## 토픽 8: auto / decltype(오토 / 디클타입, 타입 추론)

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 7

### 1. 핵심 개념

핵심 용어: `decltype(auto)`, `forwarding reference`, `추론`, `타입`.

`auto(오토, 자동 타입 추론)`와 `decltype(디클타입, 표현식 타입 추출)`은 컴파일러가 타입을 추론하도록 돕는 도구다. `auto`는 초기값에서 타입을 추론하고, `decltype`은 표현식의 타입을 그대로 읽어 온다. 핵심은 편의성보다 "의미를 흐리지 않는 범위에서 타입 표기를 줄이는 것"이다.

`auto`는 특히 긴 iterator(이터레이터, 반복자) 타입이나 템플릿 반환형에서 유용하지만, 참조와 const가 어떻게 유지되거나 제거되는지 알아야 한다. `decltype(auto)`는 표현식의 참조성까지 보존하므로 더 강력하지만 더 조심해서 써야 한다. 여기에 `auto&&`는 항상 rvalue 참조가 아니라 forwarding reference 문맥에서 값 범주를 보존하는 도구라는 점도 자주 헷갈린다.

또한 modern C++에서는 trailing return type, `inline` 함수, `decltype(auto)` 반환을 함께 보는 경우가 많다. 이때 핵심은 "문법이 짧아졌는가"가 아니라 "타입 의도와 평가 시점을 더 명확하게 만들었는가"다. 타입 추론과 컴파일 타임 계산을 남용하면 런타임 코드는 짧아져도 빌드 시간과 에러 메시지 비용이 커질 수 있다.

### 2. 코드 예시

```cpp
const int& ref = 42;
auto a = ref;
```

auto가 참조와 최상위 const를 벗겨 a는 int가 된다.
원본의 읽기 전용 참조 성질을 유지하지 못한다.

### 3. 왜 중요한가 (게임/UE5 맥락)

템플릿과 컨테이너를 많이 쓰는 코드베이스에서는 타입 이름이 길어지기 쉽다. 이때 `auto`를 적절히 쓰면 가독성이 좋아지지만, 의미가 모호한 위치까지 퍼지면 오히려 디버깅이 어려워진다.

### 4. 면접 예상 Q&A

**Q1: auto와 decltype을 쓸 때 핵심 기준은 무엇인가?**
> A: 초기화식이 타입을 충분히 설명하면 auto, 표현식의 정확한 타입 규칙을 그대로 가져와야 하면 decltype을 쓴다.

**Q2: auto의 장점과 단점을 설명하라.**
> A: 장점: 긴 타입을 줄여 가독성 향상, 리팩토링 용이. 단점: top-level const/참조 제거로 의도치 않은 복사 가능, 도메인 타입이 가려질 수 있음.

**Q3: `auto(오토, 자동 타입 추론)`의 가장 큰 장점은 무엇인가요?**
> A: 긴 타입 표기를 줄이고 리팩터링에 강한 코드를 만들 수 있다는 점이다. 타입이 바뀌어도 선언부를 여러 군데 고칠 필요가 줄어든다. 특히 템플릿 코드와 컨테이너 반복자에서 장점이 크다.

**Q4: `auto`가 항상 좋은 것은 아닌 이유는 무엇인가요?**
> A: 코드 읽는 사람이 실제 타입을 추론하기 어렵다면 오히려 의미를 숨기기 때문이다. 특히 숫자 리터럴이나 중요한 도메인 타입에서는 명시가 더 낫다. 편의성보다 전달력이 우선이다.

### 5. 흔한 실수

- `auto`가 const와 참조를 언제 지우는지 모르고 쓴다.
- 중요한 도메인 타입까지 전부 `auto`로 숨긴다.

## 토픽 9: enum vs enum class(이넘 vs 이넘 클래스, 열거형 비교)

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 없음

### 1. 핵심 개념

핵심 용어: `스코프 격리`, `기저 타입`, `스코프(scope)`, `암시적(implicit)`.

`enum(이넘, 전통적인 열거형)`은 전통적인 열거형이고, `enum class(이넘 클래스, 범위가 있는 열거형)`는 범위와 타입 안전성을 강화한 열거형이다. 일반 `enum`은 암시적 정수 변환이 일어나고 이름이 바깥 스코프에 노출되지만, `enum class`는 `Type::Value` 형태로 써야 하고 정수와 암시적으로 섞이지 않는다.

또한 `enum class : uint8`처럼 `underlying type(언더라잉 타입, 기반형)`을 명시할 수 있다. 이는 메모리 크기 통제와 직렬화, 네트워크 전송, UE 블루프린트 호환에서 중요할 수 있다. 비트 플래그 패턴은 `enum class`에 연산자를 추가해 더 안전하게 구현할 수 있다.

### 2. 코드 예시

```cpp
enum Color { Red, Green };
enum TrafficLight { Red, Yellow };
```

전통 enum은 열거자 이름을 바깥 스코프로 내보내므로 Red가 충돌한다.

### 3. 왜 중요한가 (게임/UE5 맥락)

상태 표현은 게임 코드 전역에 퍼진다. `enum class`를 이해하면 타입 안전성을 높이고, 블루프린트와 네트워크에서 필요한 크기 제어도 더 의식적으로 설계할 수 있다.

### 4. 면접 예상 Q&A

**Q1: enum class가 일반 enum보다 안전한 이유는 무엇인가?**
> A: 이름 공간 오염과 암묵적 정수 변환을 막아 타입 경계를 분명히 하기 때문이다.

**Q2: enum class가 전통 enum보다 안전한 이유를 설명하라.**
> A: enum class는 자체 스코프를 가져 이름 충돌을 방지하고, 정수 암시 변환을 차단한다.

**Q3: `enum class(이넘 클래스, 범위가 있는 열거형)`가 일반 `enum`보다 더 안전한 이유는 무엇인가요?**
> A: 이름 오염을 줄이고 정수와의 암시적 변환을 막기 때문이다. 결론적으로 잘못된 값 혼입을 컴파일 단계에서 더 많이 막아 준다. 큰 코드베이스일수록 이 차이가 눈에 띈다.

**Q4: `underlying type(언더라잉 타입, 기반형)`을 명시하는 이유는 무엇인가요?**
> A: 메모리 크기와 표현 범위를 통제하기 위해서다. `uint8`로 고정하면 직렬화와 네트워크 전송에서 의도가 더 분명해질 수 있다. UE의 리플렉션이나 블루프린트 호환에서도 자주 고려된다.

### 5. 흔한 실수

- 상태 표현에 일반 `enum`을 습관적으로 써 정수와 섞이게 둔다.
- underlying type이 필요한 상황인데 크기를 명시하지 않는다.

## 토픽 10: typedef vs using(타입디프 vs 유징, 타입 별칭)

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 8

### 1. 핵심 개념

핵심 용어: `템플릿 별칭(template alias)`, `별칭(alias)`, `template alias`, `이름(alias)`.

`typedef(타입디프, 타입 별칭 선언)`와 `using(유징, 현대식 타입 별칭)`은 별칭(alias, 앨리어스, 다른 이름)을 만드는 도구다. 둘 다 기존 타입에 다른 이름을 붙이지만, modern C++에서는 `using`이 더 읽기 쉽고 template alias(템플릿 앨리어스, 템플릿 별칭)까지 지원해 사실상 기본 선택이 됐다.

핵심은 별칭이 새 타입을 만드는 것이 아니라는 점이다. 즉 타입 이름을 바꾸는 것일 뿐, 타입 자체의 규칙은 바뀌지 않는다. 또한 별칭은 복잡한 함수 포인터, 템플릿 타입, 정책 타입을 더 읽기 쉽게 만드는 데 유용하다.

### 2. 코드 예시

```cpp
template<typename T>
typedef std::vector<T> Vec;
```

typedef는 템플릿 별칭을 직접 지원하지 않는다.
이 경우 using을 써야 한다.

### 3. 왜 중요한가 (게임/UE5 맥락)

컨테이너, 함수 포인터, 정책 타입이 많은 코드베이스에서는 타입 이름이 길어지기 쉽다. 별칭을 잘 쓰면 코드 읽기가 쉬워지지만, 의미 없는 축약만 늘리면 오히려 도메인 이해가 떨어진다.

### 4. 면접 예상 Q&A

**Q1: 현대 C++에서 using을 더 선호하는 이유는 무엇인가?**
> A: 별칭 템플릿을 지원하고 읽는 방향이 자연스러워 복잡한 타입을 더 명확하게 표현할 수 있기 때문이다.

**Q2: typedef 대신 using을 권장하는 이유는?**
> A: using은 가독성이 좋고, template alias를 지원하며, Modern C++ 스타일에 부합한다.

**Q3: `using(유징, 현대식 타입 별칭)`이 `typedef`보다 자주 권장되는 이유는 무엇인가요?**
> A: 문법이 더 읽기 쉽고 template alias를 지원하기 때문이다. 특히 복잡한 타입일수록 왼쪽이 별칭 이름이라는 점이 더 분명하다. modern C++ 문맥에서는 사실상 기본 선택처럼 쓰인다.

**Q4: 별칭(alias, 앨리어스)은 새 타입을 만드는 것인가요?**
> A: 아니다. 기존 타입에 다른 이름을 붙일 뿐이다. 타입 안정성을 새로 만들고 싶다면 wrapper struct나 class가 필요하다. 별칭과 새 타입 생성은 구분해야 한다.

### 5. 흔한 실수

- alias를 새 타입처럼 오해한다.
- 의미 없는 축약 이름을 남발한다.

## 토픽 11: 캐스팅 4종

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 7, 9

### 1. 핵심 개념

핵심 용어: `RTTI`, `reinterpret_cast`, `static_cast`, `dynamic_cast`.

C++의 명시적 캐스팅은 `static_cast(스태틱 캐스트, 정적 변환)`, `dynamic_cast(다이내믹 캐스트, 런타임 타입 변환)`, `const_cast(콘스트 캐스트, const 제거/추가)`, `reinterpret_cast(리인터프리트 캐스트, 비트 해석 변환)` 네 가지로 나뉜다. 이 구분의 핵심은 "무슨 종류의 변환을 왜 하려는가"를 분명히 드러내는 것이다. 반대로 C-style cast(씨 스타일 캐스트, 옛 문법 캐스팅)는 여러 위험한 변환을 한 문법에 숨겨 의도를 흐린다.

또한 `narrowing conversion(내로잉 컨버전, 축소 변환)`도 함께 봐야 한다. 예를 들어 큰 정수를 작은 타입으로 줄이거나 부동소수점을 정수로 바꾸는 것은 값 손실 가능성이 있다. `{}` 초기화는 이런 narrowing을 더 엄격히 막아 주는 편이다.

### 2. 코드 예시

```cpp
struct Base { virtual ~Base() = default; };
struct Derived : Base {};

Base* b = new Base();
Derived* d = static_cast<Derived*>(b);
```

문법상 상속 관계가 있어 컴파일은 되지만, 런타임 실제 객체는 Base다.
잘못된 다운캐스트로 UB가 된다.

### 3. 왜 중요한가 (게임/UE5 맥락)

타입 변환은 자주 일어나지만, 잘못된 캐스팅은 가장 위험한 버그 중 하나다. 의도를 드러내는 캐스팅을 쓰면 리뷰와 디버깅이 쉬워지고, 위험한 변환을 더 빨리 발견할 수 있다.

### 4. 면접 예상 Q&A

**Q1: C++ 캐스팅 4종을 구분하는 기준은 무엇인가?**
> A: 의도가 무엇인지에 따라 쓴다: const 제거는 const_cast, 계층 변환은 dynamic_cast, 일반 변환은 static_cast, 비트 수준 재해석은 reinterpret_cast다.

**Q2: static_cast와 dynamic_cast의 차이를 설명하라.**
> A: static_cast는 컴파일 타임 규칙 기반이고, dynamic_cast는 런타임 타입 검사를 수행한다.

**Q3: const_cast를 정당하게 사용할 수 있는 상황은?**
> A: const를 지원하지 않는 레거시 API에 const 객체를 전달해야 할 때이다.

**Q4: `static_cast(스태틱 캐스트, 정적 변환)`와 `dynamic_cast(다이내믹 캐스트, 런타임 타입 변환)`의 차이는 무엇인가요?**
> A: `static_cast`는 컴파일 타임 규칙 기반 변환이고, `dynamic_cast`는 런타임 타입 검사를 동반한다. 다형 계층에서 안전성 확인이 필요하면 `dynamic_cast`가 맞다. 성능과 안전성 trade-off를 함께 봐야 한다.

### 5. 흔한 실수

- C-style cast로 위험한 변환을 숨긴다.
- narrowing 가능성을 무시한 채 숫자 타입을 섞는다.
