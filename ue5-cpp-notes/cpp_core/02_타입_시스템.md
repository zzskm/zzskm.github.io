## 토픽 7: const correctness(콘스트 코렉트니스, const 일관성)

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 없음

### 1. 핵심 개념

`const`는 "이 값은 안 바꾼다"는 약속을 컴파일러가 대신 확인해 주는 장치다. `const correctness(콘스트 코렉트니스, const 일관성)`는 이 약속을 변수, 포인터, 참조, 매개변수, 반환값, 멤버 함수까지 일관되게 적용하는 습관을 말한다. 쉽게 말해 "읽기 전용이면 타입에서도 읽기 전용이라고 드러내자"는 규칙이다.

핵심은 `const`가 붙는 위치에 따라 의미가 달라진다는 점이다. 값 앞에 붙으면 그 값 자체를 못 바꾸고, 참조나 매개변수에 붙으면 "읽기 전용으로 받겠다"는 뜻이 된다. 포인터에서 `*` 왼쪽에 붙으면 "가리키는 대상"을 못 바꾸며, `*` 오른쪽에 붙으면 "포인터 자신"을 못 바꾼다. 멤버 함수 뒤의 `const`는 "이 함수는 이 객체의 논리적 상태를 바꾸지 않는다"는 뜻이다.

반환값에서도 해석이 조금 다르다. `const T&`나 `const T*`를 반환하면 호출자에게 "읽기 전용으로만 보라"는 의도를 전달할 수 있지만, `const T`를 값으로 반환하는 것은 실익이 크지 않은 경우가 많다. 값으로 복사되어 나간 객체는 호출자 쪽의 별도 값이기 때문에, 보통은 참조/포인터 반환에서 `const` 의미가 더 중요하다.

또 하나 중요한 개념은 `비트 수준 상수성(bitwise constness, 비트 단위 불변)`과 `논리적 상수성(logical constness, 의미상 불변)`의 차이다. C++ 컴파일러는 기본적으로 비트 수준 상수성을 기준으로 검사하지만, 실무에서는 캐시처럼 외부에서 보이는 의미가 안 바뀌는 경우를 논리적 상수성으로 다루기도 한다. 이때 필요한 도구가 `mutable(뮤터블, const 함수 안에서 바꿀 수 있는 예외 멤버)`이다.

`constexpr(컨스프렉스프, 컴파일 타임 상수 가능)`와 `consteval(컨스트이벌, 반드시 컴파일 타임 평가)`도 이 축에서 함께 이해하면 좋다. `const`가 "수정 금지 계약"에 가깝다면, `constexpr`과 `consteval`은 "컴파일 타임에 계산하거나 검증할 수 있는가"에 더 가깝다. modern C++에서는 헤더에 공유 상수와 짧은 함수를 둘 때 `inline constexpr` 패턴도 자주 등장한다.

**읽는 법 요약**
- `const int Value` : 상수 정수 값
- `const int& Ref` : 읽기 전용 정수 참조
- `const int* Ptr` : "상수 정수를 가리키는 포인터"
- `int* const Ptr` : "정수를 가리키는 상수 포인터"
- `const int* const Ptr` : "상수 정수를 가리키는 상수 포인터"
- `const int* GetPtr() const` : "읽기 전용 int를 가리키는 포인터를 반환하고, 이 함수는 객체도 바꾸지 않는다"

### 2. 코드 예시

```cpp
void ReadOnly(const int& Value)
{
    std::cout << Value << "\n";
}

void PointerExamples()
{
    int Number = 10;

    const int* PtrToConst = &Number;   // ← 가리키는 값 수정 불가
    int* const ConstPtr = &Number;     // ← 포인터 재대입 불가
    const int* const FixedBoth = &Number; // ← 둘 다 불가

    // *PtrToConst = 20;   // ❌ 값 수정 불가
    *ConstPtr = 20;        // ✅ 값 수정 가능
    // ConstPtr = nullptr; // ❌ 포인터 재대입 불가
}

class Player
{
public:
    int GetHealth() const
    {
        return Health;
    }

    int GetCachedPower() const
    {
        if (!bCached)
        {
            CachedPower = Health * 2;  // ← 논리적 상수성을 위한 캐시
            bCached = true;
        }
        return CachedPower;
    }

private:
    int Health = 100;
    mutable int CachedPower = 0;
    mutable bool bCached = false;
};

constexpr int Square(int Value)
{
    return Value * Value;
}

consteval int FixedBufferSize()
{
    return 256;
}

inline constexpr int MaxPlayers = 4;

std::array<int, FixedBufferSize()> Buffer{};
```

### 3. 왜 중요한가 (게임/UE5 맥락)

큰 프로젝트에서는 "이 함수가 값을 바꾸는가?"를 매번 구현체로 확인하면 생산성이 급격히 떨어진다. `const`를 잘 붙이면 API만 봐도 읽기 전용 함수와 상태 변경 함수를 빠르게 구분할 수 있어 코드 리뷰, 디버깅, 협업이 쉬워진다.

### 4. 면접 예상 Q&A

**Q1: `const correctness(콘스트 코렉트니스, const 일관성)`를 한 문장으로 설명해 보세요.**
> A: 객체나 값을 수정하지 않겠다는 약속을 타입에 일관되게 표현하는 습관이라고 설명하면 된다. 이렇게 하면 컴파일러가 잘못된 수정 시도를 잡아 주고, API 의도도 더 분명해진다. 결국 읽기 전용과 수정 가능 경계를 코드 차원에서 강제하는 것이다.

**Q2: `const int*`, `int* const`, `const int* const`는 어떻게 읽나요?**
> A: `const int*`는 "상수 int를 가리키는 포인터", `int* const`는 "int를 가리키는 상수 포인터", `const int* const`는 "상수 int를 가리키는 상수 포인터"다. 결론적으로 `*` 왼쪽 const는 대상, 오른쪽 const는 포인터 자신에 붙는다고 기억하면 된다.

**Q3: 멤버 함수 뒤의 `const`는 무엇을 의미하나요?**
> A: 그 함수가 객체의 논리적 상태를 바꾸지 않는다는 뜻이다. 그래서 `const` 객체에서도 호출할 수 있다. 읽기 전용 인터페이스를 분명히 한다는 점에서 매우 중요하다.

**Q4: `mutable(뮤터블, const 예외 멤버)`은 왜 필요한가요?**
> A: 외부에서 보이는 의미는 그대로인데 내부 캐시나 통계값만 갱신해야 하는 경우가 있기 때문이다. 이런 상황을 `논리적 상수성(logical constness, 의미상 불변)`으로 설명할 수 있다. 다만 진짜 상태 변경을 숨기는 용도로 남용하면 안 된다.

**Q5: `비트 수준 상수성(bitwise constness)`과 `논리적 상수성(logical constness)`의 차이는 무엇인가요?**
> A: 비트 수준 상수성은 객체 내부 비트가 하나도 바뀌면 안 된다는 관점이고, 논리적 상수성은 사용자 입장에서 의미가 안 바뀌면 내부 캐시는 바뀌어도 괜찮다는 관점이다. C++ 컴파일러는 기본적으로 앞쪽을 검사하지만, 실무 설계는 뒤쪽 개념까지 함께 본다.

**Q6: 함수 인자에 `const T&`를 자주 쓰는 이유는 무엇인가요?**
> A: 큰 객체를 복사하지 않으면서도 수정하지 않겠다는 의도를 분명히 할 수 있기 때문이다. 성능과 인터페이스 안정성을 동시에 챙길 수 있다. 그래서 읽기 전용 입력의 기본 패턴처럼 자주 쓰인다.

**Q7: 반환 타입에 붙는 `const`는 어떻게 해석해야 하나요?**
> A: `const T&`나 `const T*` 반환은 호출자가 결과를 읽기 전용으로 다루게 만드는 의미가 크다. 반면 `const T`를 값으로 반환하는 것은 복사본 자체가 넘어가므로 실익이 작거나 오히려 사용성을 떨어뜨릴 수 있다. 결론적으로 반환값 `const`는 참조/포인터에서 특히 중요하다고 보는 편이 맞다.

**Q8: `constexpr`을 사용하는 이유는 무엇인가요?**
> A: 가능한 계산을 compile-time으로 끌어올려 상수 표현식 제약을 만족하고, 런타임 실수를 더 일찍 잡기 위해서다. 배열 크기, lookup table, 정책 값처럼 "값이 바뀌면 안 되고 미리 검증되면 좋은 것"에 특히 잘 맞는다. 다만 모든 계산을 컴파일 타임으로 밀어 올리면 빌드 시간이 늘 수 있어 compile-time vs runtime tradeoff도 함께 봐야 한다.

### 5. 흔한 실수

- `const`를 문법 장식처럼만 보고, 읽기 전용 계약이라는 의미를 놓친다.
- 포인터에서 `const` 위치를 반대로 읽어 대상과 포인터 자신을 혼동한다.
- `mutable`로 실제 상태 변경을 숨겨 `const` 멤버 함수 의미를 무너뜨린다.
- `constexpr`, `consteval`, `const`를 같은 개념으로 뭉뚱그려 수정 금지와 컴파일 타임 평가를 구분하지 않는다.

## 토픽 8: auto / decltype(오토 / 디클타입, 타입 추론)

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 7

### 1. 핵심 개념

`auto(오토, 자동 타입 추론)`와 `decltype(디클타입, 표현식 타입 추출)`은 컴파일러가 타입을 추론하도록 돕는 도구다. `auto`는 초기값에서 타입을 추론하고, `decltype`은 표현식의 타입을 그대로 읽어 온다. 핵심은 편의성보다 "의미를 흐리지 않는 범위에서 타입 표기를 줄이는 것"이다.

`auto`는 특히 긴 iterator(이터레이터, 반복자) 타입이나 템플릿 반환형에서 유용하지만, 참조와 const가 어떻게 유지되거나 제거되는지 알아야 한다. `decltype(auto)`는 표현식의 참조성까지 보존하므로 더 강력하지만 더 조심해서 써야 한다. 여기에 `auto&&`는 항상 rvalue 참조가 아니라 forwarding reference 문맥에서 값 범주를 보존하는 도구라는 점도 자주 헷갈린다.

또한 modern C++에서는 trailing return type, `inline` 함수, `decltype(auto)` 반환을 함께 보는 경우가 많다. 이때 핵심은 "문법이 짧아졌는가"가 아니라 "타입 의도와 평가 시점을 더 명확하게 만들었는가"다. 타입 추론과 컴파일 타임 계산을 남용하면 런타임 코드는 짧아져도 빌드 시간과 에러 메시지 비용이 커질 수 있다.

### 2. 코드 예시

```cpp
std::vector<int> Values = {1, 2, 3};

auto It = Values.begin();
decltype(Values.size()) Count = Values.size();

int Value = 10;
auto Copy = Value;          // int
auto& Ref = Value;          // int&
const auto& ConstRef = Value;

auto Forward = [](auto&& Item) -> decltype(auto)
{
    return std::forward<decltype(Item)>(Item);
};
```

### 3. 왜 중요한가 (게임/UE5 맥락)

템플릿과 컨테이너를 많이 쓰는 코드베이스에서는 타입 이름이 길어지기 쉽다. 이때 `auto`를 적절히 쓰면 가독성이 좋아지지만, 의미가 모호한 위치까지 퍼지면 오히려 디버깅이 어려워진다.

### 4. 면접 예상 Q&A

**Q1: `auto(오토, 자동 타입 추론)`의 가장 큰 장점은 무엇인가요?**
> A: 긴 타입 표기를 줄이고 리팩터링에 강한 코드를 만들 수 있다는 점이다. 타입이 바뀌어도 선언부를 여러 군데 고칠 필요가 줄어든다. 특히 템플릿 코드와 컨테이너 반복자에서 장점이 크다.

**Q2: `auto`가 항상 좋은 것은 아닌 이유는 무엇인가요?**
> A: 코드 읽는 사람이 실제 타입을 추론하기 어렵다면 오히려 의미를 숨기기 때문이다. 특히 숫자 리터럴이나 중요한 도메인 타입에서는 명시가 더 낫다. 편의성보다 전달력이 우선이다.

**Q3: `auto`가 참조와 const를 어떻게 다루나요?**
> A: 기본 `auto`는 값으로 추론하면서 top-level const와 참조를 버릴 수 있다. 참조성을 유지하려면 `auto&`, `const auto&`, `auto&&`처럼 명시해야 한다. 이 차이를 모르면 의도치 않은 복사가 생긴다.

**Q4: `decltype(디클타입, 표현식 타입 추출)`은 언제 유용한가요?**
> A: 표현식 타입을 정확히 따라가야 할 때 유용하다. 템플릿 반환형이나 멤버 타입을 일반화할 때 자주 쓰인다. 단순 별칭용보다는 "이 식의 타입을 그대로 가져오자"는 상황에서 특히 빛난다.

**Q5: `decltype(auto)`는 왜 조심해야 하나요?**
> A: 값/참조 여부를 표현식 그대로 따라가기 때문이다. 편리하지만 의도하지 않은 참조 반환을 만들 수 있다. 반환하는 식이 lvalue인지 rvalue인지 항상 같이 봐야 한다.

**Q6: 실무에서 타입 추론을 잘 쓰는 기준은 무엇인가요?**
> A: 오른쪽 값을 보면 타입이 분명하거나, 타입 이름이 정보보다 소음일 때 쓰는 것이다. 반대로 타입 정보 자체가 중요한 문맥이면 명시형이 더 낫다. 즉 "짧아지는가"보다 "읽기 쉬워지는가"가 기준이다.

**Q7: compile-time vs runtime tradeoff를 타입 추론/상수 계산 관점에서 어떻게 보나요?**
> A: `constexpr`, `consteval`, 템플릿 기반 계산은 런타임 분기와 비용을 줄일 수 있지만, 빌드 시간과 에러 메시지 복잡도를 늘릴 수 있다. 반대로 런타임 계산은 유연하지만 실수를 늦게 발견할 수 있다. 따라서 바뀌지 않는 정책값과 검증은 compile-time으로, 입력 데이터에 따른 로직은 runtime으로 두는 식의 균형이 중요하다.

### 5. 흔한 실수

- `auto`가 const와 참조를 언제 지우는지 모르고 쓴다.
- 중요한 도메인 타입까지 전부 `auto`로 숨긴다.
- `decltype(auto)`를 반환형에 쓰고 참조 수명을 고려하지 않는다.
- `auto&&`를 무조건 rvalue 참조라고 오해해 forwarding reference 문맥을 잘못 설명한다.

## 토픽 9: enum vs enum class(이넘 vs 이넘 클래스, 열거형 비교)

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 없음

### 1. 핵심 개념

`enum(이넘, 전통적인 열거형)`은 전통적인 열거형이고, `enum class(이넘 클래스, 범위가 있는 열거형)`는 범위와 타입 안전성을 강화한 열거형이다. 일반 `enum`은 암시적 정수 변환이 일어나고 이름이 바깥 스코프에 노출되지만, `enum class`는 `Type::Value` 형태로 써야 하고 정수와 암시적으로 섞이지 않는다.

또한 `enum class : uint8`처럼 `underlying type(언더라잉 타입, 기반형)`을 명시할 수 있다. 이는 메모리 크기 통제와 직렬화, 네트워크 전송, UE 블루프린트 호환에서 중요할 수 있다. 비트 플래그 패턴은 `enum class`에 연산자를 추가해 더 안전하게 구현할 수 있다.

### 2. 코드 예시

```cpp
enum class EMovementState : uint8
{
    Idle,
    Walk,
    Run
};

enum class EAbilityFlags : uint8
{
    None = 0,
    CanJump = 1 << 0,
    CanShoot = 1 << 1
};
```

### 3. 왜 중요한가 (게임/UE5 맥락)

상태 표현은 게임 코드 전역에 퍼진다. `enum class`를 이해하면 타입 안전성을 높이고, 블루프린트와 네트워크에서 필요한 크기 제어도 더 의식적으로 설계할 수 있다.

### 4. 면접 예상 Q&A

**Q1: `enum class(이넘 클래스, 범위가 있는 열거형)`가 일반 `enum`보다 더 안전한 이유는 무엇인가요?**
> A: 이름 오염을 줄이고 정수와의 암시적 변환을 막기 때문이다. 결론적으로 잘못된 값 혼입을 컴파일 단계에서 더 많이 막아 준다. 큰 코드베이스일수록 이 차이가 눈에 띈다.

**Q2: `underlying type(언더라잉 타입, 기반형)`을 명시하는 이유는 무엇인가요?**
> A: 메모리 크기와 표현 범위를 통제하기 위해서다. `uint8`로 고정하면 직렬화와 네트워크 전송에서 의도가 더 분명해질 수 있다. UE의 리플렉션이나 블루프린트 호환에서도 자주 고려된다.

**Q3: 비트 플래그는 언제 열거형으로 표현하나요?**
> A: 여러 옵션을 조합해야 하는 상태에서 유용하다. 다만 일반 `enum`보다 `enum class`와 연산자 오버로드를 조합하는 편이 더 안전하다. 단일 상태 열거형과 섞지 않는 것도 중요하다.

**Q4: `enum class`를 정수로 바꿔야 할 때는 어떻게 하나요?**
> A: 명시적 캐스팅을 사용한다. 불편해 보이지만 그 불편함이 곧 타입 안전성이다. 의도 없이 정수와 섞이지 않도록 막아 주는 효과가 있다.

**Q5: 모든 열거형을 무조건 `enum class`로 바꾸면 되나요?**
> A: 대부분은 그렇지만, 기존 API 호환이나 비트 연산 편의성 때문에 추가 설계가 필요한 경우가 있다. 장점만 보고 전환하면 연산자 지원이 빠져 불편할 수 있다. 그래서 사용 문맥을 같이 봐야 한다.

**Q6: 실무에서 enum 설계가 나쁘다는 신호는 무엇인가요?**
> A: 의미가 겹치는 값이 많고, 정수 상수처럼 여기저기 섞이며, 비트 플래그와 단일 상태 표현이 섞여 있을 때다. 상태 모델을 먼저 정리해야 한다. 열거형은 문법보다 모델링 문제다.

### 5. 흔한 실수

- 상태 표현에 일반 `enum`을 습관적으로 써 정수와 섞이게 둔다.
- underlying type이 필요한 상황인데 크기를 명시하지 않는다.
- 단일 상태와 비트 플래그를 같은 열거형으로 섞는다.

## 토픽 10: typedef vs using(타입디프 vs 유징, 타입 별칭)

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 8

### 1. 핵심 개념

`typedef(타입디프, 타입 별칭 선언)`와 `using(유징, 현대식 타입 별칭)`은 별칭(alias, 앨리어스, 다른 이름)을 만드는 도구다. 둘 다 기존 타입에 다른 이름을 붙이지만, modern C++에서는 `using`이 더 읽기 쉽고 template alias(템플릿 앨리어스, 템플릿 별칭)까지 지원해 사실상 기본 선택이 됐다.

핵심은 별칭이 새 타입을 만드는 것이 아니라는 점이다. 즉 타입 이름을 바꾸는 것일 뿐, 타입 자체의 규칙은 바뀌지 않는다. 또한 별칭은 복잡한 함수 포인터, 템플릿 타입, 정책 타입을 더 읽기 쉽게 만드는 데 유용하다.

### 2. 코드 예시

```cpp
typedef void (*LegacyCallback)(int);

using Callback = void(*)(int);
using StringMap = std::unordered_map<std::string, int>;

template <typename T>
using PtrList = std::vector<T*>;
```

### 3. 왜 중요한가 (게임/UE5 맥락)

컨테이너, 함수 포인터, 정책 타입이 많은 코드베이스에서는 타입 이름이 길어지기 쉽다. 별칭을 잘 쓰면 코드 읽기가 쉬워지지만, 의미 없는 축약만 늘리면 오히려 도메인 이해가 떨어진다.

### 4. 면접 예상 Q&A

**Q1: `using(유징, 현대식 타입 별칭)`이 `typedef`보다 자주 권장되는 이유는 무엇인가요?**
> A: 문법이 더 읽기 쉽고 template alias를 지원하기 때문이다. 특히 복잡한 타입일수록 왼쪽이 별칭 이름이라는 점이 더 분명하다. modern C++ 문맥에서는 사실상 기본 선택처럼 쓰인다.

**Q2: 별칭(alias, 앨리어스)은 새 타입을 만드는 것인가요?**
> A: 아니다. 기존 타입에 다른 이름을 붙일 뿐이다. 타입 안정성을 새로 만들고 싶다면 wrapper struct나 class가 필요하다. 별칭과 새 타입 생성은 구분해야 한다.

**Q3: template alias는 왜 유용한가요?**
> A: 반복적으로 쓰는 템플릿 구조를 더 짧고 의미 있는 이름으로 표현할 수 있기 때문이다. 코드 가독성과 재사용성이 좋아진다. 복잡한 컨테이너 조합일수록 효과가 크다.

**Q4: 별칭을 과하게 쓰면 왜 문제가 되나요?**
> A: 실제 타입이 무엇인지 감춰 버릴 수 있기 때문이다. 특히 도메인 의미가 없는 짧은 이름은 코드를 더 읽기 어렵게 만든다. 줄였다고 항상 좋아지는 것은 아니다.

**Q5: `using`으로 함수 포인터 별칭을 만드는 장점은 무엇인가요?**
> A: 선언을 왼쪽에서 오른쪽으로 자연스럽게 읽을 수 있다. 복잡한 포인터 문법을 훨씬 단순하게 보이게 만든다. `typedef`보다 초심자에게도 덜 난해하다.

**Q6: 실무에서 좋은 alias의 기준은 무엇인가요?**
> A: 반복되는 복잡한 타입을 줄이면서도 도메인 의미를 더 잘 드러내는 것이다. 단순 축약보다 의미 전달이 우선이다. 이름이 타입 정보를 숨기기만 하면 오히려 역효과다.

### 5. 흔한 실수

- alias를 새 타입처럼 오해한다.
- 의미 없는 축약 이름을 남발한다.
- template alias가 필요한 상황에서도 구식 `typedef` 스타일만 고집한다.

## 토픽 11: 캐스팅 4종

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 7, 9

### 1. 핵심 개념

C++의 명시적 캐스팅은 `static_cast(스태틱 캐스트, 정적 변환)`, `dynamic_cast(다이내믹 캐스트, 런타임 타입 변환)`, `const_cast(콘스트 캐스트, const 제거/추가)`, `reinterpret_cast(리인터프리트 캐스트, 비트 해석 변환)` 네 가지로 나뉜다. 이 구분의 핵심은 "무슨 종류의 변환을 왜 하려는가"를 분명히 드러내는 것이다. 반대로 C-style cast(씨 스타일 캐스트, 옛 문법 캐스팅)는 여러 위험한 변환을 한 문법에 숨겨 의도를 흐린다.

또한 `narrowing conversion(내로잉 컨버전, 축소 변환)`도 함께 봐야 한다. 예를 들어 큰 정수를 작은 타입으로 줄이거나 부동소수점을 정수로 바꾸는 것은 값 손실 가능성이 있다. `{}` 초기화는 이런 narrowing을 더 엄격히 막아 주는 편이다.

### 2. 코드 예시

```cpp
double Ratio = 3.14;
int Value = static_cast<int>(Ratio);

Base* BasePtr = new Derived();
Derived* DerivedPtr = dynamic_cast<Derived*>(BasePtr);

const int Number = 10;
int* MutablePtr = const_cast<int*>(&Number);
```

### 3. 왜 중요한가 (게임/UE5 맥락)

타입 변환은 자주 일어나지만, 잘못된 캐스팅은 가장 위험한 버그 중 하나다. 의도를 드러내는 캐스팅을 쓰면 리뷰와 디버깅이 쉬워지고, 위험한 변환을 더 빨리 발견할 수 있다.

### 4. 면접 예상 Q&A

**Q1: `static_cast(스태틱 캐스트, 정적 변환)`와 `dynamic_cast(다이내믹 캐스트, 런타임 타입 변환)`의 차이는 무엇인가요?**
> A: `static_cast`는 컴파일 타임 규칙 기반 변환이고, `dynamic_cast`는 런타임 타입 검사를 동반한다. 다형 계층에서 안전성 확인이 필요하면 `dynamic_cast`가 맞다. 성능과 안전성 trade-off를 함께 봐야 한다.

**Q2: `const_cast(콘스트 캐스트, const 제거/추가)`는 언제 정당화될 수 있나요?**
> A: 원본 객체가 실제로 non-const인데 API 시그니처 때문에 const가 붙은 경우처럼 매우 제한적인 상황뿐이다. truly const 객체를 바꾸면 정의되지 않은 동작이 될 수 있다. 그래서 대부분의 경우는 설계 문제로 먼저 의심해야 한다.

**Q3: `reinterpret_cast(리인터프리트 캐스트, 비트 해석 변환)`가 위험한 이유는 무엇인가요?**
> A: 비트 패턴을 전혀 다른 타입으로 해석해 타입 시스템 보호를 거의 우회하기 때문이다. 이식성과 안전성이 크게 떨어진다. 저수준 코드가 아니면 사실상 피하는 편이 낫다.

**Q4: `C-style cast(씨 스타일 캐스트)`를 피하는 이유는 무엇인가요?**
> A: 어떤 종류의 변환인지 코드에서 드러나지 않기 때문이다. 결론적으로 리뷰 시 위험도를 판단하기 어려워진다. 의도를 명시하는 현대식 cast가 훨씬 낫다.

**Q5: `narrowing conversion(내로잉 컨버전, 축소 변환)`은 왜 문제인가요?**
> A: 값 손실이나 범위 초과가 조용히 일어날 수 있기 때문이다. 특히 게임 수치와 인덱스 계산에서 미묘한 버그를 만든다. 그래서 `{}` 초기화가 안전장치가 되기도 한다.

**Q6: 실무에서 가장 보수적인 캐스팅 원칙은 무엇인가요?**
> A: 가장 좁고 의도가 분명한 캐스팅만 쓰는 것이다. 캐스팅 자체를 줄이는 설계가 더 중요하며, C-style cast는 거의 금지에 가깝게 보는 편이 안전하다. 타입이 자주 바뀐다면 구조를 먼저 의심해야 한다.

### 5. 흔한 실수

- C-style cast로 위험한 변환을 숨긴다.
- narrowing 가능성을 무시한 채 숫자 타입을 섞는다.
- `const_cast`로 근본적인 API 설계 문제를 덮는다.
