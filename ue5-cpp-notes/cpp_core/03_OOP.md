## 토픽 12: 생성자 / 소멸자

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 4, 5

### 1. 핵심 개념

생성자는 객체가 만들어질 때 불변 조건을 세우고 자원을 획득하는 함수이고, 소멸자는 객체가 끝날 때 자원을 반납하는 함수다. 핵심은 "객체가 유효한 상태로만 존재하도록 보장하는 것"이다.

기본 생성자, 매개변수 생성자, 복사/이동 생성자, 초기화 리스트, 소멸 순서를 함께 이해해야 한다. 특히 멤버 초기화는 생성자 본문보다 초기화 리스트에서 이뤄지며, 소멸 순서는 생성과 반대 방향으로 진행된다는 점이 중요하다.

### 2. 코드 예시

```cpp
class FileHandle
{
public:
    explicit FileHandle(const std::string& Path)
        : Path(Path)
    {
        std::cout << "open: " << Path << "\n";
    }

    ~FileHandle()
    {
        std::cout << "close: " << Path << "\n";
    }

private:
    std::string Path;
};
```

### 3. 왜 중요한가 (게임/UE5 맥락)

자원 수명과 초기화 순서는 메모리, 파일, 핸들, 등록/해제 로직 전반에 영향을 준다. 생성자/소멸자를 이해해야 RAII(알에이아이아이, 자원 획득은 초기화)와 move semantics(무브 세맨틱스, 이동 의미론)도 자연스럽게 연결된다.

### 4. 면접 예상 Q&A

**Q1: 생성자에서 가장 중요한 책임은 무엇인가요?**
> A: 객체를 유효한 상태로 만드는 것이다. 결론적으로 생성 직후 객체가 사용할 수 있어야 하며, 반쯤 초기화된 상태를 허용하면 안 된다.

**Q2: 초기화 리스트를 생성자 본문보다 선호하는 이유는 무엇인가요?**
> A: 멤버를 곧바로 원하는 값으로 초기화할 수 있기 때문이다. 특히 const 멤버, 참조 멤버, 비기본 생성만 가능한 멤버는 초기화 리스트가 필수다.

**Q3: 소멸 순서는 어떻게 되나요?**
> A: 멤버는 선언 순서의 역순으로 소멸되고, 객체 계층도 생성의 반대 순서로 정리된다. 따라서 초기화 순서와 정리 순서를 함께 의식해야 한다.

**Q4: 생성자에서 예외가 발생하면 어떤 일이 생기나요?**
> A: 이미 완성된 멤버들만 정상적으로 정리되고, 완전히 만들어지지 않은 객체의 소멸자는 호출되지 않는다. 그래서 자원은 멤버 객체와 RAII에 맡기는 편이 안전하다.

**Q5: 복사 생성자와 이동 생성자는 언제 특히 중요해지나요?**
> A: 자원 소유 타입에서 객체 전달과 반환이 잦을 때 중요하다. 복사는 깊은 복사 비용을, 이동은 소유권 이전 규칙을 결정한다.

**Q6: 실무에서 생성자 설계가 나쁘다는 신호는 무엇인가요?**
> A: 생성자가 너무 많은 일을 하거나, 외부 시스템 접근과 실패 가능성이 큰 로직을 과도하게 담을 때다. 기본 불변 조건과 무거운 런타임 작업은 분리하는 편이 좋다.

### 5. 흔한 실수

- 멤버 초기화 순서를 선언 순서와 다르게 착각한다.
- 생성자 본문에서 복잡한 초기화를 하며 예외 안전을 약하게 만든다.
- 자원 정리를 소멸자 대신 수동 호출에 의존한다.

## 토픽 13: 상속

> 난이도: ★★☆ | 예상 시간: 2시간 | 선행: 토픽 12

### 1. 핵심 개념

상속은 기존 클래스의 인터페이스와 구현을 바탕으로 파생 클래스를 만드는 기능이다. 핵심은 코드 재사용보다 "is-a 관계가 실제로 성립하는가"를 먼저 보는 것이다. public 상속은 다형성 계약, private 상속은 구현 재사용 성격이 강하다.

또한 다중 상속과 diamond 문제를 이해해야 한다. virtual inheritance는 공통 기반 클래스가 한 번만 존재하도록 조정해 diamond 문제를 완화한다. 하지만 구조가 복잡해지므로, 실제 설계에서는 상속보다 조합을 먼저 고려하는 편이 안전하다.

### 2. 코드 예시

```cpp
class Animal
{
public:
    virtual void Speak() const = 0;
    virtual ~Animal() = default;
};

class Dog : public Animal
{
public:
    void Speak() const override
    {
        std::cout << "Bark\n";
    }
};
```

### 3. 왜 중요한가 (게임/UE5 맥락)

게임 코드에는 Actor, Component, Ability, Widget처럼 상속 계층이 자연스럽게 생긴다. 하지만 무분별한 상속은 God hierarchy를 만들고, 조금만 요구가 바뀌어도 구조 전체가 굳어 버린다.

### 4. 면접 예상 Q&A

**Q1: 상속의 가장 적절한 사용 기준은 무엇인가요?**
> A: 진짜로 "파생 타입이 기반 타입의 한 종류인가"가 성립할 때다. 결론적으로 단순 코드 재사용만을 위해 상속을 고르면 설계가 쉽게 틀어진다.

**Q2: public 상속과 private 상속은 어떻게 다르나요?**
> A: public 상속은 외부에 is-a 관계를 드러내고, private 상속은 구현 재사용 성격이 강하다. 인터페이스 상속인지 내부 구현 재사용인지가 기준이다.

**Q3: 다이아몬드 상속 문제는 무엇인가요?**
> A: 두 경로로 같은 기반 클래스를 물려받아 중복 기반 서브객체가 생길 수 있는 문제다. 멤버 접근과 상태 일관성이 꼬이기 쉽다.

**Q4: virtual inheritance는 왜 필요한가요?**
> A: 공통 기반 클래스를 한 번만 공유하도록 만들기 위해서다. 다이아몬드 구조에서 중복 기반 문제를 줄이는 데 쓴다.

**Q5: 상속보다 조합이 더 나은 경우는 언제인가요?**
> A: 기능을 유연하게 섞고 싶거나, 역할이 자주 바뀌는 경우다. 게임플레이 기능은 컴포넌트 조합이 더 잘 맞는 경우가 많다.

**Q6: 실무에서 상속 구조가 나쁘다는 신호는 무엇인가요?**
> A: 하위 클래스가 부모의 일부 기능을 억지로 꺼야 하거나, 깊은 계층 때문에 동작 추적이 어려울 때다. 이는 대개 상속보다 조합이 더 맞았다는 신호다.

### 5. 흔한 실수

- 코드 재사용만 보고 public 상속을 선택한다.
- 깊은 상속 계층으로 변경 영향을 키운다.
- 다중 상속과 virtual inheritance 비용을 충분히 이해하지 않고 도입한다.

## 토픽 14: virtual / vtable(버추얼 / 브이테이블, 가상 함수 / 가상 함수 테이블)

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 13

### 1. 핵심 개념

`virtual(버추얼, 가상 함수)` 함수는 파생 클래스에서 재정의 가능한 동작을 제공하며, 실행 시점에 실제 객체 타입에 맞는 함수를 호출하게 해 준다. 이 동적 디스패치를 구현하기 위해 많은 컴파일러가 `vtable(브이테이블, 가상 함수 테이블)`과 `vptr(브이포인터, 가상 함수 테이블 포인터)` 구조를 사용한다.

핵심은 "포인터 타입이 아니라 실제 객체 타입이 호출 대상을 결정한다"는 점이다. 또한 기반 클래스를 다형적으로 쓸 계획이라면 소멸자를 virtual로 두는 것이 매우 중요하다.

### 2. 코드 예시

```cpp
class Base
{
public:
    virtual void Print() const
    {
        std::cout << "Base\n";
    }

    virtual ~Base() = default;
};

class Derived : public Base
{
public:
    void Print() const override
    {
        std::cout << "Derived\n";
    }
};
```

### 3. 왜 중요한가 (게임/UE5 맥락)

엔진과 게임 코드는 다형성을 많이 사용한다. Actor, Component, Widget, Ability 같은 기반 타입 포인터로 다양한 파생 객체를 다루려면 virtual 동작과 파괴 규칙을 정확히 알아야 한다.

### 4. 면접 예상 Q&A

**Q1: `virtual(버추얼, 가상 함수)` 함수가 필요한 이유는 무엇인가요?**
> A: 기반 타입 인터페이스로 다양한 파생 객체를 다형적으로 다루기 위해서다. 결론적으로 런타임 타입에 맞는 동작을 선택하게 해 준다.

**Q2: vtable은 무엇인가요?**
> A: 가상 함수 호출을 위한 함수 포인터 테이블 구현 개념이다. 표준이 이름까지 강제하지는 않지만, 많은 구현체가 이 방식으로 동적 디스패치를 제공한다.

**Q3: 기반 클래스 소멸자를 virtual로 두지 않으면 왜 위험한가요?**
> A: 기반 포인터로 파생 객체를 삭제할 때 파생 소멸자가 호출되지 않을 수 있기 때문이다. 자원 누수나 부분 소멸이 생길 수 있다.

**Q4: virtual 호출의 비용은 어떻게 이해해야 하나요?**
> A: 간접 호출과 인라이닝 제약이 생길 수 있다. 대부분 큰 문제가 아니지만, 아주 자주 호출되는 핫패스에서는 구조 선택에 영향을 줄 수 있다.

**Q5: `override`를 왜 쓰는 것이 좋나요?**
> A: 실제로 기반 가상 함수를 재정의하는지 컴파일러가 검증해 주기 때문이다. 오타나 시그니처 불일치를 빨리 잡을 수 있다.

**Q6: 실무에서 virtual 사용이 과하다는 신호는 무엇인가요?**
> A: 계층 전체가 불필요한 다형성으로 덮여 있고, 실제로는 단일 구현만 존재하는데도 모든 함수가 virtual일 때다. 유연성보다 복잡성이 커진다.

### 5. 흔한 실수

- 다형적으로 삭제될 타입인데 virtual 소멸자를 두지 않는다.
- `override`를 생략해 시그니처 불일치를 놓친다.
- 성능 감각 없이 모든 멤버 함수를 virtual로 선언한다.

## 토픽 15: 연산자 오버로딩

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 12

### 1. 핵심 개념

연산자 오버로딩은 기존 연산자 문법에 사용자 정의 타입의 의미를 부여하는 기능이다. 잘 쓰면 자연스러운 수학 타입이나 값 타입을 만들 수 있지만, 과하면 코드 의미를 숨긴다. 핵심은 "읽는 사람이 예상 가능한 의미만 허용하라"는 것이다.

또한 `conversion operator(컨버전 오퍼레이터, 변환 연산자)`도 함께 봐야 한다. `operator bool()`처럼 특정 타입으로의 변환을 제공할 수 있지만, 암시적 변환을 너무 쉽게 열어 두면 의도치 않은 오버로드 선택과 버그가 생긴다. modern C++에서는 `explicit` 변환 연산자를 자주 고려한다.

### 2. 코드 예시

```cpp
class Vec2
{
public:
    Vec2(float X, float Y) : X(X), Y(Y) {}

    Vec2 operator+(const Vec2& Other) const
    {
        return Vec2(X + Other.X, Y + Other.Y);
    }

    explicit operator bool() const
    {
        return X != 0.0f || Y != 0.0f;
    }

private:
    float X = 0.0f;
    float Y = 0.0f;
};
```

### 3. 왜 중요한가 (게임/UE5 맥락)

벡터, 시간, 핸들, 래퍼 타입에서 연산자 오버로딩은 자주 쓰인다. 그러나 의미가 불명확한 오버로딩은 디버깅과 코드 리뷰를 어렵게 만들기 때문에, 문법적 편의보다 의미 일관성이 더 중요하다.

### 4. 면접 예상 Q&A

**Q1: 연산자 오버로딩을 좋은 설계로 보는 기준은 무엇인가요?**
> A: 해당 타입에서 그 연산이 직관적이고 수학적 의미가 자연스러울 때다. 결론적으로 사용자가 놀라지 않는 의미여야 한다.

**Q2: 멤버 함수와 비멤버 함수 오버로딩은 어떻게 구분하나요?**
> A: 좌변 타입 내부 상태에 강하게 묶이면 멤버가 자연스럽고, 좌우 대칭성과 암시적 변환 지원이 중요하면 비멤버가 더 적절할 수 있다.

**Q3: `operator bool()`은 왜 조심해야 하나요?**
> A: 암시적 변환이 너무 넓게 열리면 의도치 않은 문맥에서 타입이 bool로 바뀔 수 있기 때문이다. 그래서 modern C++에서는 `explicit operator bool()`을 자주 쓴다.

**Q4: 어떤 연산자 오버로딩이 특히 위험한가요?**
> A: 의미가 직관적이지 않은 `operator,`, `operator&&`, `operator||` 같은 경우다. 문법은 같아도 평가 규칙이 다르게 느껴져 혼란을 만들 수 있다.

**Q5: `==`와 `<` 같은 비교 연산자를 설계할 때 중요한 점은 무엇인가요?**
> A: 동치성과 정렬 규칙이 일관돼야 한다는 점이다. 컨테이너와 알고리즘은 그 계약을 믿고 동작한다.

**Q6: 실무에서 연산자 오버로딩이 과하다는 신호는 무엇인가요?**
> A: 일반 함수 호출보다 의미가 덜 명확하고, 새 팀원이 코드를 읽을 때 실제 동작을 쉽게 예측하지 못할 때다. 편의성보다 명확성이 우선이다.

### 5. 흔한 실수

- 읽는 사람이 예상 못 하는 의미로 연산자를 재정의한다.
- 암시적 conversion operator를 과하게 열어 둔다.
- 비교 연산 규칙을 불완전하게 구현해 컨테이너 계약을 깨뜨린다.
