## 토픽 12: 생성자 / 소멸자

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 4, 5

### 1. 핵심 개념

핵심 용어: `Rule of Five`, `CDO`, `획득`, `역순`.

생성자는 객체가 만들어질 때 불변 조건을 세우고 자원을 획득하는 함수이고, 소멸자는 객체가 끝날 때 자원을 반납하는 함수다. 핵심은 "객체가 유효한 상태로만 존재하도록 보장하는 것"이다.

기본 생성자, 매개변수 생성자, 복사/이동 생성자, copy/move assignment, 초기화 리스트, 소멸 순서를 함께 이해해야 한다. 특히 자원을 직접 소유하는 타입은 `Rule of 3 / 5 / 0` 관점에서 설계해야 한다. 직접 소멸자를 정의했다면 복사와 대입 정책도 같이 고민해야 하며, 가능하면 표준 타입과 RAII 멤버를 조합해 `Rule of 0`로 가는 편이 더 안전하다.

### 2. 코드 예시

```cpp
class Buffer {
    char* data;
public:
    Buffer(size_t n) : data(new char[n]) {}
    ~Buffer() { delete[] data; }
};
```

소멸자만 정의하고 복사 생성자/복사 대입을 정의하지 않으면 기본 얕은 복사가 생긴다.
복사된 두 객체가 같은 data를 가리켜 둘 다 delete[]를 시도할 수 있다.

### 3. 왜 중요한가 (게임/UE5 맥락)

자원 수명과 초기화 순서는 메모리, 파일, 핸들, 등록/해제 로직 전반에 영향을 준다. 생성자/소멸자를 이해해야 RAII(알에이아이아이, 자원 획득은 초기화)와 move semantics(무브 세맨틱스, 이동 의미론)도 자연스럽게 연결된다.

### 4. 면접 예상 Q&A

**Q1: 생성자와 소멸자의 핵심 역할을 수명 관점에서 설명하라.**
> A: 생성자는 유효한 초기 상태를 만들고, 소멸자는 객체가 소유한 자원을 정리해 클래스 불변식을 수명 전체에 걸쳐 유지한다.

**Q2: C++ 생성자의 핵심 책임 두 가지는?**
> A: 불변 조건(invariant) 수립과 자원 획득(RAII)이다.

**Q3: 생성자에서 가장 중요한 책임은 무엇인가요?**
> A: 객체를 유효한 상태로 만드는 것이다. 결론적으로 생성 직후 객체가 사용할 수 있어야 하며, 반쯤 초기화된 상태를 허용하면 안 된다.

**Q4: 초기화 리스트를 생성자 본문보다 선호하는 이유는 무엇인가요?**
> A: 멤버를 곧바로 원하는 값으로 초기화할 수 있기 때문이다. 특히 const 멤버, 참조 멤버, 비기본 생성만 가능한 멤버는 초기화 리스트가 필수다.

### 5. 흔한 실수

- 멤버 초기화 순서를 선언 순서와 다르게 착각한다.
- 생성자 본문에서 복잡한 초기화를 하며 예외 안전을 약하게 만든다.

## 토픽 13: 상속

> 난이도: ★★☆ | 예상 시간: 2시간 | 선행: 토픽 12

### 1. 핵심 개념

핵심 용어: `LSP`, `소멸자`, `is-a`, `조합`.

상속은 기존 클래스의 인터페이스와 구현을 바탕으로 파생 클래스를 만드는 기능이다. 핵심은 코드 재사용보다 "is-a 관계가 실제로 성립하는가"를 먼저 보는 것이다. public 상속은 다형성 계약, private 상속은 구현 재사용 성격이 강하다.

또한 다중 상속과 diamond 문제를 이해해야 한다. virtual inheritance는 공통 기반 클래스가 한 번만 존재하도록 조정해 diamond 문제를 완화한다. 하지만 구조가 복잡해지므로, 실제 설계에서는 상속보다 조합을 먼저 고려하는 편이 안전하다.

### 2. 코드 예시

```cpp
struct Base {
    ~Base() {}
};
struct Derived : Base {
    ~Derived() {}
};

Base* p = new Derived();
delete p;
```

Base 소멸자가 가상이 아니므로 Derived 소멸자가 호출되지 않을 수 있다.
다형성 삭제에서 자식 정리 로직이 빠져 누수가 난다.

### 3. 왜 중요한가 (게임/UE5 맥락)

게임 코드에는 Actor, Component, Ability, Widget처럼 상속 계층이 자연스럽게 생긴다. 하지만 무분별한 상속은 God hierarchy를 만들고, 조금만 요구가 바뀌어도 구조 전체가 굳어 버린다.

### 4. 면접 예상 Q&A

**Q1: 상속을 사용해야 하는 조건은 무엇인가?**
> A: 단순 코드 재사용이 아니라 'is-a' 관계와 다형성 대체 가능성이 명확할 때만 사용해야 한다.

**Q2: C++에서 public 상속이 적절한 기준은 무엇인가?**
> A: is-a 관계가 성립할 때만 public 상속을 사용해야 한다.

**Q3: 상속의 가장 적절한 사용 기준은 무엇인가요?**
> A: 진짜로 "파생 타입이 기반 타입의 한 종류인가"가 성립할 때다. 결론적으로 단순 코드 재사용만을 위해 상속을 고르면 설계가 쉽게 틀어진다.

**Q4: public 상속과 private 상속은 어떻게 다르나요?**
> A: public 상속은 외부에 is-a 관계를 드러내고, private 상속은 구현 재사용 성격이 강하다. 인터페이스 상속인지 내부 구현 재사용인지가 기준이다.

### 5. 흔한 실수

- 코드 재사용만 보고 public 상속을 선택한다.
- 깊은 상속 계층으로 변경 영향을 키운다.

## 토픽 14: virtual / vtable(버추얼 / 브이테이블, 가상 함수 / 가상 함수 테이블)

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 13

### 1. 핵심 개념

핵심 용어: `vtable`, `object slicing`, `vtable`, `vptr`.

`virtual(버추얼, 가상 함수)` 함수는 파생 클래스에서 재정의 가능한 동작을 제공하며, 실행 시점에 실제 객체 타입에 맞는 함수를 호출하게 해 준다. 이 동적 디스패치를 구현하기 위해 많은 컴파일러가 `vtable(브이테이블, 가상 함수 테이블)`과 `vptr(브이포인터, 가상 함수 테이블 포인터)` 구조를 사용한다.

핵심은 "포인터 타입이 아니라 실제 객체 타입이 호출 대상을 결정한다"는 점이다. 또한 기반 클래스를 다형적으로 쓸 계획이라면 소멸자를 virtual로 두는 것이 매우 중요하다. interface 설계에서는 "확장 포인트를 정말 열어야 하는가"와 "값, 포인터, 참조 중 어떤 형태로 API를 노출할 것인가"를 함께 판단해야 한다.

### 2. 코드 예시

```cpp
struct Enemy {
    virtual void onDeath() {}
};

struct Boss : Enemy {
    void ondeath() override {}
};
```

부모 함수 이름은 onDeath인데 자식은 ondeath로 오타가 있다.
override가 있기 때문에 컴파일러가 '재정의할 대상이 없다'고 잡아 준다.

### 3. 왜 중요한가 (게임/UE5 맥락)

엔진과 게임 코드는 다형성을 많이 사용한다. Actor, Component, Widget, Ability 같은 기반 타입 포인터로 다양한 파생 객체를 다루려면 virtual 동작과 파괴 규칙을 정확히 알아야 한다.

### 4. 면접 예상 Q&A

**Q1: virtual 함수가 동작하는 원리를 high level로 설명하라.**
> A: 객체는 보통 vptr을 통해 자신의 vtable을 가리키고, virtual 호출 시 이 테이블을 따라 실제 override된 함수를 런타임에 찾는다.

**Q2: C++에서 virtual 함수가 필요한 이유는?**
> A: 기반 클래스 포인터/참조를 통해 파생 클래스의 함수를 호출(런타임 다형성)하기 위해서이다.

**Q3: vtable(가상 함수 테이블)의 구조와 동작을 설명하시오.**
> A: 클래스마다 하나의 vtable이 존재하며, 각 가상 함수에 대한 포인터 배열이다. 객체는 vptr을 통해 자신의 클래스 vtable을 가리킨다.

**Q4: `virtual(버추얼, 가상 함수)` 함수가 필요한 이유는 무엇인가요?**
> A: 기반 타입 인터페이스로 다양한 파생 객체를 다형적으로 다루기 위해서다. 결론적으로 런타임 타입에 맞는 동작을 선택하게 해 준다.

### 5. 흔한 실수

- 다형적으로 삭제될 타입인데 virtual 소멸자를 두지 않는다.
- `override`를 생략해 시그니처 불일치를 놓친다.

## 토픽 15: 연산자 오버로딩

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 12

### 1. 핵심 개념

핵심 용어: `함수 객체(functor)`, `operator+=`, `비멤버`, `explicit`.

연산자 오버로딩은 기존 연산자 문법에 사용자 정의 타입의 의미를 부여하는 기능이다. 잘 쓰면 자연스러운 수학 타입이나 값 타입을 만들 수 있지만, 과하면 코드 의미를 숨긴다. 핵심은 "읽는 사람이 예상 가능한 의미만 허용하라"는 것이다.

또한 `conversion operator(컨버전 오퍼레이터, 변환 연산자)`도 함께 봐야 한다. `operator bool()`처럼 특정 타입으로의 변환을 제공할 수 있지만, 암시적 변환을 너무 쉽게 열어 두면 의도치 않은 오버로드 선택과 버그가 생긴다. modern C++에서는 `explicit` 변환 연산자를 자주 고려한다.

### 2. 코드 예시

```cpp
class Vec2 {
public:
    float x, y;
    Vec2& operator+(const Vec2& rhs) {
        x += rhs.x;
        y += rhs.y;
        return *this;
    }
};
```

operator+가 원본을 바꾸고 참조를 반환하면 사용자가 기대하는 수학적 의미와 어긋난다.
상태 변경은 +=에 맡기고 +는 새 값을 반환해야 한다.

### 3. 왜 중요한가 (게임/UE5 맥락)

벡터, 시간, 핸들, 래퍼 타입에서 연산자 오버로딩은 자주 쓰인다. 그러나 의미가 불명확한 오버로딩은 디버깅과 코드 리뷰를 어렵게 만들기 때문에, 문법적 편의보다 의미 일관성이 더 중요하다.

### 4. 면접 예상 Q&A

**Q1: 연산자 오버로딩을 잘하는 기준은 무엇인가?**
> A: 기존 연산자의 의미와 기대를 해치지 않고, 타입의 자연스러운 동작을 더 읽기 쉽게 만들 때만 써야 한다.

**Q2: 연산자 오버로딩을 멤버 함수로 할지 비멤버 함수로 할지 판단하는 기준은?**
> A: =, [], (), -> 는 반드시 멤버로. 대칭 이항 연산자(+, -, ==)는 비멤버(friend)가 적절하다.

**Q3: 연산자 오버로딩을 좋은 설계로 보는 기준은 무엇인가요?**
> A: 해당 타입에서 그 연산이 직관적이고 수학적 의미가 자연스러울 때다. 결론적으로 사용자가 놀라지 않는 의미여야 한다.

**Q4: 멤버 함수와 비멤버 함수 오버로딩은 어떻게 구분하나요?**
> A: 좌변 타입 내부 상태에 강하게 묶이면 멤버가 자연스럽고, 좌우 대칭성과 암시적 변환 지원이 중요하면 비멤버가 더 적절할 수 있다.

### 5. 흔한 실수

- 읽는 사람이 예상 못 하는 의미로 연산자를 재정의한다.
- 암시적 conversion operator를 과하게 열어 둔다.
