## 토픽 12: 생성자 / 소멸자

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 4, 5

### 1. 핵심 개념

핵심 용어: `Rule of Five`, `CDO`, `획득`, `역순`.

생성자는 객체가 막 만들어진 시점에 클래스 불변 조건을 세우고, 이후 모든 멤버 함수가 믿고 사용할 수 있는 출발 상태를 만든다. 소멸자는 그 반대 방향에서 객체가 소유한 자원을 정리하며 수명을 닫는다.

중요한 점은 생성과 소멸이 단순한 문법 이벤트가 아니라는 것이다. C++에서는 메모리, 파일, 락, 소켓, 등록/해제 같은 자원 관리가 객체 수명과 직접 연결되므로, 생성자와 소멸자를 '항상 유효한 상태만 존재하게 만드는 경계'로 봐야 한다.

### 2. 왜 필요한가

C++은 자원을 자동으로 추적해 주지 않는 경우가 많기 때문에, 자원의 획득과 해제를 객체 수명에 묶는 RAII가 사실상 기본 설계 전략이 된다. 생성자에서 획득하고 소멸자에서 반납하면, 코드가 여러 경로로 빠지더라도 정리 규칙을 한곳에 모을 수 있다.

이 방식의 장점은 예외 안전성이다. 함수 중간에서 예외가 발생해도 이미 완성된 멤버와 지역 객체의 소멸자가 자동으로 실행되므로, '정리 코드를 직접 호출해야 하는 부담'을 크게 줄일 수 있다.

### 3. 중요한 동작 규칙

생성자와 소멸자는 순서 규칙을 잘못 이해하면 즉시 버그로 이어진다. 특히 초기화 순서와 예외 발생 시 파괴 범위를 정확히 알아야 RAII를 믿고 설계할 수 있다.

- 멤버 초기화 순서는 초기화 리스트 작성 순서가 아니라 멤버 선언 순서를 따른다.
- 소멸자는 생성의 역순으로 호출되므로, 나중에 획득한 자원이 먼저 반납된다.
- 생성 중 예외가 발생하면 이미 완전히 생성된 멤버와 기반 클래스만 역순으로 파괴된다.
- 즉 객체가 완전히 만들어지지 못한 partially constructed 상태에서도, 이미 생성된 부분에 대해서는 RAII가 유지된다.

### 4. 비용 / 단점

생성자에 과도한 로직을 몰아 넣으면 오히려 설계가 취약해질 수 있다. 생성 시점에 외부 시스템 접근, 복잡한 분기, 실패 가능성이 큰 작업을 많이 넣을수록 예외 처리와 테스트가 어려워진다.

- 복잡한 생성자 본문은 예외 안전성을 약하게 만들고, 객체를 '반쯤 준비된 상태'로 다루게 만들기 쉽다.
- 자원을 직접 소유하는 타입에서 복사/이동 정책을 잘못 설계하면 double free, leak, use-after-free 같은 수명 버그가 발생한다.

### 5. 언제 주의해야 하는가

생성자와 소멸자는 모든 클래스에 존재하지만, 특히 직접 자원을 소유하거나 수명 규칙이 까다로운 타입에서 주의 깊게 설계해야 한다. 이 구간에서의 실수는 런타임 후반에야 드러나는 경우가 많다.

- 메모리, 파일, 락, OS 핸들처럼 자원을 직접 소유하는 클래스라면 Rule of 3/5/0 관점으로 복사와 이동 정책까지 함께 설계해야 한다.
- 생성자나 소멸자 안에서 virtual 함수를 호출하면 아직 기대한 동적 타입이 완성되지 않았거나 이미 파괴 중인 상태일 수 있어 위험하다.
- raw pointer를 직접 소유하는 설계는 누가 해제 책임을 가지는지 불분명해지기 쉬우므로 가능한 한 smart pointer나 RAII 멤버로 대체하는 편이 안전하다.

### 6. 실전 사례

#### 소멸자만 정의한 버퍼 클래스

```cpp
class Buffer {
    char* data;
public:
    Buffer(size_t n) : data(new char[n]) {}
    ~Buffer() { delete[] data; }
};
```

이 코드는 소멸자만 정의하고 복사 생성자와 복사 대입을 정의하지 않았기 때문에 기본 얕은 복사가 생긴다. 결과적으로 두 객체가 같은 `data`를 가리키게 되고, 소멸 시 둘 다 `delete[]`를 실행해 double delete가 날 수 있다.

이럴 때는 Rule of Three/Five를 지키거나, 더 나은 선택으로 `std::unique_ptr<char[]>` 같은 RAII 멤버에 소유권을 맡겨 Rule of 0에 가깝게 가는 편이 낫다.

#### 초기화 리스트 대신 본문에서 대입한 경우

```cpp
class Sensor {
    const int id_;
    std::string name_;
public:
    Sensor(int id, std::string name) {
        id_ = id;          // 에러
        name_ = name;
    }
};
```

const 멤버와 참조 멤버는 이미 만들어진 뒤 다시 대입할 수 없으므로, 생성자 본문이 아니라 초기화 리스트에서 직접 초기화해야 한다. 일반 멤버도 가능하면 초기화 리스트로 바로 만드는 편이 불필요한 기본 생성과 대입 비용을 줄인다.

### 7. 면접 심화 질문

#### Q1. 생성 중 예외가 발생하면 어떤 멤버가 정리되는가?

완전히 생성된 기반 클래스와 멤버만 역순으로 파괴된다. 아직 생성되지 않은 멤버는 파괴 대상이 아니므로, RAII 멤버를 사용하면 예외 시 정리 범위를 자연스럽게 위임할 수 있다.

#### Q2. Rule of 0가 왜 더 안전한가?

복사, 이동, 파괴를 직접 작성하지 않고 표준 라이브러리 타입에 맡기면 수명 정책이 자동으로 맞물린다. 사람이 special member function을 직접 조합할 때 생기는 누락과 불일치를 줄일 수 있다는 점이 핵심이다.

#### Q3. 생성자에서 virtual 호출이 위험한 이유는?

생성 중에는 아직 가장 파생된 타입이 완성되지 않았으므로, 기대한 override가 호출되지 않을 수 있다. 소멸 중에도 같은 이유로 이미 파괴된 하위 상태를 전제로 한 동작을 믿으면 안 된다.

## 토픽 13: 상속

> 난이도: ★★☆ | 예상 시간: 2시간 | 선행: 토픽 12

### 1. 핵심 개념

핵심 용어: `LSP`, `소멸자`, `is-a`, `조합`.

상속은 기존 타입의 인터페이스와 일부 구현을 바탕으로 새로운 타입을 정의하는 기능이다. 하지만 설계의 핵심은 코드 재사용이 아니라, 파생 타입이 기반 타입의 의미를 깨지 않고 대체될 수 있는지에 있다.

즉 public 상속은 'is-a 관계'와 다형성 계약을 의미한다. private 상속은 구현 재사용 성격이 더 강하며, 실무에서는 상속 자체보다 조합과 책임 분리가 더 좋은 선택인 경우가 많다.

### 2. 왜 필요한가

상속이 필요한 가장 대표적인 이유는 공통 인터페이스를 기준으로 여러 종류의 객체를 같은 방식으로 다루기 위해서다. 예를 들어 기반 타입 포인터 하나로 서로 다른 파생 객체를 담아 두고, 런타임에 맞는 동작을 호출하려면 상속과 virtual이 함께 쓰인다.

게임 코드에서도 Actor, Component, Widget처럼 추상적인 기반 타입 위에 여러 파생 타입을 얹는 구조가 자연스럽다. 다만 이런 편의는 강한 결합과 교환되므로, '정말 대체 가능성이 필요한가'를 먼저 확인해야 한다.

### 3. 비용 / 단점

상속은 한 번 구조가 깊어지면 변경 비용이 빠르게 커진다. 기반 클래스의 작은 수정이 여러 파생 타입에 연쇄적으로 영향을 주기 때문에, 설계 초기에 편해 보여도 유지보수 단계에서 가장 비싼 선택이 되기 쉽다.

- 깊은 상속 계층은 변경 파급 범위를 키우고, 테스트와 디버깅 지점을 늘린다.
- 다형성을 위해 virtual 경로가 늘어나면 추적이 어려워지고, 성능 민감 구간에서는 간접 호출 비용도 신경 써야 한다.
- 한 계층에 책임이 과도하게 쌓이면 God hierarchy가 만들어져 구조가 경직된다.

### 4. 언제 쓰고 언제 피해야 하는가

#### 사용해야 하는 경우

- 파생 타입이 기반 타입의 한 종류라고 자연스럽게 말할 수 있을 때
- 기반 타입 인터페이스로 여러 구현을 런타임에 교체해야 할 때
- LSP를 만족하도록 대체 가능성을 유지할 수 있을 때

#### 피해야 하는 경우

- 단순히 코드 몇 줄을 재사용하고 싶을 뿐일 때
- 기능 조합이나 위임으로 더 명확하게 표현할 수 있을 때
- 런타임 다형성이 필요 없고 타입이 사실상 고정돼 있을 때

### 5. 설계 대안

상속을 고르기 전에 먼저 조합과 인터페이스 분리를 검토하는 편이 안전하다. 이 두 방식은 공통 동작을 재사용하면서도, 타입 계층 전체를 한 축에 묶지 않아 변경에 더 유연하다.

- composition: 필요한 기능을 멤버로 포함해 has-a 관계로 조립한다.
- interface 분리: 공통 계약만 추상 인터페이스로 두고, 구현 재사용은 별도 헬퍼나 컴포넌트로 나눈다.

### 6. 실전 사례

코드 재사용만을 이유로 public 상속을 선택하면 외부에 잘못된 의미까지 노출된다. 아래 예시는 Player가 Timer의 일종이 아닌데도, 단지 `start()`와 `stop()`을 재사용하고 싶어서 상속을 택한 경우다.

```cpp
class Timer {
public:
    void start();
    void stop();
};

class Player {
    Timer timer_;
public:
    void startTimer() { timer_.start(); }
    void stopTimer()  { timer_.stop(); }
};
```

조합으로 바꾸면 Player는 여전히 타이머 기능을 쓸 수 있지만, 외부에는 'Player is-a Timer'라는 잘못된 계약을 노출하지 않는다. UE5에서도 상속 계층이 깊어질수록 기능 추가가 전체 클래스 트리를 흔들 수 있어서, 컴포넌트 기반 분리가 자주 더 좋은 해법이 된다.

### 7. 면접 심화 질문

#### Q1. 상속보다 composition이 좋은 경우는?

is-a 관계가 아니라 기능 조합이 목적일 때다. 조합은 결합도를 낮추고 런타임에 다른 구현으로 바꾸기도 쉬워, 유지보수성과 테스트성이 더 좋다.

#### Q2. public 상속과 private 상속의 차이와 사용 기준은?

public 상속은 외부에 is-a 관계와 다형성 계약을 드러낸다. private 상속은 구현을 재사용하되 기반 타입 인터페이스를 공개하지 않는 형태라서, 대부분의 경우 더 명시적인 조합으로 대체 가능하다.

#### Q3. virtual 소멸자가 필요한 이유는?

기반 타입 포인터로 파생 객체를 삭제할 가능성이 있다면, 기반 클래스 소멸자는 virtual이어야 한다. 그렇지 않으면 파생 소멸자가 호출되지 않거나 아예 미정의 동작으로 이어질 수 있다.

## 토픽 14: virtual / vtable(버추얼 / 브이테이블, 가상 함수 / 가상 함수 테이블)

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 13

### 1. 핵심 개념

핵심 용어: `vtable`, `object slicing`, `vptr`.

virtual 함수는 기반 타입 인터페이스를 통해 호출하더라도, 런타임에 실제 객체 타입에 맞는 함수를 선택하게 해 준다. 이 덕분에 '같은 메시지를 다른 객체가 각자 방식으로 처리하는' 런타임 다형성이 가능해진다.

많은 구현에서 이 동작은 `vptr`과 `vtable`로 설명된다. 중요한 포인트는 포인터나 참조의 정적 타입이 아니라, 실제 객체의 동적 타입이 호출 대상을 결정한다는 점이다.

### 2. 왜 필요한가

기반 타입 하나로 여러 파생 객체를 다뤄야 하는 순간, virtual은 코드 구조를 단순하게 만든다. 예를 들어 적, UI 위젯, 능력 객체처럼 서로 다른 구현을 하나의 인터페이스로 묶고 싶을 때 if-else 체인을 줄이고 확장 지점을 명확하게 만들 수 있다.

UE5처럼 프레임워크 코드가 많은 환경에서는 엔진이 제공하는 기반 타입에 사용자 정의 동작을 얹는 일이 흔하다. 그래서 virtual은 단순 문법이 아니라 '확장 포인트를 여는 계약'으로 이해해야 한다.

### 3. 내부 동작

일반적인 구현에서는 다형적 객체가 숨겨진 포인터 하나를 가지고 있고, 이 포인터가 자신이 속한 클래스의 vtable을 가리킨다. virtual 호출이 발생하면 컴파일러는 이 테이블에서 적절한 함수 주소를 찾아 간접 호출을 수행한다.

- 객체는 보통 `vptr`을 통해 자신의 클래스 vtable에 접근한다.
- vtable에는 가상 함수 구현 주소들이 클래스 단위로 정리된다.
- 호출 시점에는 `vptr -> vtable -> 함수 포인터` 순서로 대상 함수를 찾는다.

### 4. 성능 비용

virtual 자체가 항상 느리다고 단정할 수는 없지만, 비용이 생기는 위치는 비교적 분명하다. 간접 호출이 들어가면 분기 예측이 어려워질 수 있고, 컴파일러가 정적 타입만으로 모든 최적화를 밀어붙이기 어렵다.

- 함수 포인터를 거치는 간접 호출이 추가된다.
- devirtualization이 불가능한 경우 인라이닝 기회가 줄어들 수 있다.
- 대량 객체를 순회하는 hot path에서는 캐시 locality와 분기 예측 측면의 불이익이 누적될 수 있다.

### 5. 주요 개념

virtual을 이해할 때 같이 따라오는 개념들이 있다. 호출 규칙만 아는 것보다, 어떤 상황에서 virtual 설계가 깨지는지까지 알아야 실제 버그를 예방할 수 있다.

- object slicing: 파생 객체를 기반 타입 값으로 복사하면 자식 부분이 잘려 나가서 다형성이 사라진다.
- virtual destructor: 기반 포인터로 파생 객체를 삭제할 수 있다면 소멸자는 virtual이어야 한다.
- override/final: 재정의 의도와 확장 금지 의도를 컴파일러에 명시해 실수를 줄인다.

### 6. 언제 쓰고 언제 피해야 하는가

#### 사용해야 하는 경우

- 런타임 다형성이 꼭 필요할 때
- 프레임워크나 인터페이스 기반으로 확장 포인트를 열어야 할 때
- 기반 타입 포인터/참조로 다양한 파생 타입을 다뤄야 할 때

#### 피해야 하는 경우

- 호출 빈도가 매우 높아 작은 간접 비용도 누적되는 hot path일 때
- 타입이 컴파일 타임에 고정되어 있어 정적 다형성으로 충분할 때
- 단지 미래 확장 가능성만 막연히 열어 두고 실제 대체 시나리오는 없을 때

### 7. 설계 대안

virtual이 유일한 다형성 도구는 아니다. 호출 대상을 컴파일 타임에 확정할 수 있다면 정적 다형성이 더 단순하고 빠를 수 있다.

- CRTP: 기반 템플릿이 파생 타입을 매개변수로 받아 정적 다형성을 구현한다.
- template 기반 polymorphism: 호출 대상을 템플릿 인자로 받아 컴파일 타임 바인딩으로 해결한다.

### 8. 실전 사례

#### object slicing으로 다형성이 사라진 경우

```cpp
struct Enemy {
    virtual const char* name() const { return "Enemy"; }
};

struct Boss : Enemy {
    const char* name() const override { return "Boss"; }
};

Boss boss;
Enemy copy = boss;
// copy.name() == "Enemy"
```

값 복사가 일어나면서 `Boss` 부분이 잘려 나가고, `copy`는 더 이상 다형적 `Boss`로 동작하지 않는다. 다형성을 유지하려면 보통 값이 아니라 포인터나 참조로 객체를 전달해야 한다.

#### hot loop에서 virtual 제거를 검토하는 경우

수백만 파티클 업데이트처럼 동일한 연산이 반복되는 hot loop에서는 virtual 호출이 작은 비용이라도 누적될 수 있다. 이때는 타입을 분리된 배열로 정리하거나, CRTP나 템플릿으로 정적 바인딩을 사용해 병목을 줄이는 선택을 검토한다.

### 9. 면접 심화 질문

#### Q1. virtual 호출 비용은 어디서 발생하는가?

핵심은 간접 호출과 최적화 제약이다. `vptr`을 따라 함수 주소를 찾는 단계가 추가되고, devirtualization이 안 되면 인라이닝 같은 최적화도 제한될 수 있다.

#### Q2. vtable 구조를 설명하라.

일반적으로 클래스마다 하나의 vtable이 있고, 각 가상 함수 구현 주소가 테이블에 저장된다. 객체는 `vptr`로 자신이 속한 클래스의 vtable을 가리키며, virtual 호출 시 이 테이블을 통해 실제 함수를 찾는다.

#### Q3. virtual을 제거하는 방법은?

런타임 다형성이 꼭 필요하지 않다면 CRTP, 템플릿, 함수 객체, 분기 기반 정적 디스패치 같은 대안을 사용할 수 있다. 중요한 것은 '확장성'보다 실제 호출 패턴과 성능 요구를 기준으로 선택하는 것이다.

## 토픽 15: 연산자 오버로딩

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 12

### 1. 핵심 개념

핵심 용어: `함수 객체(functor)`, `operator+=`, `비멤버`, `explicit`.

연산자 오버로딩은 사용자 정의 타입에 기존 연산자 문법의 의미를 연결하는 기능이다. 잘 설계하면 벡터, 행렬, 시간, 핸들 같은 값 타입을 더 자연스럽게 표현할 수 있지만, 잘못 쓰면 코드 의미를 감춘다.

핵심 기준은 '읽는 사람이 기존 연산자의 의미를 크게 어긋나지 않게 이해할 수 있는가'다. 즉 편리함보다 의미 일관성과 예측 가능성이 먼저다.

### 2. 왜 필요한가

수학적 의미가 분명한 타입에서는 연산자 오버로딩이 가독성을 크게 높여 준다. `a + b`, `pos += velocity`, `lhs == rhs`처럼 도메인 의미가 곧바로 읽히면 API를 사용하는 쪽도 불필요한 잡음을 줄일 수 있다.

반대로 같은 동작을 `Add`, `Append`, `Combine` 같은 이름으로 매번 풀어 쓰는 편이 더 명확한 타입도 많다. 그래서 연산자 오버로딩은 '짧게 쓰는 기술'이 아니라 '짧게 써도 의미가 선명한 경우에만 쓰는 기술'이다.

### 3. 비용 / 단점

연산자 문법은 짧은 대신 숨기는 정보가 많다. 그래서 설계가 조금만 비틀어져도 호출자는 실제 비용과 부작용을 읽어 내기 어려워진다.

- 연산의 의미가 불명확하면 코드 리뷰와 디버깅이 어려워진다.
- 사이드 이펙트가 숨어 있으면 일반 함수보다 더 위험하다.
- 암시적 변환까지 열어 두면 의도치 않은 오버로드 선택과 버그가 발생한다.

### 4. 언제 쓰고 언제 피해야 하는가

#### 사용해야 하는 경우

- 벡터, 행렬, 복소수처럼 수학적 의미가 명확한 타입일 때
- 사용자가 해당 연산을 직관적으로 예상할 수 있을 때
- 새 값 반환과 상태 변경의 역할 분리가 자연스럽게 표현될 때

#### 피해야 하는 경우

- 연산의 의미가 애매하거나 도메인 고유 이름이 더 명확할 때
- 연산 뒤에 큰 부작용, I/O, 상태 전이가 숨어 있을 때
- 암시적 변환을 기대야만 쓰기 편한 API일 때

### 5. 설계 기준

좋은 연산자 오버로딩은 역할을 분리하고, 값 의미를 보존하며, 놀라운 동작을 피한다. 특히 산술 연산자와 변환 연산자는 사용자 기대를 깨기 쉬워서 더 보수적으로 설계해야 한다.

- `operator+`는 보통 새 값을 반환하고, `operator+=`는 현재 객체를 수정한다.
- 대칭 이항 연산자는 멤버보다 비멤버가 더 자연스러운 경우가 많다.
- `operator bool()` 같은 변환 연산자는 가능한 한 `explicit`를 우선 고려한다.

### 6. 실전 사례

#### operator+가 상태를 바꿔 버린 경우

```cpp
class Vec2 {
public:
    float x, y;
    Vec2& operator+(const Vec2& rhs) {
        x += rhs.x;
        y += rhs.y;
        return *this;
    }
};
```

이 구현은 `a + b`가 새 값을 만든다고 기대하는 사용자 직관을 깨뜨린다. 상태 변경은 `operator+=`에 맡기고, `operator+`는 복사본을 만들어 반환하는 방식으로 역할을 분리해야 한다.

#### 암시적 conversion operator가 만든 버그

```cpp
class OptionalInt {
    int val_;
    bool valid_;
public:
    OptionalInt(int v) : val_(v), valid_(true) {}
    OptionalInt() : val_(0), valid_(false) {}

    explicit operator bool() const { return valid_; }
};
```

`explicit`가 없으면 객체가 산술식이나 비교식으로 흘러 들어가면서 예상하지 못한 변환이 일어날 수 있다. boolean 문맥만 허용하고 싶다면 `explicit operator bool()`이 더 안전한 기본값이다.

### 7. 면접 심화 질문

#### Q1. 멤버와 비멤버 연산자 선택 기준은?

좌변 내부 상태와 강하게 결합된 연산은 멤버가 자연스럽고, 대칭성이 중요한 이항 연산은 비멤버가 더 적합하다. 특히 비멤버는 양쪽 피연산자 모두에 암시적 변환 기회를 줄 수 있다는 장점이 있다.

#### Q2. conversion operator의 위험성은?

호출자가 의식하지 못한 곳에서 타입 변환이 일어나 오버로드 선택이 달라질 수 있다는 점이다. 코드가 짧아지는 대신 의미 추적이 어려워져, 작은 편의가 큰 혼란으로 바뀌기 쉽다.

#### Q3. 왜 explicit를 사용하는가?

암시적 변환 범위를 좁혀 API 의도를 더 분명하게 만들기 위해서다. 특히 `operator bool()`처럼 자주 끼어드는 변환은 `explicit`를 붙여야 놀라운 부작용을 줄일 수 있다.
