## 토픽 12: 생성자 / 소멸자

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 4, 5

### 1. 문제 상황

핵심 용어: `Rule of Five`, `CDO`, `획득`, `역순`.

객체가 생성될 때 멤버가 초기화되지 않으면 UB다. 객체가 소멸될 때 자원이 해제되지 않으면 누수다. 생성자와 소멸자는 이 두 시점을 제어하는 유일한 도구다.

Rule of Five를 모르면 자원을 직접 관리하는 클래스에서 얕은 복사, 이중 해제, 소유권 분실이 발생한다.

### 2. 핵심 개념

#### 정의

생성자(constructor)는 객체 생성 시 멤버를 유효한 상태로 초기화하는 함수다. 소멸자(destructor)는 객체 소멸 시 자원을 반납하는 함수다.

특수 멤버 함수 6가지: 기본 생성자, 복사 생성자, 복사 대입, 이동 생성자, 이동 대입, 소멸자. 자원을 직접 관리하면 이 중 하나를 정의할 때 나머지도 정의해야 한다(Rule of Five). 자원을 직접 관리하지 않으면 모두 = default 또는 생략한다(Rule of Zero).

멤버 초기화 리스트(member initializer list)는 본문 진입 전에 멤버를 초기화한다. 선언 순서대로 초기화되므로 리스트 순서와 선언 순서를 일치시켜야 한다.

#### 이 개념을 이해하려면

- RAII (토픽 5): 생성자/소멸자가 RAII의 기반
- 스마트 포인터 (토픽 4): Rule of Zero를 가능하게 하는 도구

#### 이 개념이 영향을 주는 것

- 상속 (토픽 13): 기반 클래스 생성자/소멸자 호출 순서
- move semantics (토픽 16): 이동 생성자/대입
- UE5 UObject 생명주기 (카테고리 2 / 토픽 5): 엔진의 객체 생성 패턴

### 3. 내부 동작 / 메커니즘

#### 초기화 순서

기반 클래스 생성자 →멤버 변수(선언 순서) →생성자 본문. 소멸은 정확히 역순이다.

초기화 리스트에서의 순서는 무시되고, 클래스 선언에서의 멤버 순서만 의미가 있다. 순서가 다르면 초기화되지 않은 멤버를 참조하는 버그가 생길 수 있다.

#### Rule of Five vs Rule of Zero

Rule of Five: 자원을 직접 관리(raw 포인터, 파일 핸들 등)하면 복사/이동/소멸 모두 정의해야 한다. 하나만 빠져도 얕은 복사, 이중 해제, 소유권 분실이 발생한다.

Rule of Zero: 모든 멤버가 스마트 포인터, 컨테이너 등 RAII 타입이면 특수 멤버 함수를 정의할 필요가 없다. 컴파일러가 생성한 기본 동작이 올바르다. 이것이 선호되는 설계다.

### 4. 자주 틀리는 포인트

- 멤버 초기화 리스트 순서와 선언 순서가 다르다 →초기화는 선언 순서를 따른다. 리스트 순서를 맞추지 않으면 초기화 안 된 멤버를 참조하는 버그가 생긴다.
- 생성자 본문에서 대입으로 초기화한다 →기본 생성 후 대입이므로 비용이 두 배. 초기화 리스트를 사용하라.
- 자원을 직접 관리하면서 복사/이동을 정의하지 않는다 →Rule of Five 위반. 얕은 복사로 이중 해제.
- virtual 소멸자를 빠뜨린다 →상속 관계에서 기반 포인터로 delete하면 파생 소멸자가 호출되지 않는다.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

Rule of Zero를 기본으로 설계하라. 멤버를 스마트 포인터와 컨테이너로 구성하면 특수 멤버 함수를 정의할 필요가 없다.

Rule of Five가 필요한 경우: 커스텀 할당자, C API 래퍼, low-level 자원 관리 등 RAII 타입을 직접 만들어야 할 때.

explicit 생성자: 단일 인자 생성자는 explicit을 붙여 암시적 변환을 방지하라.

### 6. 코드 예시 (bad →good)

#### Bad: Rule of Five 위반

```cpp
class Buffer {
    int* data;
public:
    Buffer(int n) : data(new int[n]) {}
    ~Buffer() { delete[] data; }
    // 복사/이동 미정의 →얕은 복사 →이중 해제!
};
```

#### Good: Rule of Zero

```cpp
class Buffer {
    std::vector<int> data;  // RAII 컨테이너
public:
    Buffer(int n) : data(n) {}
    // 복사/이동/소멸 모두 컴파일러 기본값으로 올바름
};
```

### 7. 판단 기준 요약

- Rule of Zero를 기본 설계로 삼아라
- 자원 직접 관리 시 Rule of Five를 완전히 지켜라
- 초기화 리스트 순서 = 멤버 선언 순서로 맞춰라
- 상속 관계의 기반 클래스에는 virtual 소멸자를 넣어라
- 단일 인자 생성자에는 explicit을 붙여라

## 토픽 13: 상속

> 난이도: ★★☆ | 예상 시간: 2시간 | 선행: 토픽 12

### 1. 문제 상황

핵심 용어: `LSP`, `소멸자`, `is-a`, `조합`.

코드 재사용을 위해 상속부터 쓰면 깊은 상속 트리, 취약한 기반 클래스 문제, 불필요한 인터페이스 노출이 생긴다.

상속은 is-a 관계를 표현하는 도구이지 코드 재사용 도구가 아니다. 이 구분을 못 하면 유지보수 비용이 기하급수적으로 늘어난다.

### 2. 핵심 개념

#### 정의

상속(inheritance)은 기반 클래스의 인터페이스와 구현을 파생 클래스가 물려받는 메커니즘이다. public 상속은 is-a 관계를, private 상속은 구현 상속(implemented-in-terms-of)을 표현한다.

다중 상속은 여러 기반 클래스를 동시에 상속한다. diamond 문제가 발생할 수 있으며, virtual inheritance로 공통 기반을 하나만 유지하도록 해결한다.

#### inheritance vs composition decision tree

- 파생 클래스가 기반 클래스의 모든 public 인터페이스를 자연스럽게 지원하는가? →Yes: 상속 고려
- 단순히 기능을 재사용하고 싶은 것인가? →composition(멤버로 소유)이 더 안전하다
- 다형성(virtual dispatch)이 필요한가? →Yes: 상속 + virtual. No: composition
- 상속 깊이가 3단계를 넘는가? →설계를 재검토하라

#### 이 개념을 이해하려면

- 생성자/소멸자 (토픽 12): 상속 시 생성/소멸 순서

#### 이 개념이 영향을 주는 것

- virtual / vtable (토픽 14): 다형성의 구현
- UE5 Actor 계층 (카테고리 2): 엔진의 상속 구조

### 3. 내부 동작 / 메커니즘

파생 객체의 메모리 레이아웃: 기반 클래스 부분이 먼저 배치되고, 파생 멤버가 뒤에 온다. 기반 포인터로 파생 객체를 가리킬 수 있는 이유다.

virtual inheritance에서는 공유 기반 클래스가 별도 위치에 배치되고, 오프셋을 통해 접근한다. 이 때문에 추가 간접 참조 비용이 발생한다.

생성 순서: 가상 기반 →비가상 기반(선언 순서) →멤버 →본문. 소멸은 정확히 역순이다.

### 4. 자주 틀리는 포인트

- 코드 재사용만을 위해 public 상속한다 →is-a 관계가 아니면 composition을 써라.
- 다중 상속에서 diamond 문제를 무시한다 →공통 기반이 있으면 virtual inheritance가 필요하다.
- 깊은 상속 트리를 만든다 →상속 깊이가 커질수록 변경의 파급 효과가 커진다. 3단계 이상이면 설계를 재검토하라.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

상속을 쓰는 경우: is-a 관계가 자연스럽고, 다형성이 필요하며, 기반 클래스의 전체 인터페이스가 파생에서도 유효할 때.

composition을 쓰는 경우: 기능 재사용이 목적이거나, 일부 인터페이스만 필요하거나, 런타임에 구현을 교체하고 싶을 때.

UE5에서: Actor/Component 모델은 상속 + composition의 혼합. 깊은 상속보다 Component 조합을 권장한다.

### 6. 코드 예시 (bad →good)

#### Bad: 코드 재사용 목적의 상속

```cpp
class Timer : public std::vector<float> {  // is-a 관계가 아님!
public:
    float GetAverage() const;
};
```

#### Good: composition

```cpp
class Timer {
    std::vector<float> samples;  // has-a 관계
public:
    void AddSample(float t) { samples.push_back(t); }
    float GetAverage() const;
};
```

### 7. 판단 기준 요약

- is-a 관계인가? →상속. 아니면 →composition
- 다형성이 필요한가? →상속 + virtual
- 기능 재사용만 필요한가? →composition이 기본
- 상속 깊이 3 이상이면 설계를 재검토하라
- UE5: Component 조합을 깊은 상속보다 우선하라

## 토픽 14: virtual / vtable(버추얼 / 브이테이블, 가상 함수 / 가상 함수 테이블)

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 13

---

### 1. 문제 상황

기반 클래스 포인터를 통해 객체를 다룰 때,  
정적 타입이 아닌 실제 객체의 타입에 따라 다른 동작을 수행해야 하는 경우가 있다.

그러나 C++의 기본 함수 호출은 컴파일 타임에 결정되기 때문에,  
기반 포인터로 함수를 호출하면 항상 기반 클래스의 구현이 실행된다.

이 문제를 해결하기 위해 런타임에 함수 호출을 결정하는 메커니즘이 필요하며,  
이를 위해 virtual 함수가 사용된다.

---

### 2. 핵심 개념

virtual 함수는 호출 시점에 객체의 실제 타입을 기준으로  
실행할 함수를 결정하는 메커니즘이다.

이 과정은 컴파일 타임이 아닌 런타임에 이루어지며,  
이를 동적 디스패치(dynamic dispatch)라고 한다.

이 기능은 상속 구조에서 다형성을 구현하기 위한 핵심 요소이다.

---

### 3. 동작 원리 (핵심)

virtual 함수는 “숨겨진 포인터 + 함수 테이블” 구조로 구현된다.

---

#### 객체 메모리 구조

``` id="mem1"
[Derived 객체]

+----------------------+
| vptr ----------------|------+
+----------------------+      |
| Base 영역            |      |
+----------------------+      |
| Derived 멤버         |      |
+----------------------+      |
                              v
                      [ vtable (Derived) ]
                      +----------------------+
                      | &Derived::Foo        |
                      | &Base::Bar           |
                      +----------------------+
````

---

#### 구성 요소

- **vptr (virtual pointer)**
  각 객체에 포함된 숨겨진 포인터
  → 해당 객체의 vtable을 가리킨다

- **vtable (virtual table)**
  클래스별로 하나 존재하는 함수 포인터 배열
  → virtual 함수들의 실제 주소를 저장한다

---

#### 함수 호출 과정

```cpp
Base* b = new Derived();
b->Foo();
```

이 호출은 내부적으로 다음과 같이 수행된다:

```id="callflow"
1. b가 가리키는 객체의 vptr을 읽는다
2. vptr이 가리키는 vtable에 접근한다
3. Foo의 슬롯 위치를 찾는다
4. 해당 함수 포인터를 호출한다
```

따라서 실제 호출되는 함수는 `Derived::Foo`가 된다.

---

### 4. 이 구조의 의미

virtual 함수 호출은 단순한 함수 호출이 아니라
“포인터를 두 번 따라가는 간접 호출”이다.

이 구조는 다음과 같은 특징을 만든다:

- 호출 대상이 런타임에 결정된다
- 함수 인라인이 불가능하다
- 간접 참조가 발생한다

---

### 5. 성능 관점

virtual의 비용은 단순한 “느림”으로 설명되지 않는다.

#### 구성 요소

- vptr 접근 (메모리 1회)
- vtable 접근 (메모리 1회)
- 간접 호출

이 자체의 비용은 크지 않다.

#### 실제 병목

문제는 다음 상황에서 발생한다:

- 서로 다른 타입 객체를 섞어서 순회
- 매 호출마다 다른 vtable 접근

이 경우 CPU 캐시 효율이 급격히 떨어진다.

즉, virtual의 핵심 비용은
**간접 호출 자체가 아니라 캐시 미스**이다.

---

### 6. 자주 발생하는 오류

#### virtual 소멸자 누락

```cpp
class Base {
public:
    ~Base() {}
};
```

```cpp
Base* p = new Derived();
delete p;
```

이 경우 Derived 소멸자가 호출되지 않는다.

---

#### override 누락

```cpp
void Update(); 
```

시그니처가 달라도 컴파일러가 오류를 내지 않는다.
의도하지 않은 함수가 생성된다.

---

#### object slicing

```cpp
Derived d;
Base b = d;
```

Derived의 정보가 잘려나간다.
다형성이 유지되지 않는다.

---

### 7. 사용 기준

virtual은 다음 조건에서 사용한다:

- 기반 포인터 또는 참조로 객체를 다룰 때
- 실제 타입에 따라 동작이 달라야 할 때

사용하지 않는 경우:

- 정적 타입으로 충분한 경우
- 성능이 중요한 반복 루프 내부

---

### 8. 판단 기준 요약

- 런타임 다형성이 필요한가?
- 기반 타입으로 다룰 것인가?
- 호출이 성능 병목 구간에 있는가?

이 세 가지 기준으로 virtual 사용 여부를 결정한다.

## 토픽 15: 연산자 오버로딩

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 12

### 1. 문제 상황

핵심 용어: `함수 객체(functor)`, `operator+=`, `비멤버`, `explicit`.

사용자 정의 타입(벡터, 행렬, 색상 등)에 +, -, ==, << 같은 연산자를 쓰고 싶다. 연산자 오버로딩 없이는 Add(), Equals() 같은 멤버 함수를 호출해야 해서 코드가 장황해진다.

그러나 직관에 맞지 않는 의미로 연산자를 오버로딩하면 읽는 사람이 혼란스럽다. a + b가 실제로는 삭제 동작이라면 코드 신뢰가 무너진다.

### 2. 핵심 개념

#### 정의

연산자 오버로딩은 기존 연산자에 사용자 정의 타입에 대한 의미를 부여하는 기법이다. 핵심 원칙: 기존 의미와 일관된 동작만 정의하라.

멤버 함수로 정의하면 왼쪽 피연산자가 this가 된다. 비멤버(friend) 함수로 정의하면 양쪽 피연산자를 대칭적으로 처리할 수 있다.

conversion operator(변환 연산자)는 객체를 다른 타입으로 암시적/명시적 변환한다. explicit이 없으면 의도치 않은 변환이 일어난다.

#### 이 개념을 이해하려면

- 생성자/소멸자 (토픽 12): 복사/이동 대입 연산자
- const correctness (토픽 7): const 멤버 함수로서의 비교 연산자

#### 이 개념이 영향을 주는 것

- move semantics (토픽 16): 이동 대입 연산자
- STL 컨테이너 (토픽 20): 비교 연산자가 정렬/검색의 전제

### 3. 내부 동작 / 메커니즘

연산자 오버로딩은 함수 호출의 문법적 설탕이다. a + b는 a.operator+(b) 또는 operator+(a, b)로 변환된다.

오버로딩 가능한 연산자: +, -, *, /, ==, !=, <, >, <<, >>, [], (), ->, = 등. 오버로딩 불가: ::, ., .*, ?:, sizeof.

spaceship operator(<=>)는 C++20에서 도입되어 ==, !=, <, >, <=, >=를 한 번에 생성한다.

### 4. 자주 틀리는 포인트

- 읽는 사람이 예상하지 않는 의미로 오버로딩한다 →+ 연산자가 삭제를 수행하면 코드 신뢰가 무너진다.
- 암시적 conversion operator를 만든다 →예상치 못한 변환이 일어난다. explicit을 기본으로 붙여라.
- 비대칭 연산자를 멤버로만 정의한다 →1 + myObj가 컴파일 안 될 수 있다. 대칭 연산자는 비멤버 friend로 정의하라.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

오버로딩이 적절한 경우: 수학 타입(벡터, 행렬), 비교, 스트림 입출력, 스마트 포인터의 ->와 *.

오버로딩을 피해야 하는 경우: 의미가 직관적이지 않을 때. 이름 있는 함수(Merge, Append 등)가 더 명확하다.

UE5에서: FVector, FRotator 등은 연산자가 오버로딩되어 있다. 직접 만드는 타입에서는 엔진 규칙을 따르라.

### 6. 코드 예시 (bad →good)

#### Bad: 직관에 맞지 않는 오버로딩

```cpp
// + 연산자가 실제로는 원소를 제거한다면?
MyList operator+(const MyList& a, const MyList& b);
```

#### Good: 의미가 명확한 오버로딩

```cpp
FVector operator+(const FVector& A, const FVector& B) {
    return FVector(A.X + B.X, A.Y + B.Y, A.Z + B.Z);
}
```

### 7. 판단 기준 요약

- 기존 연산자 의미와 일관되게 정의하라
- conversion operator에는 explicit을 기본으로 붙여라
- 대칭 연산자(+, ==)는 비멤버 friend로 정의하라
- 의미가 애매하면 이름 있는 함수를 써라
- C++20 spaceship(<=>)로 비교 연산자를 한 번에 생성할 수 있다
