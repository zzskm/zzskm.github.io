## 토픽 29: undefined behavior(UB) & 안정성

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 2, 11, 12

### 1. 문제 상황

핵심 용어: `undefined behavior(UB, 정의되지 않은 동작)`.

디버그 빌드에서는 멀쩡한데 릴리스에서만 크래시가 난다. 또는 크래시 없이 데이터가 조용히 손상된다. 이런 문제의 상당수는 undefined behavior(UB)에서 시작한다.

UB는 '운이 나쁘면 크래시 나는 버그' 수준이 아니다. 언어 계약을 어긴 상태이므로 컴파일러가 '이 코드는 정상'이라는 가정 하에 최적화한 결과가 전혀 예측할 수 없는 형태로 나타난다. 현상만 보고 원인을 단정하면 오판한다.

### 2. 핵심 개념

#### 정의

undefined behavior(UB, 정의되지 않은 동작)는 프로그램이 언어 규칙을 어겼을 때 결과를 보장하지 않는 상태다. 크래시할 수도 있고, 겉으로 멀쩡해 보일 수도 있으며, 디버그와 릴리스에서 전혀 다른 증상으로 나타날 수 있다.

예외는 언어가 정의한 실패 전파 메커니즘이다. UB는 언어 계약 밖으로 나간 상태다. 예외는 처리 전략을 설계할 수 있지만, UB는 예방과 조기 발견이 우선이다.

#### UB 의심 체크리스트

- lifetime 위반: use-after-free, dangling pointer, 스코프 밖 참조 반환
- 캐스팅 오류: 잘못된 다운캐스트(실제 타입과 불일치), reinterpret_cast 남용
- 초기화 누락: 초기화되지 않은 변수 사용 (특히 지역 변수)
- signed integer overflow: 부호 있는 정수 오버플로우는 UB (unsigned는 wrap)
- 정렬 위반: 타입의 정렬 요구를 무시한 메모리 접근
- null 역참조: nullptr를 역참조
- data race: 동기화 없이 여러 스레드가 같은 데이터를 읽기/쓰기

#### 이 개념을 이해하려면

- 포인터 vs 참조 (토픽 2): lifetime과 소유권의 기초
- 캐스팅 4종 (토픽 11): 타입 변환의 안전성 수준
- 생성자/소멸자 (토픽 12): 객체 수명과 초기화

#### 이 개념이 영향을 주는 것

- crash 분석 (토픽 30): UB가 크래시의 근본 원인인 경우가 많다
- 에러 핸들링 (토픽 22): UB는 실패 모델이 아니라 계약 위반이다

### 3. 내부 동작 / 메커니즘

#### 왜 릴리스에서만 터지는가

최적화가 들어가면 컴파일러가 '정상 코드라면 성립해야 하는 가정'을 더 강하게 활용한다. 예를 들어 null 체크 뒤에 역참조가 있으면 '이 포인터는 null이 아니다'라고 가정하고 null 체크를 제거할 수 있다.

디버그에서는 메모리가 0으로 초기화되거나 여분 패딩이 있어 우연히 동작하던 코드가, 릴리스에서는 다른 레지스터 배치와 최적화로 깨진다. 릴리스에서만 터진다는 것은 '릴리스 전용 버그'가 아니라 '원래 숨어 있던 UB가 드러난 것'이다.

#### UB 탐지 전략

- Sanitizer: AddressSanitizer(ASan) = use-after-free/buffer overflow, UndefinedBehaviorSanitizer(UBSan) = signed overflow/정렬 위반 등, ThreadSanitizer(TSan) = data race. 빌드 플래그 하나로 런타임 검사를 추가한다. 개발 초기부터 CI에 넣는 것이 가장 효과적이다.
- 정적 분석: clang-tidy, PVS-Studio 등. 코드를 실행하지 않고 패턴으로 잠재적 UB를 찾는다. false positive가 있지만 비용이 낮다.
- assert/check: 함수 진입 시 전제 조건을 명시적으로 검증한다. UB가 발생하기 전에 중단점을 잡을 수 있다.
- 코드 설계: ownership을 타입으로 표현(스마트 포인터), raw pointer를 비소유 의미로 제한, 초기화 강제(NSDMI). '잘못 쓰기 어렵게 만드는 설계'가 가장 강력한 예방이다.

### 4. 자주 틀리는 포인트

- UB를 '운이 나쁘면 크래시 나는 버그' 정도로 가볍게 본다: 실제로는 조용한 데이터 손상과 간헐 버그로 더 오래 남는 경우가 많다. 크래시는 차라리 나은 결과다.
- 릴리스에서만 재현되면 도구 없이 로그만 늘리며 추측으로 고친다: sanitizer, 정적 분석, invalidation 경로 점검 같은 구조적 접근이 먼저다. 로그는 가설을 좁히는 보조 수단이다.
- '디버그에서 됐으니까 맞다'고 생각한다: 디버그 빌드의 초기화/패딩이 UB를 숨길 수 있다. sanitizer를 붙인 디버그 빌드와 릴리스 빌드 모두에서 테스트해야 한다.
- signed overflow를 의도적으로 사용한다: C++에서 signed integer overflow는 UB다. 의도적 wrap이 필요하면 unsigned를 사용한다.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

#### sanitizer를 쓰는 시점

- 개발 초기부터 CI에 ASan/UBSan 빌드를 포함시킨다
- 멀티스레드 코드를 작성하면 TSan을 함께 돌린다
- 간헐 크래시가 재현이 안 될 때 sanitizer 빌드로 확인한다
- 판단 기준: sanitizer는 '문제가 생기면' 쓰는 것이 아니라 '항상' 돌리는 것이다

#### 정적 분석을 쓰는 시점

- 코드 리뷰 전 자동 검사로 명백한 패턴을 잡을 때
- 레거시 코드를 처음 접할 때 잠재적 위험을 스캔할 때
- 판단 기준: false positive를 감수하더라도 비용 대비 효과가 좋다

### 6. 코드 예시 (bad → good)

#### Bad: 대표적 UB 사례들

```cpp
// 1. null 역참조
int* Ptr = nullptr;
int Val = *Ptr;          // UB

// 2. use-after-free
int* Arr = new int[4];
delete[] Arr;
int X = Arr[0];          // UB

// 3. 잘못된 다운캐스트
Base* B = new Base();
auto* D = static_cast<Derived*>(B);  // 실제 Derived가 아니면 UB

// 4. signed overflow
int Max = INT_MAX;
int Over = Max + 1;      // UB
```

이 예시는 서로 다른 형태의 UB가 모두 '가끔만 터지는 위험한 코드'라는 공통점을 가진다는 점을 보여 준다. 증상이 제각각이라도 원인은 언어 계약 위반이라는 축으로 묶어 봐야 한다.

#### Good: 타입과 도구로 UB 예방

```cpp
// 1. 스마트 포인터로 lifetime 관리
auto Ptr = std::make_unique<int>(42);
// Ptr이 스코프를 벗어나면 자동 해제, dangling 불가

// 2. dynamic_cast로 안전한 다운캐스트
if (auto* D = dynamic_cast<Derived*>(B)) {
    D->DerivedMethod();  // 실제 타입 확인 후 접근
}

// 3. unsigned로 의도적 wrap
uint32_t Counter = UINT32_MAX;
Counter += 1;  // 0으로 wrap, 정의된 동작
```

좋은 코드는 UB를 사후에 잡기보다 표현 단계에서 막는다. lifetime, 형변환, overflow 의도를 타입과 도구에 맡기면 디버깅 부담이 훨씬 줄어든다.

### 7. 판단 기준 요약

- UB는 실패 모델이 아니라 언어 계약 위반이다. 예방과 조기 탐지가 핵심이다
- 릴리스에서만 터지면 UB를 가장 먼저 의심한다. sanitizer 빌드로 확인한다
- UB 탐지 우선순위: sanitizer(ASan/UBSan/TSan) > 정적 분석 > assert > 로그
- 가장 강력한 예방은 '잘못 쓰기 어렵게 만드는 설계'다: 스마트 포인터, NSDMI, const 기본
- signed overflow는 UB, unsigned wrap은 정의된 동작. 의도적 wrap이 필요하면 unsigned를 사용한다

## 토픽 30: crash 분석 / callstack / dump / leak / dangling / 병목 분석

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 21, 29

### 1. 문제 상황

핵심 용어: `callstack(콜스택, 호출 경로)`, `dump(덤프, 프로세스 메모리 스냅샷)`.

크래시가 났는데 callstack 한 줄만 보고 원인이라고 단정한다. 메모리가 계속 늘어나는데 leak인지 정상 캐싱인지 구분하지 못한다. 느린데 어디가 느린지 모르면서 자료구조부터 바꾼다.

실전 디버깅은 증상을 나열하는 것이 아니라 재현 조건, 원인 후보, 검증 방법을 분리하는 작업이다. 면접에서도 '무슨 도구를 눌렀는가'보다 '어떤 가설을 세우고 무엇으로 검증했는가'가 더 중요하다.

### 2. 핵심 개념

#### 정의

callstack(호출 경로)은 크래시 시점에서 어떤 함수 호출 순서를 거쳐 왔는지 보여준다. '어디서 터졌는지'의 단서이지 '왜 터졌는지'의 직접적인 답은 아니다.

dump(프로세스 메모리 스냅샷)는 프로세스 종료 시점의 스레드 상태, 메모리, callstack을 기록한 파일이다. 재현이 어려운 문제에서 남아 있는 가장 구체적인 증거다.

leak은 메모리를 할당하고 해제하지 않아 점유가 계속 증가하는 문제다. dangling pointer는 이미 해제된 메모리를 다시 참조하는 문제다. 둘 다 수명 문제이지만 증상 패턴이 다르다.

#### 이 개념을 이해하려면

- UB & 안정성 (토픽 29): UB가 크래시의 근본 원인인 경우가 많다
- 멀티스레딩 심화 (토픽 28): race condition과 데드락 디버깅
- 컨테이너 / iterator invalidation (토픽 27): invalidation이 dangling의 흔한 원인

#### 이 개념이 영향을 주는 것

- 모든 실무 코드: 디버깅 역량은 코드 작성 능력만큼 중요하다
- 면접: '실제로 어떻게 문제를 좁혔는가'를 묻는 경우가 많다

### 3. 내부 동작 / 메커니즘

#### 디버깅 순서

- 1. 재현 조건 확인: 언제, 어떤 입력에서, 어떤 빈도로 발생하는가
- 2. callstack + 로그: 터진 지점과 마지막 유효 상태를 확인한다
- 3. dump 분석: 재현이 어려우면 dump에서 스레드 상태와 메모리를 본다
- 4. 가설 수립: 원인 후보를 2-3개 좁힌다 (lifetime? race? invalidation?)
- 5. 검증: sanitizer, 조건부 assert, 최소 재현 코드로 가설을 확인/반증한다
- 핵심: 코드 수정부터 시작하지 않는다. 증거를 먼저 확보한다.

#### leak vs dangling 구분

leak: 장시간 세션에서 메모리 사용량이 계속 증가한다. 메모리 그래프가 우상향이면 leak을 의심한다. 도구: Visual Studio 진단, Valgrind, UE Memreport.

dangling pointer: 간헐 크래시, 데이터 손상, 릴리스에서만 재현. callstack이 해제된 메모리 영역을 가리키거나, ASan이 use-after-free를 보고한다. invalidation 경로와 포인터 수명을 함께 확인한다.

둘 다 수명 문제이지만 접근이 다르다. leak은 '누가 해제를 안 했는가', dangling은 '누가 해제 후에도 접근하는가'를 추적한다.

#### 병목 분석 순서

- 1. profiler로 가장 비싼 구간을 확인한다 (UE: stat unit, Unreal Insights, 외부: VTune, Tracy)
- 2. CPU bound인지 GPU bound인지 나눈다
- 3. CPU라면: 연산 문제(알고리즘) vs 메모리 접근 문제(캐시 미스) vs 동기화 문제(락 경합)로 분류한다
- 4. 측정 결과를 기반으로 수정한다. '느려 보이는 부분'이 아니라 '실제로 시간이 쓰이는 부분'을 고친다

### 4. 자주 틀리는 포인트

- callstack 한 줄만 보고 원인이라고 단정한다: callstack은 터진 지점이지 근본 원인이 아닐 수 있다. 상위 프레임과 호출 맥락을 함께 본다.
- dump 없이 재현만 기다린다: 간헐 문제는 재현 조건을 모를 수 있다. dump가 남아 있으면 사후 분석이 가능하다. 반대로 dump만 보고 실행 경로 검증을 생략해도 안 된다.
- 측정 없이 성능 최적화를 시작한다: '느린 것 같다'는 감각으로 자료구조나 스레드 구조를 바꾸면 잘못된 곳을 고치게 된다. profiler로 실제 비용을 확인한 뒤 수정한다.
- leak과 정상 캐싱을 구분하지 않는다: 캐시 크기가 제한되어 있고 적절히 evict되면 정상이다. 메모리가 무한히 증가하는지, 특정 동작 후 회수되는지를 구분한다.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

#### dump를 쓰는 경우

- 현장에서 바로 재현하기 어려운 크래시
- 이미 종료된 프로세스의 상태를 나중에 분석해야 할 때
- QA나 유저 환경에서만 발생하는 문제
- 판단 기준: 재현이 어려울수록 dump의 가치가 커진다

#### profiler를 쓰는 경우

- 프레임 시간이 목표를 초과할 때
- 특정 동작에서 눈에 띄는 히치가 있을 때
- 최적화를 시작하기 전 — 반드시 먼저 측정한다
- 판단 기준: '어디가 느린지 모르면 고칠 수 없다'가 원칙이다

### 6. 코드 예시 (bad → good)

#### Bad: 컨테이너 재할당으로 dangling pointer

```cpp
std::vector<int> Values = {1, 2, 3};
int* Saved = &Values[0];
Values.push_back(4);       // 재할당 →Saved는 dangling
std::cout << *Saved;       // UB / crash
```

이 패턴은 크래시와 UB 분석에서 가장 자주 만나는 형태 중 하나다. 증상은 역참조 시점에 보이지만 실제 원인은 그보다 앞선 재할당과 참조 보관 방식에 있다.

#### Good: 인덱스로 안전하게 접근

```cpp
std::vector<int> Values = {1, 2, 3};
size_t Idx = 0;            // 포인터 대신 인덱스 저장
Values.push_back(4);
std::cout << Values[Idx];  // 항상 안전
```

인덱스는 컨테이너 내부 버퍼 주소에 직접 매달리지 않는다. 주소 안정성이 필요 없는 경우라면 포인터보다 훨씬 보수적인 선택이다.

#### 디버깅 접근 순서 요약

```cpp
// 1. 크래시 →callstack 확인
//    callstack: MyActor::Tick →UpdateHealth →*DamagedPtr
//    →DamagedPtr이 dangling인가?

// 2. ASan 빌드로 재실행
//    →"heap-use-after-free" 보고
//    →해제 시점의 callstack도 함께 출력

// 3. 해제 시점 추적
//    →다른 시스템이 먼저 파괴한 액터를 계속 참조
//    →약한 참조(TWeakObjectPtr)로 교체
```

중요한 점은 코드를 바로 고치기보다 증거를 먼저 쌓는 순서를 지키는 것이다. callstack으로 지점을 좁히고, sanitizer로 가설을 검증하고, 마지막에 lifetime 설계를 수정해야 재발을 줄일 수 있다.

### 7. 판단 기준 요약

- 디버깅은 '재현 →증거 확보 →가설 →검증' 순서를 따른다. 코드 수정부터 시작하지 않는다
- callstack은 '어디서 터졌는가'이지 '왜 터졌는가'가 아니다. 상위 프레임과 맥락을 함께 본다
- leak은 메모리 증가 패턴, dangling은 간헐 크래시/데이터 손상 패턴으로 구분한다
- 성능 최적화는 측정이 먼저다. profiler 결과 없이 자료구조를 바꾸지 않는다
- 도구 숙련보다 원인 후보를 체계적으로 줄여 가는 사고 방식이 디버깅 역량의 핵심이다
