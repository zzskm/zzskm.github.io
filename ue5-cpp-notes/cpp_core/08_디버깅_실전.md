## 토픽 29: undefined behavior(UB) & 안정성

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 2, 11, 12

### 1. 핵심 개념

핵심 용어: `undefined behavior(UB, 정의되지 않은 동작)`.

`undefined behavior(UB, 정의되지 않은 동작)`는 프로그램이 언어 규칙을 어겼을 때 결과를 보장하지 않는 상태다. null 역참조, use-after-free, 잘못된 다운캐스트, 정렬 위반, signed overflow, 초기화되지 않은 값 사용처럼 종류도 다양하다. 핵심은 "크래시할 수도 있고, 멀쩡해 보일 수도 있다"는 점이다.

안정성은 단순히 크래시를 안 내는 것이 아니라 lifetime, ownership, 타입 규칙, 동기화 계약을 지키는 것이다. 예외나 assert는 정의된 실패 경로지만, UB는 실패 경로 자체가 보장되지 않기 때문에 예방과 조기 발견이 더 중요하다.

### 2. 코드 예시

```cpp
class Base
{
public:
    virtual ~Base() = default;
};

class Derived : public Base
{
};

int* Ptr = nullptr;
// std::cout << *Ptr << "\n"; // ❌ UB: null 역참조

Base* BasePtr = new Base();
// auto* DerivedPtr = static_cast<Derived*>(BasePtr); // ❌ 실제 객체가 Derived가 아니면 UB 가능

int* Numbers = new int[4];
delete[] Numbers;
// std::cout << Numbers[0] << "\n"; // ❌ UB: use-after-free
```

게임 크래시 중 상당수는 "문법이 틀린 코드"보다 lifetime, 캐스팅, 컨테이너 invalidation, 멀티스레드 접근 같은 UB에서 시작한다. 특히 디버그에선 안 터지고 릴리스 빌드에서만 터지는 사례가 많아 면접에서도 자주 물어본다.

### 3. 왜 중요한가 (게임/UE5 맥락)

게임 크래시 중 상당수는 "문법이 틀린 코드"보다 lifetime, 캐스팅, 컨테이너 invalidation, 멀티스레드 접근 같은 UB에서 시작한다. 특히 디버그에선 안 터지고 릴리스 빌드에서만 터지는 사례가 많아 면접에서도 자주 물어본다.

### 4. 면접 예상 Q&A

**Q1: UB를 한 문장으로 설명하면 무엇인가요?**
> A: 언어 규칙을 어겨 결과가 보장되지 않는 상태라고 설명하면 된다. 예외처럼 정해진 실패 모델이 아니라서, 크래시 대신 조용한 데이터 손상으로 나타날 수도 있다.

**Q2: 왜 UB는 릴리스에서만 보이는 경우가 많나요?**
> A: 최적화가 들어가면 컴파일러가 "정상 코드라면 성립해야 하는 가정"을 더 강하게 사용하기 때문이다. 디버그에서는 우연히 동작하던 코드가 릴리스에서는 다른 순서와 레지스터 배치로 깨질 수 있다.

**Q3: 예외와 UB는 어떻게 다른가요?**
> A: 예외는 언어가 정의한 실패 전파 메커니즘이고, UB는 언어 계약 밖으로 나간 상태다. 예외는 처리 전략을 설계할 수 있지만, UB는 예방과 도구 기반 탐지가 우선이다.

**Q4: UB를 줄이는 실무 습관은 무엇인가요?**
> A: ownership을 타입으로 표현하고, raw pointer를 비소유 의미로 제한하고, sanitizer/정적 분석/assert를 조기에 적용하는 것이다. 즉 "문제가 생기면 본다"보다 "애초에 잘못 쓰기 어렵게 만든다"가 중요하다.

### 5. 흔한 실수

- UB를 "운이 나쁘면 크래시 나는 버그" 정도로 가볍게 본다.
- 릴리스에서만 재현되면 도구 없이 로그만 늘리며 추측으로 고친다.

## 토픽 30: crash 분석 / callstack / dump / leak / dangling / 병목 분석

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 21, 29

### 1. 핵심 개념

핵심 용어: `callstack(콜스택, 호출 경로)`, `dump(덤프, 프로세스 메모리 스냅샷)`.

실전 디버깅은 "증상을 설명하는 것"이 아니라 "재현 조건, 원인, 검증 방법"을 분리하는 작업이다. crash가 나면 먼저 `callstack(콜스택, 호출 경로)`과 로그로 마지막 유효 상태를 확인하고, 필요하면 `dump(덤프, 프로세스 메모리 스냅샷)`를 열어 스레드 상태와 메모리 내용을 본다. leak, dangling pointer, iterator invalidation, race는 증상이 비슷해 보여도 추적 순서가 다르다.

병목 분석도 마찬가지다. 느리다는 감각만으로 고치지 않고 profiler로 CPU/GPU 시간을 확인하고, hot path의 데이터 배치와 동기화 비용을 따로 본다. 면접에서는 "무슨 도구를 눌렀다"보다 "어떤 가설을 세우고 무엇으로 검증했는가"가 중요하다.

### 2. 코드 예시

```cpp
std::vector<int> Values = {1, 2, 3};
int* Saved = &Values[0];

Values.push_back(4); // ← 재할당이 일어나면 Saved는 dangling pointer가 될 수 있다
// std::cout << *Saved << "\n"; // ❌ UB / crash 가능
```

면접에서는 "실제로 어떻게 문제를 좁혔는가"를 묻는 경우가 많다. UE5 실무도 로그만으로 끝나지 않고 callstack, minidump, stat 명령, 트레이스, 프로파일러를 조합해 원인을 좁혀야 한다.

### 3. 왜 중요한가 (게임/UE5 맥락)

면접에서는 "실제로 어떻게 문제를 좁혔는가"를 묻는 경우가 많다. UE5 실무도 로그만으로 끝나지 않고 callstack, minidump, stat 명령, 트레이스, 프로파일러를 조합해 원인을 좁혀야 한다.

### 4. 면접 예상 Q&A

**Q1: crash가 나면 가장 먼저 무엇을 보나요?**
> A: 재현 조건을 확인한 뒤 callstack과 실패 시점의 로그를 먼저 본다고 답하는 것이 기본이다. callstack은 "어디서 터졌는가"를, 로그와 변수 상태는 "왜 그 지점까지 왔는가"를 좁히는 데 도움을 준다.

**Q2: dump는 언제 유용한가요?**
> A: 현장에서 바로 재현하기 어렵거나, 이미 종료된 프로세스 상태를 나중에 분석해야 할 때 유용하다. dump 안에는 스레드 상태, 메모리 일부, callstack 정보가 남아 있어 사후 분석의 단서가 된다.

**Q3: leak와 dangling pointer는 어떻게 구분하나요?**
> A: leak는 메모리를 놓쳐서 점유가 계속 증가하는 문제이고, dangling pointer는 이미 끝난 객체를 다시 참조하는 문제다. 전자는 장기 세션에서 메모리 증가로 보이고, 후자는 간헐 크래시나 데이터 손상으로 드러나기 쉽다.

**Q4: 성능 병목은 어떤 순서로 분석하나요?**
> A: 먼저 profiler로 가장 비싼 구간을 확인하고, 그 다음 그 구간이 연산 문제인지 메모리 접근 문제인지 동기화 문제인지 나눈다. 측정 없이 자료구조나 스레드 구조를 바꾸는 것은 가장 피해야 할 패턴이다.

### 5. 흔한 실수

- callstack 한 줄만 보고 원인이라고 단정한다.
- dump 없이 재현만 기다리거나, 반대로 dump만 보고 실행 경로 검증을 생략한다.
