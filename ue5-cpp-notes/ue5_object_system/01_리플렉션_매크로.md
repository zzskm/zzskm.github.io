## 토픽 1: UCLASS / UPROPERTY / UFUNCTION / USTRUCT / UENUM

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 없음

### 1. 문제 상황

UE 프로젝트에서는 코드와 콘텐츠가 끊임없이 왕복한다. “블루프린트에서 변수가 안 보인다”, “저장이 안 된다”, “GC가 객체를 너무 빨리 수거한다” 같은 문제가 자주 발생한다. 대부분의 원인은 **엔진이 해당 타입을 제대로 인식하지 못하기 때문**이다.

UE5의 리플렉션 시스템은 C++ 타입에 메타데이터를 붙여 에디터, 블루프린트, 직렬화, GC가 타입을 이해할 수 있게 만든다. `UCLASS`, `UPROPERTY`, `UFUNCTION`, `USTRUCT`, `UENUM`은 C++ 문법을 바꾸는 키워드가 아니라, Unreal Header Tool(UHT)이 스캔하는 표식이다.

매크로를 붙이는 행위는 단순한 편의 기능이 아니라, “이 타입을 엔진 서비스에 얼마나 노출할 것인가”를 결정하는 중요한 설계 경계 선언이다.

### 2. 핵심 개념

순수 C++ 클래스는 컴파일러에게만 의미가 있는 메모리 덩어리일 뿐이다. 반면 UE 매크로가 붙은 타입은 리플렉션 메타데이터로 변환되어 엔진 전체가 읽고 활용할 수 있게 된다.

- `UCLASS` — 클래스가 UObject 계열임을 선언하고 리플렉션에 등록  
- `UPROPERTY` — 멤버를 GC 추적, 직렬화, 에디터 노출 대상으로 지정  
- `UFUNCTION` — 함수를 블루프린트 호출, RPC, 델리게이트 대상으로 지정  
- `USTRUCT` — 값 타입 데이터 묶음을 리플렉션에 노출  
- `UENUM` — 열거형을 블루프린트와 에디터에서 타입 안전하게 사용  

핵심은 **“엔진이 알아야 하는 선언”**과 **“순수 C++ 내부 선언”**을 명확히 구분하는 것이다. 모든 멤버에 무조건 매크로를 붙이면 캡슐화가 무너지고 유지보수 비용이 증가한다.

#### 이 개념이 영향을 주는 것

- GC 추적 경로
- 블루프린트·에디터 노출
- 직렬화와 저장 시스템
- 블루프린트와 C++ 협업

### 3. 내부 동작 / 메커니즘

UHT가 헤더를 분석해 매크로를 발견하면 `.generated.h` 파일과 리플렉션 테이블(`UClass`, `UProperty`, `UFunction` 등)을 생성한다. 런타임에는 이 테이블을 통해 타입 정보를 조회한다.  
`UPROPERTY`가 붙은 `UObject*`는 자동으로 GC 참조 그래프에 포함되고, `UFUNCTION`은 블루프린트 호출 테이블에 등록되어 `ProcessEvent`로 실행될 수 있다.

### 4. 자주 틀리는 포인트

- GC 추적이 필요한 `UObject*`에 `UPROPERTY`를 빠뜨린다.
- 블루프린트에서 호출할 필요 없는 내부 헬퍼 함수에도 `UFUNCTION`을 붙인다.
- 모든 멤버를 `UPROPERTY`로 열어 캡슐화를 포기한다.
- 값 타입 데이터인데 `UCLASS`로 만든다.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

**우선 검토하는 경우**

- 엔진이 GC·직렬화·에디터·블루프린트에서 알아야 하는 선언
- 블루프린트에서 호출하거나 편집해야 하는 함수·변수
- 값 데이터 묶음을 블루프린트에서 사용하고 싶을 때 (`USTRUCT`)
- 열거형을 에디터에서 안전하게 다루고 싶을 때 (`UENUM`)

**주의하거나 분리할 경우**

- 순수 C++ 내부 헬퍼 함수와 변수
- GC나 직렬화가 전혀 필요 없는 단순 POD 타입
- 성능이 가장 중요한 핫패스 내부 데이터

### 6. 코드 예시 (bad → good)

**Bad: GC 추적 누락**

```cpp
UCLASS()
class UMyInventory : public UObject
{
    GENERATED_BODY()
public:
    UObject* CurrentItem;   // 엔진이 모르는 참조
};
```

**Good: 명시적 추적**

```cpp
UCLASS()
class UMyInventory : public UObject
{
    GENERATED_BODY()
public:
    UPROPERTY()
    UObject* CurrentItem = nullptr;
};
```

**Bad: 값 타입에 UCLASS 남용**

```cpp
UCLASS()
class UWeaponData : public UObject { ... };
```

**Good: USTRUCT 사용**

```cpp
USTRUCT(BlueprintType)
struct FWeaponData
{
    GENERATED_BODY()
    UPROPERTY(EditAnywhere, BlueprintReadOnly)
    int32 Damage = 10;
};
```

### 7. 판단 기준 요약

- 엔진 서비스(GC, 직렬화, 에디터, 블루프린트)에 노출해야 하는가? → 매크로를 붙인다.
- 순수 C++ 내부 전용인가? → 매크로를 생략한다.
- 값 데이터 묶음인가? → `USTRUCT`를 사용한다.
- “엔진이 알아야 하는 최소한”만 노출하라. 과도한 노출은 설계 부채가 된다.

## 토픽 2: 지정자 심화

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 1

### 1. 문제 상황

같은 `UPROPERTY`라도 어떤 지정자를 붙이느냐에 따라 에디터에서 수정 가능한지, 저장되는지, 서브오브젝트로 취급되는지가 완전히 달라진다. 디자이너가 바꾸면 안 되는 값을 손대거나, 반대로 꼭 조정해야 할 값이 코드에만 묶이는 실수가 자주 발생한다.

지정자는 단순한 문법 장식이 아니라, “누가, 언제, 어디서 값을 바꿀 수 있는가”를 선언하는 정책이다.

### 2. 핵심 개념

지정자는 프로퍼티와 함수의 엔진 취급 방식을 결정한다. 대표적인 예로 `EditAnywhere`, `EditDefaultsOnly`, `VisibleAnywhere`, `Transient`, `Instanced`, `meta=(AllowPrivateAccess="true")` 등이 있다.

핵심 판단 기준은 항상 **“누가, 언제, 어디서 무엇을 바꿀 수 있어야 하는가”**다.

#### 이 개념을 이해하려면

- 토픽 1

#### 이 개념이 영향을 주는 것

- 토픽 1, 토픽 4: 리플렉션 API

### 3. 내부 동작 / 메커니즘

지정자는 데이터의 성격(편집 가능성)과 수명 정책(저장 여부)을 동시에 반영한다. `Transient`는 저장에서 제외하고, `Instanced`는 서브오브젝트 소유 관계를 명시하며, `BlueprintSpawnableComponent`는 블루프린트에서 컴포넌트를 쉽게 추가할 수 있게 한다.

### 4. 자주 틀리는 포인트

- 편의만으로 모든 값을 `EditAnywhere`로 연다.
- 저장되면 안 되는 런타임 캐시를 `Transient` 없이 둔다.
- VisibleAnywhere와 Transient를 혼동한다.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

**우선 검토하는 경우**

- 인스턴스별 값 편차를 허용할지
- 저장하면 안 되는 런타임 캐시
- 서브오브젝트 소유 관계를 명확히 하고 싶을 때

**주의하거나 분리할 경우**

- 모든 값을 공개하는 편의 중심 설계
- 편집 정책과 저장 정책을 같은 축으로 판단하는 경우

### 6. 코드 예시 (bad → good)

**Bad**

```cpp
UPROPERTY(EditAnywhere)
int32 CachedRuntimeScore = 0;
```

**Good**

```cpp
UPROPERTY(Transient)
int32 CachedRuntimeScore = 0;
```

**Bad**

```cpp
UCLASS(ClassGroup=(Custom))
class UInventoryComponent : public UActorComponent {};
```

**Good**

```cpp
UCLASS(ClassGroup=(Custom), meta=(BlueprintSpawnableComponent))
class UInventoryComponent : public UActorComponent {};
```

### 7. 판단 기준 요약

- 캐시라면 `Transient`
- 디자이너가 추가해야 하는 컴포넌트라면 `BlueprintSpawnableComponent`
- 서브오브젝트 소유라면 `Instanced`
- 클래스 기본값만 관리하려면 `EditDefaultsOnly`
