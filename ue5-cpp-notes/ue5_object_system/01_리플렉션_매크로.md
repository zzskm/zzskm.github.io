## 토픽 1: UCLASS / UPROPERTY / UFUNCTION / USTRUCT / UENUM(유클래스 / 유프로퍼티 / 유펑션 / 유스트럭트 / 유이넘)

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 없음

### 1. 핵심 개념

UE5의 `reflection(리플렉션, 런타임 타입 반영 시스템)`은 "C++ 타입에 메타데이터를 붙여 에디터, 블루프린트, 직렬화, GC가 이해할 수 있게 만드는 시스템"이다. `UCLASS`, `UPROPERTY`, `UFUNCTION`, `USTRUCT`, `UENUM`은 C++ 문법 자체를 바꾸는 키워드가 아니라, UHT가 스캔할 표식이다.

순수 C++ 클래스는 엔진 입장에서 그냥 메모리 덩어리지만, UE 매크로가 붙은 타입은 에디터 노출, 블루프린트 호출, GC 추적, 직렬화 같은 엔진 서비스와 연결된다. 즉 리플렉션 매크로는 문법보다 "엔진이 이해할 정보"를 선언하는 장치다.

### 2. 코드 예시

```cpp
UENUM(BlueprintType)
enum class EWeaponType : uint8
{
    Sword,
    Bow
};

USTRUCT(BlueprintType)
struct FWeaponData
{
    GENERATED_BODY()

    UPROPERTY(EditAnywhere, BlueprintReadOnly)
    int32 Damage = 10;
};

UCLASS(Blueprintable)
class UWeaponComponent : public UActorComponent
{
    GENERATED_BODY()

public:
    UPROPERTY(EditAnywhere, BlueprintReadWrite, Category="Weapon")
    EWeaponType WeaponType = EWeaponType::Sword;

    UFUNCTION(BlueprintCallable, Category="Weapon")
    void Fire();
};
```

### 3. 왜 중요한가 (게임/UE5 맥락)

UE 프로젝트에서 코드와 콘텐츠는 강하게 연결된다. 리플렉션을 이해하지 못하면 "왜 블루프린트에서 안 보이지?", "왜 저장이 안 되지?", "왜 GC가 이 참조를 모르지?" 같은 문제를 구조적으로 설명하기 어렵다.

### 4. 면접 예상 Q&A

**Q1: UE 매크로는 단순 전처리 매크로인데 왜 특별한가요?**
> A: 핵심은 C++ 전처리 자체보다 UHT가 그 표식을 읽는다는 점이다. 즉 표준 컴파일러뿐 아니라 UE 빌드 파이프라인 전체가 이해하는 메타데이터 진입점이다.

**Q2: 모든 멤버에 `UPROPERTY`를 붙이는 것이 좋은가요?**
> A: 아니다. 엔진 시스템이 알아야 하는 멤버에만 붙여야 한다. 불필요한 노출은 캡슐화를 약하게 만들고, 에디터 노이즈와 직렬화 비용을 늘린다.

**Q3: `UCLASS`와 `USTRUCT`는 어떻게 구분하나요?**
> A: 엔진 오브젝트 수명과 리플렉션 서비스 전체가 필요하면 `UCLASS`, 값 타입 데이터 묶음이면 `USTRUCT`가 일반적이다. 결론적으로 객체 정체성이 필요한지, 값 묶음인지가 기준이다.

**Q4: `UFUNCTION`은 언제 붙이는 편이 좋나요?**
> A: 블루프린트 호출, 리플렉션 기반 호출, 델리게이트 바인딩, RPC가 필요할 때 붙인다. C++ 내부 전용 함수까지 무조건 붙일 필요는 없다.

**Q5: `UENUM(BlueprintType)`이 자주 쓰이는 이유는 무엇인가요?**
> A: 블루프린트와 에디터에서 타입 안정성 있게 상태를 다루기 쉽기 때문이다. 단순 정수 상수보다 의미 전달과 편집성이 좋다.

**Q6: 실무에서 리플렉션 매크로 남용의 문제는 무엇인가요?**
> A: 엔진이 알 필요 없는 코드까지 노출되면서 설계 경계가 흐려진다. 즉 편의성 때문에 노출 범위를 넓히면 유지보수 비용이 커진다.

### 5. 흔한 실수

- GC 추적이 필요한 `UObject*`에 `UPROPERTY`를 빼먹는다.
- 블루프린트 노출이 필요 없는 함수까지 전부 `UFUNCTION`으로 선언한다.
- 값 타입 데이터를 굳이 `UObject`로 만들어 할당 비용과 수명 복잡성을 늘린다.

## 토픽 2: 지정자 심화

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 1

### 1. 핵심 개념

지정자(specifier)는 "이 프로퍼티나 함수를 엔진이 어떻게 취급할지"를 정하는 정책이다. 같은 `UPROPERTY`라도 `EditAnywhere`, `EditDefaultsOnly`, `VisibleAnywhere`, `BlueprintReadOnly`, `Transient`, `Instanced`에 따라 완전히 다른 동작을 가진다.

핵심은 "누가, 언제, 어디서, 무엇을 바꿀 수 있어야 하는가"를 먼저 정하는 것이다. 예를 들어 `BlueprintSpawnableComponent`는 블루프린트 에디터에서 컴포넌트 추가를 쉽게 하고, `meta=(AllowPrivateAccess="true")`는 캡슐화와 에디터 접근의 절충점으로 자주 쓰인다.

### 2. 코드 예시

```cpp
UCLASS(ClassGroup=(Custom), meta=(BlueprintSpawnableComponent))
class UInventoryComponent : public UActorComponent
{
    GENERATED_BODY()
};

UCLASS()
class AMyPickup : public AActor
{
    GENERATED_BODY()

private:
    UPROPERTY(EditDefaultsOnly, BlueprintReadOnly, Category="Pickup", meta=(AllowPrivateAccess="true"))
    int32 HealAmount = 25;

    UPROPERTY(EditAnywhere, Instanced, Category="Pickup")
    TObjectPtr<UCurveFloat> RespawnCurve;

    UPROPERTY(Transient)
    float CachedDistance = 0.0f;
};
```

### 3. 왜 중요한가 (게임/UE5 맥락)

지정자는 에디터 사용성과 데이터 안전성을 동시에 결정한다. 설계를 잘못하면 디자이너가 바꾸면 안 되는 값을 건드리게 되거나, 반대로 꼭 조정해야 할 값이 코드에 묶인다.

### 4. 면접 예상 Q&A

**Q1: `EditAnywhere`와 `EditDefaultsOnly`의 차이는 무엇인가요?**
> A: `EditAnywhere`는 클래스 기본값과 인스턴스 모두에서 수정 가능하고, `EditDefaultsOnly`는 기본값만 수정 가능하다. 인스턴스별 편차를 허용할지 여부가 선택 기준이다.

**Q2: `Transient`는 언제 쓰나요?**
> A: 저장되면 안 되는 런타임 계산값에 쓴다. 결론적으로 재생성 가능한 캐시나 일시 상태는 `Transient`가 적절하다.

**Q3: `Instanced`는 어떤 의미인가요?**
> A: 단순 참조가 아니라 소유된 서브오브젝트처럼 다루고 싶을 때 쓴다. 에디터에서 내부 객체를 인라인으로 다루는 패턴과 연결된다.

**Q4: `meta=(AllowPrivateAccess="true")`는 왜 쓰나요?**
> A: 멤버는 private으로 유지하되 블루프린트 읽기 접근을 허용하고 싶을 때다. 캡슐화와 에디터 노출을 절충하는 실무 패턴이다.

**Q5: `BlueprintSpawnableComponent`는 언제 유용한가요?**
> A: 디자이너가 블루프린트 에디터에서 해당 컴포넌트를 바로 추가해야 할 때 유용하다. 코드 전용 컴포넌트와 에디터 조합형 컴포넌트를 구분하는 신호가 된다.

**Q6: 지정자 설계가 잘못됐다는 신호는 무엇인가요?**
> A: `BlueprintReadWrite`와 `EditAnywhere`가 남용돼 상태 변경 경로가 지나치게 많아질 때다. 편집 자유도보다 데이터 일관성이 먼저다.

### 5. 흔한 실수

- 편하다는 이유로 모든 값을 `EditAnywhere`로 연다.
- 저장되면 안 되는 캐시 값을 `Transient` 없이 둔다.
- 블루프린트 접근과 캡슐화 문제를 구분하지 않고 public 필드로만 해결한다.

## 토픽 3: UHT & GENERATED_BODY()(유에이치티 / 제너레이티드 바디, 헤더 툴 / 생성 코드 삽입 매크로)

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 1

### 1. 핵심 개념

UHT(Unreal Header Tool, 유에이치티, 언리얼 헤더 툴)는 C++ 컴파일 전에 헤더를 스캔해 리플렉션 정보를 추출하고 `.generated.h` 및 등록 코드를 생성하는 도구다. `GENERATED_BODY()`는 그 생성 코드가 클래스 안에 삽입될 위치를 지정하는 앵커다.

핵심 흐름은 "헤더 작성 -> UHT 스캔 -> generated 코드 생성 -> 일반 C++ 컴파일"이다. 따라서 많은 UE 빌드 에러는 사실 C++ 문법 에러보다 UHT 규칙 위반에서 시작된다.

### 2. 코드 예시

```cpp
// MyActor.h
#pragma once

#include "CoreMinimal.h"
#include "GameFramework/Actor.h"
#include "MyActor.generated.h"

UCLASS()
class AMyActor : public AActor
{
    GENERATED_BODY()

public:
    UPROPERTY(EditAnywhere)
    int32 Count = 0;
};
```

### 3. 왜 중요한가 (게임/UE5 맥락)

`.generated.h` 위치, 지원되지 않는 선언, 잘못된 매크로 사용은 IDE보다 Unreal Build 단계에서 먼저 깨진다. 이 흐름을 이해하면 빌드 에러를 "C++ 문제"와 "UHT 문제"로 빨리 나눌 수 있다.

### 4. 면접 예상 Q&A

**Q1: UHT는 왜 별도 단계로 존재하나요?**
> A: 표준 C++만으로는 UE가 원하는 수준의 메타데이터와 블루프린트 연동 정보를 얻기 어렵기 때문이다. 그래서 빌드 전에 헤더를 별도로 분석한다.

**Q2: `.generated.h`는 왜 마지막 include 근처에 두나요?**
> A: 생성 코드가 앞선 선언 상태를 전제로 하기 때문이다. 뒤에 다른 include가 끼면 선언 순서와 매크로 상태가 꼬일 수 있어 UE가 규칙화한다.

**Q3: `GENERATED_BODY()`가 빠지면 어떤 문제가 생기나요?**
> A: 생성된 정적 등록 함수와 메타데이터 연결 지점이 사라진다. 결과적으로 리플렉션 타입으로서 필요한 코드가 클래스에 붙지 않는다.

**Q4: UHT 에러와 C++ 컴파일 에러는 어떻게 구분하나요?**
> A: generated 코드 생성 전후 맥락을 보면 된다. 헤더 파싱, 매크로, 리플렉션 선언과 관련된 메시지는 UHT 가능성이 높고, 일반 문법/타입 문제는 C++ 컴파일 단계 가능성이 높다.

**Q5: 모든 C++ 문법을 UHT가 이해하나요?**
> A: 아니다. UHT는 제한된 규칙으로 리플렉션 대상을 스캔한다. 복잡한 템플릿이나 특수한 선언 패턴은 지원 제약이 있을 수 있다.

**Q6: 실무에서 generated 관련 빌드 문제를 줄이는 방법은 무엇인가요?**
> A: 헤더 구조를 단순하게 유지하고, generated include 위치 규칙을 지키며, 리플렉션 대상 선언을 과도하게 복잡하게 만들지 않는 것이다.

### 5. 흔한 실수

- `.generated.h` 뒤에 다른 include를 넣는다.
- 리플렉션 대상 선언을 과도한 템플릿이나 매크로 중첩으로 감싼다.
- UHT 규칙 위반을 일반 C++ 오류로만 보고 디버깅 시간을 낭비한다.

## 토픽 4: 리플렉션 API

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 1, 3

### 1. 핵심 개념

`API(Application Programming Interface, 에이피아이, 프로그래밍 인터페이스)`로서의 리플렉션 API는 런타임에서 `UClass`, `UFunction`, `FProperty` 같은 메타데이터를 읽고 활용하는 도구다. 에디터 확장, 디버그 툴, 자동 바인딩, 데이터 검사, 네트워크 보조 로직에서 자주 등장한다.

핵심은 "타입 정보를 코드 바깥에서 다시 읽을 수 있다"는 점이다. `StaticClass()`, `FindFunctionByName`, `TFieldIterator<FProperty>` 같은 API는 엔진이 생성해 둔 메타데이터에 접근하는 통로다.

### 2. 코드 예시

```cpp
void DumpProperties(UObject* Object)
{
    if (!Object)
    {
        return;
    }

    for (TFieldIterator<FProperty> It(Object->GetClass()); It; ++It)
    {
        UE_LOG(LogTemp, Log, TEXT("Property: %s"), *It->GetName());
    }
}
```

### 3. 왜 중요한가 (게임/UE5 맥락)

런타임 검사 도구, 커스텀 에디터, 자동 UI 생성, 디버깅 보조 기능은 리플렉션 API 이해도에 크게 의존한다. UE의 "데이터 중심" 느낌은 결국 이 메타데이터 활용 능력에서 나온다.

### 4. 면접 예상 Q&A

**Q1: `UClass`는 어떤 정보를 담고 있나요?**
> A: 타입 이름, 상속 관계, 프로퍼티, 함수, 메타데이터 같은 런타임 타입 정보를 담는다. 결론적으로 UE 세계의 타입 객체라고 볼 수 있다.

**Q2: `FProperty`를 직접 순회하는 것은 어떤 상황에서 유용한가요?**
> A: 디버그 덤프, 커스텀 검증, 자동 UI, 직렬화 보조 로직처럼 필드 목록을 동적으로 읽어야 할 때 유용하다.

**Q3: 리플렉션 API와 C++ RTTI는 어떻게 다른가요?**
> A: RTTI는 주로 타입 판별에 가깝고, UE 리플렉션은 프로퍼티와 함수, 메타데이터까지 포함한 엔진 친화적 타입 정보다. 활용 범위가 훨씬 넓다.

**Q4: `FindFunctionByName` 같은 동적 호출은 언제 조심해야 하나요?**
> A: 문자열 기반 조회는 안전성과 성능 측면에서 비용이 있다. 따라서 핫패스보다 툴링, 유연성 중심 경로에 쓰는 편이 적절하다.

**Q5: 에디터 확장에서 리플렉션 API가 중요한 이유는 무엇인가요?**
> A: 사용자가 만든 타입과 필드를 런타임에 일반화된 방식으로 읽어야 하기 때문이다. 즉 고정 타입 지식 없이도 툴이 동작하게 만든다.

**Q6: 실무에서 리플렉션 API를 남용하면 생길 수 있는 문제는 무엇인가요?**
> A: 문자열 기반 로직과 동적 분기가 늘어 정적 안정성이 떨어진다. 강한 타입 구조로 해결 가능한 문제까지 리플렉션으로 푸는 것은 과하다.

### 5. 흔한 실수

- 리플렉션 가능 대상이 아닌 멤버까지 런타임에서 찾을 수 있다고 기대한다.
- 성능 민감 경로에서 반복적으로 동적 조회를 수행한다.
- 문자열 이름 기반 호출을 남용해 리팩터링 안전성을 떨어뜨린다.

## 토픽 21: UINTERFACE / IInterface(유인터페이스 / 아이인터페이스, 언리얼 인터페이스)

> 난이도: ★★★ | 예상 시간: 1.5시간 | 선행: 토픽 1

### 1. 핵심 개념

`UINTERFACE`는 UE 리플렉션 시스템이 인식하는 인터페이스를 선언하는 매크로다. 특이한 점은 **이중 클래스 구조**를 사용한다는 것이다. `UINTERFACE` 매크로가 붙는 `UInterface` 파생 클래스(`U클래스`)는 리플렉션 등록용 껍데기이고, 실제 인터페이스 함수는 접두사 `I`가 붙은 `I클래스`에 선언한다.

블루프린트에서도 호출 가능한 인터페이스 함수를 만들려면 `BlueprintNativeEvent` 또는 `BlueprintImplementableEvent`를 쓰고, C++ 기본 구현은 `_Implementation` 접미사 함수로 제공한다. `Execute_함수명` 정적 함수는 C++에서 인터페이스 함수를 안전하게 호출하는 엔진 생성 래퍼다.

핵심은 "상속 없이 행동 계약을 정의하고, 컴포넌트 vs 인터페이스 선택을 구조적으로 할 수 있어야 한다"는 것이다.

### 2. 코드 예시

```cpp
// 이중 클래스 구조: U클래스(리플렉션용) + I클래스(실제 인터페이스)
UINTERFACE(MinimalAPI, Blueprintable)
class UDamageable : public UInterface
{
    GENERATED_BODY()
};

class IDamageable
{
    GENERATED_BODY()

public:
    // 블루프린트에서도 구현 가능한 인터페이스 함수
    UFUNCTION(BlueprintNativeEvent, Category="Damage")
    float TakeDamage(float DamageAmount, AActor* DamageCauser);

    // C++ 전용 인터페이스 함수 (순수 가상)
    virtual bool IsDead() const = 0;
};

// ---- 구현 측 ----
UCLASS()
class AMyCharacter : public ACharacter, public IDamageable
{
    GENERATED_BODY()

public:
    // BlueprintNativeEvent의 C++ 기본 구현
    virtual float TakeDamage_Implementation(float DamageAmount, AActor* DamageCauser) override;

    virtual bool IsDead() const override { return Health <= 0.f; }

private:
    UPROPERTY()
    float Health = 100.f;
};

// ---- 호출 측 ----
void ApplyDamage(AActor* Target, float Amount, AActor* Causer)
{
    // 방법 1: Implements 체크 후 Execute_ 래퍼로 안전 호출
    if (Target->GetClass()->ImplementsInterface(UDamageable::StaticClass()))
    {
        IDamageable::Execute_TakeDamage(Target, Amount, Causer);
    }

    // 방법 2: C++ 전용 함수는 Cast로 직접 호출
    if (IDamageable* Damageable = Cast<IDamageable>(Target))
    {
        bool bDead = Damageable->IsDead();
    }
}
```

### 3. 왜 중요한가 (게임/UE5 맥락)

"컴포넌트 vs 인터페이스 선택 기준"은 구조 설계 면접에서 거의 빠지지 않는 질문이다. 인터페이스는 상속 계층을 오염시키지 않고 행동 계약을 추가할 수 있어, 다중 상속이 제한된 환경에서 유연성을 확보하는 핵심 도구다. 대미지 처리, 상호작용, 저장/로드 가능 표식 같은 패턴에서 자주 쓰인다.

### 4. 면접 예상 Q&A

**Q1: `UINTERFACE`의 이중 클래스 구조(`U클래스` + `I클래스`)는 왜 필요한가요?**
> A: `U클래스`는 리플렉션 시스템 등록과 블루프린트 연동용이고, `I클래스`가 실제 인터페이스 함수를 담는다. UE 리플렉션이 UObject 기반으로 동작하기 때문에 등록 껍데기가 필요하다.

**Q2: `Execute_함수명`은 왜 쓰나요? 직접 가상 함수를 호출하면 안 되나요?**
> A: `BlueprintNativeEvent` 함수는 블루프린트에서 오버라이드될 수 있으므로, 블루프린트 구현까지 포함해 올바르게 호출하려면 엔진이 생성한 `Execute_` 래퍼를 써야 한다. C++ 전용 순수 가상 함수는 `Cast` 후 직접 호출해도 된다.

**Q3: 컴포넌트 vs 인터페이스, 어떤 기준으로 고르나요?**
> A: 데이터와 로직이 함께 재사용되면 컴포넌트, 행동 계약만 정의하고 구현은 각 클래스에 맡기면 인터페이스가 적절하다. "무엇을 할 수 있는가"를 선언하는 것은 인터페이스, "어떻게 동작하는가"를 캡슐화하는 것은 컴포넌트에 가깝다.

**Q4: `ImplementsInterface`와 `Cast<IInterface>`는 어떻게 다른가요?**
> A: `ImplementsInterface`는 블루프린트 전용 구현까지 감지하고, `Cast<IDamageable>`은 C++ 구현만 감지한다. 블루프린트에서만 인터페이스를 구현한 액터라면 `Cast`로는 잡히지 않는다.

**Q5: 인터페이스에 `UPROPERTY`를 넣을 수 있나요?**
> A: `I클래스`에는 `UPROPERTY`를 넣을 수 없다. 인터페이스는 행동 계약이지 데이터 보관용이 아니다. 데이터가 필요하면 구현 클래스나 컴포넌트에 둔다.

**Q6: 실무에서 인터페이스 남용의 신호는 무엇인가요?**
> A: 인터페이스 함수가 너무 많아지거나, 거의 모든 클래스가 같은 인터페이스를 구현하게 될 때다. 그때는 컴포넌트나 기반 클래스가 더 적절한지 재검토해야 한다.

### 5. 흔한 실수

- `I클래스`에 `GENERATED_BODY()`를 빼먹는다.
- `BlueprintNativeEvent` 함수를 직접 호출하고 `Execute_` 래퍼를 쓰지 않는다.
- 블루프린트 전용 인터페이스 구현을 `Cast`로 감지하려 한다.
- 인터페이스에 데이터(`UPROPERTY`)를 넣으려 한다.
- 컴포넌트가 더 적절한 상황에서 인터페이스로 데이터와 로직을 동시에 강요한다.
