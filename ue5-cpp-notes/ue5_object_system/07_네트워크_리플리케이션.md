## 토픽 26: 리플리케이션 기초

> 난이도: ★★★ | 예상 시간: 1시간 | 선행: 토픽 5

### 1. 핵심 개념

`replication(리플리케이션, 복제)`은 서버의 상태를 클라이언트에 자동으로 동기화하는 시스템이다. UE5는 서버 권위(server-authoritative) 모델을 따르므로, 서버만 진실을 가지고 클라이언트는 그 사본을 본다고 이해하는 편이 맞다. 즉 리플리케이션은 "값을 복사한다"보다 "서버의 권위 있는 상태를 각 머신에 투영한다"는 개념에 가깝다.

프로퍼티 리플리케이션의 핵심은 세 단계다. Actor에 `bReplicates = true`를 설정하고, 복제할 프로퍼티에 `UPROPERTY(Replicated)` 또는 `UPROPERTY(ReplicatedUsing = OnRep_함수명)`을 붙인 뒤, `GetLifetimeReplicatedProps`에서 등록해야 한다. `OnRep_` 함수는 클라이언트에서 값이 바뀔 때 자동으로 호출되며 서버에서는 자동 호출되지 않는다는 점도 함께 기억해야 한다.

### 2. 코드 예시

```cpp
UCLASS()
class AMyCharacter : public ACharacter
{
    GENERATED_BODY()

public:
    AMyCharacter();

    // 변경 시 클라이언트에서 OnRep_Health 자동 호출
    UPROPERTY(ReplicatedUsing = OnRep_Health)
    float Health = 100.0f;

    // 단순 복제 (OnRep 없음)
    UPROPERTY(Replicated)
    int32 Score = 0;

protected:
    // GetLifetimeReplicatedProps에서 복제 대상 등록 필수
    virtual void GetLifetimeReplicatedProps(
        TArray<FLifetimeProperty>& OutLifetimeProps) const override;

    // 클라이언트에서만 호출됨
    UFUNCTION()
    void OnRep_Health();
};

void AMyCharacter::GetLifetimeReplicatedProps(
    TArray<FLifetimeProperty>& OutLifetimeProps) const
{
    Super::GetLifetimeReplicatedProps(OutLifetimeProps);

    DOREPLIFETIME(AMyCharacter, Health);
    DOREPLIFETIME(AMyCharacter, Score);

    // 조건부 복제 예시: 소유 클라이언트에만 전송
    DOREPLIFETIME_CONDITION(AMyCharacter, Score, COND_OwnerOnly);
}

void AMyCharacter::OnRep_Health()
{
    // 체력 변경에 따른 UI 업데이트, 이펙트 재생 등
    UpdateHealthUI();
}
```

### 3. 왜 중요한가 (게임/UE5 맥락)

멀티플레이어 게임에서 가장 많은 버그가 나오는 곳이 리플리케이션이다. "서버에서는 되는데 클라이언트에서 반영이 안 된다", "클라이언트 값이 잠깐 바뀌었다가 다시 돌아온다" 같은 현상은 대부분 등록 누락이나 권위 모델 오해에서 시작된다. 그래서 리플리케이션은 문법보다 서버가 진실을 갖는다는 전제를 먼저 붙잡고 읽는 편이 훨씬 안전하다.

### 4. 면접 예상 Q&A

**Q1: `UPROPERTY(Replicated)`와 `UPROPERTY(ReplicatedUsing=...)`의 차이는 무엇인가요?**
> A: `Replicated`는 값만 동기화하고, `ReplicatedUsing`은 값 동기화 후 클라이언트에서 지정한 `OnRep_` 함수를 추가로 호출한다. 변경에 반응해야 하는 로직이 있으면 `ReplicatedUsing`을 쓴다.

**Q2: `GetLifetimeReplicatedProps`는 왜 구현해야 하나요?**
> A: 어떤 프로퍼티를 어떤 조건으로 복제할지 엔진에 알려주는 등록 함수다. 이 함수 없이는 `UPROPERTY(Replicated)`를 붙여도 실제 복제가 일어나지 않는다.

**Q3: `OnRep_` 함수는 서버에서도 호출되나요?**
> A: 아니다. `OnRep_` 함수는 클라이언트에서만 자동 호출된다. 서버에서 같은 반응이 필요하면 값을 변경하는 코드 바로 아래에 직접 호출해야 한다.

**Q4: `DOREPLIFETIME_CONDITION`은 언제 유용한가요?**
> A: 모든 클라이언트가 아니라 특정 조건의 클라이언트에만 복제할 때 유용하다. `COND_OwnerOnly`는 소유 클라이언트에만, `COND_InitialOnly`는 최초 한 번만 전송하는 식이다. 불필요한 네트워크 트래픽을 줄이는 데 중요하다.

**Q5: 리플리케이션 빈도를 제어하는 방법은 무엇인가요?**
> A: Actor의 `NetUpdateFrequency`와 `MinNetUpdateFrequency`를 조정한다. 빠르게 움직이는 액터는 높게, 거의 안 변하는 액터는 낮게 설정해 네트워크 부하를 분산한다.

**Q6: 리플리케이션 버그를 진단할 때 가장 먼저 볼 것은 무엇인가요?**
> A: `bReplicates = true` 설정 여부, `GetLifetimeReplicatedProps` 등록 여부, 값을 변경하는 쪽이 서버인지부터 확인한다. 또한 `HasAuthority()`로 서버/클라이언트 흐름을 추적하면 원인을 좁힐 수 있다.

### 5. 흔한 실수

- `bReplicates = true` 없이 `UPROPERTY(Replicated)`만 붙인다.
- `GetLifetimeReplicatedProps`에 `DOREPLIFETIME`을 추가하지 않는다.
- 클라이언트에서 상태를 직접 변경한다(서버 권위 위반).
- `OnRep_` 함수를 서버에서도 호출되길 기대한다.

## 토픽 27: RPC 패턴(Remote Procedure Call, 원격 프로시저 호출)

> 난이도: ★★★ | 예상 시간: 1시간 | 선행: 토픽 26

### 1. 핵심 개념

`RPC(Remote Procedure Call, 원격 프로시저 호출)`는 다른 머신에서 함수를 실행하는 메커니즘이다. UE5에서는 `Server`, `Client`, `NetMulticast` 세 가지 RPC를 `UFUNCTION` 지정자로 선언한다. 핵심은 함수 이름을 외우는 것보다 "누가 요청하고 누가 실행하며, 그 호출이 권위 있는 상태 변경인지 단순 알림인지"를 먼저 구분하는 데 있다.

- `Server`: 클라이언트가 호출 → 서버에서 실행. 클라이언트 요청을 서버가 처리하는 패턴.
- `Client`: 서버가 호출 → 소유 클라이언트에서 실행. 특정 클라이언트에만 보내는 이벤트.
- `NetMulticast`: 서버가 호출 → 모든 클라이언트에서 실행. 시각/청각 이펙트 브로드캐스트에 유용.

`Reliable`은 반드시 전달을 보장하지만 지연과 재전송 비용이 붙고, 기본(Unreliable)은 더 가볍지만 패킷 손실이 있을 수 있다. 그래서 게임플레이에 중요한 호출은 `Reliable`, 이펙트처럼 놓쳐도 되는 것은 `Unreliable`로 설계한다. 결국 RPC 설계는 호출 방향만이 아니라 비용과 중요도를 함께 보는 문제다.

### 2. 코드 예시

```cpp
UCLASS()
class AMyCharacter : public ACharacter
{
    GENERATED_BODY()

public:
    // 클라이언트 → 서버. 입력 처리를 서버에 요청.
    UFUNCTION(Server, Reliable, WithValidation)
    void ServerFire(FVector Direction);

    // 서버 → 모든 클라이언트. 시각 이펙트 재생.
    UFUNCTION(NetMulticast, Unreliable)
    void MulticastPlayFireEffect();

    // 서버 → 소유 클라이언트. 개인 UI 업데이트.
    UFUNCTION(Client, Reliable)
    void ClientShowDamageNumber(float DamageAmount);
};

// 서버 실행 구현부: 반드시 _Implementation 접미사
void AMyCharacter::ServerFire_Implementation(FVector Direction)
{
    // 서버에서만 실행. 실제 발사 로직.
    GetWorld()->SpawnActor<AMyProjectile>(ProjectileClass, GetActorLocation(), Direction.Rotation());
    MulticastPlayFireEffect();
}

// Validation: 악의적인 호출 필터링. false 반환 시 클라이언트 연결 차단 가능.
bool AMyCharacter::ServerFire_Validate(FVector Direction)
{
    return Direction.IsNormalized();
}
```

### 3. 왜 중요한가 (게임/UE5 맥락)

멀티플레이어 게임에서 클라이언트가 직접 상태를 바꾸면 권위 모델이 바로 깨지고, 그 틈이 곧 치트와 동기화 오류로 이어진다. 서버 RPC 패턴은 중요한 상태 변경을 서버로 모아 권위를 유지하기 위한 기본 구조다. 그래서 이 토픽을 이해하면 RPC를 "원격 함수 호출"이 아니라 "권위 있는 상태 변경을 어디서 확정할 것인가"의 문제로 볼 수 있다.

### 4. 면접 예상 Q&A

**Q1: `Server`, `Client`, `NetMulticast` RPC는 각각 어떻게 다른가요?**
> A: `Server`는 클라이언트가 서버에 요청할 때, `Client`는 서버가 특정 클라이언트에게 보낼 때, `NetMulticast`는 서버가 모든 클라이언트에 브로드캐스트할 때 쓴다. 호출 방향과 수신 대상이 다르다.

**Q2: `Reliable`과 `Unreliable`은 어떻게 구분해서 쓰나요?**
> A: 게임플레이에 직접 영향을 주는 호출(발사, 아이템 사용)은 `Reliable`, 이펙트나 사운드처럼 놓쳐도 게임이 망가지지 않는 것은 `Unreliable`로 설계한다. `Reliable`은 재전송 비용이 있으므로 남용하면 안 된다.

**Q3: `WithValidation`은 왜 필요한가요?**
> A: 서버 RPC에 악의적인 호출이 들어올 수 있기 때문이다. `_Validate` 함수에서 `false`를 반환하면 해당 클라이언트를 연결 차단할 수 있다. 보안에 중요한 RPC에는 항상 추가하는 것이 좋다.

**Q4: RPC가 실행되지 않는 흔한 원인은 무엇인가요?**
> A: `Actor->HasAuthority()` 조건을 잘못 잡았거나, Actor의 Owner가 설정되지 않아 Client RPC가 라우팅되지 않는 경우가 많다. 또한 리플리케이션이 꺼진 Actor(`bReplicates = false`)에서는 RPC가 동작하지 않는다.

**Q5: `NetMulticast`는 서버에서도 실행되나요?**
> A: 서버에서 호출되면 서버와 모든 클라이언트에서 실행된다. 따라서 서버에서 이펙트가 중복 실행되는 것을 피하려면 서버 쪽 처리를 별도로 분기해야 할 수 있다.

**Q6: 클라이언트가 서버 RPC를 너무 자주 호출하면 어떤 문제가 있나요?**
> A: 대역폭과 서버 처리 비용이 증가한다. 입력 기반 RPC는 프레임마다 보내지 않도록 변경이 있을 때만 호출하거나, 서버 측 검증으로 DoS 공격을 방어해야 한다.

### 5. 흔한 실수

- `_Implementation` 접미사를 빠뜨려 링크 에러를 낸다.
- 게임플레이 중요 RPC에 `Reliable` 없이 `Unreliable`만 쓴다.
- `NetMulticast`를 클라이언트에서 호출한다(서버에서만 의미 있음).
- Owner가 없는 Actor에서 `Client` RPC를 시도한다.

## 토픽 28: 네트워크 역할 모델

> 난이도: ★★★ | 예상 시간: 1시간 | 선행: 토픽 26, 27

### 1. 핵심 개념

UE5 멀티플레이어에서 각 Actor는 머신마다 다른 역할(role)을 가진다. `ENetRole(이넷롤, 네트워크 역할)`로 표현하며, `HasAuthority()`가 가장 자주 쓰이는 확인 방법이다. 같은 Actor라도 서버에서는 권위 있는 원본이고, 소유 클라이언트에서는 입력을 반영하는 프록시이며, 다른 클라이언트에서는 복제된 표현일 뿐이라는 점을 먼저 받아들이는 것이 중요하다.

- `ROLE_Authority`: 서버에서의 역할. 해당 Actor의 진짜 상태를 갖는다.
- `ROLE_AutonomousProxy`: 소유 클라이언트에서의 역할. 예측 이동 등을 직접 처리.
- `ROLE_SimulatedProxy`: 다른 클라이언트에서의 역할. 서버 복제 값을 보간.

`GetLocalRole()`은 현재 머신에서 이 Actor가 어떤 위치에 있는지 알려 주고, `GetRemoteRole()`은 상대방 머신에서의 기대 역할을 나타낸다. `IsLocallyControlled()`는 권위 여부와는 별개로 현재 머신이 이 Actor를 직접 제어하는지 확인하는 함수다. 그래서 역할 모델을 읽을 때는 권위와 로컬 제어를 같은 것으로 섞지 않는 습관이 중요하다.

```
서버
  └─ AMyCharacter: GetLocalRole() = ROLE_Authority
클라이언트 A (소유자)
  └─ AMyCharacter: GetLocalRole() = ROLE_AutonomousProxy
클라이언트 B (다른 플레이어)
  └─ AMyCharacter: GetLocalRole() = ROLE_SimulatedProxy
```

### 2. 코드 예시

```cpp
void AMyCharacter::Tick(float DeltaTime)
{
    Super::Tick(DeltaTime);

    // 서버에서만 체력 회복 처리
    if (HasAuthority())
    {
        RegenerateHealth(DeltaTime);
    }

    // 소유 클라이언트에서만 입력 처리
    if (IsLocallyControlled())
    {
        HandleInput();
    }

    // 역할 확인 상세 버전
    if (GetLocalRole() == ROLE_SimulatedProxy)
    {
        // 다른 플레이어의 움직임 보간
        InterpolateMovement(DeltaTime);
    }
}

// Authority 판별 예시
void AMyCharacter::TakeDamageCustom(float Amount)
{
    if (!HasAuthority())
    {
        return;  // 서버가 아니면 무시
    }

    Health -= Amount;
    // Health는 Replicated이므로 클라이언트에 자동 동기화됨
}
```

### 3. 왜 중요한가 (게임/UE5 맥락)

역할 모델을 이해하지 못하면 "서버에서는 작동하는데 클라이언트에서 두 번 실행된다", "클라이언트에서 직접 상태를 바꿨는데 바로 되돌아온다" 같은 현상을 설명하기 어렵다. 멀티플레이어 코드의 많은 분기는 결국 현재 이 머신이 권위를 가진 쪽인지, 단지 보여 주는 쪽인지 구분하는 데서 시작한다. 그래서 역할 모델은 네트워크 API 세부사항보다 먼저 잡아야 하는 기본 좌표계다.

### 4. 면접 예상 Q&A

**Q1: `ROLE_AutonomousProxy`와 `ROLE_SimulatedProxy`는 어떻게 다른가요?**
> A: `AutonomousProxy`는 소유 클라이언트로, 직접 입력을 처리하고 예측 이동을 실행할 수 있다. `SimulatedProxy`는 다른 클라이언트 머신에서의 복제 사본으로, 서버 값을 보간하며 표현한다.

**Q2: `HasAuthority()`와 `IsLocallyControlled()`는 어떻게 다른가요?**
> A: `HasAuthority()`는 현재 머신이 서버인지 확인한다. `IsLocallyControlled()`는 현재 머신이 이 Pawn/Character를 직접 제어하는지 확인한다. 서버에서도 AI가 제어하면 `IsLocallyControlled()`가 false일 수 있다.

**Q3: 클라이언트에서 상태를 직접 변경하면 어떻게 되나요?**
> A: 서버의 다음 리플리케이션 주기에 서버 값으로 덮어씌워진다. 이것이 "클라이언트 예측(client prediction)"과 "서버 보정(server reconciliation)"이 필요한 이유다.

**Q4: Listen Server와 Dedicated Server의 차이는 무엇인가요?**
> A: Listen Server는 서버 역할을 하면서 동시에 클라이언트로도 플레이한다. 그러므로 특정 Actor의 `GetLocalRole()`이 `ROLE_Authority`이면서 동시에 그 클라이언트가 소유자일 수 있다. Dedicated Server는 플레이어가 없이 순수 서버 역할만 한다.

**Q5: 클라이언트 예측(client prediction)은 왜 필요한가요?**
> A: 네트워크 지연(latency) 때문이다. 서버 응답을 기다리면 입력이 지연되어 조작감이 나빠진다. 클라이언트가 먼저 결과를 예측해 반영하고, 서버 보정이 오면 맞추는 방식이 일반적이다.

**Q6: 역할 조건을 잘못 쓴 코드의 대표적인 예는 무엇인가요?**
> A: 게임플레이 상태 변경을 `if (!HasAuthority())` 안에서 처리하거나, 반대로 이펙트 재생 같은 시각 처리를 서버에서만 하는 경우다. 권위 있는 상태 변경은 서버에서, 화면 표현은 각 클라이언트에서 하는 원칙을 지켜야 한다.

### 5. 흔한 실수

- 서버와 클라이언트 역할을 구분하지 않고 모든 머신에서 상태를 직접 변경한다.
- `HasAuthority()`를 빠뜨려 클라이언트가 직접 게임플레이 상태를 바꾼다.
- `GetLocalRole()`과 `GetRemoteRole()`을 혼동한다.
- Listen Server 환경에서 서버 클라이언트의 특수한 상황을 고려하지 않는다.
