## 토픽 23: Actor 라이프사이클

> 난이도: ★★★ | 예상 시간: 1시간 | 선행: 토픽 5, 7

### 1. 핵심 개념

`AActor(에이액터, 언리얼 액터)`의 생명주기는 생성자 → `PostInitializeComponents` → `BeginPlay` → `Tick` → `EndPlay` → `Destroyed` 순서로 흐른다. 각 단계에서 할 수 있는 것과 하면 안 되는 것이 명확히 나뉜다.

핵심 포인트는 세 가지다. 첫째, 생성자는 CDO 경로에서도 실행되므로 월드 문맥 접근은 금지다. 둘째, 컴포넌트 초기화는 `PostInitializeComponents`까지 끝나야 완전해진다. 셋째, `BeginPlay`는 월드에 등록된 후 호출되므로 여기서부터 게임플레이 로직을 안전하게 쓸 수 있다.

```
생성자
  └─ CDO 경로 포함. 월드/플레이어 접근 금지. 컴포넌트 구조 정의만.
PostInitializeComponents
  └─ 컴포넌트 초기화 완료 직후. 컴포넌트 간 참조 설정에 적합.
BeginPlay
  └─ 월드 등록 후 최초 호출. 게임플레이 로직 시작 지점.
Tick(DeltaTime)
  └─ 매 프레임 호출. 성능 민감. 필요한 Actor만 Tick 활성화.
EndPlay(EEndPlayReason)
  └─ 게임 종료, 레벨 언로드, Destroy 등 다양한 이유로 호출. 정리 로직.
Destroyed
  └─ 메모리 해제 직전. EndPlay 이후 추가 정리가 필요하면 여기서.
```

### 2. 코드 예시

```cpp
UCLASS()
class AMyActor : public AActor
{
    GENERATED_BODY()

public:
    AMyActor();

protected:
    virtual void PostInitializeComponents() override;
    virtual void BeginPlay() override;
    virtual void Tick(float DeltaTime) override;
    virtual void EndPlay(const EEndPlayReason::Type EndPlayReason) override;
};

AMyActor::AMyActor()
{
    // CDO 경로에서도 실행됨 - 월드 접근 금지
    PrimaryActorTick.bCanEverTick = true;
    MeshComponent = CreateDefaultSubobject<UStaticMeshComponent>(TEXT("Mesh"));
}

void AMyActor::PostInitializeComponents()
{
    Super::PostInitializeComponents();
    // 컴포넌트 초기화 완료 후 컴포넌트 간 참조 설정 가능
}

void AMyActor::BeginPlay()
{
    Super::BeginPlay();
    // 여기서부터 GetWorld(), GetPlayerController() 등 안전하게 사용 가능
}

void AMyActor::EndPlay(const EEndPlayReason::Type EndPlayReason)
{
    // 델리게이트 해제, 타이머 클리어 등 정리 작업
    Super::EndPlay(EndPlayReason);
}
```

### 3. 왜 중요한가 (게임/UE5 맥락)

"생성자에서 왜 GetWorld()가 null인가?", "컴포넌트 설정은 어디서 해야 하나?", "EndPlay와 Destroyed는 언제 쓰나?" 같은 질문은 Actor 라이프사이클을 모르면 답할 수 없다. 거의 모든 UE5 코드에 등장하는 구조다.

### 4. 면접 예상 Q&A

**Q1: 생성자와 BeginPlay의 역할 차이는 무엇인가요?**
> A: 생성자는 클래스 기본 구조를 설정하는 곳으로 CDO 경로에서도 실행된다. BeginPlay는 실제 월드에 배치된 후 실행되므로 게임플레이 로직의 진짜 시작점이다.

**Q2: `PostInitializeComponents`는 어떤 경우에 유용한가요?**
> A: 컴포넌트 초기화가 모두 끝난 직후, 컴포넌트 간 참조를 설정하거나 컴포넌트 상태를 구성할 때 유용하다. 생성자보다 늦고 BeginPlay보다 이르다.

**Q3: Tick은 모든 Actor에 켜야 하나요?**
> A: 아니다. Tick이 필요 없는 Actor는 생성자에서 `PrimaryActorTick.bCanEverTick = false`로 끄는 것이 성능상 좋다. 필요한 시점에만 `SetActorTickEnabled`로 켤 수도 있다.

**Q4: EndPlay와 Destroyed의 차이는 무엇인가요?**
> A: EndPlay는 레벨 언로드, 게임 종료, 명시적 Destroy 등 다양한 이유로 호출되며 이유를 파라미터로 받는다. Destroyed는 EndPlay 이후 메모리 해제 직전에 호출된다. 대부분의 정리는 EndPlay에서 한다.

**Q5: `Destroy()`를 호출하면 즉시 삭제되나요?**
> A: 아니다. `Destroy()` 호출 후 해당 프레임의 처리가 끝나면 GC가 수거하는 방식에 가깝다. 즉 `Destroy()` 직후에도 참조는 잠시 살아 있을 수 있으므로 `IsValid()` 체크가 여전히 중요하다.

**Q6: SpawnActor 후 BeginPlay까지의 흐름은 어떻게 되나요?**
> A: `SpawnActor` → 생성자 → `PostInitializeComponents` → (bAutoBeginPlay라면) `BeginPlay` 순서다. 스폰 직후 바로 게임플레이 상태에 접근하려면 BeginPlay 시점을 기다려야 한다.

### 5. 흔한 실수

- 생성자에서 `GetWorld()`, `GetGameInstance()` 등에 접근한다.
- BeginPlay에서 `Super::BeginPlay()`를 빠뜨린다.
- 필요 없는 Actor에도 Tick을 켜 두어 성능을 낭비한다.
- EndPlay에서 델리게이트나 타이머를 정리하지 않아 dangling binding이 남는다.

## 토픽 24: 컴포넌트 모델

> 난이도: ★★★ | 예상 시간: 1시간 | 선행: 토픽 23

### 1. 핵심 개념

UE5 컴포넌트 모델은 Actor에 기능을 조합(composition)하는 방식이다. `UActorComponent`는 씬 트랜스폼 없이 순수 로직을 담는 베이스이고, `USceneComponent`는 트랜스폼을 갖는 컴포넌트, `UPrimitiveComponent`는 렌더링과 충돌까지 포함한다. 루트 컴포넌트가 Actor의 트랜스폼을 대표하고, 나머지는 계층으로 붙는다.

선택 기준은 단순하다. 트랜스폼이 필요 없는 순수 로직이면 `UActorComponent`, 씬 안에 위치를 가져야 하면 `USceneComponent`, 렌더링과 물리까지 필요하면 `UPrimitiveComponent`다.

컴포넌트 vs 인터페이스 선택 기준도 자주 나온다. 기능을 독립적으로 조합하고 데이터도 필요하면 컴포넌트, 행동 계약만 강제하면 인터페이스가 더 맞다.

### 2. 코드 예시

```cpp
// 기본 서브오브젝트로 생성자에서 등록
UCLASS()
class AMyCharacter : public ACharacter
{
    GENERATED_BODY()

public:
    AMyCharacter();

private:
    UPROPERTY(VisibleAnywhere)
    TObjectPtr<UHealthComponent> HealthComponent;

    UPROPERTY(VisibleAnywhere)
    TObjectPtr<UStaticMeshComponent> WeaponMesh;
};

AMyCharacter::AMyCharacter()
{
    HealthComponent = CreateDefaultSubobject<UHealthComponent>(TEXT("Health"));

    WeaponMesh = CreateDefaultSubobject<UStaticMeshComponent>(TEXT("WeaponMesh"));
    WeaponMesh->SetupAttachment(GetMesh(), TEXT("hand_r"));  // 소켓에 부착
}

// 런타임에 컴포넌트 동적 추가
UHealthComponent* NewComp = NewObject<UHealthComponent>(this);
NewComp->RegisterComponent();
```

### 3. 왜 중요한가 (게임/UE5 맥락)

UE5 게임플레이 코드는 대부분 Actor와 Component의 조합이다. 컴포넌트 계층 이해 없이는 에디터에서 안 보이거나, 트랜스폼이 안 따라오거나, 충돌이 안 작동하는 문제를 설명하기 어렵다.

### 4. 면접 예상 Q&A

**Q1: `UActorComponent`와 `USceneComponent`의 차이는 무엇인가요?**
> A: `UActorComponent`는 트랜스폼 없는 순수 로직 컴포넌트이고, `USceneComponent`는 씬 내 위치(트랜스폼)를 갖는다. 위치가 필요한지 여부가 선택 기준이다.

**Q2: 루트 컴포넌트는 무엇인가요?**
> A: Actor의 트랜스폼을 대표하는 컴포넌트다. `SetRootComponent`로 설정하며, 루트가 움직이면 붙은 모든 자식이 따라간다.

**Q3: 기본 서브오브젝트와 런타임 동적 컴포넌트는 어떻게 다른가요?**
> A: `CreateDefaultSubobject`로 생성한 기본 서브오브젝트는 에디터에서 보이고 CDO에 등록된다. 런타임에 `NewObject` + `RegisterComponent`로 추가한 컴포넌트는 동적이며 에디터 직렬화 대상이 아니다.

**Q4: `SetupAttachment`는 어디에서 호출해야 하나요?**
> A: 생성자에서 호출해야 CDO와 에디터가 계층을 올바르게 인식한다. 런타임 동적 컴포넌트는 `AttachToComponent`를 사용한다.

**Q5: 컴포넌트와 인터페이스를 어떻게 선택하나요?**
> A: 기능을 조합하고 데이터도 갖는 독립 단위가 필요하면 컴포넌트, 행동 계약만 강제할 목적이면 인터페이스가 맞다. 컴포넌트는 조합, 인터페이스는 계약이다.

**Q6: 컴포넌트 없이 Actor에 모든 기능을 넣으면 어떤 문제가 생기나요?**
> A: Actor가 비대해져 재사용성이 떨어지고, 다른 Actor 타입에 같은 기능을 추가할 때 중복 코드가 생긴다. 컴포넌트 분리는 기능 재사용과 단일 책임 원칙을 지키는 방법이다.

### 5. 흔한 실수

- 런타임에 `CreateDefaultSubobject`를 호출한다(생성자 전용 API).
- `RegisterComponent` 없이 동적 컴포넌트를 추가해 엔진이 인식하지 못한다.
- 루트 컴포넌트를 설정하지 않아 Actor 트랜스폼이 예상대로 동작하지 않는다.
- 트랜스폼이 필요 없는 로직에 `USceneComponent`를 습관적으로 사용한다.

## 토픽 25: GameInstance & Subsystem 패턴

> 난이도: ★★★ | 예상 시간: 1시간 | 선행: 토픽 5

### 1. 핵심 개념

`UGameInstance(유게임인스턴스, 게임 인스턴스)`는 레벨 전환 사이에서도 살아남는 단일 객체다. 세션 정보, 사용자 데이터, 레벨 독립적인 서비스를 담는 곳으로, 게임 실행 내내 단 하나만 존재한다.

`Subsystem(서브시스템)` 패턴은 UE5에서 권장하는 "기능 모듈화" 방식이다. 싱글턴 대신 엔진이 관리하는 수명을 가진 서비스 객체를 `UGameInstanceSubsystem`, `UWorldSubsystem`, `UEngineSubsystem` 등으로 나누어 등록한다. 자동 생성, 자동 소멸, 의존성 주입 없이 접근 가능하다는 장점이 있다.

```
UEngineSubsystem   → 엔진 수명 (에디터 포함)
UGameInstanceSubsystem → GameInstance 수명 (레벨 독립)
UWorldSubsystem    → World 수명 (레벨 종속)
ULocalPlayerSubsystem → 로컬 플레이어 수명
```

### 2. 코드 예시

```cpp
// GameInstance 서브시스템 정의
UCLASS()
class UInventorySubsystem : public UGameInstanceSubsystem
{
    GENERATED_BODY()

public:
    virtual void Initialize(FSubsystemCollectionBase& Collection) override;
    virtual void Deinitialize() override;

    void AddItem(FName ItemId);
    int32 GetItemCount(FName ItemId) const;

private:
    TMap<FName, int32> Inventory;
};

// 어디서든 접근
void AMyActor::UseInventory()
{
    UGameInstance* GI = GetGameInstance();
    if (UInventorySubsystem* InvSys = GI->GetSubsystem<UInventorySubsystem>())
    {
        InvSys->AddItem(FName("Sword"));
    }
}
```

### 3. 왜 중요한가 (게임/UE5 맥락)

"레벨이 바뀌어도 인벤토리를 유지하려면 어디에 두나요?", "싱글턴 대신 무엇을 쓰나요?"는 면접에서 자주 나오는 설계 질문이다. `GameInstance`와 `Subsystem` 패턴을 모르면 전역 변수나 싱글턴으로 대답하게 되는데, UE5 방식과는 거리가 멀다.

### 4. 면접 예상 Q&A

**Q1: GameInstance는 왜 레벨 전환 시에도 살아남나요?**
> A: 레벨(World)이 아니라 게임 프로세스 수명에 묶여 있기 때문이다. 레벨을 바꿔도 GameInstance는 파괴되지 않아 레벨 독립적인 상태를 유지할 수 있다.

**Q2: 싱글턴 대신 Subsystem 패턴을 권장하는 이유는 무엇인가요?**
> A: Subsystem은 엔진이 수명을 관리하므로 수동 생성/파괴가 필요 없고, 적절한 타입을 선택하면 자동으로 올바른 시점에 초기화된다. 또한 테스트와 모듈화에도 유리하다.

**Q3: `UWorldSubsystem`과 `UGameInstanceSubsystem`은 어떻게 구분하나요?**
> A: `UWorldSubsystem`은 레벨과 수명을 같이 하므로 레벨마다 새로 생성된다. `UGameInstanceSubsystem`은 게임 전체에서 하나만 유지된다. 레벨 종속 서비스인지 여부가 기준이다.

**Q4: Subsystem에 접근하는 방법은 무엇인가요?**
> A: `GetSubsystem<T>()` 또는 `GetWorld()->GetSubsystem<T>()` 형태로 접근한다. 타입 파라미터만으로 자동 등록된 서브시스템을 찾아준다.

**Q5: GameInstance에 너무 많은 것을 넣으면 어떤 문제가 생기나요?**
> A: 레벨 종속이어야 하는 상태가 레벨 독립으로 유지되어 버그가 생기거나, 너무 많은 책임이 한 곳에 몰려 유지보수가 어려워진다. 수명 범위에 맞게 책임을 분산해야 한다.

**Q6: 레벨마다 리셋되어야 하는 게임 상태는 어디에 두나요?**
> A: `AGameMode`, `AGameState`, 또는 `UWorldSubsystem`이 적합하다. 레벨이 끝나면 함께 사라져야 하는 상태는 GameInstance가 아니라 레벨 수명에 묶어야 한다.

### 5. 흔한 실수

- 레벨 종속 상태를 GameInstance에 두어 레벨 전환 후에도 남는다.
- 전역 싱글턴(`static` 변수)으로 해결하려다 수명 관리와 GC 추적을 놓친다.
- `UWorldSubsystem`을 GameInstance 수명으로 오해한다.
- `Initialize`/`Deinitialize`를 오버라이드하지 않아 초기화 로직이 없다.
