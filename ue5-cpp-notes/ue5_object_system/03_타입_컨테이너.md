## 토픽 12: UE5 문자열 체계

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 없음

### 1. 핵심 개념

UE5 문자열은 용도에 따라 `FString(에프스트링, 가변 문자열)`, `FName(에프네임, 이름 ID)`, `FText(에프텍스트, 로컬라이즈 가능 텍스트)`, `FStringView(에프스트링뷰, 비소유 문자열 뷰)`를 다르게 쓴다. `FString`은 가변 문자열 데이터, `FName`은 빠른 비교를 위한 이름 ID, `FText`는 로컬라이제이션 가능한 사용자 표시 텍스트, `FStringView`는 비소유 읽기 전용 뷰에 가깝다.

핵심은 "사람에게 보여 줄 텍스트인가, 내부 키인가, 문자열 가공인가, 복사 없이 읽기만 할 것인가"를 먼저 구분하는 것이다. 변환 유틸리티로는 `LexToString`, `LexFromString`도 자주 등장한다.

### 2. 코드 예시

```cpp
FString RawName = TEXT("Player_01");
FName RowName = FName(TEXT("Weapon.Sword"));
FText DisplayText = FText::FromString(TEXT("게임 시작"));
FStringView View = RawName;

FString NumberText = LexToString(42);
int32 Parsed = 0;
LexFromString(Parsed, TEXT("42"));
```

### 3. 왜 중요한가 (게임/UE5 맥락)

UI 텍스트를 `FString`으로만 다루면 로컬라이제이션이 약해지고, 키 비교를 문자열로 처리하면 비용이 커진다. 또한 불필요한 복사를 줄이려면 `FStringView` 같은 도구를 이해해야 한다.

### 4. 면접 예상 Q&A

**Q1: `FString`, `FName`, `FText`의 차이를 설명해 보세요.**
> A: `FString`은 가공용 문자열, `FName`은 식별자와 비교용 이름, `FText`는 사용자 표시와 번역용 텍스트다. 결론적으로 용도가 다르므로 서로 완전한 대체 관계가 아니다.

**Q2: `FName`이 빠른 이유는 무엇인가요?**
> A: 내부 이름 테이블과 인덱스 기반 비교를 사용하기 때문이다. 자주 비교되는 식별자에 적합하다.

**Q3: `FStringView`는 언제 유용한가요?**
> A: 문자열을 소유할 필요 없이 읽기만 할 때 유용하다. 불필요한 복사를 줄여 함수 인터페이스를 더 가볍게 만들 수 있다.

**Q4: `LexToString`과 `LexFromString`은 어떤 장점이 있나요?**
> A: 다양한 타입을 일관된 방식으로 문자열과 변환할 수 있다는 점이다. 포매팅과 파싱 로직을 통일하기 좋다.

**Q5: UI 문자열에 `FName`을 쓰면 왜 안 좋은가요?**
> A: `FName`은 표시용 텍스트가 아니라 식별자다. 사용자에게 보여 줄 텍스트는 문화권과 로컬라이제이션을 고려하는 `FText`가 맞다.

**Q6: 실무에서 문자열 타입 선택이 틀렸다는 신호는 무엇인가요?**
> A: UI 번역이 안 되거나, 키 비교 때문에 문자열 할당이 과도하거나, 읽기 전용 인자에 매번 `FString` 복사가 발생할 때다.

### 5. 흔한 실수

- UI 표시 문자열을 전부 `FString`으로 처리한다.
- 문자열 가공이 많은 경로에 `FName`을 억지로 쓴다.
- 읽기 전용 함수 파라미터에서 불필요한 `FString` 복사를 반복한다.

## 토픽 13: UE5 컨테이너

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 없음

### 1. 핵심 개념

UE5는 `TArray(티어레이, 동적 배열)`, `TMap(티맵, 키-값 맵)`, `TSet(티셋, 집합)`, `TQueue(티큐, 큐)` 같은 기본 컨테이너 외에도 `TInlineAllocator(티인라인알로케이터, 소형 버퍼 최적화 할당자)`, `TSparseArray(티스파스어레이, 빈 슬롯 허용 배열)`, `TMultiMap(티멀티맵, 중복 키 허용 맵)` 같은 특수 도구를 제공한다. 표준 라이브러리와 비슷해 보이지만, 엔진 메모리 정책과 리플렉션/직렬화 친화성에 맞춘 구조가 많다.

핵심은 "UE 시스템과 얼마나 자연스럽게 붙는가"다. 에디터 노출, 직렬화, GameplayTag, DataTable, 델리게이트와 함께 쓸 데이터라면 UE 컨테이너 쪽이 더 자연스러운 경우가 많다.

### 2. 코드 예시

```cpp
TArray<int32, TInlineAllocator<8>> SmallBuffer;
SmallBuffer.Add(10);

TSparseArray<FString> SparseNames;
int32 Index = SparseNames.Add(TEXT("Enemy"));

TMultiMap<FName, int32> ScoresByTeam;
ScoresByTeam.Add(TEXT("Red"), 10);
ScoresByTeam.Add(TEXT("Red"), 15);
```

### 3. 왜 중요한가 (게임/UE5 맥락)

컨테이너 선택은 성능과 사용성 둘 다에 영향을 준다. 예를 들어 작은 크기에서 할당을 피하고 싶으면 `TInlineAllocator`, 중간에 빈 슬롯을 허용하는 구조가 필요하면 `TSparseArray`, 같은 키에 여러 값을 두려면 `TMultiMap`이 자연스럽다.

### 4. 면접 예상 Q&A

**Q1: UE 프로젝트에서 `std::vector`보다 `TArray`를 더 자주 쓰는 이유는 무엇인가요?**
> A: 엔진 API, 직렬화, 리플렉션, 메모리 정책과의 호환성이 좋기 때문이다. UE 시스템과 붙는 데이터는 `TArray`가 더 자연스럽다.

**Q2: `TInlineAllocator`는 언제 고려하나요?**
> A: 작은 크기 원소가 자주 생성되고 힙 할당을 줄이고 싶을 때 고려한다. 다만 실제 원소 수 분포를 모른 채 습관적으로 쓰면 오히려 메모리 낭비가 날 수 있다.

**Q3: `TSparseArray`는 어떤 문제를 해결하나요?**
> A: 중간 삭제가 있어도 인덱스 관리가 중요한 구조를 다루기 좋다. 빈 슬롯을 허용하는 대신 연속 메모리 특성은 약해진다.

**Q4: `TMap`과 `TMultiMap`은 어떻게 구분하나요?**
> A: 키 하나에 값 하나면 `TMap`, 같은 키에 여러 값이 필요하면 `TMultiMap`이다. 데이터 모델을 먼저 보고 선택해야 한다.

**Q5: 순회 중 삭제는 왜 조심해야 하나요?**
> A: 인덱스와 내부 구조가 변하기 때문이다. `RemoveAll`, 역순 순회, 수집 후 삭제 같은 패턴이 안전하다.

**Q6: 실무에서 컨테이너 선택이 잘못됐다는 신호는 무엇인가요?**
> A: 잦은 재할당, 불필요한 검색 비용, 데이터 모델과 맞지 않는 키 구조가 반복될 때다. 성능뿐 아니라 코드 의미도 흐려진다.

### 5. 흔한 실수

- 기본 `TArray`만 만능처럼 사용한다.
- 삭제가 잦은 구조에 연속 배열만 고집한다.
- 순회 중 요소 삭제로 인덱스 꼬임을 만든다.

## 토픽 14: UE5 캐스팅

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 1

### 1. 핵심 개념

UE5는 `dynamic_cast(다이내믹 캐스트, 런타임 타입 변환)`보다 `Cast<T>(캐스트, 안전한 UE 캐스팅)`, `CastChecked<T>(캐스트체크드, 실패 시 체크되는 캐스팅)`, `ExactCast<T>(이그잭트 캐스트, 정확한 타입만 허용)`, `IsA()(이즈에이, 타입 판별)`를 주로 쓴다. 이유는 UE 리플렉션 타입 정보와 더 잘 통합되고, RTTI 의존 없이 엔진 타입 체계 위에서 동작하기 때문이다.

핵심은 안전성과 의도를 나누는 것이다. 실패 가능성이 열려 있으면 `Cast`, 논리적으로 반드시 맞아야 하면 `CastChecked`, 정확히 동일 타입만 허용하면 `ExactCast`, 판별만 하면 `IsA()`를 쓴다.

### 2. 코드 예시

```cpp
void HandleActor(AActor* Actor)
{
    if (ACharacter* Character = Cast<ACharacter>(Actor))
    {
        Character->Jump();
    }

    if (Actor && Actor->IsA(AMyBoss::StaticClass()))
    {
        UE_LOG(LogTemp, Log, TEXT("Boss detected"));
    }
}
```

### 3. 왜 중요한가 (게임/UE5 맥락)

게임플레이 코드는 액터와 컴포넌트 타입 분기가 많다. UE 캐스팅을 이해하면 블루프린트 클래스, 리플렉션 타입, 런타임 판별을 일관된 방식으로 설명할 수 있다.

### 4. 면접 예상 Q&A

**Q1: UE는 왜 `dynamic_cast`보다 `Cast<T>`를 더 쓰나요?**
> A: UE 자체 리플렉션 타입 정보와 잘 맞기 때문이다. RTTI 의존을 줄이면서 엔진 타입 체계 안에서 일관된 판별을 할 수 있다.

**Q2: `CastChecked<T>`는 언제 써야 하나요?**
> A: 실패가 논리적으로 불가능해야 하는 지점에서만 쓴다. 결론적으로 계약 위반을 빨리 잡고 싶을 때 유용하지만, 열려 있는 경로에서는 위험하다.

**Q3: `ExactCast<T>`는 왜 필요한가요?**
> A: 파생 클래스까지 허용하는 `Cast`와 달리 정확히 해당 타입만 받아야 하는 경우가 있기 때문이다. 툴링과 검증 코드에서 자주 유용하다.

**Q4: `IsA()`와 `Cast<T>`는 어떻게 구분하나요?**
> A: 타입 판별만 필요하면 `IsA()`, 실제 포인터를 해당 타입으로 쓰려면 `Cast<T>`가 자연스럽다. 의미에 맞게 고르면 의도가 더 명확하다.

**Q5: 캐스팅이 너무 많다는 것은 어떤 신호인가요?**
> A: 설계상 인터페이스, 컴포넌트, 델리게이트가 더 적절할 수 있다는 신호다. 타입 분기 체인이 길어질수록 확장성이 떨어진다.

**Q6: 실무에서 가장 흔한 캐스팅 실수는 무엇인가요?**
> A: 성공 가능성이 열려 있는데도 `CastChecked`를 쓰거나, null 체크 없는 후속 접근을 하는 것이다. 안전성보다 편의성을 앞세우면 런타임 버그가 바로 난다.

### 5. 흔한 실수

- 실패 가능 경로에 `CastChecked`를 남용한다.
- 캐스팅으로 설계 문제를 덮는다.
- 타입 판별만 하면 되는데 불필요하게 여러 번 캐스팅한다.
