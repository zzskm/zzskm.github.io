## 토픽 12: UE5 문자열 체계

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 없음

### 1. 문제 상황

핵심 용어: `FStringView`, `LexToString`, `FText`, `FName`.

문자열 타입 선택은 단순 취향 문제가 아니라 로컬라이제이션, 비교 비용, API 설계와 직접 연결된다. 초반에는 전부 `FString`으로 밀어도 돌아가지만, UI 번역이 들어오거나 데이터 키 비교가 많아지는 순간 잘못된 선택이 유지보수 비용으로 드러난다. `FStringView`까지 익혀 두면 읽기 전용 경로에서 불필요한 복사를 줄이는 설계도 자연스럽게 할 수 있다.

UE5 문자열 타입은 이름만 보면 비슷해 보여도 설계 의도가 분명히 다르다. `FString(에프스트링, 가변 문자열)`은 문자열을 만들고 수정하는 작업용 버퍼에 가깝고, `FName(에프네임, 이름 ID)`은 반복 비교되는 식별자를 싸게 다루기 위한 타입이다. `FText(에프텍스트, 로컬라이즈 가능 텍스트)`는 사용자에게 보여 줄 문구를 위한 타입이고, `FStringView(에프스트링뷰, 비소유 문자열 뷰)`는 소유권 없이 읽기만 하는 경량 뷰다.

### 2. 핵심 개념

#### 정의

UE5 문자열 타입은 이름만 보면 비슷해 보여도 설계 의도가 분명히 다르다. `FString(에프스트링, 가변 문자열)`은 문자열을 만들고 수정하는 작업용 버퍼에 가깝고, `FName(에프네임, 이름 ID)`은 반복 비교되는 식별자를 싸게 다루기 위한 타입이다. `FText(에프텍스트, 로컬라이즈 가능 텍스트)`는 사용자에게 보여 줄 문구를 위한 타입이고, `FStringView(에프스트링뷰, 비소유 문자열 뷰)`는 소유권 없이 읽기만 하는 경량 뷰다.

실무 기준은 "이 값이 UI 문구인가, 내부 키인가, 가공 중인 문자열인가, 읽기 전용 입력인가"를 먼저 정하는 것이다. UI 문구를 `FName`처럼 다루면 번역 경로가 막히고, 읽기 전용 인자를 매번 `FString`으로 받으면 복사 비용이 눈에 덜 띄게 누적된다. `LexToString`, `LexFromString`은 이런 타입 경계를 오갈 때 표현을 통일하는 도구로 자주 등장한다.

### 3. 내부 동작 / 메커니즘

#### 동작 관점

실무 기준은 "이 값이 UI 문구인가, 내부 키인가, 가공 중인 문자열인가, 읽기 전용 입력인가"를 먼저 정하는 것이다. UI 문구를 `FName`처럼 다루면 번역 경로가 막히고, 읽기 전용 인자를 매번 `FString`으로 받으면 복사 비용이 눈에 덜 띄게 누적된다. `LexToString`, `LexFromString`은 이런 타입 경계를 오갈 때 표현을 통일하는 도구로 자주 등장한다.

#### 핵심 규칙

- 표시용 문자열과 식별자 문자열을 구분한다.
- FStringView는 읽기 전용 단기 참조로 쓰고, 원본문자열보다 오래 살게 두지 않는다.

### 4. 자주 틀리는 포인트

- UI 표시 문자열을 전부 `FString`으로 처리한다.
- 문자열 가공이 많은 경로에 `FName`을 억지로 쓴다.
- 식별자 비교에 FString을 남용하거나, 로컬라이즈 UI 문자열을 FName/FString으로 처리하면 의도가 어긋난다.

#### 우선 검토하는 경우

- 표시용 문자열과 식별자 문자열을 구분한다.

#### 주의하거나 분리할 경우

- 문자열 가공이 많은 경로에 `FName`을 억지로 쓴다.

### 6. 대안 비교

- FString은 내용 조작, FName은 식별, FText는 표시와 현지화에 강하다.

### 7. 코드 예시 (bad → good)

#### Bad: 이 코드가 식별자 비교용으로 비효율적일 수 있는 이유는

```cpp
if (CurrentState == FString(TEXT("Idle")))
{
    // ...
}
```

반복 비교되는 상태 이름이라면 FString보다 FName이 더 식별자적이고 비교 비용도 더 적절한 경우가 많다.

#### Good: 식별자 비교라면 FName 사용을 우선 검토한다.

```cpp
if (CurrentState == FName(TEXT("Idle")))
{
    // ...
}
```

식별자 비교라면 FName 사용을 우선 검토한다.

표시용 문자열과 식별자 문자열을 구분한다.

#### Bad: 이 캐싱이 왜 위험한가

```cpp
FStringView CachedView;

void BuildLabel()
{
    FString Temp = FString::Printf(TEXT("HP_%d"), 10);
    CachedView = Temp;
}
```

`FStringView`는 메모리를 소유하지 않는데, 원본 `Temp`는 함수가 끝나면 사라진다. 뷰를 오래 보관하면 댕글링 뷰가 된다.

#### Good: 장기 보관이 필요하면 `FString` 자체를 멤버로 저장한다.

```cpp
FString CachedLabel;

void BuildLabel()
{
    CachedLabel = FString::Printf(TEXT("HP_%d"), 10);
}
```

장기 보관이 필요하면 `FString` 자체를 멤버로 저장한다.

FStringView는 읽기 전용 단기 참조로 쓰고, 원본문자열보다 오래 살게 두지 않는다.

### 8. 판단 기준 요약

- 표시용 문자열과 식별자 문자열을 구분한다.
- FStringView는 읽기 전용 단기 참조로 쓰고, 원본문자열보다 오래 살게 두지 않는다.
- 식별자 비교라면 FName 사용을 우선 검토한다.
- 장기 보관이 필요하면 `FString` 자체를 멤버로 저장한다.
- 식별은 enum이나 `FName`으로 하고, 화면 표시만 `FText`로 분리한다.
- 수정이 없으면 `const FString&` 또는 짧은 참조면 `FStringView`를 고려한다.

## 토픽 13: UE5 컨테이너

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 없음

### 1. 문제 상황

핵심 용어: `TInlineAllocator`, `TMultiMap`, `TSparseArray`, `TSet`.

컨테이너는 단순히 데이터를 담는 그릇이 아니라 성능과 코드 의미를 동시에 결정하는 선택이다. 재할당이 잦거나 삭제 패턴이 맞지 않으면 병목이 생기고, 키 구조가 문제와 안 맞으면 코드 자체가 부자연스러워진다. 그래서 UE 컨테이너를 외우는 것보다 어떤 비용을 감수하고 무엇을 얻는지 이해하는 편이 훨씬 중요하다.

UE5 컨테이너는 표준 라이브러리의 대체품이라기보다 엔진 생태계에 맞춘 자료구조 묶음에 가깝다. `TArray(티어레이, 동적 배열)`, `TMap(티맵, 키-값 맵)`, `TSet(티셋, 집합)`처럼 자주 쓰는 타입 외에도 `TInlineAllocator(티인라인알로케이터, 소형 버퍼 최적화 할당자)`, `TSparseArray(티스파스어레이, 빈 슬롯 허용 배열)`, `TMultiMap(티멀티맵, 중복 키 허용 맵)`처럼 특정 비용 구조를 겨냥한 도구가 함께 있다.

### 2. 핵심 개념

TArray<FName> UniqueTags;

## 토픽 14: UE5 캐스팅

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 1

### 1. 문제 상황

핵심 용어: `Cast<T>`, `IsA()`, `CastChecked`, `ExactCast`.

## 토픽 22: GameplayTag 시스템

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 12

### 1. 문제 상황

핵심 용어: `FGameplayTag`, `FGameplayTagContainer`, `RequestGameplayTag`, `HasTagExact`.

상태, 버프, 아이템 분류, UI 조건을 문자열이나 enum 몇 개로만 관리하기 시작하면 규칙이 늘어날수록 분기문과 하드코딩 키가 빠르게 불어난다. 특히 "기절 계열 상태 전체"처럼 상위 의미를 기준으로 묶고 싶을 때 닫힌 enum만으로는 표현이 뻣뻣해진다.

GameplayTag는 이런 의미 분류를 등록 기반 계층 체계로 다루기 위한 도구다. 핵심은 문자열을 예쁘게 감싸는 것이 아니라, 의미와 질의 규칙을 데이터로 옮기는 데 있다.

### 2. 핵심 개념

`FGameplayTag`는 단일 태그를, `FGameplayTagContainer`는 여러 태그의 집합을 표현한다. 태그는 등록된 이름 체계를 바탕으로 동작하고, `Character.State.Stunned`처럼 상하위 관계를 가진다.

`HasTag`, `HasAny`, `HasAll`, `HasTagExact`를 조합하면 정확한 일치, 상위 분류 포함, 집합 조건 같은 규칙을 코드보다 데이터에 가깝게 표현할 수 있다. 그래서 GameplayTag는 상태 관리라기보다 의미 분류와 질의 시스템으로 보는 편이 맞다.

### 3. 내부 동작 / 메커니즘

태그는 중앙 등록을 통해 이름 체계를 관리하고, 코드에서는 `RequestGameplayTag`로 등록된 태그를 가져와 사용한다. 이 경로를 쓰면 오타와 임의 문자열 사용을 줄이고, 에디터/설정과 코드가 같은 의미 체계를 공유할 수 있다.

계층 매칭 여부에 따라 같은 컨테이너 질의라도 결과가 달라진다. 예를 들어 `HasTag(Character.State)`는 `Character.State.Stunned`를 포함하지만, `HasTagExact(Character.State)`는 포함하지 않는다.

- `RequestGameplayTag` = 등록 태그 조회
- `FGameplayTag` = 단일 의미 태그
- `FGameplayTagContainer` = 태그 집합
- `HasTag` = 계층 포함 검사 / `HasTagExact` = 정확한 일치 검사

### 4. 자주 틀리는 포인트

- 등록하지 않은 태그 이름을 문자열로 하드코딩한다.
- `HasTag`와 `HasTagExact`의 계층 의미 차이를 구분하지 않는다.
- 모든 상태를 태그로 밀어 넣어 태그 수와 계층 깊이가 과도하게 커진다.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

#### 우선 검토하는 경우

- 상태, 능력, 속성 분류가 계속 늘어나고 조합 질의가 많다.
- 에디터/설정과 코드가 같은 의미 체계를 공유해야 한다.
- 상위 분류를 기준으로 조건을 묶고 싶다.

#### 주의하거나 분리할 경우

- 상태 집합이 아주 작고 컴파일 타임 분기만 필요하면 enum이 더 단순할 수 있다.
- 임시 문자열 키만으로 충분한 내부 캐시까지 전부 태그로 바꾸면 과설계가 된다.
- 태그 계층이 지나치게 깊어지면 오히려 의미 체계가 흐려진다.

### 6. 대안 비교

- enum은 닫힌 컴파일 타임 집합이고, GameplayTag는 데이터 기반 확장형 의미 집합에 가깝다.
- `FName`은 식별자 비교에 가깝고, GameplayTag는 등록과 계층 매칭까지 포함한 의미 분류 도구다.
- 단순 문자열 배열은 빠르게 시작할 수 있지만 등록 관리, 오타 방지, 컨테이너 질의 같은 장점이 없다.

### 7. 코드 예시 (bad → good)

#### Bad: 문자열 비교는 오타와 의미 손실에 취약하다.

```cpp
if (Tag.ToString() == TEXT("Character.State.Stuned"))
{
    // ...
}
```

문자열 오타를 컴파일 타임에 잡기 어렵고, 계층형 태그 시스템의 등록·매칭 이점을 잃는다.

#### Good: 등록 태그를 요청해 비교한다.

```cpp
const FGameplayTag StunTag = FGameplayTag::RequestGameplayTag(TEXT("Character.State.Stunned"));
if (Tag == StunTag)
{
    // ...
}
```

의미 분류 키는 등록 태그로 가져오고, 비교는 태그 타입 수준에서 수행하는 편이 안전하다.

#### Bad: 상위 태그 포함 조건인데 `HasTagExact`를 쓰고 있다.

```cpp
if (ActiveTags.HasTagExact(FGameplayTag::RequestGameplayTag(TEXT("Character.State"))))
{
    // 상태 계열이면 처리
}
```

정확한 일치만 보므로 `Character.State.Stunned` 같은 하위 태그는 잡히지 않는다.

#### Good: 상위 분류 포함이 목적이면 `HasTag`를 사용한다.

```cpp
const FGameplayTag StateTag = FGameplayTag::RequestGameplayTag(TEXT("Character.State"));
if (ActiveTags.HasTag(StateTag))
{
    // 상태 계열이면 처리
}
```

정확한 일치인지, 계층 포함 매칭인지 기준을 먼저 정해야 질의가 의도대로 동작한다.

### 8. 판단 기준 요약

- 상태 분류가 계속 늘어나고 조합 질의가 많으면 GameplayTag를 먼저 본다.
- 등록 기반 태그 조회와 컨테이너 질의를 기본 패턴으로 둔다.
- 단순 식별자 비교면 `FName`, 의미 분류 체계면 GameplayTag로 구분한다.
- 상위 분류 포함 조건이면 `HasTag`, 정확한 일치만 필요하면 `HasTagExact`를 쓴다.
- 태그 계층은 의미가 드러날 정도로만 유지한다.
- GAS가 없어도 상태·아이템·UI 분류에 독립적으로 활용할 수 있다.
