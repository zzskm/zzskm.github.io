## 토픽 5: UObject(유오브젝트, 언리얼 객체) 생명주기

> 난이도: ★★★ | 예상 시간: 1시간 | 선행: 없음

### 1. 핵심 개념

`UObject(유오브젝트, 언리얼 객체)`는 일반 C++ 객체처럼 `new/delete`로 수동 관리하는 대상이 아니다. 생성은 `NewObject`, `CreateDefaultSubobject`, `SpawnActor` 같은 엔진 경로를 따르고, 파괴는 GC와 엔진 수명 주기에 의해 관리된다. 따라서 생성 순간보다 "누가 소유하고, 엔진이 어떻게 추적하는가"가 더 중요하다.

특히 `Outer` 개념은 중요하다. `Outer`는 단순 부모 포인터가 아니라 이름 공간과 수명 관련 컨텍스트를 제공한다. 다만 `Outer`만 설정했다고 곧바로 GC 보호가 되는 것은 아니라는 점도 같이 기억해야 한다.

### 2. 코드 예시

```cpp
UObject* MyObject = NewObject<UObject>(GetTransientPackage());

UHealthComponent* HealthComponent =
    CreateDefaultSubobject<UHealthComponent>(TEXT("HealthComponent"));
```

### 3. 왜 중요한가 (게임/UE5 맥락)

UE 프로젝트에서 객체 생성과 파괴 버그는 대부분 "일반 C++ 객체처럼 생각한 것"에서 시작된다. `UObject` 생명주기를 이해해야 GC, 직렬화, 에디터, 블루프린트 흐름이 모두 연결된다.

### 4. 면접 예상 Q&A

**Q1: `UObject`를 왜 `new/delete`로 다루면 안 되나요?**
> A: 엔진의 등록, 리플렉션, GC 추적, 이름 관리 경로를 우회하게 되기 때문이다. 결론적으로 `UObject`는 엔진이 아는 방식으로 생성해야 한다.

**Q2: `Outer`는 무엇인가요?**
> A: 객체의 소속 컨텍스트이자 이름 공간 관계를 표현하는 포인터다. 다만 `Outer`는 수명 힌트일 뿐, 참조 추적과 동일하지는 않다.

**Q3: `CreateDefaultSubobject`와 `NewObject`는 어떻게 다른가요?**
> A: 전자는 생성자에서 기본 서브오브젝트를 만들 때, 후자는 런타임에 일반 UObject를 생성할 때 주로 쓴다. 클래스 기본 구조인지 런타임 동적 생성인지가 기준이다.

**Q4: `UObject`의 파괴는 언제 일어나나요?**
> A: 더 이상 GC가 도달할 수 없을 때 수거된다. 즉 즉시 삭제보다 "참조 그래프에서 끊겼는가"가 핵심이다.

**Q5: `Outer`를 지정하면 GC에서 안전한가요?**
> A: 아니다. `Outer`만으로 모든 참조가 보장되지는 않는다. GC가 실제로 볼 수 있는 경로, 예를 들어 `UPROPERTY`나 root set 연결이 중요하다.

**Q6: 실무에서 수명 버그를 줄이는 기준은 무엇인가요?**
> A: 생성 API, 소유 컨텍스트, 참조 추적 경로를 한 세트로 보는 것이다. 객체를 만든 뒤 누가 들고 있고 엔진이 그 사실을 아는지까지 확인해야 한다.

### 5. 흔한 실수

- `UObject`를 일반 힙 객체처럼 생각해 수동 삭제를 시도한다.
- `Outer`만 지정하면 GC가 무조건 보호한다고 오해한다.
- 기본 서브오브젝트와 런타임 동적 객체 생성을 같은 패턴으로 다룬다.

## 토픽 6: UE5 GC(Garbage Collection, 가비지 컬렉션, 자동 메모리 수거) 심화

> 난이도: ★★★ | 예상 시간: 1.5시간 | 선행: 토픽 5

### 1. 핵심 개념

UE5 GC(Garbage Collection, 가비지 컬렉션, 자동 메모리 수거)는 `UObject` 참조 그래프를 따라 도달 가능한 객체를 유지하고, 도달 불가능한 객체를 수거하는 `mark-and-sweep(마크 앤 스윕, 표시 후 수거)` 방식에 가깝다. 핵심은 root set, `UPROPERTY`, `AddToRoot`, `FGCObject`, 약한 참조 개념을 함께 이해하는 것이다.

또한 `PendingKill`, `MarkAsGarbage` 같은 상태는 "곧 쓸 수 없게 될 객체"를 다루는 실무 포인트다. 즉 GC는 단순히 메모리를 지우는 시스템이 아니라, 객체 유효성 상태와 참조 추적 정책을 함께 다룬다.

**`IsValid()` vs `nullptr` 체크**: `UPROPERTY` 없는 raw 포인터가 GC에 수거돼도 포인터 변수 자체가 null로 바뀌지는 않는다. 이 때문에 `nullptr` 체크만으로는 이미 수거된 객체에 접근하는 것을 막을 수 없다. `IsValid()`는 포인터가 null이 아닌지뿐 아니라 `PendingKill`이나 `MarkAsGarbage` 상태까지 걸러주므로, UObject 포인터 유효성을 확인할 때는 `IsValid()`가 더 안전하다.

### 2. 코드 예시

```cpp
UObject* ImportantObject = NewObject<UObject>();
ImportantObject->AddToRoot();

// 나중에 필요 없으면
ImportantObject->RemoveFromRoot();

// --- IsValid() vs nullptr ---
// 위험: raw 포인터가 GC 수거 후에도 주소를 들고 있을 수 있음
UObject* RawPtr = SomeObject; // UPROPERTY 아님
// GC 수거 후...
if (RawPtr != nullptr)  // true일 수 있지만 이미 수거된 객체
{
    RawPtr->GetName(); // 정의되지 않은 동작!
}

// 안전: IsValid()는 null 체크 + PendingKill/MarkAsGarbage 상태 체크
if (IsValid(RawPtr))
{
    RawPtr->GetName(); // 유효한 객체임이 보장됨
}
```

### 3. 왜 중요한가 (게임/UE5 맥락)

GC를 이해하지 못하면 "분명 포인터가 있었는데 사라졌다", "해제된 줄 알았는데 아직 살아 있다" 같은 문제를 반복하게 된다. 특히 블루프린트와 C++를 섞는 프로젝트일수록 참조 추적 경로를 설명할 수 있어야 한다. `IsValid()`와 `nullptr` 체크의 차이를 모르면 간헐적 크래시를 추적하기 매우 어렵다.

### 4. 면접 예상 Q&A

**Q1: UE GC는 무엇을 기준으로 객체를 살려 두나요?**
> A: root set에서 시작해 추적 가능한 참조 그래프에 연결된 객체를 살려 둔다. 결론적으로 포인터가 존재하는지만이 아니라, 엔진이 그 참조를 아는지가 중요하다.

**Q2: `UPROPERTY`가 중요한 이유는 무엇인가요?**
> A: GC가 해당 참조를 추적할 수 있게 하기 때문이다. 엔진이 보지 못하는 raw 포인터는 객체를 살려 두지 못한다.

**Q3: `FGCObject`는 언제 쓰나요?**
> A: `UObject`가 아닌 일반 C++ 객체가 `UObject` 참조를 들고 있으면서 GC에 그 참조를 알려야 할 때 쓴다. 즉 리플렉션 밖의 참조를 GC에 연결하는 다리다.

**Q4: `AddToRoot`는 언제 조심해서 써야 하나요?**
> A: 객체를 강제로 root set에 넣으므로 해제 시점을 놓치면 영구적으로 살아남을 수 있다. 편리하지만 남용하면 메모리 누수성 구조가 된다.

**Q5: `IsValid()`와 `nullptr` 체크는 왜 다른가요?**
> A: `nullptr` 체크는 포인터 주소만 본다. GC가 객체를 수거해도 `UPROPERTY`가 아닌 raw 포인터는 null로 바뀌지 않아 dangling pointer가 된다. `IsValid()`는 null 체크에 더해 `PendingKill`/`MarkAsGarbage` 상태까지 검사하므로 "정말 사용 가능한 객체인가"를 판단할 수 있다.

**Q6: 실무에서 GC 문제를 진단할 때 가장 먼저 볼 것은 무엇인가요?**
> A: 해당 객체로 이어지는 추적 가능한 참조 경로가 있는지다. 대부분의 문제는 포인터 자체보다 추적 경로 누락에서 나온다.

### 5. 흔한 실수

- raw `UObject*`만 들고 있으면서 객체가 살아 있을 것이라 기대한다.
- `AddToRoot`를 임시 해결책으로 쓰고 `RemoveFromRoot`를 놓친다.
- `MarkAsGarbage` 이후 객체를 계속 사용한다.
- GC 수거된 객체에 대해 `nullptr` 체크만 하고 안전하다고 판단한다.

## 토픽 7: CDO(Class Default Object, 씨디오, 클래스 기본 객체)

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 5

### 1. 핵심 개념

CDO(Class Default Object, 씨디오, 클래스 기본 객체)는 클래스 기본값을 대표하는 특수 객체다. UE는 클래스마다 CDO를 만들어 기본 프로퍼티 값을 보관하고, 새 인스턴스는 이 기본값을 바탕으로 초기화된다. 즉 생성자는 인스턴스만이 아니라 CDO 생성 경로에서도 호출될 수 있다.

이 때문에 생성자에서 월드 접근, 플레이어 접근, 동적 로딩을 무심코 하면 CDO 생성 시점에 문제가 날 수 있다. CDO는 기본값 템플릿이라는 관점으로 이해하는 것이 좋다.

### 2. 코드 예시

```cpp
UClass* Class = AMyActor::StaticClass();
const AMyActor* DefaultObject = Class->GetDefaultObject<AMyActor>();
```

### 3. 왜 중요한가 (게임/UE5 맥락)

UE 생성자 규칙, 블루프린트 기본값, 에디터 미리보기, 기본 서브오브젝트 구조는 모두 CDO와 연결된다. 면접에서 "생성자에서 월드 접근을 왜 조심해야 하나요?"라는 질문은 사실 CDO 이해를 묻는 경우가 많다.

### 4. 면접 예상 Q&A

**Q1: CDO는 무엇인가요?**
> A: 클래스 기본값을 담는 대표 객체다. 결론적으로 새 인스턴스가 초기화될 때 참조하는 기본 템플릿으로 이해하면 된다.

**Q2: 생성자에서 월드 의존 로직을 조심해야 하는 이유는 무엇인가요?**
> A: 생성자가 CDO 생성 경로에서도 실행될 수 있기 때문이다. CDO에는 실제 월드 문맥이 없을 수 있어 안전하지 않다.

**Q3: CDO는 어디에 유용하게 쓰이나요?**
> A: 기본값 조회, 에디터 툴, 클래스 비교, 기본 설정 검증에 유용하다. 인스턴스를 만들지 않고도 클래스 기본 상태를 읽을 수 있다.

**Q4: 블루프린트 기본값과 CDO는 어떤 관계인가요?**
> A: 블루프린트 클래스도 자체 CDO를 가진다. 따라서 블루프린트에서 바꾼 기본값은 그 클래스의 CDO에 반영된다고 볼 수 있다.

**Q5: CDO와 인스턴스를 혼동하면 어떤 문제가 생기나요?**
> A: 기본값을 바꾸려는 코드와 런타임 상태를 바꾸려는 코드가 섞인다. 결과적으로 의도치 않게 전체 클래스 기본 설정에 영향을 줄 수도 있다.

**Q6: 실무에서 CDO를 설명할 때 한 문장으로 요약하면 무엇인가요?**
> A: "클래스 기본 상태를 담는 원본 객체"라고 설명하면 가장 직관적이다.

### 5. 흔한 실수

- 생성자에서 인스턴스 전용 월드 로직을 실행한다.
- CDO와 런타임 인스턴스 상태를 구분하지 않는다.
- 기본값 조회가 필요한데 인스턴스를 불필요하게 생성한다.

## 토픽 8: 객체 생성 방식

> 난이도: ★★★ | 예상 시간: 1시간 | 선행: 토픽 5, 7

### 1. 핵심 개념

UE 객체 생성은 대상 타입과 생성 시점에 따라 API가 다르다. `CreateDefaultSubobject`는 생성자에서 기본 서브오브젝트용, `NewObject`는 런타임 `UObject`용, `SpawnActor`는 월드 배치형 `AActor`용, `DuplicateObject`는 기존 객체 복제용으로 자주 쓰인다.

핵심은 "무엇을 만들고 어디에 붙일 것인가"다. UObject와 Actor, 기본 구조와 런타임 생성, 복제와 신규 생성을 구분해야 수명과 초기화가 꼬이지 않는다.

### 2. 코드 예시

```cpp
// 생성자 내부
MeshComponent = CreateDefaultSubobject<UStaticMeshComponent>(TEXT("Mesh"));

// 런타임 UObject
UInventoryItem* Item = NewObject<UInventoryItem>(this);

// 런타임 Actor
AMyProjectile* Projectile = GetWorld()->SpawnActor<AMyProjectile>(ProjectileClass, SpawnTransform);
```

### 3. 왜 중요한가 (게임/UE5 맥락)

잘못된 생성 API 선택은 초기화 순서, 등록, 소유권, 월드 연결을 모두 망칠 수 있다. 특히 기본 서브오브젝트와 동적 컴포넌트를 구분하지 못하면 에디터와 런타임 동작이 달라진다.

### 4. 면접 예상 Q&A

**Q1: `CreateDefaultSubobject`와 `NewObject`는 어떻게 다르나요?**
> A: 전자는 클래스 기본 구조를 만드는 생성자 전용 API이고, 후자는 런타임 UObject 생성 API다. 결론적으로 기본 구조인지 동적 생성인지가 기준이다.

**Q2: `SpawnActor`가 별도 API인 이유는 무엇인가요?**
> A: Actor는 월드 등록, transform, 네트워크, 충돌 초기화 같은 추가 단계가 필요하기 때문이다. 일반 UObject 생성과 같은 경로로 처리할 수 없다.

**Q3: `DuplicateObject`는 언제 유용한가요?**
> A: 기존 설정을 복제한 새 인스턴스를 만들고 싶을 때 유용하다. 에디터 툴이나 템플릿 기반 생성에서 자주 쓴다.

**Q4: 생성 API 선택이 잘못됐을 때 대표 증상은 무엇인가요?**
> A: 컴포넌트가 에디터에서 안 보이거나, 월드 등록이 안 되거나, 기본값이 예상과 다르게 적용되는 식이다. 수명과 등록 시점 오류가 많다.

**Q5: `Outer`는 객체 생성 시 왜 중요하나요?**
> A: 소속 컨텍스트와 이름 공간을 정하기 때문이다. 또한 이후 추적과 정리 흐름을 이해하는 데 단서가 된다.

**Q6: 실무에서 생성 시점을 정할 때 가장 먼저 보는 기준은 무엇인가요?**
> A: 이 객체가 클래스 기본 구조인지, 월드 런타임 상태인지, 일회성 임시 객체인지다. 그 판단이 생성 API 선택을 거의 결정한다.

### 5. 흔한 실수

- 생성자 밖에서 `CreateDefaultSubobject`와 비슷한 패턴을 기대한다.
- Actor를 `NewObject`처럼 생각한다.
- 기본 서브오브젝트와 동적 생성 컴포넌트를 같은 수명 규칙으로 다룬다.

## 토픽 9: UE5 스마트 포인터

> 난이도: ★★☆ | 예상 시간: 1.5시간 | 선행: 카테고리 1 / 토픽 4

### 1. 핵심 개념

UE는 `TSharedPtr`, `TWeakPtr`, `TUniquePtr` 같은 스마트 포인터를 제공하지만, 이들은 일반 C++ 객체용이지 `UObject` 소유권 관리용이 아니다. `UObject`는 GC가 관리하므로 `TObjectPtr`, raw `UObject*` + `UPROPERTY`, `TWeakObjectPtr`, `TSoftObjectPtr` 같은 별도 체계를 써야 한다.

핵심은 "스마트 포인터가 다 같은 문제가 아니다"라는 점이다. 비UObject 수명 문제와 UObject GC 추적 문제는 완전히 다른 축이다.

**`TObjectPtr<T>` 심화**: UE5에서 `UObject*` raw 포인터를 헤더 파일의 `UPROPERTY` 멤버에 한해 `TObjectPtr<T>`로 전환하는 방향이 권장된다. `TObjectPtr`는 디버그/에디터 빌드에서 접근 추적(access tracking)을 지원해 어떤 코드가 어떤 참조에 접근하는지 진단할 수 있게 한다. 쉬핑 빌드에서는 raw 포인터와 동일한 성능으로 자동 전환된다. `.cpp` 파일의 지역 변수나 함수 파라미터에서는 raw 포인터를 그대로 쓴다. `TObjectPtr`는 소유권 모델을 바꾸는 것이 아니라 기존 `UPROPERTY` + GC 체계 위에 진단 도구를 더한 것이다.

### 2. 코드 예시

```cpp
// 비UObject 스마트 포인터
TSharedPtr<FString> SharedName = MakeShared<FString>(TEXT("Player"));
TWeakPtr<FString> WeakName = SharedName;

// UObject 참조 — 헤더 UPROPERTY에는 TObjectPtr 사용 (UE5 권장)
UPROPERTY()
TObjectPtr<UTexture2D> IconTexture;

// .cpp 파일 지역 변수에서는 raw 포인터 사용
UTexture2D* LocalTexture = IconTexture; // TObjectPtr → raw 포인터 암묵 변환

// TObjectPtr와 raw 포인터의 호환
void ProcessTexture(UTexture2D* Tex); // 함수 시그니처는 raw 포인터
ProcessTexture(IconTexture);          // TObjectPtr가 자동 변환
```

### 3. 왜 중요한가 (게임/UE5 맥락)

UE 입문자에게 가장 흔한 오해 중 하나는 "스마트 포인터면 다 안전하다"는 생각이다. 하지만 `TSharedPtr<UObject>` 같은 구조는 GC와 충돌하므로 사용 기준을 분명히 알아야 한다. UE5 전환 이후 `TObjectPtr`를 쓰는 이유와 범위를 모르면 코드 리뷰에서 자주 지적받는다.

### 4. 면접 예상 Q&A

**Q1: `TSharedPtr`를 `UObject`에 쓰지 않는 이유는 무엇인가요?**
> A: `UObject`는 참조 카운팅이 아니라 GC로 수명 관리되기 때문이다. 결론적으로 소유권 모델이 달라 섞으면 오히려 혼란이 커진다.

**Q2: `TWeakPtr`와 `TWeakObjectPtr`는 어떻게 다른가요?**
> A: 전자는 shared ownership 체계 위의 약한 참조이고, 후자는 GC 관리 UObject를 위한 약한 참조다. 대상 수명 모델 자체가 다르다.

**Q3: `TObjectPtr`는 어떤 역할을 하고, 어디에 쓰나요?**
> A: 헤더 파일의 `UPROPERTY` 멤버에서 `UObject*` 대신 사용한다. 디버그 빌드에서 접근 추적을 지원하고, 쉬핑 빌드에서는 raw 포인터와 동일하게 동작한다. `.cpp` 지역 변수나 함수 파라미터에는 쓰지 않는다.

**Q4: `TObjectPtr`는 GC 동작을 바꾸나요?**
> A: 아니다. GC와 소유권 모델은 기존과 같다. `TObjectPtr`는 진단과 추적 도구이지 수명 관리 정책을 바꾸는 것이 아니다.

**Q5: UObject를 약하게 참조하고 싶을 때 무엇을 쓰나요?**
> A: `TWeakObjectPtr`를 쓴다. 필요 시 유효성 체크 후 접근하는 패턴이 일반적이다.

**Q6: 실무에서 포인터 선택이 잘못됐다는 신호는 무엇인가요?**
> A: 수명 버그를 스마트 포인터 종류만 바꿔 해결하려 하거나, UObject와 비UObject를 같은 규칙으로 다룰 때다. 먼저 수명 모델을 구분해야 한다.

### 5. 흔한 실수

- `TSharedPtr<UObject>`를 사용한다.
- 약한 참조를 저장해 놓고 유효성 체크 없이 바로 역참조한다.
- 비UObject 헬퍼와 UObject 참조를 한 컨테이너에서 같은 규칙으로 관리한다.
- `.cpp` 지역 변수에까지 `TObjectPtr`를 사용해 불필요한 복잡성을 더한다.
- 기존 프로젝트에서 `TObjectPtr` 전환 시 함수 시그니처까지 바꿔 호환성 문제를 만든다.

## 토픽 10: 소프트 참조 & 비동기 로드

> 난이도: ★★★ | 예상 시간: 1시간 | 선행: 토픽 9

### 1. 핵심 개념

`soft reference(소프트 레퍼런스, 경로 기반 지연 참조)`는 객체를 즉시 메모리에 올리지 않고 경로 기반으로 참조하는 방식이다. `TSoftObjectPtr`, `TSoftClassPtr`, `FSoftObjectPath`가 대표적이며, 필요 시점에 동기 또는 비동기로 로드한다. 큰 프로젝트일수록 하드 참조를 줄이는 데 중요하다.

비동기 로드는 `UAssetManager`나 `FStreamableManager`와 함께 쓰는 경우가 많다. 핵심은 "참조와 로드를 분리한다"는 것이다. 즉 소프트 참조는 메모리와 로딩 비용 제어 도구다.

### 2. 코드 예시

```cpp
UPROPERTY(EditAnywhere)
TSoftObjectPtr<UTexture2D> IconTexture;

void LoadIcon()
{
    IconTexture.LoadSynchronous();
}
```

### 3. 왜 중요한가 (게임/UE5 맥락)

모든 에셋을 하드 참조하면 로딩 시간과 메모리 사용량이 빠르게 커진다. UI 아이콘, 큰 스켈레탈 메시, 희귀 아이템 데이터처럼 필요 시점 로드가 적합한 자산은 소프트 참조가 큰 차이를 만든다.

### 4. 면접 예상 Q&A

**Q1: 하드 참조와 소프트 참조의 차이는 무엇인가요?**
> A: 하드 참조는 대상이 로드 그래프에 직접 포함되기 쉽고, 소프트 참조는 경로만 저장해 필요 시점에 로드한다. 결론적으로 메모리와 로딩 결합도가 다르다.

**Q2: 비동기 로드를 쓰는 이유는 무엇인가요?**
> A: 큰 에셋을 메인 흐름을 멈추지 않고 로드하기 위해서다. 로딩 스파이크와 사용자 체감 지연을 줄이는 데 중요하다.

**Q3: `UAssetManager`는 언제 고려하나요?**
> A: 자산 분류와 로딩 정책을 체계적으로 관리해야 하는 프로젝트에서 고려한다. 규모가 커질수록 단순 `LoadSynchronous` 호출보다 장점이 커진다.

**Q4: 소프트 참조가 만능이 아닌 이유는 무엇인가요?**
> A: 접근 전에 로드 완료를 보장해야 하고, 로딩 실패 처리도 필요하기 때문이다. 즉 결합은 줄지만 흐름 제어가 복잡해진다.

**Q5: 실무에서 소프트 참조가 특히 유용한 예시는 무엇인가요?**
> A: 인벤토리 아이콘, DLC성 콘텐츠, 드물게 등장하는 보스 자산, 메뉴 진입 시 필요한 UI 세트 등이다. 자주 쓰지 않는 큰 자산일수록 효과가 크다.

**Q6: 비동기 로드 설계에서 가장 중요한 것은 무엇인가요?**
> A: 로드 완료 전 상태를 어떻게 처리할지 정하는 것이다. placeholder, 로딩 화면, 취소 처리까지 설계해야 실제 사용성이 좋다.

### 5. 흔한 실수

- 소프트 참조를 써 놓고 매번 즉시 동기 로드만 호출한다.
- 로드 완료 전에 객체 사용을 가정한다.
- 하드 참조와 소프트 참조를 섞어 로딩 구조를 더 복잡하게 만든다.

## 토픽 11: 메모리 할당자 & 오브젝트 풀링

> 난이도: ★★★ | 예상 시간: 1시간 | 선행: 카테고리 1 / 토픽 25

### 1. 핵심 개념

UE는 전역 할당 계층과 플랫폼별 할당자를 통해 메모리를 관리한다. 개발자는 주로 `FMemory::Malloc`, `FMemory::Free`, 컨테이너 할당 정책, Binned 계열 할당자 개념을 이해하면 된다. 핵심은 "게임 엔진은 일반 앱보다 할당 패턴이 훨씬 공격적"이라는 점이다.

작은 객체가 빈번히 생성되고 파괴되는 환경에서는 파편화와 캐시 효율이 중요하다. 그래서 엔진은 범용 malloc만 믿지 않고 자체 할당 전략을 둔다.

**오브젝트 풀링 패턴**: 빈번한 Spawn/Destroy를 줄이는 실무 최적화 기법이다. `TArray` 기반 풀에 미리 생성한 Actor를 넣어 두고, 필요 시 `SetActorHiddenInGame(false)` + `SetActorEnableCollision(true)` + 컴포넌트 활성화로 꺼내 쓴 뒤, 반환 시 다시 비활성화한다. `SpawnActor`/`Destroy` 비용(월드 등록, 네트워크 채널, 컴포넌트 초기화)을 피할 수 있어, 투사체, 이펙트, 반복 스폰 AI 같은 곳에서 큰 차이를 만든다. Niagara 파티클 풀은 엔진이 자체 관리하므로, 직접 풀링하는 것은 주로 Actor와 컴포넌트 대상이다.

### 2. 코드 예시

```cpp
// --- 저수준 메모리 할당 ---
void* RawMemory = FMemory::Malloc(256);
FMemory::Memzero(RawMemory, 256);
FMemory::Free(RawMemory);

// --- 오브젝트 풀링 패턴 예시 ---
UCLASS()
class AProjectilePool : public AActor
{
    GENERATED_BODY()

public:
    void InitializePool(TSubclassOf<AActor> PooledClass, int32 PoolSize)
    {
        for (int32 i = 0; i < PoolSize; ++i)
        {
            AActor* Obj = GetWorld()->SpawnActor<AActor>(PooledClass);
            DeactivateActor(Obj);
            Pool.Add(Obj);
        }
    }

    AActor* GetPooledActor()
    {
        for (AActor* Actor : Pool)
        {
            if (Actor->IsHidden())
            {
                ActivateActor(Actor);
                return Actor;
            }
        }
        return nullptr; // 풀 소진
    }

    void ReturnToPool(AActor* Actor)
    {
        DeactivateActor(Actor);
    }

private:
    void ActivateActor(AActor* Actor)
    {
        Actor->SetActorHiddenInGame(false);
        Actor->SetActorEnableCollision(true);
        Actor->SetActorTickEnabled(true);
    }

    void DeactivateActor(AActor* Actor)
    {
        Actor->SetActorHiddenInGame(true);
        Actor->SetActorEnableCollision(false);
        Actor->SetActorTickEnabled(false);
    }

    UPROPERTY()
    TArray<TObjectPtr<AActor>> Pool;
};
```

### 3. 왜 중요한가 (게임/UE5 맥락)

프레임 단위 할당 스파이크, 파편화, 플랫폼별 메모리 제약은 게임에서 치명적이다. 엔진 할당 개념을 알아야 컨테이너 선택, 오브젝트 풀링, 프레임 버짓 관점에서 더 정확한 판단이 가능하다. 특히 "투사체를 매 프레임 Spawn하면 왜 느린지" 같은 실무 역량 질문에는 풀링 구현까지 설명할 수 있어야 한다.

### 4. 면접 예상 Q&A

**Q1: UE가 자체 할당 전략을 두는 이유는 무엇인가요?**
> A: 게임은 작은 객체의 빈번한 생성/파괴와 실시간 성능 요구가 강하기 때문이다. 범용 할당자만으로는 파편화와 예측 가능성을 제어하기 어렵다.

**Q2: `FMemory` 계층은 언제 직접 쓰나요?**
> A: 아주 낮은 수준의 메모리 제어나 성능 민감 코드에서 쓴다. 일반 게임플레이 로직에서는 컨테이너와 UObject 경로를 우선하는 편이 더 안전하다.

**Q3: 오브젝트 풀링은 왜 효과가 큰가요?**
> A: `SpawnActor`는 월드 등록, 컴포넌트 초기화, 네트워크 채널 설정 등 무거운 작업을 동반한다. 풀링은 이 비용을 최초 1회로 줄이고 이후에는 활성/비활성 전환만 한다.

**Q4: 풀링 시 `Destroy` 대신 비활성화하는 이유는 무엇인가요?**
> A: `Destroy`는 GC 대상으로 만들고 모든 등록을 해제한다. 비활성화(`SetActorHiddenInGame`, `SetActorEnableCollision(false)`, `SetActorTickEnabled(false)`)는 월드에 등록된 상태를 유지하면서 렌더링과 로직만 끈다.

**Q5: Niagara 파티클과 Actor 풀링은 어떻게 다른가요?**
> A: Niagara는 엔진이 파티클 인스턴스를 자체 풀링한다. Actor 풀링은 개발자가 직접 관리하는 패턴이다. Niagara 이펙트를 위해 Actor 풀을 만들 필요는 없다.

**Q6: 실무에서 메모리 문제를 볼 때 가장 먼저 보는 지표는 무엇인가요?**
> A: 할당 빈도, 피크 시점, 객체 수명 길이, 파편화 양상이다. 총 사용량만 보는 것은 부족하다.

### 5. 흔한 실수

- 일반 게임플레이 코드에 불필요하게 저수준 할당을 직접 도입한다.
- 풀링이 필요한 대상을 매 프레임 새로 생성한다.
- 할당 경로와 해제 경로를 일관되게 관리하지 않는다.
- 풀에서 꺼낸 Actor의 상태(위치, 속도, 타이머)를 리셋하지 않는다.
- 풀 크기를 고정해 놓고 소진 시 예외 처리를 하지 않는다.

## 토픽 22: Actor 라이프사이클(액터 생명주기)

> 난이도: ★★★ | 예상 시간: 1시간 | 선행: 토픽 5, 7

### 1. 핵심 개념

`AActor`는 `UObject`의 생명주기 위에 월드와 게임플레이에 특화된 추가 단계를 가진다. 핵심 흐름은 다음과 같다:

**생성자** → `PostInitializeComponents()` → `BeginPlay()` → `Tick()` (매 프레임) → `EndPlay()` → `Destroyed()` → GC 수거

각 단계의 역할을 구분하는 것이 중요하다:
- **생성자**: CDO 생성에도 호출되므로, `CreateDefaultSubobject`로 기본 컴포넌트를 만들고 기본값만 설정한다. 월드, 게임 상태, 다른 액터에 의존하면 안 된다.
- **`PostInitializeComponents()`**: 모든 컴포넌트가 초기화된 직후 호출된다. 컴포넌트 간 연결이나 설정이 필요한 로직을 여기에 둔다.
- **`BeginPlay()`**: 게임이 시작되고 액터가 월드에서 활성화될 때 호출된다. 게임플레이 초기화, 타이머 설정, 다른 액터 참조 등을 여기서 한다.
- **`EndPlay()`**: 액터가 월드에서 제거되기 직전에 호출된다. 정리 작업, 델리게이트 해제, 외부 참조 정리를 여기서 한다. `EEndPlayReason`으로 이유를 구분할 수 있다.
- **`Destroyed()`**: `EndPlay` 이후 파괴 과정에서 호출된다.

`SpawnActorDeferred`를 쓰면 스폰과 `BeginPlay` 사이에 프로퍼티를 설정할 수 있고, `FinishSpawning`으로 완료한다.

### 2. 코드 예시

```cpp
UCLASS()
class AMyCharacter : public ACharacter
{
    GENERATED_BODY()

public:
    AMyCharacter()
    {
        // 생성자: CDO에서도 호출됨 — 기본 구조만 설정
        HealthComponent = CreateDefaultSubobject<UHealthComponent>(TEXT("Health"));
        PrimaryActorTick.bCanEverTick = true;
        // GetWorld() 호출 금지! CDO에는 월드가 없을 수 있음
    }

    virtual void PostInitializeComponents() override
    {
        Super::PostInitializeComponents();
        // 컴포넌트가 모두 준비된 후 — 컴포넌트 간 연결
        HealthComponent->OnDeath.AddDynamic(this, &AMyCharacter::HandleDeath);
    }

    virtual void BeginPlay() override
    {
        Super::BeginPlay();
        // 게임플레이 시작 — 월드, 게임모드, 다른 액터 접근 가능
        if (AMyGameMode* GM = Cast<AMyGameMode>(GetWorld()->GetAuthGameMode()))
        {
            GM->RegisterCharacter(this);
        }
    }

    virtual void EndPlay(const EEndPlayReason::Type EndPlayReason) override
    {
        // 정리: 델리게이트 해제, 외부 참조 정리
        if (AMyGameMode* GM = Cast<AMyGameMode>(GetWorld()->GetAuthGameMode()))
        {
            GM->UnregisterCharacter(this);
        }
        Super::EndPlay(EndPlayReason);
    }

private:
    UPROPERTY()
    TObjectPtr<UHealthComponent> HealthComponent;

    UFUNCTION()
    void HandleDeath();
};

// --- SpawnActorDeferred 패턴 ---
AMyProjectile* Proj = GetWorld()->SpawnActorDeferred<AMyProjectile>(
    ProjectileClass, SpawnTransform);
Proj->DamageAmount = 50.f;      // BeginPlay 전에 프로퍼티 설정
Proj->FinishSpawning(SpawnTransform); // 이후 BeginPlay 호출됨
```

### 3. 왜 중요한가 (게임/UE5 맥락)

생성자와 `BeginPlay`의 차이, CDO 맥락에서의 제약, 정리 시점 설계는 UE 면접에서 기초 체력으로 여겨진다. "생성자에서 왜 `GetWorld()`를 쓰면 안 되나요?"는 CDO 토픽(7번)과 연결되고, "`PostInitializeComponents`는 언제 쓰나요?"는 컴포넌트 모델과 연결된다. 이 흐름을 알아야 초기화 버그를 구조적으로 추적할 수 있다.

### 4. 면접 예상 Q&A

**Q1: 생성자에서 게임플레이 초기화를 하면 안 되는 이유는 무엇인가요?**
> A: 생성자는 CDO 생성에서도 실행되므로 월드, 게임모드, 다른 액터가 없는 환경에서도 안전해야 한다. 게임플레이 초기화는 `BeginPlay`에서 해야 한다.

**Q2: `PostInitializeComponents`와 `BeginPlay`는 어떻게 구분하나요?**
> A: `PostInitializeComponents`는 컴포넌트가 모두 준비된 직후이므로 컴포넌트 간 연결에 적합하다. `BeginPlay`는 게임이 시작된 후이므로 월드와 게임 상태에 의존하는 초기화에 적합하다.

**Q3: `EndPlay`에서 꼭 해야 하는 것은 무엇인가요?**
> A: 외부 시스템에 등록한 참조 해제, 델리게이트 언바인드, 타이머 클리어 같은 정리 작업이다. 이걸 빠뜨리면 dangling 참조나 중복 호출이 발생한다.

**Q4: `SpawnActorDeferred`는 왜 필요한가요?**
> A: 스폰 후 `BeginPlay` 전에 프로퍼티를 설정해야 하는 경우가 있기 때문이다. 일반 `SpawnActor`는 즉시 `BeginPlay`까지 진행하므로 설정 시점이 없다.

**Q5: `Tick`을 끄는 것이 왜 중요한가요?**
> A: 모든 액터가 매 프레임 `Tick`하면 성능 비용이 크다. 필요 없는 액터는 `PrimaryActorTick.bCanEverTick = false`로 끄는 것이 기본이다.

**Q6: 에디터에서 액터를 배치할 때와 런타임에 스폰할 때 라이프사이클이 다른가요?**
> A: 핵심 흐름은 같지만, 에디터 배치 액터는 레벨 로드 시 `BeginPlay`가 호출되고, 런타임 스폰은 `SpawnActor` 시점에 즉시 진행된다. `PostInitializeComponents` → `BeginPlay` 순서는 동일하다.

### 5. 흔한 실수

- 생성자에서 `GetWorld()`, `GetGameInstance()`, `GetPlayerController()` 같은 런타임 의존 호출을 한다.
- `BeginPlay`에서 해야 할 초기화를 `PostInitializeComponents`에서 하거나 그 반대.
- `EndPlay`에서 델리게이트 해제를 빠뜨려 파괴 후 콜백 크래시가 발생한다.
- `Super::` 호출을 빼먹어 부모 클래스의 초기화/정리 로직이 누락된다.
- `SpawnActorDeferred` 후 `FinishSpawning` 호출을 잊는다.
