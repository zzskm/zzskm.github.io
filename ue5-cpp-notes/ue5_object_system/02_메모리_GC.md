## 토픽 5: UObject 생명주기

> 난이도: ★★★ | 예상 시간: 1시간 | 선행: 없음

### 1. 문제 상황

UE 프로젝트에서 객체 생성과 파괴 관련 버그는 대부분 “일반 C++ 객체처럼 생각했다”는 데서 시작된다. `UObject` 생명주기를 제대로 이해해야 GC, 블루프린트, 에디터, 서브오브젝트 생성 규칙이 하나의 일관된 흐름으로 연결된다.

`UObject`는 일반 C++ 객체처럼 `new`/`delete`로 다루는 대상이 아니다. `NewObject`, `CreateDefaultSubobject`, `SpawnActor` 같은 생성 API는 단순 편의 함수가 아니라, 엔진이 객체를 이름 공간에 등록하고 GC 추적 대상에 편입시키는 정식 입구다. 따라서 객체를 설계할 때 가장 중요한 질문은 “생성 순간”이 아니라 **“누가 참조하고, 어떤 컨텍스트에 속하는가”**다.

### 2. 핵심 개념

`UObject`의 수명은 프로그래머가 직접 관리하는 것이 아니라, 엔진이 인식하는 참조 그래프에 의해 결정된다. 특히 `Outer`는 자주 오해되는 개념이다. 겉으로는 부모 포인터처럼 보이지만, 실제로는 이름 공간과 소속 컨텍스트를 제공하는 관계에 가깝다. `Outer`만 지정했다고 해서 GC 보호가 자동으로 완성되는 것은 아니다. `UPROPERTY`, root set, 명시적 참조 경로를 함께 고려해야 한다.

### 3. 내부 동작 / 메커니즘

엔진 생성 API(`NewObject` 등)를 사용하면 객체는 자동으로 리플렉션 시스템에 등록되고, GC가 추적할 수 있는 그래프에 편입된다. `Outer`는 객체의 계층적 소속을 정의하지만, GC가 객체를 유지하는 결정적인 요소는 **추적 가능한 참조 경로**다. `UPROPERTY`가 붙지 않은 raw 포인터는 GC가 볼 수 없는 참조로 취급된다.

#### 핵심 규칙

- `UObject`는 엔진 생성 경로를 사용하고, 수동 `delete`를 호출하지 않는다.
- 수명은 참조 경로와 GC 도달 가능성에 의해 결정된다.

### 4. 자주 틀리는 포인트

- `UObject`를 일반 힙 객체처럼 `new`/`delete`로 관리한다.
- `Outer`만 지정하면 GC가 자동으로 보호해 준다고 오해한다.
- 로컬 변수에 담겨 있다는 이유만으로 안전하다고 판단한다.
- 엔진 API로 생성했더라도 참조 경로를 명시하지 않으면 예상보다 빨리 수거된다.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

**우선 검토하는 경우**

- UE 프로젝트에서 객체를 다룰 때 (거의 모든 경우)
- 수명 관리가 중요한 UObject를 생성할 때
- 서브오브젝트, 컴포넌트, 에셋 관련 객체를 다룰 때

**주의하거나 분리할 경우**

- `UObject`를 일반 C++ 힙 객체처럼 `new`/`delete`로 관리하려 할 때
- `Outer`만으로 수명이 보장된다고 착각할 때
- 참조 경로 없이 단순 포인터만 들고 있을 때

### 6. 코드 예시 (bad → good)

**Bad: 일반 C++ 방식으로 UObject 관리**

```cpp
UObject* Obj = new UObject();
delete Obj;        // 엔진이 모르는 객체
```

**Good: 엔진 생성 경로 사용**

```cpp
UObject* Obj = NewObject<UObject>(GetTransientPackage());
// 수명은 참조 경로와 GC에 맡긴다
```

**Bad: 엔진 생성 후 수동 delete**

```cpp
UObject* Obj = NewObject<UObject>(GetTransientPackage());
delete Obj;        // UE 수명 규칙 위반
```

**Good: 참조 경로로 관리**

```cpp
UObject* Obj = NewObject<UObject>(GetTransientPackage());
// 추적 가능한 UPROPERTY나 소유자 멤버로 연결
```

### 7. 판단 기준 요약

- `UObject`는 반드시 엔진 생성 API(`NewObject`, `CreateDefaultSubobject`, `SpawnActor`)로 만든다.
- 수동 `delete`는 절대 사용하지 않는다.
- 수명 관리는 `Outer`보다 **추적 가능한 참조 경로**(주로 `UPROPERTY`)가 핵심이다.
- 로컬 변수에만 담아두는 것은 안전하지 않다. 명시적인 소유 관계를 설계해야 한다.

## 토픽 6: UE5 GC 심화

> 난이도: ★★★ | 예상 시간: 1시간 | 선행: 토픽 5

### 1. 문제 상황

“포인터가 있는데 객체가 사라졌다”, “해제된 줄 알았는데 아직 살아 있다” 같은 버그가 반복된다. 특히 C++와 블루프린트를 함께 쓰는 프로젝트에서는 어떤 참조가 엔진에 의해 추적되는지 명확히 알아야 문제를 빠르게 해결할 수 있다.

UE5 GC는 `mark-and-sweep` 방식에 가깝다. root set에서 시작해 참조 그래프를 따라 도달 가능한 `UObject`를 유지하고, 도달 불가능한 객체를 수거한다. 중요한 점은 “포인터가 존재하는가”가 아니라 **“엔진이 그 참조를 알고 있는가”**다.

### 2. 핵심 개념

GC의 핵심은 **도달 가능성(reachability)**이다. root set, `UPROPERTY`, `AddToRoot`, `FGCObject` 등이 엔진이 참조를 인정하는 연결 장치다. `PendingKill` 상태도 이 맥락에서 이해해야 한다. GC 후에도 raw 포인터는 `nullptr`이 되지 않으므로 `IsValid()`를 사용하는 것이 안전하다.

#### 이 개념을 이해하려면

- 토픽 5: UObject 생명주기

#### 이 개념이 영향을 주는 것

- 토픽 5, 토픽 9: UE5 스마트 포인터

### 3. 내부 동작 / 메커니즘

GC는 root set에서 시작해 `UPROPERTY`로 연결된 참조 그래프를 탐색한다. 일반 C++ 클래스 안의 raw `UObject*`는 기본적으로 추적되지 않으므로 `FGCObject`를 상속해 `AddReferencedObjects`를 구현해야 한다. `IsValid()`는 `nullptr` 체크뿐 아니라 `PendingKill` 상태까지 확인해준다.

#### 핵심 규칙

- UObject 참조는 엔진이 볼 수 있는 경로(`UPROPERTY`, `FGCObject` 등)로 연결해야 한다.

### 4. 자주 틀리는 포인트

- raw `UObject*`만 들고 있으면서 객체가 살아 있을 것이라 기대한다.
- GC 후 포인터가 자동으로 `nullptr`이 된다고 오해한다.
- `IsValid()` 없이 raw 포인터를 직접 사용한다.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

**우선 검토하는 경우**

- UObject 참조를 다룰 때
- 블루프린트와 C++를 섞어 사용할 때
- 비UObject 클래스에서 UObject를 참조할 때 (`FGCObject`)

**주의하거나 분리할 경우**

- raw 포인터만으로 수명을 기대할 때
- GC 후에도 포인터를 그대로 역참조할 때

### 6. 코드 예시 (bad → good)

**Bad: GC가 모르는 참조**

```cpp
UCLASS()
class UMyHolder : public UObject
{
    GENERATED_BODY()
public:
    UObject* CachedTarget = nullptr;   // GC 추적 불가
};
```

**Good: UPROPERTY로 명시**

```cpp
UPROPERTY()
UObject* CachedTarget = nullptr;
```

**Bad: 일반 C++ 클래스에서 raw 참조**

```cpp
class FNativeCache
{
public:
    UObject* Cached = nullptr;
};
```

**Good: FGCObject 사용**

```cpp
class FNativeCache : public FGCObject
{
public:
    UObject* Cached = nullptr;

    virtual void AddReferencedObjects(FReferenceCollector& Collector) override
    {
        Collector.AddReferencedObject(Cached);
    }
};
```

### 7. 판단 기준 요약

- UObject 참조는 반드시 엔진이 인식할 수 있는 경로로 연결한다.
- raw `UObject*`는 비소유 관찰용으로만 사용한다.
- GC 후 유효성 확인은 `IsValid()`를 사용한다.
- 비UObject에서 UObject를 참조할 때는 `FGCObject`를 활용한다.

## 토픽 7: CDO (Class Default Object)

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 5

### 1. 문제 상황

UE의 생성자 규칙, 블루프린트 기본값, 에디터 미리보기, 기본 서브오브젝트 구조는 모두 CDO와 깊이 연결되어 있다. “생성자에서 왜 월드를 접근하면 안 되나요?”라는 질문은 사실 CDO 이해를 묻는 경우가 많다.

CDO는 각 클래스마다 하나씩 존재하는 특수 객체로, 클래스 기본값을 보관한다. 새 인스턴스는 이 CDO를 복사해 초기화된다. 따라서 생성자는 **인스턴스 생성 시점뿐 아니라 CDO 생성 시점**에도 호출될 수 있다.

### 2. 핵심 개념

CDO는 클래스 차원의 기본값 템플릿이다. 생성자에서 월드나 플레이어 같은 런타임 의존성을 접근하면 CDO 생성 단계에서 문제가 발생할 수 있다.

- 토픽 5

#### 이 개념이 영향을 주는 것

- 토픽 5, 토픽 8

### 3. 내부 동작 / 메커니즘

생성자는 CDO를 만들 때와 실제 인스턴스를 만들 때 모두 호출된다. 따라서 CDO 생성 시점에는 월드가 없을 수도 있다. `GetMutableDefault`로 CDO를 수정하는 것은 클래스 전체의 기본값을 바꾸는 행위다.

#### 핵심 규칙

- CDO 수정은 클래스 기본값을 바꾸는 행위임을 명확히 인지해야 한다.
- 월드 의존 로직은 생성자가 아닌 `BeginPlay` 등 런타임 단계로 미룬다.

### 4. 자주 틀리는 포인트

- 생성자에서 월드 접근이나 동적 로딩을 수행한다.
- CDO와 런타임 인스턴스 상태를 구분하지 않는다.
- `GetMutableDefault`를 인스턴스 수정처럼 사용한다.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

**우선 검토하는 경우**

- 클래스 기본값을 조회하거나 수정할 때
- 생성자에서 기본 서브오브젝트를 만들 때

**주의하거나 분리할 경우**

- 생성자에서 런타임 의존 로직을 실행할 때
- CDO를 인스턴스처럼 수정할 때

### 6. 코드 예시 (bad → good)

**Bad: 생성자에서 월드 접근**

```cpp
UMyActor::UMyActor()
{
    auto* PC = GetWorld()->GetFirstPlayerController();  // CDO 생성 시 위험
}
```

**Good: 런타임 단계로 이동**

```cpp
void UMyActor::BeginPlay()
{
    Super::BeginPlay();
    auto* PC = GetWorld()->GetFirstPlayerController();
}
```

**Bad: CDO를 인스턴스처럼 수정**

```cpp
GetMutableDefault<UMySettings>()->Damage = 999;
```

**Good: 인스턴스 수정**

```cpp
UMySettings* Instance = NewObject<UMySettings>();
Instance->Damage = 999;
```

### 7. 판단 기준 요약

- 생성자에서는 CDO 생성도 고려하여 월드 의존 로직을 피한다.
- 월드 의존 로직은 `BeginPlay` 등으로 미룬다.
- CDO 수정은 클래스 기본값 변경임을 인지한다.
- 인스턴스 상태 변경은 실제 인스턴스에서 수행한다.

## 토픽 8: 객체 생성 방식

> 난이도: ★★★ | 예상 시간: 1시간 | 선행: 토픽 5, 7

### 1. 문제 상황

잘못된 생성 API를 선택하면 초기화 순서, 등록, 소유권, 월드 연결이 모두 꼬인다. 특히 기본 서브오브젝트와 런타임 동적 컴포넌트를 구분하지 못하면 에디터와 런타임 동작이 달라지는 버그가 발생한다.

UE 객체 생성은 대상 타입과 시점에 따라 API가 다르다. `CreateDefaultSubobject`는 생성자에서 기본 구조용, `NewObject`는 일반 UObject 런타임 생성용, `SpawnActor`는 Actor 월드 배치용이다.

### 2. 핵심 개념

생성 API 선택의 핵심은 “무엇을 만들고, 어디에 붙일 것인가”다. 기본 구조와 런타임 동적 생성, UObject와 Actor를 구분해야 수명과 초기화 흐름이 올바르게 유지된다.

#### 이 개념을 이해하려면

- 토픽 5, 토픽 7

#### 이 개념이 영향을 주는 것

- 토픽 5, 토픽 7

### 3. 내부 동작 / 메커니즘

`CreateDefaultSubobject`는 생성자에서만 안전하게 사용되며, 기본 서브오브젝트 체인을 구축한다. `NewObject`는 런타임 UObject 생성에 적합하고, `SpawnActor`는 Actor의 월드 등록과 초기화를 함께 처리한다.

#### 핵심 규칙

- 생성자 기본 구조와 런타임 동적 생성은 API를 구분해서 사용한다.

### 4. 자주 틀리는 포인트

- 생성자 밖에서 `CreateDefaultSubobject`와 유사한 패턴을 기대한다.
- Actor를 `NewObject`로 생성한다.
- 생성자에서 `NewObject`로 컴포넌트를 만들어 기본 체인이 깨진다.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

**우선 검토하는 경우**

- 클래스 기본 구조를 만들 때 (`CreateDefaultSubobject`)
- 일반 UObject를 런타임에 동적으로 만들 때 (`NewObject`)
- Actor를 월드에 배치할 때 (`SpawnActor`)

**주의하거나 분리할 경우**

- 생성자 밖에서 기본 서브오브젝트 생성 패턴을 사용할 때
- Actor를 일반 UObject처럼 취급할 때

### 6. 코드 예시 (bad → good)

**Bad: 생성자에서 NewObject로 컴포넌트 생성**

```cpp
AMyActor::AMyActor()
{
    Mesh = NewObject<UStaticMeshComponent>(this);
}
```

**Good: CreateDefaultSubobject 사용**

```cpp
AMyActor::AMyActor()
{
    Mesh = CreateDefaultSubobject<UStaticMeshComponent>(TEXT("Mesh"));
}
```

**Bad: Actor를 NewObject로 생성**

```cpp
AMyProjectile* Proj = NewObject<AMyProjectile>(this);
```

**Good: SpawnActor 사용**

```cpp
AMyProjectile* Proj = GetWorld()->SpawnActor<AMyProjectile>(ProjectileClass, SpawnTransform);
```

### 7. 판단 기준 요약

- 생성자에서 기본 구조 → `CreateDefaultSubobject`
- 런타임 일반 UObject → `NewObject`
- 월드에 배치하는 Actor → `SpawnActor`
- 생성 목적과 시점에 맞는 API를 선택하라.

## 토픽 9: UE5 스마트 포인터

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 카테고리 1 토픽 4

### 1. 문제 상황

많은 개발자가 “스마트 포인터면 안전하다”고 생각하지만, UE에서는 상황이 다르다. `TSharedPtr<UObject>`는 GC와 충돌할 수 있으며, UE5에서는 `TObjectPtr`가 점차 표준이 되고 있다.

UE 스마트 포인터는 비UObject와 UObject를 완전히 다른 규칙으로 다룬다. UObject는 GC가 주도하므로 일반 C++ 스마트 포인터와 혼용하면 수명 모델이 깨진다.

### 2. 핵심 개념

- 비UObject → `TSharedPtr`, `TUniquePtr`, `TWeakPtr`
- UObject → `TObjectPtr`, `TWeakObjectPtr`, `UPROPERTY`, `TSoftObjectPtr`

`TObjectPtr<T>`는 raw `UObject*`를 대체하는 명시적 래퍼로, 디버그 빌드에서 잘못된 접근을 조기 감지한다.

#### 이 개념을 이해하려면

- 카테고리 1 토픽 4

#### 이 개념이 영향을 주는 것

- 토픽 6: UE5 GC 심화
- 토픽 10: 소프트 참조 & 비동기 로드

### 3. 내부 동작 / 메커니즘

UObject는 GC가 관리하므로 참조 카운트 기반 스마트 포인터와 충돌한다. `TObjectPtr`는 UObject 참조를 더 안전하고 명시적으로 다루기 위한 UE5의 방향이다.

#### 핵심 규칙

- UObject와 비UObject는 스마트 포인터 선택 기준을 명확히 분리한다.

### 4. 자주 틀리는 포인트

- `TSharedPtr<UObject>`를 사용한다.
- `TWeakObjectPtr`를 사용하면서 유효성 체크를 생략한다.
- UObject를 일반 스마트 포인터로 소유하려 한다.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

**우선 검토하는 경우**

- 비UObject 공유 소유 → `TSharedPtr`
- UObject 비소유 관찰 → `TWeakObjectPtr`
- UObject 참조 멤버 → `TObjectPtr` + `UPROPERTY`

**주의하거나 분리할 경우**

- UObject를 `TSharedPtr`/`TUniquePtr`로 관리하려 할 때

### 6. 코드 예시 (bad → good)

**Bad**

```cpp
TSharedPtr<UObject> SharedObj;
```

**Good**

```cpp
TWeakObjectPtr<UObject> WeakObj;
UPROPERTY()
TObjectPtr<UObject> OwnedObj;
```

### 7. 판단 기준 요약

- UObject는 GC 모델을 따른다 → `TObjectPtr`, `TWeakObjectPtr`, `UPROPERTY`
- 비UObject는 일반 스마트 포인터를 사용한다.
- UObject 참조는 엔진이 추적할 수 있는 형태로 유지한다.

## 토픽 10: 소프트 참조 & 비동기 로드

> 난이도: ★★★ | 예상 시간: 1시간 | 선행: 토픽 9

### 1. 문제 상황

모든 에셋을 하드 참조하면 초기 로딩 시간과 메모리 사용량이 급격히 증가한다. 반대로 필요할 때만 로드하는 소프트 참조를 적절히 사용하면 큰 성능 이득을 얻을 수 있다.

소프트 참조(`TSoftObjectPtr`, `TSoftClassPtr`, `FSoftObjectPath`)는 객체를 즉시 메모리에 올리지 않고 경로만 보관한다. 실제 객체는 필요 시점에 동기 또는 비동기로 로드한다.

### 2. 핵심 개념

항상 함께 사용해야 하는 에셋은 하드 참조가 단순하지만, 크고 드물게 사용하는 에셋은 소프트 참조가 메모리와 로딩 비용을 효과적으로 제어한다. `UAssetManager`와 `FStreamableManager`는 이 분리를 체계적으로 관리하는 도구다.

#### 이 개념을 이해하려면

- 토픽 9

#### 이 개념이 영향을 주는 것

- 토픽 19: UPackage & 에셋 시스템

### 3. 내부 동작 / 메커니즘

소프트 참조는 경로만 저장하고, `LoadSynchronous()` 또는 비동기 로드로 실제 객체를 가져온다. 로드 완료 전에 객체를 사용하려 하면 null이나 미로드 상태가 된다.

#### 핵심 규칙

- 대형·희귀 에셋은 소프트 참조를 고려한다.
- 비동기 로드 시 완료 상태를 반드시 확인한다.

### 4. 자주 틀리는 포인트

- 소프트 참조를 사용하면서 매번 `LoadSynchronous()`만 호출한다.
- 로드 완료 전에 객체를 사용한다.
- 핵심 에셋까지 소프트 참조로만 관리한다.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

**우선 검토하는 경우**

- 대형 에셋, UI 아이콘, 희귀 아이템 등 필요 시점 로드가 적합한 경우

**주의하거나 분리할 경우**

- 항상 함께 로드되어야 하는 핵심 에셋
- 로드 완료를 기다리지 않고 바로 사용하는 경우

### 6. 코드 예시 (bad → good)

**Bad: 하드 참조 과다 사용**

```cpp
UPROPERTY(EditAnywhere)
UStaticMesh* BossMesh;
```

**Good: 소프트 참조로 지연 로드**

```cpp
UPROPERTY(EditAnywhere)
TSoftObjectPtr<UStaticMesh> BossMesh;
```

**Bad: 매 프레임 동기 로드**

```cpp
void TickUI() {
    IconTexture.LoadSynchronous();
    DrawIcon(IconTexture.Get());
}
```

**Good: 초기화 시 한 번 로드**

```cpp
void PrepareUI() {
    LoadedIcon = IconTexture.LoadSynchronous();
}
void TickUI() {
    DrawIcon(LoadedIcon);
}
```

### 7. 판단 기준 요약

- 대형·희귀 에셋 → 소프트 참조 고려
- 로드 완료 전에 사용하지 않는다.
- 핵심 에셋은 하드 참조가 더 안전하다.
- 비동기 로드 시 완료 상태를 반드시 처리한다.

## 토픽 11: 메모리 할당자

> 난이도: ★★★ | 예상 시간: 1시간 | 선행: 카테고리 1 토픽 25

### 1. 문제 상황

게임은 프레임마다 작은 객체가 빈번하게 생성·파괴되므로, 일반 앱과 할당 패턴이 매우 다르다. 할당 비용과 메모리 단편화가 직접 프레임 드롭으로 이어진다.

UE는 자체 할당 계층(`FMemory`, Binned allocator 등)을 두고 있으며, 개발자는 주로 할당 패턴을 이해하고 적절한 전략(풀링, 프레임 메모리 등)을 선택해야 한다.

### 2. 핵심 개념

게임 엔진은 할당 빈도가 높고 수명이 짧은 객체가 많다. 따라서 범용 malloc만으로는 파편화와 성능 문제를 제어하기 어렵다. UE는 Binned allocator 등 자체 전략을 사용한다.

#### 이 개념을 이해하려면

- 카테고리 1 토픽 25

#### 이 개념이 영향을 주는 것

- 토픽 6: UE5 GC 심화

### 3. 내부 동작 / 메커니즘

작은 블록은 크기 구간별로 관리(Binned)하여 할당 속도를 높인다. 반복 생성·파괴가 많은 객체는 풀링이나 일괄 해제 전략이 효과적이다.

#### 핵심 규칙

- 짧고 반복적인 객체 수명에는 풀링이나 프레임 메모리 전략을 우선 검토한다.
- 할당과 해제 경로는 반드시 일치시켜야 한다.

### 4. 자주 틀리는 포인트

- 일반 게임플레이 코드에 저수준 할당을 불필요하게 도입한다.
- 매 프레임 필요한 객체를 새로 생성한다.
- `new`/`delete`와 `FMemory::Malloc`/`Free`를 섞어 사용한다.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

**우선 검토하는 경우**

- 프레임 내 임시 데이터
- 반복 생성·파괴가 많은 객체 (투사체, 파티클 등)

**주의하거나 분리할 경우**

- 일반 로직에서 저수준 할당 직접 사용
- 병목 분석 없이 할당자부터 변경

### 6. 코드 예시 (bad → good)

**Bad: 반복 new/delete**

```cpp
for (int32 i = 0; i < 5000; ++i) {
    FMyTemp* Item = new FMyTemp();
    Use(Item);
    delete Item;
}
```

**Good: 풀링 사용**

```cpp
FMyTemp* Item = TempPool.Acquire();
Use(Item);
TempPool.Release(Item);
```

**Bad: 할당/해제 경로 불일치**

```cpp
void* Raw = FMemory::Malloc(256);
delete[] Raw;
```

**Good: 일관된 경로**

```cpp
void* Raw = FMemory::Malloc(256);
FMemory::Free(Raw);
```

### 7. 판단 기준 요약

- 반복 생성·파괴 패턴 → 풀링 또는 프레임 메모리 검토
- 할당과 해제는 반드시 같은 API로 수행
- 일반 로직에서는 고수준 컨테이너나 UE 객체 생성 API를 우선한다.
- 성능 병목이 확인된 후에 저수준 할당자를 고려한다.
