## 토픽 15: delegate - 싱글캐스트 / 멀티캐스트

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 없음

### 1. 문제 상황

핵심 용어: `BindWeakLambda`, `FDelegateHandle`, `BindSP`, `BindUObject`.

체력 변화, 상태 전환, 로딩 완료, UI 갱신처럼 "어떤 일이 일어났음을 알리는" 흐름에서 델리게이트는 매우 자주 등장한다. 직접 호출보다 결합을 낮출 수 있어 구조는 유연해지지만, 해제 정책이 흐릿하면 그 유연성이 곧 크래시와 중복 호출로 돌아온다. 그래서 UE에서 델리게이트를 잘 쓴다는 말은 바인딩보다 언바인드 시점을 먼저 설명할 수 있다는 뜻에 가깝다.

`delegate(델리게이트, 콜백 객체)`는 "나중에 호출할 함수 하나"를 담는 도구처럼 보이지만, UE에서는 호출 대상의 수명과 바인딩 정책까지 함께 표현하는 연결 장치에 가깝다. 싱글캐스트는 하나의 응답자에게, 멀티캐스트는 여러 구독자에게 알리는 구조라는 점에서 함수 직접 호출과 출발점이 다르다.

### 2. 핵심 개념

#### 정의

`delegate(델리게이트, 콜백 객체)`는 "나중에 호출할 함수 하나"를 담는 도구처럼 보이지만, UE에서는 호출 대상의 수명과 바인딩 정책까지 함께 표현하는 연결 장치에 가깝다. 싱글캐스트는 하나의 응답자에게, 멀티캐스트는 여러 구독자에게 알리는 구조라는 점에서 함수 직접 호출과 출발점이 다르다.

실무 기준은 호출 횟수보다 수명 경계를 먼저 보는 것이다. `BindRaw`, `BindSP`, `BindUObject`, `BindWeakLambda`가 나뉘는 이유도 누가 소유하고 언제 사라질지가 다르기 때문이다. 그래서 델리게이트는 "어떻게 호출하나"보다 "누가 언제 해제하나"까지 포함해 설계해야 안전하다.

### 3. 내부 동작 / 메커니즘

#### 동작 관점

실무 기준은 호출 횟수보다 수명 경계를 먼저 보는 것이다. `BindRaw`, `BindSP`, `BindUObject`, `BindWeakLambda`가 나뉘는 이유도 누가 소유하고 언제 사라질지가 다르기 때문이다. 그래서 델리게이트는 "어떻게 호출하나"보다 "누가 언제 해제하나"까지 포함해 설계해야 안전하다.

#### 핵심 규칙

- 델리게이트는 바인딩보다 해제 정책을 먼저 정한다.

### 4. 자주 틀리는 포인트

- `BindRaw`를 남용해 수명 보장이 없는 객체를 묶는다.
- `FDelegateHandle`을 저장하지 않아 선택적 해제가 어렵다.
- 수명 보장 없는 객체에 BindRaw를 남용하면 호출 시점 크래시가 생기기 쉽다.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

#### 우선 검토하는 경우

- 계약이 필요한 직접 호출은 인터페이스가 더 맞고, 알림 브로드캐스트는 델리게이트가 더 맞다.
- raw 수명은 BindRaw, shared pointer 수명은 BindSP, UObject 수명은 BindUObject가 더 자연스럽다.
- 체력 변화, 상태 전환, 로딩 완료, UI 갱신처럼 "어떤 일이 일어났음을 알리는" 흐름에서 델리게이트는 매우 자주 등장한다. 직접 호출보다 결합을 낮출 수 있어 구조는 유연해지지만, 해제 정책이 흐릿하면 그 유연성이 곧 크래시와 중복 호출로 돌아온다. 그래서 UE에서 델리게이트를 잘 쓴다는 말은 바인딩보다 언바인드 시점을 먼저 설명할 수 있다는 뜻에 가깝다.
- 델리게이트는 바인딩보다 해제 정책을 먼저 정한다.

#### 주의하거나 분리할 경우

- `BindRaw`를 남용해 수명 보장이 없는 객체를 묶는다.
- `FDelegateHandle`을 저장하지 않아 선택적 해제가 어렵다.
- 수명 보장 없는 객체에 BindRaw를 남용하면 호출 시점 크래시가 생기기 쉽다.

### 6. 대안 비교

- 계약이 필요한 직접 호출은 인터페이스가 더 맞고, 알림 브로드캐스트는 델리게이트가 더 맞다.
- raw 수명은 BindRaw, shared pointer 수명은 BindSP, UObject 수명은 BindUObject가 더 자연스럽다.

### 7. 코드 예시 (bad → good)

#### Bad: 이 바인딩이 왜 위험할 수 있는가

```cpp
OnHealthChanged.AddRaw(this, &SMyWidget::HandleHealthChanged);
```

위젯 수명이 델리게이트 발행자보다 짧으면 해제 누락 시 죽은 객체를 호출할 수 있다. BindRaw는 수명 보장을 직접 책임져야 한다.

#### Good: 약한 바인딩이나 명시적 해제 핸들 관리를 사용한다.

```cpp
Handle = OnHealthChanged.AddWeakLambda(this, [this](float NewValue)
{
    HandleHealthChanged(NewValue);
});
```

약한 바인딩이나 명시적 해제 핸들 관리를 사용한다.

델리게이트는 바인딩보다 해제 정책을 먼저 정한다.

#### Bad: 이 바인딩이 왜 위험한가

```cpp
UMyWidget* Widget = GetWidget();
OnChanged.BindRaw(Widget, &UMyWidget::Refresh);
```

위젯은 UObject 수명 모델을 따르는데 `BindRaw`는 그런 수명 정보를 반영하지 못한다. 파괴 뒤 호출되면 댕글링 호출이 될 수 있다.

#### Good: UObject 멤버 함수는 `BindUObject`나 필요 시 weak 바인딩을 쓴다.

```cpp
UMyWidget* Widget = GetWidget();
OnChanged.BindUObject(Widget, &UMyWidget::Refresh);
```

UObject 멤버 함수는 `BindUObject`나 필요 시 weak 바인딩을 쓴다.

### 8. 판단 기준 요약

- 델리게이트는 바인딩보다 해제 정책을 먼저 정한다.
- 약한 바인딩이나 명시적 해제 핸들 관리를 사용한다.
- UObject 멤버 함수는 `BindUObject`나 필요 시 weak 바인딩을 쓴다.
- 개별 해제가 필요하면 `FDelegateHandle`을 저장하는 패턴을 쓴다.
- 수명 보장이 약하면 `BindWeakLambda` 같은 약한 연결을 고려한다.
- 계약이 필요한 직접 호출은 인터페이스가 더 맞고, 알림 브로드캐스트는 델리게이트가 더 맞다.

## 토픽 16: dynamic delegate

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 15

### 1. 문제 상황

핵심 용어: `BlueprintAssignable`, `UFUNCTION`, `DECLARE_DYNAMIC_MULTICAST_DELEGATE`.

블루프린트에서 UI나 레벨 스크립트가 이벤트를 받도록 하려면 다이나믹 델리게이트가 거의 기본 통로가 된다. 반대로 순수 C++ 핫패스에서는 일반 델리게이트가 더 가볍고 제약도 적다. 결국 이 토픽의 핵심은 "이 이벤트를 엔진 콘텐츠 계층까지 열어야 하는가"를 판단하는 데 있다.

`dynamic delegate(다이나믹 델리게이트, 리플렉션 기반 델리게이트)`는 일반 델리게이트에 리플렉션 계층을 얹어, 블루프린트와 직렬화 경로가 이해할 수 있게 만든 형태다. 콜백이라는 본질은 같지만 호출 성능보다 엔진 통합성을 우선하는 선택지라는 점이 핵심이다.

### 2. 핵심 개념

#### 정의

`dynamic delegate(다이나믹 델리게이트, 리플렉션 기반 델리게이트)`는 일반 델리게이트에 리플렉션 계층을 얹어, 블루프린트와 직렬화 경로가 이해할 수 있게 만든 형태다. 콜백이라는 본질은 같지만 호출 성능보다 엔진 통합성을 우선하는 선택지라는 점이 핵심이다.

실무 판단은 비교적 분명하다. C++ 내부에서만 빠르게 도는 이벤트면 일반 델리게이트가 단순하고, 블루프린트나 에디터 콘텐츠가 받아야 하는 이벤트면 다이나믹 델리게이트가 자연스럽다. `UFUNCTION` 제약과 오버헤드를 감수하는 대신, 협업 경계가 매끄러워지는 셈이다.

- 블루프린트 연동이 필요 없는데도 습관적으로 다이나믹 델리게이트를 쓴다.

### 6. 대안 비교

- 일반 delegate는 가볍고 C++ 중심이며, dynamic delegate는 엔진 통합성이 강하다.
- 핫패스 C++ 콜백은 일반 delegate, BP 노출과 리플렉션 연동은 dynamic delegate가 더 맞다.

### 7. 코드 예시 (bad → good)

#### Bad: 이 바인딩 코드가 실패할 가능성이 높은 이유는

```cpp
DECLARE_DYNAMIC_MULTICAST_DELEGATE(FOnDone);

UCLASS()
class UMyObj : public UObject
{
    GENERATED_BODY()

public:
    FOnDone OnDone;

    void HandleDone();
};
```

dynamic delegate 대상 함수가 리플렉션에 노출되지 않았다. 블루프린트/이름 기반 바인딩을 기대한다면 UFUNCTION이 필요하다.

#### Good: 대상 함수를 UFUNCTION으로 선언하고 필요하면 BlueprintAssignable로 노출한다.

```cpp
UFUNCTION()
void HandleDone();
```

대상 함수를 UFUNCTION으로 선언하고 필요하면 BlueprintAssignable로 노출한다.

dynamic delegate는 리플렉션 규칙을 함께 만족해야 한다.

#### Bad: 이 바인딩이 왜 실패할 수 있는가

```cpp
DECLARE_DYNAMIC_DELEGATE(FOnDone);

UCLASS()
class UMyObj : public UObject
{
    GENERATED_BODY()

public:
    void HandleDone();
};

void Setup(UMyObj* Obj, FOnDone& Delegate)
{
    Delegate.BindDynamic(Obj, &UMyObj::HandleDone);
}
```

dynamic delegate가 함수 메타데이터를 찾을 수 있도록 보통 `UFUNCTION`이 필요하다.

#### Good: 동적 바인딩 대상 함수에 `UFUNCTION`을 붙인다.

```cpp
DECLARE_DYNAMIC_DELEGATE(FOnDone);

UCLASS()
class UMyObj : public UObject
{
    GENERATED_BODY()

public:
    UFUNCTION()
    void HandleDone();
};
```

동적 바인딩 대상 함수에 `UFUNCTION`을 붙인다.

- dynamic delegate는 리플렉션 규칙을 함께 만족해야 한다.
- 대상 함수를 UFUNCTION으로 선언하고 필요하면 BlueprintAssignable로 노출한다.
- 동적 바인딩 대상 함수에 `UFUNCTION`을 붙인다.
- 핫패스 내부 알림은 일반 delegate로 분리하고, BP 노출이 필요한 경로만 dynamic으로 둔다.
- BP 노출 목적이면 dynamic multicast 선언으로 바꾼다.
- 일반 delegate는 가볍고 C++ 중심이며, dynamic delegate는 엔진 통합성이 강하다.

## 토픽 17: 이벤트 디스패처

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 16

### 1. 문제 상황

핵심 용어: `broadcast`, `binding`, `RemoveAll`, `AddDynamic`.

블루프린트 기반 UI와 게임플레이 연결은 이벤트 디스패처가 가장 많이 쓰이는 구간 중 하나다. 상태를 매 프레임 읽어 오는 방식보다 훨씬 깔끔하지만, 남용하면 "어디서 누가 반응하는지"를 찾는 비용이 커진다. 그래서 좋은 이벤트 디스패처 설계는 발행 지점보다 구독 범위와 해제 타이밍이 먼저 보이는 구조다.

이벤트 디스패처는 블루프린트 사용자가 다이나믹 멀티캐스트 델리게이트를 더 직관적으로 다루도록 노출한 표현이다. 본질적으로는 "이 객체에서 이런 사건이 발생하면 외부가 구독해 반응한다"는 공개 이벤트 포인트라고 이해하면 된다.

### 2. 핵심 개념

#### 정의

이벤트 디스패처는 블루프린트 사용자가 다이나믹 멀티캐스트 델리게이트를 더 직관적으로 다루도록 노출한 표현이다. 본질적으로는 "이 객체에서 이런 사건이 발생하면 외부가 구독해 반응한다"는 공개 이벤트 포인트라고 이해하면 된다.

장점은 폴링을 줄이고 객체 간 결합을 낮춘다는 데 있지만, 대신 제어 흐름이 코드에서 덜 보인다는 비용이 생긴다. 그래서 실무에서는 누가 언제 바인딩하고 언제 해제하는지까지 함께 설계하지 않으면, 구조는 느슨해졌는데 추적은 더 어려워지는 상황이 생긴다.

### 3. 내부 동작 / 메커니즘

#### 동작 관점

장점은 폴링을 줄이고 객체 간 결합을 낮춘다는 데 있지만, 대신 제어 흐름이 코드에서 덜 보인다는 비용이 생긴다. 그래서 실무에서는 누가 언제 바인딩하고 언제 해제하는지까지 함께 설계하지 않으면, 구조는 느슨해졌는데 추적은 더 어려워지는 상황이 생긴다.

#### 핵심 규칙

- 디스패처 바인딩은 보통 해제 지점까지 한 쌍으로 설계한다.

### 4. 자주 틀리는 포인트

- 구독 해제 없이 Broadcast만 설계한다.
- 디버그 가능한 직접 호출보다 이벤트를 습관적으로 우선한다.
- Construct에서 바인딩만 하고 해제를 잊으면 위젯 재생성/파괴 주기에서 중복 호출과 누수성 버그가 생길 수 있다.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

#### 우선 검토하는 경우

- 인터페이스는 구현 계약을 강제하고, 디스패처는 알림 통로를 제공한다.
- 느슨한 broadcast는 event dispatcher, 고정 계약은 interface가 더 맞다.
- 블루프린트 기반 UI와 게임플레이 연결은 이벤트 디스패처가 가장 많이 쓰이는 구간 중 하나다. 상태를 매 프레임 읽어 오는 방식보다 훨씬 깔끔하지만, 남용하면 "어디서 누가 반응하는지"를 찾는 비용이 커진다. 그래서 좋은 이벤트 디스패처 설계는 발행 지점보다 구독 범위와 해제 타이밍이 먼저 보이는 구조다.
- 디스패처 바인딩은 보통 해제 지점까지 한 쌍으로 설계한다.

#### 주의하거나 분리할 경우

- 구독 해제 없이 Broadcast만 설계한다.
- 디버그 가능한 직접 호출보다 이벤트를 습관적으로 우선한다.
- Construct에서 바인딩만 하고 해제를 잊으면 위젯 재생성/파괴 주기에서 중복 호출과 누수성 버그가 생길 수 있다.

### 7. 코드 예시 (bad → good)

#### Bad: 이 위젯 코드에서 수명 관리상 놓치기 쉬운 부분은

```cpp
void UMyWidget::NativeConstruct()
{
    Super::NativeConstruct();
    Source->OnHealthChanged.AddDynamic(this, &UMyWidget::HandleHealthChanged);
}
```

바인딩은 했지만 NativeDestruct 등에서 해제하지 않으면 위젯 재구성이나 수명 종료 뒤에도 호출 경로가 남을 수 있다.

#### Good: 해제 시점을 명시적으로 추가한다.

```cpp
void UMyWidget::NativeDestruct()
{
    if (Source)
    {
        Source->OnHealthChanged.RemoveDynamic(this, &UMyWidget::HandleHealthChanged);
    }
    Super::NativeDestruct();
}
```

해제 시점을 명시적으로 추가한다.

디스패처 바인딩은 보통 해제 지점까지 한 쌍으로 설계한다.

#### Bad: 이 위젯 코드는 왜 수명 관리상 취약한가

```cpp
void UMyWidget::NativeConstruct()
{
    Super::NativeConstruct();
    SourceActor->OnChanged.AddDynamic(this, &UMyWidget::Refresh);
}
```

구독은 추가하지만 위젯이 사라질 때 해제하지 않으면 중복 바인딩이나 파괴 뒤 호출 문제가 남는다.

#### Good: 바인딩한 쪽에서 해제 지점도 같이 만든다.

```cpp
void UMyWidget::NativeConstruct()
{
    Super::NativeConstruct();
    SourceActor->OnChanged.AddDynamic(this, &UMyWidget::Refresh);
}

void UMyWidget::NativeDestruct()
{
    if (SourceActor)
    {
        SourceActor->OnChanged.RemoveAll(this);
    }
    Super::NativeDestruct();
}
```

바인딩한 쪽에서 해제 지점도 같이 만든다.

### 8. 판단 기준 요약

- 디스패처 바인딩은 보통 해제 지점까지 한 쌍으로 설계한다.
- 해제 시점을 명시적으로 추가한다.
- 바인딩한 쪽에서 해제 지점도 같이 만든다.
- 등록 시점을 한 번으로 제한하거나, 재등록 전에 기존 바인딩을 정리한다.
- 디스패처를 쓰기로 했다면 수신자는 구독자로만 두고 직접 호출을 섞지 않는다.
- 인터페이스는 구현 계약을 강제하고, 디스패처는 알림 통로를 제공한다.
