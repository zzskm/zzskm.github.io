## 토픽 15: delegate(델리게이트, 콜백 객체) - 싱글캐스트 / 멀티캐스트

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 없음

### 1. 핵심 개념

`delegate(델리게이트, 콜백 객체)`는 "나중에 호출할 함수를 안전한 형식으로 보관하는 콜백 시스템"이다. 싱글캐스트는 하나의 대상만 바인딩하고, 멀티캐스트는 여러 대상을 브로드캐스트한다. 핵심은 함수 포인터보다 수명과 바인딩 정책을 더 명시적으로 다룬다는 점이다.

특히 UE에서는 `BindRaw`, `BindSP`, `BindUObject`, `BindWeakLambda`처럼 소유 대상 수명에 따라 바인딩 방식을 고른다. 또한 해제 관리를 위해 `FDelegateHandle`을 저장하는 패턴이 중요하다.

### 2. 코드 예시

```cpp
DECLARE_MULTICAST_DELEGATE_OneParam(FOnHealthChanged, float);

FOnHealthChanged OnHealthChanged;
FDelegateHandle Handle;

Handle = OnHealthChanged.AddLambda([](float NewValue)
{
    UE_LOG(LogTemp, Log, TEXT("Health: %f"), NewValue);
});
```

### 3. 왜 중요한가 (게임/UE5 맥락)

체력 변화, 상태 전환, 로딩 완료, UI 갱신처럼 "어떤 일이 일어났음을 알리는" 흐름에서 델리게이트는 매우 자주 등장한다. 직접 호출보다 결합을 낮출 수 있지만, 수명 관리 실수가 곧 크래시로 이어지기도 한다.

### 4. 면접 예상 Q&A

**Q1: 싱글캐스트와 멀티캐스트는 어떻게 구분하나요?**
> A: 호출 대상이 하나면 싱글캐스트, 여러 청자를 동시에 알릴 필요가 있으면 멀티캐스트가 적절하다. 이벤트 브로드캐스트는 대개 멀티캐스트에 가깝다.

**Q2: `BindRaw`, `BindSP`, `BindUObject`는 왜 나뉘나요?**
> A: 바인딩 대상의 수명 모델이 다르기 때문이다. raw 포인터, shared pointer, UObject는 안전하게 추적하는 방식이 다르므로 같은 바인딩으로 처리할 수 없다.

**Q3: `FDelegateHandle`은 왜 저장하나요?**
> A: 나중에 특정 바인딩을 정확히 해제하기 위해서다. 멀티캐스트 델리게이트에서는 특히 개별 구독 해제가 중요하다.

**Q4: `BindWeakLambda`는 어떤 상황에서 유용한가요?**
> A: 람다를 쓰고 싶지만 소유 객체 수명이 보장되지 않을 때 유용하다. weak 바인딩으로 해제 누락 위험을 줄일 수 있다.

**Q5: 델리게이트와 인터페이스는 어떻게 선택하나요?**
> A: 이벤트 알림이면 델리게이트, 호출 대상이 특정 행동 계약을 구현해야 하면 인터페이스가 더 맞다. 결론적으로 이벤트냐 계약이냐를 구분한다.

**Q6: 실무에서 가장 위험한 델리게이트 실수는 무엇인가요?**
> A: 수명이 끝난 객체가 계속 바인딩된 상태로 남는 것이다. 바인딩보다 해제 정책을 먼저 정하는 습관이 중요하다.

### 5. 흔한 실수

- `BindRaw`를 남용해 수명 보장이 없는 객체를 묶는다.
- `FDelegateHandle`을 저장하지 않아 선택적 해제가 어렵다.
- 멀티캐스트 브로드캐스트가 호출 순서에 의존하도록 설계한다.

## 토픽 16: dynamic delegate(다이나믹 델리게이트, 리플렉션 기반 델리게이트)

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 15

### 1. 핵심 개념

`dynamic delegate(다이나믹 델리게이트, 리플렉션 기반 델리게이트)`는 리플렉션 시스템과 연결된 델리게이트다. 블루프린트 바인딩, 직렬화 가능성, 이름 기반 호출과 연동된다는 장점이 있지만 일반 델리게이트보다 오버헤드가 있다. 즉 성능보다 엔진 통합성이 우선인 선택지다.

리플렉션 기반이므로 대상 함수는 보통 `UFUNCTION`이어야 한다. 그래서 일반 C++ 이벤트와 블루프린트 이벤트를 이어 주는 다리 역할을 많이 한다.

### 2. 코드 예시

```cpp
DECLARE_DYNAMIC_MULTICAST_DELEGATE_OneParam(FOnPickupChanged, int32, NewCount);

UPROPERTY(BlueprintAssignable)
FOnPickupChanged OnPickupChanged;
```

### 3. 왜 중요한가 (게임/UE5 맥락)

블루프린트에서 UI나 레벨 스크립트가 이벤트를 받도록 하려면 다이나믹 델리게이트가 자주 필요하다. 반대로 순수 C++ 핫패스에서는 일반 델리게이트가 더 적절한 경우가 많다.

### 4. 면접 예상 Q&A

**Q1: 다이나믹 델리게이트와 일반 델리게이트의 차이는 무엇인가요?**
> A: 다이나믹 델리게이트는 리플렉션 기반으로 블루프린트와 직렬화 친화적이지만 비용이 더 크다. 일반 델리게이트는 더 가볍고 C++ 중심이다.

**Q2: 왜 `UFUNCTION`이 필요한가요?**
> A: 리플렉션 시스템이 대상 함수를 찾아야 하기 때문이다. 이름과 메타데이터를 통해 호출 가능한 함수여야 블루프린트와 연결된다.

**Q3: 다이나믹 델리게이트를 모든 곳에 쓰면 안 되는 이유는 무엇인가요?**
> A: 리플렉션 오버헤드와 제약이 있기 때문이다. 순수 C++ 내부 이벤트는 일반 델리게이트가 더 단순하고 빠르다.

**Q4: `BlueprintAssignable`은 어떤 의미인가요?**
> A: 블루프린트에서 해당 델리게이트에 이벤트를 바인딩할 수 있게 한다는 의미다. 협업 지점으로 자주 쓰인다.

**Q5: 다이나믹 멀티캐스트는 어떤 상황에 특히 잘 맞나요?**
> A: UI, 레벨 블루프린트, 디자이너 스크립트가 같은 이벤트를 구독해야 하는 상황이다. 여러 청자가 콘텐츠 계층에 퍼져 있을 때 유용하다.

**Q6: 실무에서 선택 기준을 한 줄로 말하면 무엇인가요?**
> A: C++ 내부 고성능 이벤트면 일반 델리게이트, 블루프린트와 연결돼야 하면 다이나믹 델리게이트다. 성능보다 통합성이 필요한지 먼저 본다.

### 5. 흔한 실수

- 블루프린트 연동이 필요 없는데도 습관적으로 다이나믹 델리게이트를 쓴다.
- 대상 함수를 `UFUNCTION`으로 선언하지 않는다.
- 리플렉션 비용이 큰 경로에 다이나믹 이벤트를 과하게 둔다.

## 토픽 17: 이벤트 디스패처

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 16

### 1. 핵심 개념

이벤트 디스패처는 블루프린트 문맥에서 다이나믹 멀티캐스트 델리게이트를 쉽게 다루는 표현이다. 본질은 동일하지만, 블루프린트 사용자에게는 "이 객체가 외부에 알릴 수 있는 이벤트 포인트"로 보인다.

중요한 것은 바인딩만큼 해제 정책이다. `Remove`, `RemoveAll`, 위젯 파괴 시 언바인드 같은 루틴이 없으면 오래된 구독자가 남아 중복 호출이나 잘못된 참조를 만들 수 있다.

### 2. 코드 예시

```cpp
UPROPERTY(BlueprintAssignable)
FOnPickupChanged OnPickupChanged;

void APickupActor::NotifyPickupChanged(int32 NewCount)
{
    OnPickupChanged.Broadcast(NewCount);
}
```

### 3. 왜 중요한가 (게임/UE5 맥락)

블루프린트 기반 UI와 게임플레이 연결은 이벤트 디스패처를 통해 많이 이뤄진다. 폴링 없이 상태 변화를 전달할 수 있어 구조가 깔끔해지지만, 수명 관리가 부실하면 오히려 디버깅이 어려워진다.

### 4. 면접 예상 Q&A

**Q1: 이벤트 디스패처와 함수 직접 호출의 차이는 무엇인가요?**
> A: 직접 호출은 호출 대상이 고정돼 있지만, 이벤트 디스패처는 누가 구독할지 열려 있다. 결론적으로 결합도를 낮추고 다수 청자를 지원한다.

**Q2: 블루프린트 UI에서 이벤트 디스패처가 유용한 이유는 무엇인가요?**
> A: 상태 변화가 있을 때만 반응하도록 만들 수 있기 때문이다. 매 프레임 값 바인딩보다 의도가 명확하고 비용도 통제하기 쉽다.

**Q3: `RemoveAll(this)` 같은 해제 패턴은 왜 중요하나요?**
> A: 위젯이나 액터가 사라진 뒤에도 바인딩이 남으면 중복 호출이나 invalid 참조가 발생할 수 있기 때문이다. 바인딩 지점과 해제 지점을 항상 쌍으로 본다.

**Q4: 이벤트 디스패처가 설계를 나쁘게 만들 수 있는 경우는 언제인가요?**
> A: 너무 많은 이벤트를 무분별하게 뿌려서 흐름 추적이 어려워질 때다. 이벤트는 느슨한 연결이지만, 남용하면 제어 흐름이 숨는다.

**Q5: 이벤트 디스패처와 `GameplayMessageSubsystem`은 어떻게 다른가요?**
> A: 이벤트 디스패처는 객체 중심 이벤트에 가깝고, `GameplayMessageSubsystem`은 시스템 간 브로드캐스트 허브에 가깝다. 범위와 결합 방식이 다르다.

**Q6: 실무에서 이벤트 구조가 복잡하다는 신호는 무엇인가요?**
> A: 누가 언제 바인딩하고 해제하는지 설명하기 어려울 때다. 이벤트 수보다 수명 경계가 더 중요하다.

### 5. 흔한 실수

- 구독 해제 없이 Broadcast만 설계한다.
- 디버그 가능한 직접 호출보다 이벤트를 습관적으로 우선한다.
- 하나의 상태 변화에 이벤트를 중복 발행해 UI가 여러 번 갱신된다.
