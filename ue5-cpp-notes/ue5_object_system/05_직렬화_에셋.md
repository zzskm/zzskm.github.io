## 토픽 18: serialization

> 난이도: ★★★ | 예상 시간: 1시간 | 선행: 토픽 1

### 1. 문제 상황

핵심 용어: `FArchive`, `custom versioning`, `StructuredArchive`, `NetSerialize`.

세이브 데이터, 에셋, 복제, 에디터 저장은 전부 직렬화에 닿아 있다. 직렬화 원리를 이해하면 버전 호환, 데이터 손상, 네트워크 전송 비용 문제를 하나의 관점으로 묶어 설명할 수 있다. 결국 이 토픽은 "저장한다"를 넘어서 어떤 상태를 영속화할 가치가 있는지 판단하는 기준과 연결된다.

`serialization(시리얼라이제이션, 직렬화)`은 객체 상태를 저장하거나 전송 가능한 형태로 읽고 쓰는 과정이다. UE5에서는 `FArchive(에프아카이브, 기본 직렬화 스트림)`, `StructuredArchive(스트럭처드 아카이브, 구조화 직렬화 스트림)`, 프로퍼티 직렬화, `NetSerialize(넷시리얼라이즈, 네트워크 직렬화)`, `custom versioning(커스텀 버저닝, 사용자 정의 버전 관리)` 같은 개념이 함께 등장한다.

### 2. 핵심 개념

#### 정의

`serialization(시리얼라이제이션, 직렬화)`은 객체 상태를 저장하거나 전송 가능한 형태로 읽고 쓰는 과정이다. UE5에서는 `FArchive(에프아카이브, 기본 직렬화 스트림)`, `StructuredArchive(스트럭처드 아카이브, 구조화 직렬화 스트림)`, 프로퍼티 직렬화, `NetSerialize(넷시리얼라이즈, 네트워크 직렬화)`, `custom versioning(커스텀 버저닝, 사용자 정의 버전 관리)` 같은 개념이 함께 등장한다.

핵심 판단은 "무엇을 누구를 위해 어떤 수명으로 보존하는가"다. 디스크 저장, 에셋 저장, 네트워크 전송은 모두 직렬화지만 요구 조건이 다르다. 네트워크용은 효율과 대역폭이 중요하고, 에셋 저장은 버전 업그레이드와 편집 툴 호환성이 더 중요하다.

### 4. 자주 틀리는 포인트

- 캐시와 영속 데이터를 구분하지 않고 전부 저장한다.
- 버전 업그레이드 경로 없이 포맷을 변경한다.
- 필드 순서 변경과 타입 변경을 버전 처리 없이 밀어 넣으면 과거 데이터가 깨진다.

### 6. 대안 비교

- 디스크 저장은 안정성과 호환성이 중요하고, 네트워크 직렬화는 크기와 속도가 더 중요하다.

### 7. 코드 예시 (bad → good)

#### Bad: 이 직렬화 변경이 왜 위험한가

```cpp
void Serialize(FArchive& Ar)
{
    Ar << Coins;
    Ar << Level;
    Ar << Name;
}
// 이후 순서를 Name, Coins, Level로 변경
```

저장 순서를 바꾸면 이전 버전 데이터 해석이 달라질 수 있다. 버전 처리 없이 구조를 바꾸면 로드 호환성이 깨진다.

#### Good: 버전 분기를 두거나 구조화된 아카이브와 명시적 버전 값을 사용한다.

```cpp
// 개념 예시
if (Version < 2) { Ar << Coins; Ar << Level; Ar << Name; }
else { Ar << Name; Ar << Coins; Ar << Level; }
```

버전 분기를 두거나 구조화된 아카이브와 명시적 버전 값을 사용한다.

직렬화 포맷 변경은 버전 관리와 함께 진행한다.

#### Bad: 이 변경이 왜 위험한가

```cpp
void FMySaveData::Serialize(FArchive& Ar)
{
    Ar << Level;
    Ar << Coins;
    Ar << Nickname; // 뒤늦게 추가
}
```

이미 배포된 저장 포맷에 필드를 추가하면서 버전 관리가 없으면 이전 데이터와 읽기 계약이 어긋날 수 있다.

#### Good: 버전 정보를 두고, 버전별로 읽기 경로를 분기한다.

```cpp
void FMySaveData::Serialize(FArchive& Ar)
{
    Ar << Version;
    Ar << Level;
    Ar << Coins;
    if (Version >= 2)
    {
        Ar << Nickname;
    }
}
```

버전 정보를 두고, 버전별로 읽기 경로를 분기한다.

### 8. 판단 기준 요약

- 직렬화 포맷 변경은 버전 관리와 함께 진행한다.
- 버전 분기를 두거나 구조화된 아카이브와 명시적 버전 값을 사용한다.
- 버전 정보를 두고, 버전별로 읽기 경로를 분기한다.
- 영속적으로 복원해야 하는 상태만 골라서 직렬화한다.
- 복제에 꼭 필요한 필드만 `NetSerialize`에 포함한다.
- 기존 순서를 유지하거나 버전 분기에서 새 순서를 별도로 처리한다.

## 토픽 19: UPackage & 에셋 시스템

> 난이도: ★★★ | 예상 시간: 1시간 | 선행: 토픽 18

### 1. 문제 상황

핵심 용어: `UPackage`, `redirector`, `cook`.

프로젝트가 커질수록 에셋 경로, 참조, 패키징 문제는 빌드 안정성과 직결된다. "에셋을 옮겼는데 왜 참조가 남지?", "에디터에서는 되는데 패키징 후 실패하지?" 같은 질문은 결국 패키지와 cook 규칙을 이해하고 있느냐의 문제다. 이 구조를 알면 에셋 문제를 단순 경로 오타가 아니라 자산 그래프 설계 문제로 볼 수 있다.

UE 에셋은 보통 `UPackage` 안에 저장되고, 경로와 이름을 통해 식별된다. 중요한 것은 에셋을 단순 파일이 아니라 패키지, 오브젝트, 참조 관계를 가진 엔진 자산으로 봐야 한다는 점이다. 그래서 로드, 리디렉터, 쿠킹까지 함께 이해해야 비로소 에셋 시스템 전체가 보인다.

### 6. 코드 예시 (bad → good)

#### Bad: 이 접근이 에셋 이동 뒤 취약한 이유는

```cpp
const TCHAR* Path = TEXT("/Game/Weapons/BP_Sword.BP_Sword_C");
```

경로 문자열을 하드코딩하면 에셋 이동/이름 변경 뒤 깨지기 쉽다. 리디렉터 청소나 자산 관리 체계와도 잘 맞지 않는다.

```cpp
UPROPERTY(EditAnywhere)
```

```cpp
UObject* Obj = StaticLoadObject(

    TEXT("/Game/Data/OldPath/DA_Weapon.DA_Weapon")
```

## 토픽 20: Config 시스템

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 18

### 1. 문제 상황

핵심 용어: `DefaultConfig`, `Config`, `DeveloperSettings`, `GlobalConfig`.

플랫폼별 옵션, 디버그 플래그, 서비스 주소, 기본 밸런스 값은 빌드나 실행 환경마다 달라질 수 있다. Config를 이해하면 코드 수정 없이 정책을 조정할 수 있고, 환경별 차이를 구조적으로 관리할 수 있다. 반대로 Config와 콘텐츠 데이터를 구분하지 못하면 편집성과 추적성이 급격히 떨어진다.

`Config(콘피그, 설정) 시스템`은 `.ini` 기반 설정을 클래스와 변수에 연결하는 구조다. 코드 기본값, 프로젝트 설정, 플랫폼 설정, 사용자 설정을 계층적으로 합쳐 최종 값을 만들므로, 단순 파일 파싱보다 "설정 계층을 어떻게 병합할 것인가"에 더 가깝다.

### 2. 핵심 개념

#### 정의

`Config(콘피그, 설정) 시스템`은 `.ini` 기반 설정을 클래스와 변수에 연결하는 구조다. 코드 기본값, 프로젝트 설정, 플랫폼 설정, 사용자 설정을 계층적으로 합쳐 최종 값을 만들므로, 단순 파일 파싱보다 "설정 계층을 어떻게 병합할 것인가"에 더 가깝다.

핵심은 하드코딩 대신 환경별 정책을 데이터로 빼되, 콘텐츠 데이터와는 분리하는 것이다. `Config`, `DefaultConfig`, `GlobalConfig` 같은 지정자를 통해 어떤 설정 파일과 연결할지 정할 수 있으므로, Config는 단순 텍스트 파일이 아니라 클래스 메타데이터와 연결된 정책 시스템이라고 보는 편이 맞다.

### 3. 내부 동작 / 메커니즘

#### 동작 관점

핵심은 하드코딩 대신 환경별 정책을 데이터로 빼되, 콘텐츠 데이터와는 분리하는 것이다. `Config`, `DefaultConfig`, `GlobalConfig` 같은 지정자를 통해 어떤 설정 파일과 연결할지 정할 수 있으므로, Config는 단순 텍스트 파일이 아니라 클래스 메타데이터와 연결된 정책 시스템이라고 보는 편이 맞다.

#### 핵심 규칙

- Config는 정책/설정, SaveGame은 플레이 상태 저장에 가깝다.

### 4. 자주 틀리는 포인트

- 사용자 저장 데이터를 Config로 처리한다.
- 설정 계층 구조를 이해하지 않고 한 파일만 수정한다.
- 플레이어 진행 데이터까지 Config에 넣으면 설정과 저장 책임이 섞여 관리가 어려워진다.

### 5. 언제 쓰는가 / 언제 쓰지 않는가

#### 우선 검토하는 경우

- Config는 설정 정책에, DataAsset은 콘텐츠 데이터에, SaveGame은 플레이어 진행 저장에 더 가깝다.
- 환경 설정은 Config, 콘텐츠 자산은 DataAsset이 더 자연스럽다.
- 플랫폼별 옵션, 디버그 플래그, 서비스 주소, 기본 밸런스 값은 빌드나 실행 환경마다 달라질 수 있다. Config를 이해하면 코드 수정 없이 정책을 조정할 수 있고, 환경별 차이를 구조적으로 관리할 수 있다. 반대로 Config와 콘텐츠 데이터를 구분하지 못하면 편집성과 추적성이 급격히 떨어진다.
- Config는 정책/설정, SaveGame은 플레이 상태 저장에 가깝다.

#### 주의하거나 분리할 경우

- 사용자 저장 데이터를 Config로 처리한다.
- 설정 계층 구조를 이해하지 않고 한 파일만 수정한다.
- 플레이어 진행 데이터까지 Config에 넣으면 설정과 저장 책임이 섞여 관리가 어려워진다.

### 6. 대안 비교

- Config는 설정 정책에, DataAsset은 콘텐츠 데이터에, SaveGame은 플레이어 진행 저장에 더 가깝다.
- 환경 설정은 Config, 콘텐츠 자산은 DataAsset이 더 자연스럽다.

### 7. 코드 예시 (bad → good)

#### Bad: 이 선택이 설정과 저장 책임을 왜 섞는가

```cpp
UCLASS(Config=Game, DefaultConfig)
class UMyPlayerSettings : public UObject
{
    GENERATED_BODY()

public:
    UPROPERTY(Config)
    int32 CurrentGold = 0;
};
```

CurrentGold는 플레이어 진행 데이터에 가까운데 Config에 두면 설정과 세이브 책임이 섞인다. 환경 정책과 진행 상태는 수명과 용도가 다르다.

#### Good: 플레이어 진행 상태는 SaveGame 또는 별도 저장 시스템으로 분리한다.

```cpp
UCLASS()
class UMySaveData : public USaveGame
{
    GENERATED_BODY()

public:
    UPROPERTY()
    int32 CurrentGold = 0;
};
```

플레이어 진행 상태는 SaveGame 또는 별도 저장 시스템으로 분리한다.

Config는 정책/설정, SaveGame은 플레이 상태 저장에 가깝다.

#### Bad: 이 설계가 왜 설정과 세이브 책임을 섞는가

```cpp
UCLASS(Config=Game, DefaultConfig)
class UMySettings : public UObject
{
    GENERATED_BODY()

public:
    UPROPERTY(Config)
    TArray<FName> InventoryItems;
};
```

플레이어 인벤토리처럼 사용자 진행 상태는 환경 설정보다 세이브 데이터에 가깝다.

#### Good: Config에는 환경 설정만 두고, 진행 상태는 별도 저장 구조로 분리한다.

Config에는 환경 설정만 두고, 진행 상태는 별도 저장 구조로 분리한다.

### 8. 판단 기준 요약

- Config는 정책/설정, SaveGame은 플레이 상태 저장에 가깝다.
- 플레이어 진행 상태는 SaveGame 또는 별도 저장 시스템으로 분리한다.
- Config에는 환경 설정만 두고, 진행 상태는 별도 저장 구조로 분리한다.
- 세션별 가변 상태는 Config가 아니라 런타임 상태나 세이브 시스템으로 분리한다.
- 콘텐츠성 데이터는 DataAsset이나 테이블 계열로 분리한다.
- Config는 설정 정책에, DataAsset은 콘텐츠 데이터에, SaveGame은 플레이어 진행 저장에 더 가깝다.
