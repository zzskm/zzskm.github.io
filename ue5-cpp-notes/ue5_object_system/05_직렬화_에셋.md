## 토픽 18: serialization(시리얼라이제이션, 직렬화)

> 난이도: ★★★ | 예상 시간: 1시간 | 선행: 토픽 1

### 1. 핵심 개념

`serialization(시리얼라이제이션, 직렬화)`은 객체 상태를 저장하거나 전송 가능한 형태로 읽고 쓰는 과정이다. UE5에서는 `FArchive(에프아카이브, 기본 직렬화 스트림)`, `StructuredArchive(스트럭처드 아카이브, 구조화 직렬화 스트림)`, 프로퍼티 직렬화, `NetSerialize(넷시리얼라이즈, 네트워크 직렬화)`, `custom versioning(커스텀 버저닝, 사용자 정의 버전 관리)` 같은 개념이 함께 등장한다. 핵심은 "무엇을 어떤 맥락에서 저장하는가"를 구분하는 것이다.

디스크 저장, 에셋 저장, 네트워크 전송은 모두 직렬화지만 요구 조건이 다르다. 네트워크용은 효율과 호환성이 중요하고, 에셋 저장은 버전 업그레이드와 편집 툴 호환성이 중요하다.

### 2. 코드 예시

```cpp
void FMySaveData::Serialize(FArchive& Ar)
{
    Ar << PlayerLevel;
    Ar << Coins;
}
```

### 3. 왜 중요한가 (게임/UE5 맥락)

세이브 데이터, 에셋, 복제, 에디터 저장은 전부 직렬화에 닿아 있다. 직렬화 원리를 이해하면 버전 호환과 데이터 손상, 네트워크 전송 비용 문제를 더 잘 설명할 수 있다.

### 4. 면접 예상 Q&A

**Q1: `FArchive`와 `StructuredArchive`는 어떻게 다른가요?**
> A: `FArchive`는 전통적인 읽기/쓰기 스트림이고, `StructuredArchive`는 더 구조화된 형태를 표현하기 좋다. 복잡한 데이터와 버전 대응에서는 `StructuredArchive`가 읽기 쉬운 경우가 많다.

**Q2: custom versioning은 왜 필요한가요?**
> A: 직렬화 포맷이 바뀌어도 오래된 데이터를 읽어야 하기 때문이다. 결론적으로 저장 형식 진화를 안전하게 관리하기 위한 장치다.

**Q3: `NetSerialize`는 일반 직렬화와 무엇이 다른가요?**
> A: 네트워크 전송을 위한 효율적 직렬화에 초점을 둔다. 대역폭과 비트 단위 최적화가 중요해 일반 저장 직렬화와 목적이 다르다.

**Q4: 모든 필드를 그대로 저장하면 안 되는 이유는 무엇인가요?**
> A: 재계산 가능한 캐시나 플랫폼 의존 상태까지 저장하면 포맷이 불안정해지고 데이터가 불필요하게 커진다. 저장 가치가 있는 상태만 선별해야 한다.

**Q5: 버전 호환 문제는 언제 터지나요?**
> A: 필드 추가, 타입 변경, 의미 변경이 일어났는데 이전 버전 데이터 처리를 안 했을 때 터진다. 세이브와 에셋은 특히 민감하다.

**Q6: 실무에서 직렬화 디버깅은 어떻게 접근하나요?**
> A: 어떤 필드가 언제 저장되고 어떤 버전으로 읽히는지 순서를 추적한다. 많은 문제는 값 자체보다 순서와 버전 분기에서 나온다.

### 5. 흔한 실수

- 캐시와 영속 데이터를 구분하지 않고 전부 저장한다.
- 버전 업그레이드 경로 없이 포맷을 변경한다.
- 네트워크 직렬화와 디스크 저장 직렬화를 같은 기준으로 생각한다.

## 토픽 19: UPackage(유패키지, 에셋 패키지) & 에셋 시스템

> 난이도: ★★★ | 예상 시간: 1시간 | 선행: 토픽 18

### 1. 핵심 개념

UE 에셋은 보통 `UPackage` 안에 저장되고, 경로와 이름을 통해 식별된다. 중요한 것은 "에셋은 단순 파일이 아니라 패키지, 오브젝트, 참조 관계를 가진 엔진 자산"이라는 점이다. 로드, 리디렉터, 쿠킹 과정까지 함께 봐야 한다.

리디렉터는 에셋 이동/이름 변경 후 참조를 연결해 주는 임시 다리이고, cook은 런타임용 데이터로 가공하는 과정이다. 즉 에셋 시스템은 저장만이 아니라 빌드 파이프라인 일부다.

### 2. 코드 예시

```cpp
FSoftObjectPath AssetPath(TEXT("/Game/Data/DA_Weapon.DA_Weapon"));
UObject* Loaded = AssetPath.TryLoad();
```

### 3. 왜 중요한가 (게임/UE5 맥락)

프로젝트가 커질수록 에셋 경로, 참조, 패키징 문제는 빌드 안정성과 직결된다. "에셋을 옮겼는데 왜 참조가 남지?", "에디터에서는 되는데 패키징 후 실패하지?" 같은 질문은 이 시스템 이해를 요구한다.

### 4. 면접 예상 Q&A

**Q1: `UPackage`는 무엇인가요?**
> A: 에셋 오브젝트를 담는 컨테이너이자 저장 단위다. 결론적으로 에셋은 파일 하나가 아니라 패키지 컨텍스트 속 오브젝트로 이해하는 편이 맞다.

**Q2: 리디렉터는 왜 생기나요?**
> A: 에셋을 이동하거나 이름을 바꿨을 때 기존 참조를 잠시 이어 주기 위해서다. 정리하지 않으면 프로젝트에 불필요한 경로 의존이 쌓인다.

**Q3: cook 단계는 어떤 역할을 하나요?**
> A: 에디터 친화적인 자산을 런타임 배포용 데이터로 가공하는 역할이다. 플랫폼별 최적화와 필요 데이터 추출이 여기서 일어난다.

**Q4: 에디터에서는 로드되는데 패키징 후 실패하는 이유는 무엇인가요?**
> A: 하드 참조가 없어서 cook 대상에서 빠졌거나, 경로 의존이 잘못됐거나, 에디터 전용 자산을 참조했기 때문이다. 런타임 자산 관리 규칙을 따로 봐야 한다.

**Q5: 소프트 참조와 에셋 시스템은 왜 자주 같이 언급되나요?**
> A: 모든 에셋을 즉시 로드하지 않고 필요 시점에 로드하려면 경로 기반 참조 체계가 중요하기 때문이다. 자산 규모가 커질수록 차이가 커진다.

**Q6: 실무에서 에셋 구조가 나쁘다는 신호는 무엇인가요?**
> A: 순환 참조, 과도한 하드 참조, 정리되지 않은 리디렉터, cook 누락 문제가 반복될 때다. 결국 자산 구조가 빌드와 메모리 문제로 번진다.

### 5. 흔한 실수

- 에셋 이동 후 리디렉터를 정리하지 않는다.
- 런타임 로드가 필요한 자산을 cook 경로에 올리지 않는다.
- 하드 참조만으로 자산 그래프를 비대하게 만든다.

## 토픽 20: Config(콘피그, 설정) 시스템

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 18

### 1. 핵심 개념

`Config(콘피그, 설정) 시스템`은 `.ini` 기반 설정을 클래스와 변수에 연결하는 구조다. 코드 기본값, 프로젝트 설정, 플랫폼 설정, 사용자 설정을 계층적으로 합쳐 최종 값을 만든다. 핵심은 "하드코딩 대신 환경별 정책을 데이터로 빼는 것"이다.

`Config`, `DefaultConfig`, `GlobalConfig` 같은 지정자를 통해 어떤 설정 파일과 연결할지 정할 수 있다. 즉 Config는 단순 텍스트 파일이 아니라 클래스 메타데이터와 연결된 설정 시스템이다.

### 2. 코드 예시

```cpp
UCLASS(Config=Game, DefaultConfig)
class UMyGameSettings : public UObject
{
    GENERATED_BODY()

public:
    UPROPERTY(Config, EditAnywhere, Category="Balance")
    float DefaultMoveSpeed = 600.0f;
};
```

### 3. 왜 중요한가 (게임/UE5 맥락)

플랫폼별 옵션, 디버그 플래그, 서비스 주소, 기본 밸런스 값은 빌드마다 달라질 수 있다. Config를 이해하면 코드 수정 없이 정책을 조정할 수 있고, 환경별 차이를 구조적으로 관리할 수 있다.

### 4. 면접 예상 Q&A

**Q1: Config 시스템을 쓰는 이유는 무엇인가요?**
> A: 코드 재빌드 없이 환경별 설정을 바꿀 수 있기 때문이다. 프로젝트 설정과 런타임 정책을 분리하는 데 유용하다.

**Q2: `Config`와 `DefaultConfig`는 어떤 역할을 하나요?**
> A: `Config`는 프로퍼티가 설정 파일과 연동됨을 뜻하고, `DefaultConfig`는 기본 설정 파일 저장 대상 클래스를 나타낸다. 즉 클래스와 필드 둘 다 연결 규칙을 가져야 한다.

**Q3: 어떤 값이 Config에 적합하고 어떤 값이 적합하지 않나요?**
> A: 환경별 정책, 플랫폼 옵션, 기본값은 적합하다. 반면 세이브 데이터처럼 사용자 진행 상태는 Config보다 별도 저장 시스템이 맞다.

**Q4: Config와 DataAsset은 어떻게 구분하나요?**
> A: 환경 설정과 전역 정책이면 Config, 콘텐츠 데이터나 디자이너 조정 자산이면 DataAsset이 더 적절하다. 결론적으로 설정과 콘텐츠를 나눠 본다.

**Q5: Config 값이 예상과 다르게 나올 때 어디를 먼저 보나요?**
> A: 어떤 `.ini` 계층이 최종값을 덮었는지부터 본다. 설정은 병합 구조이므로 파일 하나만 보면 오해하기 쉽다.

**Q6: 실무에서 Config 남용의 문제는 무엇인가요?**
> A: 콘텐츠 데이터까지 설정 파일로 몰아 넣으면 검색성과 편집성이 떨어진다. Config는 정책용이지 범용 데이터베이스가 아니다.

### 5. 흔한 실수

- 사용자 저장 데이터를 Config로 처리한다.
- 설정 계층 구조를 이해하지 않고 한 파일만 수정한다.
- 콘텐츠 데이터와 환경 설정을 구분하지 않는다.

## 토픽 24: 네트워크 리플리케이션(Network Replication, 네트워크 복제) 기초

> 난이도: ★★★ | 예상 시간: 1.5시간 | 선행: 토픽 1, 18

### 1. 핵심 개념

UE5 네트워크 리플리케이션은 서버의 게임 상태를 클라이언트에 동기화하는 시스템이다. 핵심 구성 요소는 세 가지다:

**프로퍼티 리플리케이션**: `UPROPERTY(Replicated)` 또는 `UPROPERTY(ReplicatedUsing=OnRep_함수)`로 선언한 프로퍼티를 서버에서 클라이언트로 자동 동기화한다. `GetLifetimeReplicatedProps`에서 `DOREPLIFETIME` 매크로로 리플리케이션 대상과 조건을 등록해야 한다.

**RPC(Remote Procedure Call)**: 서버와 클라이언트 간 함수 호출 방향을 지정한다.
- `Server`: 클라이언트 → 서버 호출 (입력 처리, 요청)
- `Client`: 서버 → 소유 클라이언트 호출 (결과 통보)
- `NetMulticast`: 서버 → 모든 클라이언트 호출 (이펙트, 사운드)

**역할 모델**: `ROLE_Authority`(서버, 게임 로직 결정권), `ROLE_AutonomousProxy`(소유 클라이언트, 입력 예측), `ROLE_SimulatedProxy`(비소유 클라이언트, 서버 결과 수신)로 각 액터의 네트워크 역할을 구분한다.

`AActor::bReplicates = true`를 설정해야 액터가 리플리케이션 대상이 된다.

### 2. 코드 예시

```cpp
// --- 헤더 ---
UCLASS()
class AMyCharacter : public ACharacter
{
    GENERATED_BODY()

public:
    AMyCharacter();

    // 리플리케이션 대상 프로퍼티
    UPROPERTY(ReplicatedUsing=OnRep_Health)
    float Health = 100.f;

    // 서버에서 Health가 바뀌면 클라이언트에서 호출되는 콜백
    UFUNCTION()
    void OnRep_Health();

    // 프로퍼티 리플리케이션 등록
    virtual void GetLifetimeReplicatedProps(
        TArray<FLifetimeProperty>& OutLifetimeProps) const override;

    // RPC: 클라이언트 → 서버 (입력 기반 요청)
    UFUNCTION(Server, Reliable, WithValidation)
    void ServerRequestAttack(FVector_NetQuantize TargetLocation);

    // RPC: 서버 → 모든 클라이언트 (이펙트 재생)
    UFUNCTION(NetMulticast, Unreliable)
    void MulticastPlayHitEffect(FVector_NetQuantize HitLocation);
};

// --- 구현 ---
AMyCharacter::AMyCharacter()
{
    bReplicates = true; // 리플리케이션 활성화
}

void AMyCharacter::GetLifetimeReplicatedProps(
    TArray<FLifetimeProperty>& OutLifetimeProps) const
{
    Super::GetLifetimeReplicatedProps(OutLifetimeProps);

    // Health를 모든 클라이언트에 리플리케이트
    DOREPLIFETIME(AMyCharacter, Health);

    // 조건부 리플리케이션 예시: 소유자에게만
    // DOREPLIFETIME_CONDITION(AMyCharacter, Ammo, COND_OwnerOnly);
}

void AMyCharacter::OnRep_Health()
{
    // 클라이언트에서 Health 변경 시 UI 갱신 등
    UpdateHealthBar(Health);
}

// RPC 구현 — _Implementation 접미사 필수
void AMyCharacter::ServerRequestAttack_Implementation(FVector_NetQuantize TargetLocation)
{
    // 서버에서 실행: 공격 로직 처리
    ApplyDamageAtLocation(TargetLocation);
    MulticastPlayHitEffect(TargetLocation);
}

bool AMyCharacter::ServerRequestAttack_Validate(FVector_NetQuantize TargetLocation)
{
    // 치트 방지: 서버에서 요청 유효성 검사
    return TargetLocation.Size() < MaxAttackRange;
}

void AMyCharacter::MulticastPlayHitEffect_Implementation(FVector_NetQuantize HitLocation)
{
    // 모든 클라이언트에서 실행: 시각 이펙트
    UGameplayStatics::SpawnEmitterAtLocation(GetWorld(), HitFX, HitLocation);
}
```

### 3. 왜 중요한가 (게임/UE5 맥락)

멀티플레이어 프로젝트 면접에서 리플리케이션은 독립적으로 물어보는 핵심 주제다. "서버에서 체력이 바뀌면 클라이언트에 어떻게 반영되나요?", "클라이언트 입력은 서버에 어떻게 전달되나요?", "Authority는 무엇인가요?" 같은 질문은 이 시스템 이해 없이 답하기 어렵다. 직렬화(토픽 18)의 `NetSerialize`와 연결되며, Actor 라이프사이클(토픽 22)의 네트워크 채널 초기화와도 관련된다.

### 4. 면접 예상 Q&A

**Q1: `Replicated`와 `ReplicatedUsing`은 어떻게 다른가요?**
> A: `Replicated`는 값만 동기화하고, `ReplicatedUsing`은 값이 바뀔 때 지정한 `OnRep_` 콜백을 클라이언트에서 호출한다. UI 갱신이나 부수 효과가 필요하면 `ReplicatedUsing`을 쓴다.

**Q2: `GetLifetimeReplicatedProps`는 왜 필요한가요?**
> A: 리플리케이션 대상 프로퍼티와 조건을 엔진에 등록하는 함수다. `UPROPERTY(Replicated)`만 붙여서는 부족하고, 이 함수에서 `DOREPLIFETIME`으로 등록해야 실제로 동기화된다.

**Q3: Server, Client, NetMulticast RPC는 어떻게 구분하나요?**
> A: 호출 방향으로 구분한다. `Server`는 클라이언트가 서버에 요청할 때, `Client`는 서버가 소유 클라이언트에 알릴 때, `NetMulticast`는 서버가 모든 클라이언트에 브로드캐스트할 때 쓴다.

**Q4: `Reliable`과 `Unreliable`은 언제 구분하나요?**
> A: 게임 로직에 필수적인 호출(공격 요청, 상태 변경)은 `Reliable`, 누락돼도 괜찮은 시각적 효과는 `Unreliable`이 적절하다. `Reliable`을 남용하면 대역폭 문제가 생긴다.

**Q5: Authority와 Proxy 역할은 어떻게 다른가요?**
> A: `Authority`(서버)가 게임 로직 결정권을 가지고 상태를 변경한다. `AutonomousProxy`(소유 클라이언트)는 입력 예측을 하고, `SimulatedProxy`(비소유 클라이언트)는 서버 결과를 받아 보간한다.

**Q6: 실무에서 리플리케이션 설계가 잘못됐다는 신호는 무엇인가요?**
> A: 클라이언트에서 직접 상태를 변경하거나, 모든 프로퍼티를 무조건 리플리케이트하거나, `Reliable` RPC를 매 프레임 호출하는 구조다. 대역폭과 Authority 원칙을 항상 함께 고려해야 한다.

### 5. 흔한 실수

- `bReplicates = true` 설정을 빼먹어 액터가 리플리케이트되지 않는다.
- `GetLifetimeReplicatedProps`에 프로퍼티 등록을 빠뜨린다.
- 클라이언트에서 Authority 없이 게임 상태를 직접 변경한다.
- `Reliable` RPC를 시각 효과에까지 남용해 대역폭을 소모한다.
- `OnRep_` 콜백이 서버에서는 호출되지 않는다는 점을 모르고 서버 로직을 넣는다.

## 토픽 25: GameInstance(게임인스턴스) & Subsystem(서브시스템) 패턴

> 난이도: ★★☆ | 예상 시간: 1.5시간 | 선행: 토픽 5

### 1. 핵심 개념

**`UGameInstance`**는 게임 프로세스 전체 수명 동안 유지되는 객체다. 레벨 전환(`OpenLevel`, 심리스 트래블)에도 파괴되지 않으므로 월드를 넘나드는 상태(로비 정보, 매치 설정, 세이브 슬롯 참조 등)를 보관하기에 적합하다. 반대로 레벨 종속 데이터를 `GameInstance`에 넣으면 정리 시점을 놓쳐 메모리 누수 위험이 생긴다.

**Subsystem 패턴**은 UE5에서 "싱글턴 대신 무엇을 쓰나요?"에 대한 엔진 공식 답이다. 수명 범위에 따라 네 가지 계열이 있다:

- **`UGameInstanceSubsystem`**: `GameInstance`와 같은 수명. 프로세스 전역 서비스에 적합하다.
- **`UWorldSubsystem`**: 월드(레벨)와 같은 수명. 레벨별 시스템에 적합하다.
- **`ULocalPlayerSubsystem`**: 로컬 플레이어와 같은 수명.
- **`UEngineSubsystem`**: 엔진과 같은 수명. 에디터 도구 등에 적합하다.

Subsystem의 핵심 장점은 **의존성 없이 시스템을 추가할 수 있다**는 것이다. 기존 클래스를 수정하지 않고 `UWorldSubsystem`을 상속한 클래스를 만들기만 하면 엔진이 자동으로 생성하고 수명을 관리한다. 생성자에서 `ShouldCreateSubsystem`을 오버라이드해 조건부 생성도 가능하다.

### 2. 코드 예시

```cpp
// --- GameInstance 활용: 월드 간 상태 유지 ---
UCLASS()
class UMyGameInstance : public UGameInstance
{
    GENERATED_BODY()

public:
    // 레벨 전환에도 유지되는 데이터
    UPROPERTY()
    FString CurrentPlayerName;

    UPROPERTY()
    int32 SelectedDifficulty = 1;

    virtual void Init() override
    {
        Super::Init();
        // 게임 시작 시 초기화
    }

    virtual void Shutdown() override
    {
        // 게임 종료 시 정리
        Super::Shutdown();
    }
};

// --- GameInstanceSubsystem: 프로세스 전역 서비스 ---
UCLASS()
class UOnlineSessionSubsystem : public UGameInstanceSubsystem
{
    GENERATED_BODY()

public:
    virtual void Initialize(FSubsystemCollectionBase& Collection) override
    {
        Super::Initialize(Collection);
        // 세션 매니저 초기화
    }

    virtual void Deinitialize() override
    {
        // 정리
        Super::Deinitialize();
    }

    void CreateSession(const FString& SessionName);
    void JoinSession(const FString& SessionId);
};

// --- WorldSubsystem: 레벨별 시스템 ---
UCLASS()
class UCombatLogSubsystem : public UWorldSubsystem
{
    GENERATED_BODY()

public:
    virtual bool ShouldCreateSubsystem(UObject* Outer) const override
    {
        // 에디터 프리뷰에서는 생성하지 않음
        return !IsRunningDedicatedServer() || HasAuthority();
    }

    void LogCombatEvent(const FString& EventDescription);

private:
    TArray<FString> CombatLog;
};

// --- 접근 방법 ---
void SomeFunction(UObject* WorldContextObject)
{
    // GameInstanceSubsystem 접근
    UGameInstance* GI = UGameplayStatics::GetGameInstance(WorldContextObject);
    UOnlineSessionSubsystem* SessionSys = GI->GetSubsystem<UOnlineSessionSubsystem>();
    SessionSys->CreateSession(TEXT("MyMatch"));

    // WorldSubsystem 접근
    UWorld* World = WorldContextObject->GetWorld();
    UCombatLogSubsystem* CombatLog = World->GetSubsystem<UCombatLogSubsystem>();
    CombatLog->LogCombatEvent(TEXT("Player attacked Boss"));
}
```

### 3. 왜 중요한가 (게임/UE5 맥락)

"싱글턴 대신 무엇을 쓰나요?"는 최근 UE5 면접에서 매우 자주 나오는 질문이다. 전역 매니저를 수동 싱글턴으로 만들면 초기화 순서, 수명, 테스트 가능성이 모두 문제가 된다. Subsystem은 엔진이 수명을 자동 관리하고, 범위(프로세스/월드/플레이어)를 명시적으로 구분하며, 기존 코드 수정 없이 시스템을 추가할 수 있다.

`GameInstance`는 월드 전환 시 상태 유지가 필요한 곳에서 핵심이며, Subsystem과 함께 쓰면 깔끔한 아키텍처를 만들 수 있다.

### 4. 면접 예상 Q&A

**Q1: "싱글턴 대신 무엇을 쓰나요?"**
> A: UE5 Subsystem을 쓴다. 수명 범위에 따라 `GameInstanceSubsystem`, `WorldSubsystem`, `LocalPlayerSubsystem`, `EngineSubsystem`을 선택하면 엔진이 자동으로 생성과 파괴를 관리한다. 수동 싱글턴보다 수명이 명확하고 테스트하기 쉽다.

**Q2: `GameInstance`는 언제 쓰나요?**
> A: 레벨 전환에도 유지돼야 하는 상태(로비 설정, 플레이어 선택 정보, 매치 메타데이터)를 보관할 때 쓴다. 월드가 바뀌어도 살아남는 유일한 게임플레이 객체다.

**Q3: `WorldSubsystem`과 `GameInstanceSubsystem`은 어떻게 구분하나요?**
> A: 레벨과 함께 생성/파괴돼야 하면 `WorldSubsystem`, 프로세스 전체에서 유지돼야 하면 `GameInstanceSubsystem`이다. 전투 로그는 레벨별, 세션 관리는 프로세스별이 자연스럽다.

**Q4: Subsystem의 가장 큰 장점은 무엇인가요?**
> A: 기존 클래스를 수정하지 않고 시스템을 추가할 수 있다는 점이다. 클래스를 만들기만 하면 엔진이 자동 등록하므로 의존성 주입 없이도 느슨한 결합이 가능하다.

**Q5: `GameInstance`에 너무 많은 것을 넣으면 왜 문제인가요?**
> A: 레벨과 함께 정리돼야 할 데이터까지 프로세스 수명으로 유지되면 메모리 누수와 stale 상태가 발생한다. `GameInstance`는 진짜 월드를 넘는 데이터만 넣어야 한다.

**Q6: `ShouldCreateSubsystem`은 어떤 상황에서 오버라이드하나요?**
> A: 특정 조건(데디케이티드 서버 전용, 에디터 제외, 특정 게임 모드)에서만 서브시스템을 만들고 싶을 때다. 모든 월드에 불필요한 시스템을 띄우지 않는 최적화 수단이다.

### 5. 흔한 실수

- 레벨 종속 데이터를 `GameInstance`에 넣어 정리 시점을 놓친다.
- 수동 싱글턴 패턴을 고집하면서 초기화 순서 버그에 시달린다.
- `WorldSubsystem`에 프로세스 수명 데이터를 넣어 레벨 전환 시 손실된다.
- `GetSubsystem<T>()`에서 null 체크를 빠뜨린다 (조건부 생성 시 null 가능).
- 서브시스템 간 의존 관계를 고려하지 않고 `Initialize` 순서를 가정한다.
