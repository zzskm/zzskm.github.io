## 토픽 5: Enhanced Input System(인핸스드 인풋 시스템, 입력 액션/컨텍스트 기반 입력 체계)

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 1

### 1. 핵심 개념

Enhanced Input System(인핸스드 인풋 시스템, 입력 액션/컨텍스트 기반 입력 체계)은 입력을 "키 바인딩"이 아니라 "의미 있는 액션과 컨텍스트"로 관리하는 시스템이다. `UInputAction`이 의도를 표현하고, `UInputMappingContext`가 실제 장치 바인딩을 담당한다. 핵심은 키가 아니라 플레이어가 무엇을 하려는지를 데이터로 다루는 것이다.

또한 컨텍스트 우선순위, 입력 소비(input consumption), 트리거와 모디파이어를 통해 같은 버튼도 상황에 따라 다르게 해석할 수 있다. 예를 들어 일반 상태에서는 점프, 조준 상태에서는 상호작용으로 바꾸는 식이다.

**판단 기준**
- 조작 상태가 바뀌면 매핑 컨텍스트를 전환한다.
- 입력 값 보정은 모디파이어, 발동 조건은 트리거에 둔다.
- UI와 게임플레이가 겹칠 때는 우선순위와 소비 여부를 같이 본다.

### 2. 코드 예시

```cpp
void AMyCharacter::SetupPlayerInputComponent(UInputComponent* PlayerInputComponent)
{
    Super::SetupPlayerInputComponent(PlayerInputComponent);

    if (UEnhancedInputComponent* EIC = Cast<UEnhancedInputComponent>(PlayerInputComponent))
    {
        EIC->BindAction(MoveAction, ETriggerEvent::Triggered, this, &AMyCharacter::Move);
        EIC->BindAction(JumpAction, ETriggerEvent::Started, this, &ACharacter::Jump);
        EIC->BindAction(JumpAction, ETriggerEvent::Completed, this, &ACharacter::StopJumping);
    }
}

void AMyCharacter::ActivateBattleInput()
{
    if (APlayerController* PC = Cast<APlayerController>(GetController()))
    {
        if (ULocalPlayer* LP = PC->GetLocalPlayer())
        {
            if (auto* Subsystem = LP->GetSubsystem<UEnhancedInputLocalPlayerSubsystem>())
            {
                Subsystem->AddMappingContext(BattleContext, 10);
            }
        }
    }
}
```

### 3. 왜 중요한가 (게임/UE5 맥락)

요즘 게임은 상황에 따라 입력 해석이 계속 바뀐다. 탑승, 조준, 메뉴, 잠금 타깃, 빌드 모드가 들어오면 기존 축/액션 바인딩 방식은 빠르게 한계가 온다. Enhanced Input은 이 상태 전환을 구조적으로 다루기 위한 기반이다.

### 4. 면접 예상 Q&A

**Q1: Enhanced Input의 핵심 이점은 무엇인가요?**
> A: 입력을 하드코딩된 키 처리에서 분리해 액션과 컨텍스트로 관리할 수 있다는 점이다. 그래서 상태 전환, 장치별 대응, 리매핑이 훨씬 유연해진다.

**Q2: `InputAction`과 `InputMappingContext`는 어떻게 다르나요?**
> A: `InputAction`은 "점프", "이동" 같은 의미이고, `InputMappingContext`는 그 의미를 실제 키나 패드 입력에 연결하는 바인딩 묶음이다. 결론적으로 액션은 의도, 컨텍스트는 배치다.

**Q3: 컨텍스트 우선순위는 왜 중요한가요?**
> A: 같은 입력이 여러 컨텍스트에 존재할 수 있기 때문이다. UI가 열린 상태에서 게임플레이 입력이 같이 처리되면 의도치 않은 동작이 나오므로 우선순위와 소비 정책을 함께 설계해야 한다.

**Q4: 트리거와 모디파이어는 언제 구분하나요?**
> A: 입력 값 자체를 바꾸면 모디파이어, 발동 조건을 제어하면 트리거다. 예를 들어 데드존 보정은 모디파이어, 길게 누르기 판정은 트리거에 가깝다.

**Q5: 실무에서 입력 소비를 놓치면 어떤 문제가 생기나요?**
> A: UI 버튼을 눌렀는데 캐릭터도 동시에 점프하는 식의 중복 처리 문제가 생긴다. 특히 Common UI나 메뉴 시스템과 함께 쓸 때 자주 드러난다.

**Q6: 기존 입력 시스템보다 Enhanced Input이 더 적합한 프로젝트는 어떤 유형인가요?**
> A: 상태별 입력 전환, 다양한 장치 대응, 리매핑, 복잡한 액션 조건이 많은 프로젝트다. 반대로 매우 단순한 조작만 있는 소규모 프로토타입은 기존 방식도 충분할 수 있다.

### 5. 흔한 실수

- 매핑 컨텍스트를 추가만 하고 제거하지 않아 입력 충돌을 만든다.
- 트리거와 모디파이어 책임을 섞어 입력 자산이 읽기 어려워진다.
- UI 입력과 게임플레이 입력의 우선순위, 소비 여부를 설계하지 않는다.

## 토픽 6: CharacterMovementComponent(캐릭터 무브먼트 컴포넌트, 캐릭터 이동 컴포넌트)

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 5

### 1. 핵심 개념

`CharacterMovementComponent(캐릭터 무브먼트 컴포넌트, 캐릭터 이동 컴포넌트)`는 단순 이동 함수 묶음이 아니라 가속, 마찰, 중력, 점프, 네트워크 예측, 보정까지 포함한 이동 프레임워크다. 핵심은 `Character`가 직접 이동을 구현하는 것이 아니라 이동 규칙을 컴포넌트가 책임진다는 점이다.

또한 이동 모드(`Walking`, `Falling`, `Flying`, `Swimming`, `Custom`)와 네트워크 smoothing, prediction 개념을 같이 봐야 한다. 멀티플레이에서 캐릭터 이동이 자연스럽게 보이는 이유는 로컬 예측과 서버 보정이 함께 동작하기 때문이다.

### 2. 코드 예시

```cpp
void AMyCharacter::Move(const FInputActionValue& Value)
{
    const FVector2D Axis = Value.Get<FVector2D>();

    AddMovementInput(GetActorForwardVector(), Axis.Y);
    AddMovementInput(GetActorRightVector(), Axis.X);
}

void AMyCharacter::StartClimb()
{
    GetCharacterMovement()->SetMovementMode(MOVE_Custom, 1);
}
```

### 3. 왜 중요한가 (게임/UE5 맥락)

이동은 거의 모든 게임플레이의 중심이다. 캐릭터 조작감, 점프 감각, 경사면 처리, 네트워크에서의 부드러움은 대부분 이 컴포넌트 이해도에 달려 있다. 직접 구현보다 기존 구조를 잘 활용하는 능력이 중요하다.

### 4. 면접 예상 Q&A

**Q1: `Character`와 `CharacterMovementComponent`의 역할은 어떻게 나뉘나요?**
> A: `Character`는 캐릭터라는 게임 오브젝트의 틀이고, 실제 이동 규칙과 물리성 판단은 `CharacterMovementComponent`가 맡는다. 결론적으로 이동 로직의 중심은 컴포넌트 쪽이다.

**Q2: `Pawn`이 아니라 `Character`를 쓰는 이유는 무엇인가요?**
> A: 보행형 캐릭터에 필요한 캡슐, 메시, 이동 컴포넌트, 네트워크 이동 지원이 이미 갖춰져 있기 때문이다. 사람형 이동이 중심이라면 `Character`가 훨씬 빠르다.

**Q3: 이동 모드는 왜 중요한가요?**
> A: 걷기, 낙하, 수영, 비행처럼 이동 규칙이 달라질 때 물리 계산과 제어 방식이 바뀌기 때문이다. 커스텀 이동도 결국 이동 모드 전환이라는 틀 안에서 관리하는 것이 안정적이다.

**Q4: 네트워크 prediction과 smoothing을 간단히 설명해 보세요.**
> A: 로컬 클라이언트는 입력에 즉시 반응하도록 예측 이동하고, 서버가 권위 있는 결과를 보낸 뒤 오차를 보정한다. 이때 화면상 튐을 줄이는 과정이 smoothing이다.

**Q5: 커스텀 이동을 구현할 때 자주 하는 실수는 무엇인가요?**
> A: 기본 이동 모드와 충돌하는 상태 전환을 대충 처리하는 것이다. 예를 들어 커스텀 모드로 들어가 놓고 낙하 전환, 충돌 처리, 복귀 시점을 설계하지 않으면 버그가 누적된다.

**Q6: 실무에서 이동 관련 디버깅은 어떻게 접근하나요?**
> A: 우선 이동 모드, 속도, 가속, 네트워크 역할을 로그와 디버그 드로잉으로 확인한다. 많은 문제는 수식보다 상태 전환과 권한 판단에서 발생한다.

### 5. 흔한 실수

- `AddMovementInput`만 호출하면 모든 이동 규칙이 해결된다고 생각한다.
- 커스텀 이동 모드를 추가하면서 전환 조건과 해제 조건을 명확히 두지 않는다.
- 네트워크 보정 문제를 단순 애니메이션 문제로 오해한다.
