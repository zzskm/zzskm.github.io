## 토픽 7: 월드 구성

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 1

### 1. 핵심 개념

UE5에서 월드는 단순 맵 파일이 아니라 `UWorld(유월드, 실행 월드)`, 레벨, 액터, 스트리밍 상태가 묶인 실행 컨텍스트다. 하나의 게임은 `GameInstance` 위에서 여러 `UWorld`를 거칠 수 있고, 하나의 `UWorld` 안에는 persistent level과 streamed sublevel이 함께 있을 수 있다.

핵심은 "무엇이 월드와 같이 생기고 사라지는가"를 이해하는 것이다. `OpenLevel`은 보통 월드를 새로 열어 갈아끼우고, seamless travel은 일부 객체를 유지한 채 전환하는 흐름이다. 대규모 맵은 레벨 스트리밍과 월드 파티션 개념까지 같이 본다.

### 2. 코드 예시

```cpp
UGameplayStatics::OpenLevel(this, TEXT("BattleMap"));

UWorld* World = GetWorld();
if (World)
{
    UE_LOG(LogTemp, Log, TEXT("World Name: %s"), *World->GetName());
}
```

### 3. 왜 중요한가 (게임/UE5 맥락)

월드 전환과 레벨 스트리밍을 이해하지 못하면 참조 수명, 세이브 복원, 로딩 화면, 매치 전환 흐름이 모두 불안정해진다. 면접에서도 "레벨을 바꾸면 무엇이 살아남나요?" 같은 질문으로 자주 연결된다.

### 4. 면접 예상 Q&A

**Q1: `UWorld`와 레벨은 어떻게 다른가요?**
> A: `UWorld`는 실행 컨텍스트 전체이고, 레벨은 그 안에 로드되는 콘텐츠 단위다. 결론적으로 월드가 상위 개념이고 레벨은 그 구성 요소다.

**Q2: `OpenLevel`과 seamless travel의 차이는 무엇인가요?**
> A: `OpenLevel`은 보통 새 월드를 열어 기존 월드를 교체한다. seamless travel은 플레이어 연결과 일부 객체를 유지한 채 더 자연스럽게 전환하는 흐름이다. 멀티플레이 전환에서는 이 차이가 매우 크다.

**Q3: 레벨 스트리밍은 언제 유용한가요?**
> A: 넓은 월드를 구역 단위로 나눠 메모리와 로딩 비용을 제어할 때 유용하다. 전부 한 번에 올리기 어려운 오픈월드나 대형 스테이지에서 특히 중요하다.

**Q4: 레벨 전환 시 `GameInstance`가 유지되는 이유는 무엇인가요?**
> A: `GameInstance`는 월드보다 더 긴 수명을 갖기 때문이다. 그래서 로그인 세션, 글로벌 서비스, 전역 설정 같은 데이터는 레벨을 넘어 유지할 수 있다.

**Q5: 월드 종속 참조를 오래 들고 있으면 왜 위험한가요?**
> A: 레벨 전환이나 스트리밍 언로드 후에는 그 객체가 더 이상 유효하지 않을 수 있기 때문이다. 월드 경계를 넘는 참조는 항상 수명 검사를 함께 해야 한다.

**Q6: 실무에서 seamless travel을 고려해야 하는 대표 상황은 무엇인가요?**
> A: 로비에서 전투 맵으로 넘어갈 때 플레이어 연결을 유지해야 하는 멀티플레이 흐름이다. 이때 어떤 객체가 유지되고 어떤 객체가 재생성되는지 명확히 알아야 한다.

### 5. 흔한 실수

- 월드가 바뀌어도 이전 액터 참조가 계속 유효할 것이라 가정한다.
- `GameInstance`와 `GameState`의 수명 범위를 혼동한다.
- 레벨 스트리밍과 완전한 월드 전환을 같은 것으로 생각한다.

## 토픽 8: 충돌 & 피직스

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 7

### 1. 핵심 개념

UE5 충돌 시스템은 `collision(콜리전, 충돌)`이 "무엇과 부딪히는가"만이 아니라 "쿼리용인가, 물리 시뮬레이션용인가"까지 포함한다. 대표 개념은 collision channel, object type, response, trace, overlap, hit이다. 즉 충돌은 단순한 on/off가 아니라 여러 정책 조합이다.

특히 trace channel과 object channel 차이, `QueryOnly`와 `PhysicsOnly` 차이는 반드시 구분해야 한다. trace channel은 "이번 검사에서 무엇을 볼지"를 정하는 쿼리 기준이고, object channel은 "이 객체가 어떤 종류인지"를 정하는 분류 기준이다.

### 2. 코드 예시

```cpp
FHitResult Hit;
FCollisionQueryParams Params(SCENE_QUERY_STAT(WeaponTrace), false);

const FVector Start = GetActorLocation();
const FVector End = Start + GetActorForwardVector() * 500.0f;

bool bHit = GetWorld()->LineTraceSingleByChannel(
    Hit,
    Start,
    End,
    ECC_Visibility,
    Params);
```

### 3. 왜 중요한가 (게임/UE5 맥락)

총알 판정, 상호작용 탐색, 시야 체크, 피격 감지, ragdoll, 물리 오브젝트까지 거의 모든 게임플레이가 충돌 시스템을 지난다. 충돌 채널 설계를 잘못하면 기능 충돌이 연쇄적으로 생긴다.

### 4. 면접 예상 Q&A

**Q1: hit와 overlap의 차이는 무엇인가요?**
> A: hit는 막힘(block)과 물리 접촉에 가까운 개념이고, overlap은 겹침 감지다. 결론적으로 막아야 하는지, 감지만 하면 되는지에 따라 선택이 갈린다.

**Q2: trace channel과 object channel은 어떻게 다르나요?**
> A: trace channel은 검사 쪽 기준이고, object channel은 대상 객체의 분류다. 예를 들어 "시야 검사"는 trace channel, "이 액터는 월드 정적 오브젝트인가"는 object channel에 가깝다.

**Q3: `QueryOnly`와 `PhysicsOnly`는 언제 구분하나요?**
> A: line trace나 overlap 검사만 필요하면 `QueryOnly`, 실제 물리 시뮬레이션과 충돌 반응이 필요하면 `PhysicsOnly` 또는 둘 다 켠다. 필요 이상의 물리 설정은 비용과 혼동을 늘린다.

**Q4: 총알 판정에 overlap보다 trace를 많이 쓰는 이유는 무엇인가요?**
> A: 순간적인 발사체 판정은 한 프레임에서 경로를 검사하는 trace가 단순하고 정확하기 때문이다. 반대로 지속 영역 감지는 overlap이 더 자연스럽다.

**Q5: 충돌 프리셋 설계를 잘못하면 어떤 문제가 생기나요?**
> A: UI용 트레이스가 월드 오브젝트까지 전부 맞거나, AI 시야와 무기 판정이 같은 채널을 공유해 예기치 않은 상호작용이 생긴다. 결국 시스템별 의도를 분리해 채널을 설계해야 한다.

**Q6: 물리 문제인지 충돌 설정 문제인지 어떻게 구분하나요?**
> A: 먼저 collision response, enabled 상태, 채널 설정, generate overlap events를 확인한다. 많은 문제는 물리 엔진보다 설정 조합에서 나온다.

### 5. 흔한 실수

- trace channel과 object channel을 같은 개념처럼 사용한다.
- 모든 콜리전을 기본 채널 몇 개로 우겨 넣어 시스템 충돌을 만든다.
- 쿼리만 필요한 컴포넌트에 물리 시뮬레이션 설정을 과하게 켠다.
