## 토픽 9: Actor(액터, 월드 오브젝트) 간 통신

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 4

### 1. 핵심 개념

Actor(액터, 월드 오브젝트) 간 통신 방식은 직접 참조, 인터페이스, 델리게이트, 메시지 브로드캐스트로 나눠 생각하면 정리된다. 중요한 것은 "누가 누구를 알아야 하는가"와 "수명 경계가 얼마나 안전한가"다. 통신이 잦다고 해서 항상 직접 캐스팅으로 해결하면 결합도가 급격히 커진다.

UE에서는 `UInterface`, `TScriptInterface`, 델리게이트, 그리고 상황에 따라 `UGameplayMessageSubsystem` 같은 브로드캐스트 도구를 함께 고려한다. 즉 통신은 호출 방법보다 결합도와 소유권 문제다.

### 2. 코드 예시

```cpp
UINTERFACE(BlueprintType)
class UInteractable : public UInterface
{
    GENERATED_BODY()
};

class IInteractable
{
    GENERATED_BODY()

public:
    virtual void Interact(AActor* InstigatorActor) = 0;
};

void TryInteract(AActor* Target, AActor* InstigatorActor)
{
    if (Target && Target->GetClass()->ImplementsInterface(UInteractable::StaticClass()))
    {
        IInteractable::Execute_Interact(Target, InstigatorActor);
    }
}
```

### 3. 왜 중요한가 (게임/UE5 맥락)

상호작용, 트리거, 아이템 획득, 퀘스트 진행, UI 알림은 전부 객체 간 통신 위에서 움직인다. 여기서 직접 참조만 남용하면 레벨 의존성, 캐스팅 체인, null 처리 비용이 빠르게 커진다.

### 4. 면접 예상 Q&A

**Q1: 직접 참조, 인터페이스, 델리게이트는 언제 구분하나요?**
> A: 강한 소유 관계와 빈번한 직접 호출이면 참조, 공통 행동 계약이면 인터페이스, 이벤트 알림이면 델리게이트가 적절하다. 결론적으로 호출 패턴과 결합도에 따라 고른다.

**Q2: `TScriptInterface`는 왜 쓰나요?**
> A: 인터페이스 포인터를 리플렉션 친화적으로 들고 다니기 위해서다. 블루프린트와 C++ 경계를 넘나드는 경우 일반 C++ 인터페이스 포인터보다 안전하게 다룰 수 있다.

**Q3: `Cast`를 반복하는 구조는 왜 좋지 않나요?**
> A: 호출 대상 구체 타입에 강하게 의존하기 때문이다. 타입 분기가 늘수록 확장성이 떨어지고, 새로운 타입 추가 때마다 기존 호출부를 계속 수정하게 된다.

**Q4: `GameplayMessageSubsystem`은 어떤 상황에 어울리나요?**
> A: 발신자와 수신자를 직접 엮고 싶지 않은 브로드캐스트성 이벤트에 어울린다. 예를 들어 전투 로그, 글로벌 알림, 시스템 간 느슨한 이벤트 전달에 유용하다.

**Q5: 인터페이스보다 델리게이트가 나은 경우는 언제인가요?**
> A: 특정 이벤트 시점에 여러 청자를 알리고 싶을 때다. 인터페이스는 행동 계약, 델리게이트는 이벤트 통지라는 성격 차이를 기억하면 선택이 쉬워진다.

**Q6: 실무에서 가장 안전한 통신 설계 기준은 무엇인가요?**
> A: 소유권이 분명하면 직접 참조, 다형성 행동이면 인터페이스, 다수 청자 알림이면 이벤트로 분리하는 것이다. 한 가지 방식으로 모든 문제를 풀려고 하면 곧 구조가 꼬인다.

### 5. 흔한 실수

- `GetAllActorsOfClass`로 매번 통신 대상을 찾는다.
- 캐스팅 사슬로 객체 간 행동을 분기한다.
- 수명 관리가 불명확한 객체를 델리게이트에 바인딩하고 해제하지 않는다.

## 토픽 10: 블루프린트 & C++ 협업

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 9

### 1. 핵심 개념

블루프린트와 C++ 협업의 핵심은 "정책과 성능은 C++, 조립과 반복 조정은 블루프린트"라는 역할 분리다. C++이 모든 것을 잠그면 디자이너 생산성이 떨어지고, 반대로 블루프린트에 핵심 규칙을 다 몰아 넣으면 추적과 리팩터링이 어려워진다.

실무에서는 `BlueprintCallable`, `BlueprintImplementableEvent`, `BlueprintNativeEvent`, `BlueprintReadOnly` 같은 노출 지점을 신중하게 고른다. 즉 협업은 단순 노출이 아니라 변경 권한 설계다.

### 2. 코드 예시

```cpp
UCLASS()
class AMyChest : public AActor
{
    GENERATED_BODY()

public:
    UFUNCTION(BlueprintCallable, Category="Chest")
    void OpenChest();

    UFUNCTION(BlueprintImplementableEvent, Category="Chest")
    void PlayOpenEffect();
};
```

### 3. 왜 중요한가 (게임/UE5 맥락)

UE 프로젝트는 코드와 콘텐츠가 함께 움직인다. 협업 경계를 잘못 잡으면 작은 연출 변경에도 프로그래머 수정이 필요하거나, 반대로 중요한 게임 규칙이 블루프린트 여러 곳에 흩어진다.

### 4. 면접 예상 Q&A

**Q1: `BlueprintImplementableEvent`와 `BlueprintNativeEvent` 차이는 무엇인가요?**
> A: 전자는 C++ 기본 구현 없이 블루프린트 구현을 기대하고, 후자는 C++ 기본 구현을 제공하면서 블루프린트 확장을 허용한다. 기본 동작이 필요하면 `BlueprintNativeEvent`가 더 안전하다.

**Q2: 어떤 로직을 C++에 두고 어떤 로직을 블루프린트에 두나요?**
> A: 규칙, 성능 민감 로직, 공통 코어는 C++에 두고, 연출, 조정값 연결, 에디터 친화적 조립은 블루프린트에 두는 편이 일반적이다. 즉 핵심과 변동성이 높은 부분을 분리한다.

**Q3: `BlueprintReadWrite`를 남용하면 왜 문제가 되나요?**
> A: 상태 변경 경로가 늘어나 디버깅이 어려워진다. 가능한 한 읽기 전용으로 노출하고, 변경은 함수 API를 통해 통제하는 편이 유지보수에 낫다.

**Q4: 블루프린트와 C++ 협업에서 가장 중요한 설계 포인트는 무엇인가요?**
> A: 누가 어떤 값을 바꿀 수 있는지 명확히 정하는 것이다. 권한 경계가 불분명하면 결국 데이터 일관성이 깨진다.

**Q5: 블루프린트에서 성능 문제가 생기면 어떻게 대응하나요?**
> A: 먼저 병목이 실제로 블루프린트인지 프로파일링으로 확인한 뒤, 반복 실행되는 핵심 경로만 C++로 내리는 편이 좋다. 모든 블루프린트를 무조건 C++로 바꾸는 것은 과잉 대응일 수 있다.

**Q6: 협업 구조가 잘못됐다는 신호는 무엇인가요?**
> A: 같은 규칙이 여러 블루프린트에 복제되거나, 디자이너가 자주 건드리는 값을 C++ 재컴파일 없이 조정할 수 없을 때다. 반대로 중요한 상태가 블루프린트 임의 수정에 그대로 노출돼도 문제다.

### 5. 흔한 실수

- 내부 전용 변수까지 `BlueprintReadWrite`로 열어 둔다.
- 연출 이벤트와 게임 규칙을 한 함수에 섞는다.
- 블루프린트 확장 포인트를 만들지 않아 디자이너 작업이 코드 수정에 묶인다.

## 토픽 11: DataAsset / DataTable(데이터 에셋 / 데이터 테이블, 자산형 데이터 / 표 형식 데이터)

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 10

### 1. 핵심 개념

`DataAsset(데이터 에셋, 자산형 데이터)`과 `DataTable(데이터 테이블, 표 형식 데이터)`은 코드에서 하드코딩하던 값을 콘텐츠로 분리하는 대표 수단이다. `UPrimaryDataAsset`은 구조가 자유로운 자산형 데이터에 적합하고, `UDataTable`은 동일한 행 구조가 반복되는 표 형식 데이터에 적합하다.

핵심은 "동작"이 아니라 "데이터"를 밖으로 빼는 것이다. 이 구조를 잘 쓰면 디자이너는 값을 조정하고, 코드는 그 값을 소비하는 역할에 집중할 수 있다.

### 2. 코드 예시

```cpp
USTRUCT(BlueprintType)
struct FWeaponRow : public FTableRowBase
{
    GENERATED_BODY()

    UPROPERTY(EditAnywhere, BlueprintReadOnly)
    int32 Damage = 10;

    UPROPERTY(EditAnywhere, BlueprintReadOnly)
    float Cooldown = 1.0f;
};
```

### 3. 왜 중요한가 (게임/UE5 맥락)

무기 수치, 아이템 정보, 적 설정, UI 문구를 코드에 하드코딩하면 밸런싱 속도가 급격히 떨어진다. 데이터 중심 구조는 협업 효율뿐 아니라 패치와 운영에도 유리하다.

### 4. 면접 예상 Q&A

**Q1: `DataAsset`과 `DataTable`은 어떻게 구분하나요?**
> A: 행 구조가 동일한 대량 데이터면 `DataTable`, 자산 단위로 개별 데이터 구성이 필요하면 `DataAsset`이 적절하다. 결론적으로 표인지, 개별 객체형 데이터인지가 기준이다.

**Q2: `PrimaryDataAsset`을 쓰는 이유는 무엇인가요?**
> A: 자산 매니지먼트와 식별을 체계적으로 다루기 좋기 때문이다. 로딩 정책, 분류, 자산 참조 구조가 커질수록 장점이 커진다.

**Q3: 데이터 중심 설계의 장점은 무엇인가요?**
> A: 밸런싱과 콘텐츠 수정이 코드 변경 없이 가능해진다. 또한 여러 시스템이 같은 데이터를 일관되게 소비하기 쉬워진다.

**Q4: DataTable의 약점은 무엇인가요?**
> A: 행 구조가 고정되어 복잡한 계층형 데이터를 담기 어렵다. 또 row 이름 문자열 의존이 강해 관리 규칙이 필요하다.

**Q5: 실무에서 row 이름을 어떻게 다루는 편이 좋나요?**
> A: 문자열 상수 남발보다 `FDataTableRowHandle`, `FName` 상수, 태그 기반 식별을 섞어 관리하는 편이 안전하다. 오타 한 번으로도 런타임 실패가 날 수 있기 때문이다.

**Q6: 데이터 자산 구조가 나쁘다는 신호는 무엇인가요?**
> A: 코드가 자산 내부 세부 구조를 너무 많이 알거나, 자산끼리 순환 참조가 많아질 때다. 데이터는 외부 설정이어야지 또 다른 코드 복잡성 원천이 되면 안 된다.

### 5. 흔한 실수

- 행 이름을 문자열 리터럴로 여기저기 흩뿌린다.
- 자산 데이터를 읽는 코드가 직접 타입 캐스팅과 null 분기를 반복한다.
- 데이터 변경이 필요한데도 값을 코드 상수에 남겨 둔다.

## 토픽 12: 디자인 패턴 in UE5

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 9, 11

### 1. 핵심 개념

UE5에서는 순수 GoF 패턴 이름보다 엔진 문맥에 맞는 구조 선택이 더 중요하다. 대표적으로 컴포넌트 조합, Subsystem 기반 서비스, 상태 패턴, 커맨드 패턴, 오브젝트 풀링, 데이터 중심 패턴이 자주 등장한다. 중요한 것은 패턴 이름보다 "UE 수명 주기와 잘 맞는가"다.

예를 들어 object pooling은 투사체, 이펙트, UI 위젯 재사용에 유용하고, state 패턴은 캐릭터 상태 전환에 유용하며, command 패턴은 입력 기록과 재실행, AI 명령 큐에 어울린다. `GameplayMessageSubsystem(게임플레이 메시지 서브시스템, 느슨한 브로드캐스트 허브)`은 옵저버를 더 느슨하게 구현하는 도구로 생각할 수 있다.

### 2. 코드 예시

```cpp
class FMoveCommand
{
public:
    explicit FMoveCommand(const FVector& InDirection) : Direction(InDirection) {}

    void Execute(ACharacter* Character) const
    {
        Character->AddMovementInput(Direction);
    }

private:
    FVector Direction;
};
```

### 3. 왜 중요한가 (게임/UE5 맥락)

게임 코드는 상태가 많고 이벤트가 빈번하다. 패턴을 엔진 수명과 연결해 이해하면 "왜 이 기능을 컴포넌트로 뺐는지", "왜 이 매니저가 Subsystem인지"를 설명할 수 있게 된다.

### 4. 면접 예상 Q&A

**Q1: UE5에서 싱글턴 대신 Subsystem을 선호하는 이유는 무엇인가요?**
> A: 엔진이 수명과 초기화 순서를 관리해 주기 때문이다. 결론적으로 전역 접근 편의성은 유지하면서도 무질서한 전역 객체 난립을 줄일 수 있다.

**Q2: object pooling은 어떤 경우에 유용한가요?**
> A: 생성과 파괴가 매우 잦은 객체가 있을 때 유용하다. 투사체, 이펙트, 반복 생성 UI 같은 대상에 적용하면 할당 비용과 파편화를 줄일 수 있다.

**Q3: state 패턴을 UE에서 어떻게 적용하나요?**
> A: 캐릭터 상태를 enum 하나로만 처리하지 않고 상태 객체나 상태별 로직 분리로 관리할 수 있다. 이동 모드, 전투 상태, 보스 패턴 전환에서 특히 효과적이다.

**Q4: command 패턴은 입력 시스템과 어떤 관계가 있나요?**
> A: 입력을 즉시 실행하지 않고 명령 객체로 표현하면 리플레이, 입력 기록, AI 재사용, undo 같은 확장이 쉬워진다. Enhanced Input과도 자연스럽게 연결된다.

**Q5: `GameplayMessageSubsystem`은 옵저버 패턴과 어떻게 닮았나요?**
> A: 발신자와 수신자를 직접 연결하지 않고 메시지를 통해 느슨하게 연결한다는 점에서 닮았다. 시스템 간 결합을 낮추고 전역 이벤트 흐름을 만들 때 유용하다.

**Q6: 패턴을 과하게 적용하면 어떤 문제가 생기나요?**
> A: 구조는 멋져 보여도 코드가 불필요하게 추상화된다. 실무에서는 패턴 이름보다 디버깅 가능성, 성능, 팀 이해도를 먼저 봐야 한다.

### 5. 흔한 실수

- 패턴 자체를 목표로 삼아 과한 추상화를 만든다.
- object pool을 도입하고도 초기화 리셋 규칙을 정의하지 않는다.
- Subsystem, 컴포넌트, Actor 책임을 패턴 이름만 보고 억지로 맞춘다.
