## 토픽 1: 5대 클래스 역할 분리

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 없음

### 1. 핵심 개념

UE5 게임플레이 프레임워크의 기본 축은 `GameMode`, `GameState`, `PlayerController`, `PlayerState`, `Pawn/Character`다. 중요한 것은 이름 암기가 아니라 "서버 전용인지", "복제 대상인지", "입력 주체인지", "몸체인지"를 역할로 구분하는 것이다.

쉽게 말하면 `GameMode`는 경기 규칙을 결정하는 심판, `GameState`는 모두가 공유하는 전광판, `PlayerController`는 플레이어의 의사결정과 입력 창구, `PlayerState`는 플레이어별 공개 기록, `Pawn`은 월드에서 실제로 움직이는 몸체에 가깝다. 멀티플레이에서는 이 분리가 특히 중요하다.

**판단 기준**
- 규칙 집행과 승패 판정은 `GameMode`
- 모든 클라이언트가 봐야 하는 경기 상태는 `GameState`
- 입력, 카메라, UI 상호작용은 `PlayerController`
- 점수, 팀, 닉네임처럼 플레이어별 공유 상태는 `PlayerState`
- 위치, 이동, 애니메이션은 `Pawn` 또는 `Character`

### 2. 코드 예시

```cpp
UCLASS()
class AMyGameMode : public AGameModeBase
{
    GENERATED_BODY()
};

UCLASS()
class AMyGameState : public AGameStateBase
{
    GENERATED_BODY()

public:
    UPROPERTY(Replicated)
    int32 RemainingTime = 300;
};

UCLASS()
class AMyPlayerState : public APlayerState
{
    GENERATED_BODY()

public:
    UPROPERTY(Replicated)
    int32 KillCount = 0;
};
```

### 3. 왜 중요한가 (게임/UE5 맥락)

면접에서 "체력은 어디에 두나요?", "점수는 어디에 두나요?", "매치 타이머는 어디에 두나요?"라는 질문은 결국 역할 분리를 묻는 것이다. 싱글에서는 대충 동작해도 멀티플레이나 리스폰, 관전, 재접속이 들어오면 잘못된 배치는 바로 드러난다.

### 4. 면접 예상 Q&A

**Q1: `GameMode`와 `GameState`의 차이는 무엇인가요?**
> A: `GameMode`는 서버에만 존재하며 규칙을 집행한다. `GameState`는 경기 진행 결과나 남은 시간처럼 모든 클라이언트가 공유해야 하는 상태를 복제한다. 즉 결론적으로 규칙은 `GameMode`, 공개 상태는 `GameState`다.

**Q2: 점수는 왜 `PlayerController`보다 `PlayerState`에 두는 편이 낫나요?**
> A: 점수는 플레이어별 공유 상태이기 때문이다. `PlayerController`는 소유 클라이언트 중심이고, `PlayerState`는 복제와 관전자 관점에 더 적합하다. 멀티플레이를 고려하면 `PlayerState`가 자연스럽다.

**Q3: `PlayerController`와 `Pawn`을 굳이 나누는 이유는 무엇인가요?**
> A: 입력 주체와 몸체를 분리하면 몸체 교체, 관전, 빙의, 사망 후 리스폰 같은 흐름이 유연해진다. 예를 들어 플레이어가 차량을 탑승해도 컨트롤러는 유지하고 Pawn만 바꿀 수 있다.

**Q4: 체력은 `PlayerState`와 `Pawn` 중 어디에 두는 게 맞나요?**
> A: 기본 결론은 현재 몸체의 체력이면 `Pawn` 쪽이 더 자연스럽다. 다만 영구 스탯이나 플레이어 계정 기반 수치라면 `PlayerState` 또는 별도 시스템이 적절하다. 무엇이 몸체 종속 상태인지 먼저 판단해야 한다.

**Q5: 멀티플레이에서 `GameMode`를 클라이언트에서 직접 참조하면 안 되는 이유는 무엇인가요?**
> A: 클라이언트에는 `GameMode`가 존재하지 않기 때문이다. 서버 전용 규칙 객체를 클라이언트 로직에서 전제로 하면 null 접근이나 구조 오해가 발생한다. 클라이언트는 보통 `GameState`나 RPC를 통해 결과를 본다.

**Q6: 실무에서 역할 분리가 잘못됐다는 신호는 무엇인가요?**
> A: 캐릭터가 죽었더니 점수도 사라지거나, 클라이언트에서만 보이는 값이 서버 규칙처럼 취급되거나, 관전자 UI가 정보를 못 읽는 경우다. 이런 문제는 대부분 상태의 소유 위치를 잘못 잡았다는 신호다.

### 5. 흔한 실수

- 서버 권한 로직을 `GameState`에 넣어 상태 컨테이너와 규칙 엔진을 섞는다.
- 플레이어 영속 데이터를 `Pawn`에 넣어 사망 또는 몸체 교체 시 상태를 잃는다.
- 클라이언트 코드에서 `GetGameMode()`를 당연하게 호출해 멀티 환경에서 null 처리를 놓친다.

## 토픽 2: GameInstance & Subsystem(게임 인스턴스 / 서브시스템, 전역 실행 컨텍스트 / 엔진 관리 서비스)

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 1

### 1. 핵심 개념

`GameInstance(게임 인스턴스, 게임 전체 실행 컨텍스트)`는 레벨 전환을 넘어 유지되는 런타임 컨텍스트다. `GameInstanceSubsystem`, `WorldSubsystem`, `LocalPlayerSubsystem`은 "수명 범위가 다른 서비스 객체"를 안전하게 배치하기 위한 장치다. 핵심은 싱글턴처럼 보이지만 실제로는 엔진 수명에 맞춘 서비스 컨테이너라는 점이다.

`GameInstance`는 게임 전체, `WorldSubsystem`은 월드 단위, `LocalPlayerSubsystem`은 로컬 플레이어 단위로 생각하면 이해가 쉽다. 전역 데이터라고 해서 전부 `GameInstance`에 몰아넣는 순간 God Object가 되기 쉽다.

**판단 기준**
- 레벨을 넘어 유지돼야 하면 `GameInstance`
- 월드와 같이 생성되고 같이 끝나야 하면 `WorldSubsystem`
- 플레이어별 입력/UI 서비스면 `LocalPlayerSubsystem`

### 2. 코드 예시

```cpp
UCLASS()
class UInventorySubsystem : public UGameInstanceSubsystem
{
    GENERATED_BODY()

public:
    void AddItem(FName ItemId) { Items.Add(ItemId); }
    bool HasItem(FName ItemId) const { return Items.Contains(ItemId); }

private:
    TArray<FName> Items;
};

UInventorySubsystem* Inventory =
    GetGameInstance()->GetSubsystem<UInventorySubsystem>();
```

### 3. 왜 중요한가 (게임/UE5 맥락)

세션 전역 데이터, 월드 전용 서비스, 로컬 플레이어 전용 서비스가 섞이면 라이프사이클 버그가 생긴다. 레벨 전환 뒤에도 남아야 하는데 사라지거나, 반대로 월드가 끝났는데 이전 참조를 계속 붙잡는 문제가 대표적이다.

### 4. 면접 예상 Q&A

**Q1: `GameInstance`에는 어떤 데이터를 두는 것이 적절한가요?**
> A: 로그인 세션, 계정 정보, 전역 매니저, 매치 전후를 넘나드는 상태가 적절하다. 반대로 레벨마다 달라지는 월드 상태는 `GameState`나 `WorldSubsystem`에 두는 편이 맞다.

**Q2: Subsystem을 쓰는 이유는 무엇인가요?**
> A: 엔진이 생성과 소멸 시점을 관리해 주기 때문이다. 덕분에 전통적인 싱글턴보다 수명 경계가 명확하고, 의존성도 엔진 컨텍스트를 통해 읽을 수 있다.

**Q3: `GameInstanceSubsystem`과 `WorldSubsystem`의 차이는 무엇인가요?**
> A: 전자는 게임 인스턴스 전체에서 하나고, 후자는 월드마다 하나다. 결론적으로 레벨 전환을 넘어 유지할지 여부가 가장 큰 기준이다.

**Q4: `LocalPlayerSubsystem`은 어떤 상황에서 특히 유용한가요?**
> A: 입력 매핑, 로컬 UI, 플랫폼 사용자 정보처럼 플레이어별 로컬 서비스가 필요할 때 유용하다. 스플릿 스크린이나 여러 로컬 플레이어를 고려하면 차이가 분명해진다.

**Q5: `GameInstance`를 너무 많이 쓰면 어떤 문제가 생기나요?**
> A: 수명은 편하지만 결합도가 높아지고 테스트가 어려워진다. 결국 모든 시스템이 거대한 전역 객체에 의존하게 되며 변경 영향이 커진다.

**Q6: 실무에서는 Subsystem과 일반 UObject 매니저를 어떻게 구분하나요?**
> A: 엔진 수명에 자연스럽게 얹힐 서비스라면 Subsystem이 낫다. 반대로 임시 객체이거나 명확한 소유자가 있는 경우는 일반 UObject나 컴포넌트가 더 단순하다.

### 5. 흔한 실수

- 월드 종속 데이터를 `GameInstance`에 저장해 레벨 전환 후 정리 시점이 꼬인다.
- Subsystem을 도입했지만 의존성을 분리하지 않아 사실상 다른 형태의 전역 변수로 쓴다.
- 로컬 플레이어별 상태를 `GameInstanceSubsystem`에 넣어 멀티 로컬 환경을 깨뜨린다.

## 토픽 3: Actor(액터, 월드 오브젝트) 라이프사이클(lifecycle, 생명주기)

> 난이도: ★★★ | 예상 시간: 2시간 | 선행: 토픽 1

### 1. 핵심 개념

`Actor(액터, 월드 오브젝트)`는 생성자만으로 끝나는 객체가 아니라 `PostInitializeComponents`, `BeginPlay`, `Tick`, `EndPlay`, 파괴 단계까지 이어지는 긴 수명 주기를 가진다. 핵심은 "어느 시점에 컴포넌트가 준비됐는가", "월드에 실제 등록됐는가", "플레이가 시작됐는가"를 구분하는 것이다.

컴포넌트 단위에서는 `OnRegister`, `InitializeComponent`, `BeginPlay` 같은 훅도 함께 본다. 생성자에서 월드 의존 로직을 돌리면 CDO나 에디터 경로까지 같이 타므로 시점 혼동이 생기기 쉽다.

### 2. 코드 예시

```cpp
AMyActor::AMyActor()
{
    PrimaryActorTick.bCanEverTick = true;
}

void AMyActor::PostInitializeComponents()
{
    Super::PostInitializeComponents();
}

void AMyActor::BeginPlay()
{
    Super::BeginPlay();
}

void AMyActor::EndPlay(const EEndPlayReason::Type EndPlayReason)
{
    Super::EndPlay(EndPlayReason);
}
```

### 3. 왜 중요한가 (게임/UE5 맥락)

UE 버그 상당수는 "생성은 됐는데 플레이 시작 전이다", "컴포넌트는 만들었지만 아직 등록되지 않았다" 같은 시점 오해에서 온다. 초기화 코드와 런타임 연결 코드를 올바른 훅에 배치해야 재현 어려운 버그를 줄일 수 있다.

### 4. 면접 예상 Q&A

**Q1: 생성자와 `BeginPlay`는 어떻게 다르게 써야 하나요?**
> A: 생성자는 기본값과 기본 서브오브젝트 설정에 집중하고, `BeginPlay`는 월드 진입 후 외부 시스템 연결에 쓰는 것이 안전하다. 결론적으로 월드 의존 로직은 대개 `BeginPlay` 이후가 맞다.

**Q2: `PostInitializeComponents`는 언제 유용한가요?**
> A: 모든 컴포넌트가 준비된 뒤 액터 내부 관계를 맞출 때 유용하다. 생성자보다 실제 컴포넌트 상태가 더 갖춰져 있어 초기화 순서 문제를 줄일 수 있다.

**Q3: 컴포넌트의 `OnRegister`와 `InitializeComponent`는 어떤 차이가 있나요?**
> A: `OnRegister`는 월드 등록과 연결된 훅이고, `InitializeComponent`는 컴포넌트 초기화 의도에 가깝다. 둘 다 생성자보다 늦지만 역할이 같지는 않다.

**Q4: `EndPlay`에서 정리해야 할 대표 항목은 무엇인가요?**
> A: 타이머, 델리게이트 바인딩, 외부 서비스 등록, 비동기 요청 핸들 같은 연결성 자원이다. 이유는 월드 종료와 액터 제거 시점 모두에서 유령 참조를 남기기 쉽기 때문이다.

**Q5: seamless travel과 라이프사이클은 어떤 관계가 있나요?**
> A: 객체마다 여행 중 유지 여부가 다르므로 단순히 레벨 전환만 생각하면 안 된다. 예를 들어 `PlayerController`는 유지될 수 있지만 월드 종속 Actor는 재생성될 수 있다.

**Q6: 실무에서 라이프사이클 버그를 어떻게 진단하나요?**
> A: 로그를 각 훅에 심어 실제 호출 순서를 먼저 확인한다. 문제의 대부분은 로직 자체보다 "너무 이르거나 너무 늦게 실행됐다"는 데 있다.

### 5. 흔한 실수

- 생성자에서 월드, 플레이어, 게임 상태를 직접 참조한다.
- `BeginPlay`에서 등록한 델리게이트를 `EndPlay`에서 해제하지 않는다.
- 컴포넌트 초기화와 런타임 시작을 같은 시점으로 착각한다.

## 토픽 4: 컴포넌트 모델

> 난이도: ★★☆ | 예상 시간: 1시간 | 선행: 토픽 3

### 1. 핵심 개념

UE5는 상속만으로 기능을 쌓기보다 컴포넌트 조합으로 기능을 분리하는 구조를 선호한다. `UActorComponent`는 로직, `USceneComponent`는 transform, `UPrimitiveComponent`는 렌더링/충돌처럼 역할이 나뉜다. 즉 Actor는 그릇이고, 실제 기능은 컴포넌트가 나눠 가진다.

이 구조를 이해하면 재사용과 책임 분리가 쉬워진다. 반대로 Actor 하나에 이동, 감지, 전투, UI, 오디오를 전부 넣기 시작하면 상속보다 더 빨리 비대해진다.

### 2. 코드 예시

```cpp
UCLASS(ClassGroup=(Custom), meta=(BlueprintSpawnableComponent))
class UHealthComponent : public UActorComponent
{
    GENERATED_BODY()

public:
    UPROPERTY(EditAnywhere, BlueprintReadOnly, Category="Health")
    float MaxHealth = 100.0f;
};

AMyEnemy::AMyEnemy()
{
    HealthComponent = CreateDefaultSubobject<UHealthComponent>(TEXT("HealthComponent"));
}
```

### 3. 왜 중요한가 (게임/UE5 맥락)

게임플레이 기능은 반복된다. 체력, 인벤토리, 타겟 감지, 상호작용, 능력 시스템 같은 기능을 컴포넌트로 분리하면 다른 Actor에 재사용하기 쉬워지고 테스트 포인트도 명확해진다.

### 4. 면접 예상 Q&A

**Q1: 상속보다 컴포넌트 조합을 선호하는 이유는 무엇인가요?**
> A: 기능을 독립 단위로 분리해 재사용성과 변경 유연성을 높일 수 있기 때문이다. 결론적으로 "무엇을 할 수 있는가"를 조합으로 표현하는 편이 "무엇의 하위 타입인가"만으로 설계하는 것보다 실용적이다.

**Q2: `UActorComponent`와 `USceneComponent`는 어떻게 구분하나요?**
> A: transform이 필요 없고 순수 로직이면 `UActorComponent`, 위치와 부모-자식 관계가 필요하면 `USceneComponent`를 쓴다. 시각/충돌 요소는 보통 `UPrimitiveComponent`까지 이어진다.

**Q3: `BlueprintSpawnableComponent`는 언제 붙이나요?**
> A: 디자이너가 블루프린트 에디터에서 해당 컴포넌트를 쉽게 추가해야 할 때 붙인다. 즉 코드 전용 컴포넌트인지, 에디터 조합형 컴포넌트인지에 대한 의도를 드러낸다.

**Q4: 기본 서브오브젝트와 런타임 동적 컴포넌트 추가는 어떻게 다르나요?**
> A: 기본 서브오브젝트는 생성자에서 구조 일부로 생성되고, 런타임 동적 추가는 상황에 따라 생성된다. 전자는 클래스 기본 구조, 후자는 유연한 조립이라고 보면 된다.

**Q5: 컴포넌트가 많아지면 생길 수 있는 문제는 무엇인가요?**
> A: 의존 관계가 숨어 복잡해질 수 있다. 예를 들어 A 컴포넌트가 B 컴포넌트 초기화를 전제로 하면 조합은 유연해 보여도 실제 결합도는 높다.

**Q6: 실무에서 Actor와 컴포넌트 책임을 어떻게 나누나요?**
> A: Actor는 외부 조율과 소유권, 컴포넌트는 개별 기능과 상태에 집중시키는 편이 좋다. 컴포넌트끼리 직접 강하게 엮기보다 Actor가 중재하는 구조가 디버깅에 유리하다.

### 5. 흔한 실수

- 재사용 가능한 기능을 Actor 상속 계층에 묻어 둔다.
- transform이 필요 없는 로직 컴포넌트를 불필요하게 `USceneComponent`로 만든다.
- 여러 컴포넌트 간 초기화 순서를 암묵적으로 가정한다.
